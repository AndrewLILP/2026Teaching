CPD Descriptor 1

"Demonstrates an extensive understanding of the influence of enterprise, innovation and automation on the evolution of computing technology."

CT5-COM-01CT5-THI-01
AExtensive — enterprise, innovation and automation
BThorough — enterprise, innovation and automation
CSound — enterprise, innovation and automation
DBasic — enterprise and innovation only (automation absent)
EIdentifies evolution of / innovations in computing technology
✦ Strong fit
→Enterprise: Unity Technologies' public listing and monetisation strategy (runtime fee controversy, Unity Ads, Unity Gaming Services) — student analyses the business model and its impact on indie vs enterprise developers
→Innovation: DOTS (Data-Oriented Technology Stack), Shader Graph, VFX Graph, real-time GI, WebGL export — student identifies two innovations that changed what small teams can build
→Automation: Identifies at least two automations in their own project (e.g. NavMesh baking automates pathfinding; Animator Controller automates state transitions; Physics engine automates collision resolution) and explains what manual work each replaces
→Unity Asset Store as a marketplace: parallels to App Store economics — creator income, curation, licence terms — is explored as an enterprise system
Unity's controversial 2023 runtime fee announcement is an excellent case study for enterprise decision-making and its impact on the developer community — use it.
✦ Strong fit
→Enterprise: Godot Foundation's non-profit model — sustained by donations and corporate sponsors (W4 Games, Re-Logic). Student analyses how FOSS changes the economics of engine development and who controls the roadmap
→Innovation: The W4 Games partnership enabling Godot on consoles; GDExtension replacing GDNative; the community-driven feature pipeline. How open-source creates innovation cycles impossible in commercial engines
→Automation: AnimationPlayer automating cutscene and UI transitions; NavigationAgent2D/3D automating pathfinding; TileMap auto-tiling; built-in physics — student identifies two in their own game and explains what they replace
→The migration wave from Unity to Godot following the runtime fee announcement is a direct, student-relevant case study of how enterprise decisions reshape the computing industry
The Unity runtime fee controversy and the subsequent developer migration to Godot is one of the most vivid recent examples of enterprise decisions reshaping the computing technology landscape — Godot students have a richer contemporary narrative to analyse.

CPD Descriptor 2

"Skilfully applies appropriate iterative processes to produce computing solutions."

CT5-DPM-01CT5-EVL-01
ASkilfully applies appropriate iterative processes
BApplies appropriate iterative processes
CApplies iterative processes
DUses processes
EIdentifies processes that may produce a solution
✦ Strong fit
→Skilful = each cycle shows observed problem → justified decision → implementation → evidence → next step. Not "I made changes."
→Minimum three documented cycles with Console window screenshots as objective evidence of what was observed
→Project plan revised with annotations — what changed from the original and why
→Appropriate = student can justify why they iterated here but not there — not everything needs iteration, and recognising that demonstrates understanding
→Unity Profiler used at least once to measure performance — documents a data-driven iteration decision (not just "it felt slow")
✦ Strong fit
→Same requirements as Unity. Godot's Output panel and Debugger provide equivalent Console evidence — screenshot each meaningful output
→Signal architecture is a natural iteration target: plan in Phase 2, build in Phase 3, discover signal connection issues, fix and document — this is genuine iterative process evidence
→Scene tree restructuring between builds is a Godot-specific iteration artefact — document why a node moved or was refactored into its own scene
→HTML5 export testing creates a distinct iteration milestone — what broke in the browser that worked in-editor? Document the diagnosis and fix
Godot's faster iteration cycle (no compilation step for GDScript) means students can run more test cycles in the same time. This can produce richer build-log evidence if the student documents each cycle.

CPD Descriptor 3

"Develops highly effective computing solutions using computational, design and systems thinking skills."

CT5-THI-01CT5-DPM-01
AHighly effective — all three thinking types
BEffective — all three thinking types
CSound — all three thinking types
DBasic — computational or design or systems (one only)
ENot described at E level
✦ Strong fit
→Computational: Pseudocode shows deliberate decomposition — problem broken into named sub-problems before code is written. Algorithm selection justified (why a List<T> not an array; why Coroutine not Update())
→Design: Canvas UI wireframes annotated with design rationale tied to player profile. UX decisions iterated based on peer feedback with documented "why"
→Systems: The MonoBehaviour/Component architecture and C# event system are the systems thinking evidence. Student documents how GameObjects communicate without direct references — decoupling as a system design choice
→Architecture reflection explains a specific design pattern used (e.g. Observer via C# events, Singleton GameManager) — naming and justifying a pattern demonstrates systems-level understanding
✦ Strong fit
→Computational: Same — pseudocode, decomposition, algorithm selection justification in GDScript annotations
→Design: Control node wireframes annotated with design rationale. Theme resource (.tres) design as a systematic approach to visual consistency
→Systems: Godot's Signal system is one of the strongest systems thinking frameworks available to Stage 5 students. The Signal table (who emits → who receives → what changes) is an explicit systems model of the game. Student explains why they used signals instead of direct references
→Autoload singletons decision documented: "I used Autoload for ScoreManager because score state must persist across scene changes — this is a system-level architectural decision"
Godot's Signal table — planned in Phase 2 and verified in Phase 3 — is one of the most explicit systems thinking artefacts in any Stage 5 game project. It maps the event-driven architecture as a system diagram in its own right.

CPD Descriptor 4

"Selects and applies safe, secure and ethical practices in the use of data."

CT5-SAF-01
ASelects and applies — active, justified choices
BSelects and applies (quality distinguishes from A)
CApplies safe, secure and ethical practices
DUses data safely and responsibly
EWorks safely with data
✦ Strong fit
→Asset licence compliance: Every Asset Store asset licence type recorded (Standard / Extended / Custom). Student identifies at least one constraint — e.g. "Standard Licence prohibits redistributing source assets, so I cannot share my Assets/ folder publicly"
→Data minimisation: PlayerPrefs usage documented — every key stored is justified. "I store 'high_score' and 'sfx_volume' only. I do not store player name or any identifying information"
→First-run safety: Code handles missing PlayerPrefs keys gracefully on first launch — documented in test table with deliberate first-run test
→Content ethics: Every asset and mechanic reviewed against target player's age and context. Specific decisions documented — not just "I followed the rules"
→Unity licence terms reviewed: "My game is eligible for Unity Personal because I meet the revenue/funding threshold" — this is an active licence compliance decision
✦ Strong fit
→FOSS licence as ethical choice: Student documents what MIT licence means for their project and players — "Anyone can inspect, modify, and redistribute Godot. My game is built on software with no hidden telemetry from the engine itself"
→Asset Library licence register: Every asset's licence recorded. CC BY assets have attribution added to Credits scene before Phase 4 — not as an afterthought
→FileAccess / JSON security: Save file stored in user:// (OS-appropriate location). Student documents what data is saved, why it's minimal, and what the JSON file contains — shown in portfolio with redacted screenshot
→First-run safety: Code checks if save file exists before loading — missing file handled gracefully. Test table entry documents deliberate first-run test with save file deleted
→HTML5 export: student documents what browser storage (if any) the game uses — IndexedDB via Godot's file system abstraction — and whether player data could persist beyond the session
Godot's MIT licence and FOSS ecosystem create a richer ethical discussion than Unity's commercial licence terms. Students building on Godot can address engine transparency, community governance, and data collection as genuine ethical dimensions — not just asset compliance.

CPD Descriptor 5

"Skilfully develops, tests and implements technically concise algorithms in a general-purpose programming language."

CT5-OPL-01
ASkilfully develops, tests and implements — technically concise
BDevelops, tests and implements — functional algorithms
CDevelops common algorithms and implements in a GPL
DDevelops basic algorithms and/or implements
EFollows / partially implements algorithms
✦ Strong fit
→C# is enterprise-grade GPL with static typing, generics, interfaces, delegates, async/await, LINQ — the full algorithmic toolkit is available and appropriate for "technically concise"
→Technically concise: refactoring pass documented — "I had duplicate collision handling in three MonoBehaviours; I extracted an interface IInteractable and a single handler script"
→[SerializeField] and single-responsibility scripts demonstrate concise, well-structured code design — not just "it works"
→Pseudocode precedes every C# implementation; test table maps test cases to algorithm intent
→XML summary comments on every class and method — code is self-documenting as well as annotated
C# gives the widest algorithmic vocabulary of any platform in this comparison set. Interfaces, generics, delegates, and LINQ all contribute to "technically concise" evidence that GDScript cannot yet match.
⚡ Achievable
→GDScript 2.0 (Godot 4) is a typed, interpreted GPL with static typing support (@export, typed arrays, typed dictionaries) — qualifies as a general-purpose programming language for NESA purposes
→Technically concise: typed GDScript with @export, explicit return types, and single-responsibility scripts demonstrates conciseness — refactoring pass where a large script is broken into smaller scripts documents this explicitly
→Signal-based decoupling is an algorithmic design decision — "I used emit_signal() instead of a direct method call to keep this algorithm reusable"
→Pseudocode precedes GDScript; test table with Output panel evidence maps test results to algorithm intent
GDScript's dynamic typing (even with optional static types) and lack of interfaces/generics mean the algorithmic vocabulary is narrower than C#. A student who uses typed GDScript throughout and documents a refactoring pass can reach A, but the path is narrower. Godot 4 C# mode removes this limitation entirely — an alternative worth knowing.

CPD Descriptor 6

"Demonstrates creativity and innovation in the design and implementation of user interfaces to create engaging user experiences."

CT5-DES-01
ACreativity and innovation — engaging
BCreativity — engaging
CDesigns and implements user interfaces
DImplements basic elements of UI design
EIdentifies elements of UI that contribute to UX
✦ Strong fit
→Unity Canvas system: Image, Text (TextMeshPro), Button, Slider, CanvasGroup — full UI toolkit. Animator component on UI elements enables animated transitions (fade in, slide, scale)
→Innovation in Unity UI: DOTween or LeanTween for fluid animations; Shader Graph for custom UI effects; Screen Space UI layered over world-space elements for depth
→Canvas wireframes annotated with design rationale tied to player profile — every font choice, colour, and layout decision explained
→Peer feedback on UI specifically collected and iterated — "Tester couldn't read score at a glance; I increased font size and added a contrasting background panel"
→Accessibility: TextMeshPro allows dyslexia-friendly fonts; colour contrast checked; UI tested at different resolutions using Canvas Scaler
✦ Strong fit
→Godot Control nodes: Label, Button, ProgressBar, TextureRect, NinePatchRect, VBoxContainer/HBoxContainer — full UI system built for games. CanvasLayer ensures UI renders above game world without parenting issues
→Innovation: Tween for animated UI transitions (AnimationPlayer or create_tween()); Theme resource (.tres) for consistent visual language across all scenes; custom StyleBox for button states
→Control node wireframe planned before building — hierarchy annotated to show which Labels are updated by which signals
→Peer feedback on UI iterated with documented before/after — font size, ProgressBar visibility, button contrast
→Theme resource as a design system: student creates a custom Theme.tres that defines fonts, colours, and margins — documents this as an intentional design system decision, not visual decoration
Godot's Theme resource (.tres) creates a natural design-systems thinking opportunity — building a consistent UI theme is evidence of both design thinking and systems thinking simultaneously.
