Phase 1 — Identifying & Defining

Define the problem and player before opening Godot Editor. Everything you design connects back to this.

Problem & audience

Problem StatementRequired

Write a specific, testable problem statement — not "I want to make a game in Godot."

Address each of these
  • Who is your player? (age, gaming background, context)
  • What specific need, challenge, or opportunity does your game address?
  • Why is an interactive game the right solution — not a video, app, or website?
  • Why is Godot the right tool for this particular game?
Player ProfileRequired
Cover these points
  • Age range and gaming experience
  • Target platform: PC, Mac, browser (HTML5 export), or Android? Godot supports all — justify your choice
  • Context: solo play, classroom, home?
  • Accessibility needs to consider

Requirements

Functional RequirementsRequired
Example functional requirements
  • The game must detect collision between the player CharacterBody2D and enemies using Area2D signals
  • The score must persist between sessions using FileAccess and JSON serialisation
  • The game must transition between scenes using SceneManager without memory leaks
  • Players must receive audio-visual feedback within one frame of any scored action
Non-Functional RequirementsRequired
Consider
  • Performance: target 60fps on school hardware; HTML5 export adds constraints
  • Session length: completable in how many minutes?
  • File/build size: HTML5 export has loading time implications
  • Aesthetic tone and feel
Legal, Ethical & Safety ConsiderationsRequired
Godot-specific considerations
  • Godot Engine licence: Godot is released under the MIT licence — you own your game and can distribute it freely. Document this explicitly as an ethical advantage over proprietary engines.
  • Godot Asset Library: assets have varying licences (MIT, CC-BY, CC0, custom). Check and record the licence for every asset before use. CC-BY requires attribution in your credits.
  • FOSS ethics: What does it mean to build your project on open-source software? Who maintains Godot, and how? Consider contributing a bug report or asset back to the community.
  • FileAccess / JSON data: What player data will you save, and why is each field necessary? Where is the file saved on the user's device?
  • HTML5 export and browser storage: If your game runs in a browser, what data could be cached or accessible?
  • Content appropriateness for target audience
Enterprise, Innovation & Automation ContextEvidence

Grade A requires "extensive" understanding across enterprise, innovation, and automation — all three. Begin this research in Phase 1 and build on it throughout.

Research and respond
  • Enterprise: Godot is developed by a non-profit foundation (Godot Foundation) funded by donations and corporate sponsors. How does this FOSS enterprise model differ from Unity's commercial model — and what are the implications for developers?
  • Innovation: What has Godot 4 enabled that wasn't possible in Godot 3? (Vulkan renderer, improved 3D, GDExtension, C# improvements). How does open-source development create innovation cycles different from commercial engines?
  • Automation: Identify at least two automations in your game provided by Godot (physics engine, NavigationAgent, AnimationPlayer, TileMap auto-tiling, built-in collision shapes). Explain what manual work they replace.

Phase 2 — Researching & Planning

Analyse existing games, design your node architecture, and write pseudocode before writing a line of GDScript.

NSW pseudocode must precede all GDScript implementation. This is required evidence for Grade C and above on the algorithms descriptor.

Research

Existing Game Analysis #1Evidence
Analyse these elements
  • Core mechanic: what does the player do, and how is it taught without instructions?
  • Input → process → output flow
  • UI/UX: how is game state communicated to the player?
  • What design decisions would you adopt or reject for your game, and why?
Existing Game Analysis #2Evidence

Architecture & design

Scene Tree Plan & Node ArchitectureRequired

Plan your Godot scene tree before building it. The best Godot games have a clear scene hierarchy where each scene has a single responsibility and communicates through signals.

Godot architecture — what to plan
  • Scene breakdown: Which .tscn files will you have? (MainMenu.tscn, GameWorld.tscn, Player.tscn, Enemy.tscn, HUD.tscn, GameOver.tscn)
  • Node hierarchy: For each scene, draw the node tree. What is the root node type and why? (Node2D, CharacterBody2D, Control, CanvasLayer)
  • Script assignment: Which nodes get scripts? Each script should have one responsibility — name it and describe its single job.
  • Signals: Where will you emit and connect signals? A signal table: which node emits, which node receives, what event triggers it. This is evidence of systems thinking.
  • Resources (.tres): Will you use Resource files for reusable data? (enemy stats, level data, item definitions)
Pseudocode — Core Mechanic(s)Code

NSW pseudocode format. Language-independent — do not write GDScript syntax. Include which node/script will own each block.

Must include
  • A variable storing changing game state
  • A selection structure (IF / ELSE IF / ELSE)
  • An iteration structure (FOR or WHILE)
  • A note on which signal(s) trigger this logic
UI / Control Node Design SketchesEvidence

Sketch your UI scenes. In Godot, UI is built with Control nodes inside a CanvasLayer — plan this hierarchy before building it.

Sketch and annotate
  • HUD layout — which Control nodes? (Label, ProgressBar, TextureRect) and their anchors
  • Main menu layout and button signals
  • Win/lose/pause screen design
  • Which Labels/UI elements update at runtime via signal connections?
  • Theme resource: will you create a custom Theme (.tres) for consistent fonts and colours?
Asset Library Register & Licence RecordGodot-specific

List every asset from the Godot Asset Library, itch.io, OpenGameArt, Freesound, or elsewhere. Many require attribution — record it before you forget.

Asset register policy
  • MIT and CC0 assets: free to use without attribution (but record it anyway)
  • CC BY assets: require attribution in your game credits or README
  • Scripts from tutorials: rewrite and annotate in your own words — you must be able to explain every line
  • Godot Asset Library addons (e.g. Dialogue Manager, Phantom Camera): document what they do and why you chose them over writing your own

Phase 3 — Producing & Implementing

Build your game. Your signal connections and scene architecture are as important to document as what the game does.

Use Godot's Output panel and print() / push_warning() / push_error() as test evidence. Screenshot your Output panel alongside test table entries. The Debugger panel shows stack traces — include these when documenting bugs.

Code & architecture

Annotated GDScript FilesCode

Every script must have comments explaining what AND why. Uncommented code earns no credit for thinking skills.

Required in your GDScript
  • ## docstring comment at the top of each script describing its single responsibility
  • @export variables used for inspector-configurable values — explain why @export vs a hard-coded constant
  • Signals defined and documented: ## what event this signal announces and who should connect to it
  • Variables, selection (if/elif/else), and iteration (for/while) all present
  • At least one typed Array or Dictionary with a purpose comment
  • A function that emits a signal — annotated to show the decoupling intent
  • _ready(), _process() or _physics_process() used appropriately — comment explains which and why
Scene Tree Screenshot & Signal MapGodot-specific

Document your actual scene tree and signal connections from Godot's Node panel. This is evidence that your architecture matches your Phase 2 plan — and that you understand what each connection does.

Document at minimum
  • Scene tree screenshot for your main game scene — annotate which nodes have scripts and what each script's job is
  • Node panel screenshot showing signal connections for your Player or core mechanic node
  • Where your scene tree changed from your Phase 2 plan — explain why
  • Any autoloaded singletons (Autoload in Project Settings) — justify why Autoload was necessary

Phase 4 — Testing & Evaluating

Structured testing, genuine peer feedback, and honest evaluation. Identifying what you'd change is a sign of mastery.

Structured Test TableRequired
Godot-specific tests to include
  • Signal chain test: does the expected signal fire when its source condition is met? (Check Node → Signals tab for connection count)
  • Physics test: does CharacterBody2D movement behave correctly at edge conditions? (zero gravity, maximum speed, wall collision)
  • File persistence: does JSON save/load correctly across sessions? (Close game, reopen — is data present?)
  • Scene transition: does get_tree().change_scene_to_file() load the correct scene without node leaks?
  • HTML5 export (if applicable): does the game run in a browser without console errors?
  • Deliberate failure: what happens if the save file is missing on first load?
Peer Playtest — Tester 1Evidence
Tester must respond to
  • Did you understand what to do without being told? (Y/N + comment)
  • Did you encounter bugs, visual glitches, or unexpected behaviour?
  • What was the most enjoyable moment?
  • What was confusing or frustrating?
  • Was the difficulty appropriate?
  • On a scale of 1–5, how likely would you be to play again?

Evaluation Against RequirementsRequired
Signal Architecture ReflectionGodot-specific

Reflect on your signal design decisions — this is strong evidence for the systems thinking and algorithms descriptors.

Address
  • Where did signals work exactly as planned to decouple nodes?
  • Where did you end up using direct node references (get_node(), $Node) instead of signals — was this the right call?
  • Did any signal connection cause unexpected behaviour? How did you diagnose and fix it?
  • What would you restructure in your scene tree if you rebuilt from scratch?
Final ReflectionRequired
Reflect on
  • What was the most significant technical challenge you overcame, and how?
  • What design decision are you most proud of?
  • What would you do differently in scene architecture, GDScript structure, or process?
  • How did Godot's open-source nature and community resources affect your development process?
  • What skills did you build that you didn't have at the start?
