Phase 1 — Identifying & Defining

Define your problem and player before you open Roblox Studio. Everything connects back to this.

Problem & Audience

Problem Statement Required

Write a specific, testable statement of the design problem your game solves. "Make a fun game" is not a problem statement.

Address each of these
  • Who is your player? (age, context, prior gaming experience)
  • What specific need, challenge or opportunity does your game address?
  • Why is Roblox the right platform for this game?
Player Profile Required

Describe the specific person you are designing for. Consider how they will access the game — Roblox is primarily played in a browser or via the desktop app, which affects design decisions.

Cover these points
  • Age range and Roblox experience level
  • Will they play solo, in a group, or via Roblox's multiplayer?
  • Device — PC, Mac, or mobile? (affects UI sizing decisions)
  • Any accessibility needs to consider

Requirements

Functional Requirements Required

List what your game must do. These are testable — you will use them directly in Phase 4. Aim for 5–8 specific, measurable requirements.

Examples of functional requirements
  • The game must have a clear win condition communicated to the player via a ScreenGui
  • The scoring mechanic must update in real time and persist between rounds using DataStore
  • The game must support 2–6 simultaneous players without performance issues
  • Players must be able to respawn without leaving the server
Non-Functional Requirements Required

These describe how your game should feel or perform — qualities that matter but are harder to test directly.

Consider
  • Playability — completable in how many minutes?
  • Performance — should run at acceptable frame rate on school computers
  • Onboarding — can a new player understand the rules within 60 seconds?
  • Tone — competitive, casual, educational, horror?
Legal, Ethical & Safety Considerations Required

Roblox creates richer ethical discussion than most platforms — address each of these areas specifically.

Roblox-specific considerations
  • Toolbox assets — have you checked the creator's licence before using free models?
  • DataStore — what player data will you store, and why is each piece necessary?
  • Client-server security — why can't game logic that affects outcomes run on the client?
  • Appropriate content — does your game comply with Roblox's Community Standards?
  • Monetisation — if you add gamepasses, what are the ethical implications?
Enterprise & Innovation Context Evidence

Roblox is itself a case study in enterprise and innovation. This section feeds directly into one of the seven CPD descriptors — don't leave it until Phase 4.

Research and respond to these questions
  • How has Roblox changed the way games are developed and distributed compared to traditional studios?
  • What role does automation play in Roblox Studio? (e.g. Physics engine, character animation, terrain generation)
  • How do successful Roblox developers monetise their games via the Developer Exchange program?
  • How does the platform economy influence what games get made?

Phase 2 — Researching & Planning

Analyse existing games, design your system architecture, and write pseudocode before you script anything.

⚠ Pseudocode first — always You must write NSW pseudocode for your custom mechanic(s) BEFORE opening the Luau script editor. This is required evidence for a C grade or above. Pseudocode written after the fact and reverse-engineered from your code is not acceptable.

Research

Existing Game Analysis #1 Evidence

Choose an existing game — Roblox or otherwise — and analyse its design and technical decisions. Go beyond description: explain why developers made each choice.

Analyse these elements
  • What is the core mechanic? How is it taught to the player?
  • What inputs, processes and outputs does the game use?
  • How does the UI communicate game state to the player?
  • If it's a Roblox game: can you identify whether logic runs client-side or server-side? How do you know?
  • What would you borrow or change for your own design?
Existing Game Analysis #2 Evidence

Repeat the analysis for a second game. Comparing two gives you richer design vocabulary and shows broader research.

Architecture & Design

System Diagram — Including Client/Server Split Required

Draw your game as a system. In Roblox, this diagram must show which logic runs on the client (LocalScript) and which runs on the server (Script), and how they communicate.

Roblox architecture — what goes where
  • Server Script (Script): game state, scoring, win conditions, DataStore reads/writes, anything that must be trusted
  • LocalScript: player input handling, local UI updates, camera, animations
  • RemoteEvent / RemoteFunction: the bridge — client sends player action → server validates → server updates state → server fires event back to all clients
  • ModuleScript: shared logic or data structures used by both sides
Your diagram must show
  • Inputs (player actions, time, game events)
  • Processes, labelled as client or server
  • Outputs (world changes, UI updates, DataStore writes)
  • At least one RemoteEvent communication path
Pseudocode — Custom Mechanic(s) Code

Write pseudocode using NSW conventions (BEGIN/END, IF/THEN/ELSE, FOR/WHILE). This must be language-independent — do not write Luau syntax here. Write it as if explaining the logic to someone who has never coded.

Your pseudocode must demonstrate
  • A variable storing changing game state (score, lives, timer, etc.)
  • A selection structure — IF / ELSE IF / ELSE
  • An iteration structure — FOR or WHILE loop
  • Note in a comment whether each step runs server-side or client-side
UI / ScreenGui Design Sketches Evidence

Sketch and annotate your in-game UI elements. In Roblox, all player-facing UI is built via ScreenGui in StarterGui — your sketches should map to this structure.

Sketch and annotate at least
  • The HUD (score, timer, lives, round number — whatever applies)
  • Any prompt or instruction shown on spawn/round start
  • The win/lose/end screen the player sees
  • Any menu or settings UI (if applicable)
Toolbox Asset Register Roblox-specific

List every free model, plugin, or asset you plan to use from the Roblox Toolbox. You must check and record the licence or terms for each one before using it.

Toolbox policy for this project
  • You may use Toolbox assets for visual elements (terrain, props, decorations)
  • You may NOT submit Toolbox scripts as your own programming work
  • Any Toolbox code you use must be read, understood, and commented in your own words
  • If asked "explain this code", you must be able to — line by line
Project Plan with Milestones Required

Plan your build schedule. Leave time for testing, fixing, and playtesting — most students underestimate how long debugging takes in Roblox Studio.

3

Phase 3 — Producing & Implementing

Build and script your game. Document every meaningful decision and iteration — the log is the evidence.

Studio output window is your best friend Use print() to log variable states during testing, and warn() for conditions you want to flag. Include screenshots of your output window in your iteration log — this is real testing evidence.

Code

Annotated Luau Script(s) Code

Your scripts must be commented to explain what each section does AND why you made that design choice. Uncommented code earns no credit for thinking skills.

Required elements in your code
  • At least one variable with a meaningful name (not x, val, temp)
  • At least one IF / ELSEIF / ELSE selection structure
  • At least one FOR or WHILE loop
  • At least one function with a clear name and purpose comment
  • At least one RemoteEvent (if multiplayer) with server-side validation
  • DataStore read/write with error handling (pcall) if you are saving data
DataStore — required pattern for data persistence
  • Always wrap DataStore calls in pcall() to handle failures gracefully
  • Save data on PlayerRemoving AND game:BindToClose() to avoid data loss
  • Never trust client-sent data — always validate on the server before saving
Explorer Structure Map Roblox-specific

Document your Roblox Explorer hierarchy. A screenshot is ideal — annotate it to explain why each script or asset sits where it does. This is evidence of systems thinking.

For each script, explain
  • Is it a Script, LocalScript, or ModuleScript?
  • Where does it live in the hierarchy, and why?
  • What is its single responsibility?

  • What problem or decision prompted this?
  • Output window screenshot or test result
  • Did it work as expected? What did you learn?

Iteration Cycle 2 Evidence
Iteration Cycle 3 Evidence

4

Phase 4 — Testing & Evaluating

Structured testing, honest peer feedback, and a genuine evaluation. Identifying what you'd do differently is a sign of mastery, not failure.

Testing

Structured Test Table Required

For each mechanic or feature, document what you expected, what actually happened, and what you did as a result. Use Roblox Studio's Output window as evidence — screenshot print/warn logs in your table.

Roblox-specific tests to include
  • Test your RemoteEvent chain: does the client action reach the server correctly?
  • Test DataStore: does data save and load correctly across sessions? (use pcall and log success/failure)
  • Test with multiple simulated players using Studio's local server mode
  • Test a deliberate failure: what happens if DataStore is unavailable?
Peer Playtest — Tester 1 Evidence

A real player tests your published or LAN-shared game and gives written feedback. You cannot objectively test your own game — you already know how it works.

Your tester must respond to
  • Did you understand the objective without being told? (Y/N + comment)
  • Did you encounter any bugs, lag, or unexpected behaviour?
  • What was the most enjoyable moment?
  • What was confusing or frustrating?
  • On a scale of 1–5, how likely would you be to play this again? Why?
Peer Playtest — Tester 2 Evidence

Evaluation

Evaluation Against Requirements Required

Return to your Phase 1 requirements. For each one, state whether it was met, partially met, or not met — with evidence from your testing or playtest feedback.

Security & Ethics Reflection Roblox-specific

This feeds directly into the "safe, secure and ethical" CPD descriptor. Reflect specifically on the decisions you made, not just generic statements about online safety.

Address each of these
  • What game logic did you deliberately move to the server and why?
  • How did you handle DataStore failure — what happens to the player's data if the save fails?
  • Were there any Toolbox assets you decided NOT to use? Why?
  • Does your game content comply with Roblox Community Standards? How do you know?
Final Reflection Required

This is your most personal piece of writing. Analyse how you built it and what you learned — don't just summarise what you built.

Reflect on
  • What was the most significant technical challenge, and how did you solve it?
  • What design decision are you most proud of, and why?
  • What would you do differently — in architecture, code, or process?
  • How did the client/server split change the way you thought about your game as a system?
  • What skills did you build that you didn't have at the start?
