Phase 1 — Identifying & Defining

Define the problem and player before opening Unity. Every design decision connects back to this foundation.

Problem & audience

Problem StatementRequired

Write a specific, testable problem statement. "Make a fun game in Unity" is not a problem statement — it describes a tool, not a design challenge.

Address each of these
  • Who is your player? (age, context, prior experience with games)
  • What specific need, gap, or opportunity does your game address?
  • Why does this problem require an interactive game solution rather than another medium?
  • Why is Unity the right tool for this particular solution?
Player ProfileRequired

Describe the specific person you are designing for. This profile should influence every subsequent design decision.

Cover these points
  • Age, gaming experience, and familiarity with the genre
  • Device: PC, Mac, WebGL browser, or mobile? (Unity supports all — choose deliberately)
  • Context of play: solo, in class, at home?
  • Accessibility considerations: colour blindness, motion sensitivity, motor ability

Requirements

Functional RequirementsRequired

List what your game must do. These are testable — you will map each one to test cases in Phase 4. Aim for 5–8 specific, measurable requirements.

Example functional requirements
  • The game must detect player collision with the goal object and trigger the win state
  • The score must persist between scenes using PlayerPrefs
  • The game must support keyboard and mouse input on PC without controller required
  • The player must receive feedback within 0.5 seconds of any significant action
Non-Functional RequirementsRequired
Consider
  • Performance: maintain 60fps on target hardware (school computers)
  • Build size: acceptable for WebGL delivery if browser-based
  • Session length: completable in how many minutes?
  • Tone and aesthetic: what feeling should the player have?
Legal, Ethical & Safety ConsiderationsRequired
Unity-specific considerations
  • Unity Asset Store licences vary — Standard, Extended, and custom licences have different redistribution rights. Check every asset's licence before using it.
  • Unity's Personal licence terms: free for students, but commercial use thresholds apply if the game earns revenue
  • Synty / third-party asset packs: may prohibit redistribution of the raw assets — check if you can include them in a build shared publicly
  • PlayerPrefs stores data locally — what data are you saving, and why is each field necessary?
  • If collecting any user input: consent, privacy, and data minimisation apply
  • Content appropriateness: ESRB/Classification Board equivalent for your target audience
Enterprise, Innovation & Automation ContextEvidence

This feeds directly into CPD Descriptor 1. Grade A requires "extensive" understanding — address all three: enterprise, innovation, and automation.

Research and respond
  • Enterprise: Unity Technologies is a publicly listed company — how has Unity's business model (engine + services) shaped the games industry? How does the Asset Store function as an enterprise marketplace?
  • Innovation: What innovations has Unity enabled that weren't previously possible for small teams? (DOTS, real-time VFX Graph, WebGL deployment, XR)
  • Automation: How does Unity automate work that previously required specialists? (Physics engine, NavMesh, Animator controller, Cinemachine, ProBuilder) Identify at least two automations in your own game and explain what they replace.

Phase 2 — Researching & Planning

Analyse existing games, design your architecture, and write pseudocode before writing a single line of C#.

Pseudocode in NSW format must be written before any C# implementation. This is non-negotiable evidence for Grade C and above on the algorithms descriptor. Pseudocode written after the fact and reverse-engineered from your code is not acceptable.

Research

Existing Game Analysis #1Evidence

Choose a game (Unity-made or otherwise) and analyse its design and technical decisions. Go beyond description — explain why the developer made each choice.

Analyse these elements
  • Core mechanic: what does the player do, and how is it taught?
  • Input → process → output flow for the key mechanic
  • UI/UX decisions: how does the game communicate state to the player?
  • What design choices would you adopt or reject for your own game, and why?
Existing Game Analysis #2Evidence

Architecture & design

System Diagram & Scene ArchitectureRequired

Draw your game as a system AND plan your Unity scene/GameObject hierarchy before building. A student who plans architecture before building avoids the most common Unity trap: one massive script that does everything.

Unity architecture — what to plan
  • Scene breakdown: which scenes do you need? (MainMenu, GameScene, UIScene, GameOver?) How do scenes transition?
  • GameObject/Component plan: for each major object, what components does it need? (Rigidbody, Collider, custom MonoBehaviour, Animator)
  • Script responsibility: one script = one responsibility. Name each script and describe its single job. (e.g. PlayerController handles movement only; ScoreManager handles scoring only; UIManager updates Canvas elements)
  • Input → Process → Output: draw the flow for your core mechanic
  • Events/delegates: where will you use UnityEvent or C# events to decouple components?
Pseudocode — Core Mechanic(s)Code

Write pseudocode in NSW format for each custom mechanic. Language-independent — do not write C# syntax here.

Your pseudocode must include
  • At least one variable storing changing game state
  • A selection structure (IF / ELSE IF / ELSE)
  • An iteration structure (FOR or WHILE)
  • Note which class/script will own each section
UI / Canvas Design SketchesEvidence

Sketch and annotate your Canvas UI before building it. In Unity, UI lives in a Canvas component — plan the hierarchy before you build it.

Sketch and annotate
  • HUD layout: score, health, timer, minimap — position and visual hierarchy justified
  • Main menu screen: buttons, background, layout
  • Win/lose/pause screen design
  • TextMeshPro font choices: why this typeface for this player/context?
  • Which UI elements update at runtime via script, and how?
Asset Register & Licence RecordUnity-specific

List every third-party asset used (Unity Asset Store, itch.io, Mixamo, Freesound, etc.) and confirm its licence before using it in your project.

Asset register policy
  • Asset Store assets under Standard Licence may be used in your game but raw source files cannot be redistributed — check before sharing your project folder
  • Scripts from tutorials or forums must be rewritten and understood — you must be able to explain every line
  • Free assets from itch.io vary widely — some require attribution in your credits
  • Sounds and music from Freesound may require Creative Commons attribution
Project Plan with MilestonesRequired

Plan your build schedule. Unity projects almost always take longer than expected — leave at least two weeks for testing, bug fixing, and polishing UI.

3

Phase 3 — Producing & Implementing

Build your game. Every architectural decision must be documented — the code structure is as important as what the game does.

Use Unity's Console window as test evidence. Debug.Log() for expected state, Debug.LogWarning() for edge cases, Debug.LogError() for failures. Screenshot your Console alongside test table entries.

Code architecture

Annotated C# ScriptsCode

Every script must have comments explaining what each section does AND why that design decision was made. Uncommented code earns no credit for thinking skills.

Required in your code
  • XML summary comment on every public class and method (///)
  • [SerializeField] used instead of public for Inspector-exposed fields — explain why in a comment
  • Single responsibility: each MonoBehaviour has one job, stated in its class summary
  • C# events or UnityEvents used to decouple at least two components — the decoupling decision annotated
  • Variables, selection (if/else if/else), and iteration (for/foreach/while) all present
  • At least one use of a C# collection (List<T>, Dictionary<K,V>, or array) with purpose comment
  • Coroutines used where appropriate (timed delays, sequences) — explain why Coroutine vs Update()
Hierarchy & Inspector Screenshot LogUnity-specific

Take annotated screenshots of your Hierarchy window and key Inspector views. This is evidence that your architecture matches your Phase 2 plan — and that you understand each component you have attached.

Document at minimum
  • Hierarchy view of your main game scene — annotate which GameObjects own which scripts
  • Inspector view of your Player GameObject — annotate each component and why it's there
  • Inspector view of your GameManager/ScoreManager — annotate [SerializeField] values and what they control
  • Any Prefabs used — explain the Prefab pattern and why you used it here

Phase 4 — Testing & Evaluating

Structured testing, genuine peer feedback, and honest evaluation. Grade A students identify what they'd change — that's mastery, not failure.

Structured Test TableRequired

For each mechanic or feature: expected behaviour, actual behaviour, pass/fail, action taken. Unity's Console window provides timestamped evidence — include screenshots.

Unity-specific tests to include
  • Collision/trigger tests: does OnTriggerEnter/OnCollisionEnter fire at the correct moment?
  • Physics test: does Rigidbody behaviour match expectations at edge velocities?
  • PlayerPrefs: does data save correctly between play sessions? (Exit Play Mode, re-enter — does value persist?)
  • Scene transition: does SceneManager.LoadScene() load the correct scene and does state reset as expected?
  • Frame rate: does the game maintain target fps with all assets loaded? (Unity Profiler screenshot)
  • Deliberate failure: what happens if PlayerPrefs data is corrupted or missing on first load?
Peer Playtest — Tester 1Evidence
Tester must respond to
  • Did you understand the objective without being told? (Y/N + comment)
  • Did you encounter any bugs, glitches, or unexpected behaviour?
  • What was the most satisfying moment?
  • What was confusing or frustrating?
  • Did the game feel fair? Too easy, too hard, or well-balanced?
  • On a scale of 1–5, how likely would you be to play again?
Peer Playtest — Tester 2Evidence
Evaluation Against RequirementsRequired

Return to your Phase 1 requirements. For each one: met / partially met / not met, with evidence from your test table or playtesting.

Architecture ReflectionUnity-specific

Reflect specifically on your code architecture decisions — this is evidence for the algorithms and thinking skills descriptors that goes beyond "my game works".

Address
  • Which script responsibilities were well-defined, and which became "too big" during development?
  • Where did you use events/delegates to decouple components? Did this work as planned?
  • Did you use Coroutines? Were they the right choice vs Update()?
  • What would you refactor if you had another week — and why?
Final ReflectionRequired
Reflect on
  • What was the most significant technical challenge you overcame? How?
  • What design decision are you most proud of and why?
  • What would you do differently in architecture, code structure, or process?
  • How did Unity's automation (physics, animation, NavMesh, etc.) change what was possible for a solo student developer?
  • What skills did you develop that you didn't have at the start?
