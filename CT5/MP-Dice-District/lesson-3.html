Part 1 · Concept (10 min)

How Unity Organises a Scene

Every Unity scene is organised as a hierarchy — a tree of GameObjects where some objects are parents that contain children. Understanding this structure before you build saves hours of confusion later.

For Dice District, your scene will have four main groups. Study the hierarchy below — this is the target structure you will build in Lessons 4–5.

📁 Unity Scene Hierarchy — Dice District
🎬 DiceDistrictScene Scene Root

└ ⚙ GameManager GameManager.cs

├ ⬜ Space_00_Go BoardSpace.cs
├ ⬜ Space_01_HarbourSt BoardSpace.cs
├ ⬜ Space_02_CityTax BoardSpace.cs
… Space_03 through Space_23 (all 24 spaces in order)

└ ▭ GameOverPanel

Why does naming and order matter?

The boardSpaces[] array in GameManager is populated by dragging objects from the Hierarchy into the Inspector. Unity preserves the order you drag them — so Space_00_Go must be index 0, Space_01 must be index 1, and so on. A space in the wrong position will cause players to land on the wrong square. Consistent, numbered naming prevents this.

Unity tip: Use the naming format Space_00_Name from the start. The two-digit number keeps the Hierarchy sorted correctly (Space_09 before Space_10). Rename objects in the Hierarchy before attaching scripts — renaming afterwards also renames the GameObject reference, which can break Inspector links.

Board positioning formula

For a rectangular loop (like Monopoly), calculate each space's position based on which side it's on. For a simple circular loop, use trigonometry. Here is the simplest approach — a rectangle with 6 spaces per side:

Board Positioning — Inspector Values (Rectangle Loop, 24 spaces) 
// Bottom row (spaces 0–5): Z = -5, X = -2.5 to +2.5
// Right column (spaces 6–11): X = +2.5, Z = -5 to +5
// Top row (spaces 12–17): Z = +5, X = +2.5 to -2.5
// Left column (spaces 18–23): X = -2.5, Z = +5 to -5

// Spacing between tiles: 1.1 units (tile width 1.0 + 0.1 gap)
// Y position: 0 for all spaces (flat on the ground plane)

// Example positions for a 6×6 grid loop:
Space_00_Go         Position: (-2.75, 0, -2.75)   // corner
Space_01_HarbourSt  Position: (-1.65, 0, -2.75)

Efficiency tip: Once you have one space correctly configured (scale, script attached, Inspector fields filled), duplicate it with Ctrl+D and only change the position and the Inspector fields. Do not recreate the GameObject from scratch — you will lose the script attachment and material.

① Player HUD (top bar)

Shows all players simultaneously — name, money, and a visual indicator of whose turn it is. Essential for hot-seat play.

② 3D board view (left/centre)

The main camera view of the board. Tokens move here. Players always need to see the board to understand game state.

③ Space info panel (right)

Shows details about the space the current player just landed on — type, price, rent, owner. Driven by the active BoardSpace.

④ Message / action area (bottom)

Contextual message explaining what just happened, plus action buttons (Roll, Buy, Skip). Changes every turn based on game state.

⑤ Dice result display (right panel)

Shows the result of each roll. Keeps the game transparent — players can verify the movement was correct.

⑥ Game Over overlay (not shown)

A full-screen panel that appears when the game ends, showing the winner. Hidden during play; activated by GameManager.

Part 5 · Class Diagram (remainder of lesson)

Finalising Your Class Blueprint

Take the class cards from Lesson 2 and finalise them now — adding your theme's custom attributes and confirming the methods each class needs. This is your build contract: the diagram your group agrees to implement in Lessons 4–8.

What makes a good class diagram at this stage? Every attribute has a data type. Every method has a return type and parameter list (even if just in pseudocode). The diagram is stable enough that a new group member could pick it up and know what to build.

Pseudocode: one method your group must write

Before finishing today, agree as a group on what MovePlayer() should do and write it in pseudocode. Here is the starter — your version should adapt it to your theme and board size.

NSW Pseudocode — MovePlayer() 
BEGIN MovePlayer(player, rollResult)

  // Calculate new position using modulo to wrap around the board
  newPosition ← (player.position + rollResult) MOD boardLength

  // Check if the player has passed or landed on Go
  IF newPosition < player.position OR newPosition = 0 THEN
    player.money ← player.money + goSalary
    OUTPUT player.playerName + " passed Go! Collect $" + goSalary
  END IF

  // Update the player's position
  player.position ← newPosition

  // Move the token GameObject to the new space's world position
  player.token.position ← boardSpaces[newPosition].transform.position

  // Resolve the space the player has landed on
  ResolveSpace(player, boardSpaces[newPosition])

END MovePlayer

Extension Work

Tiered Tasks

Basic
  • Label every space in your board sketch with a number (0–23), space name, type, and price
  • Write one sentence per UI element explaining why the player needs to see it during gameplay
Intermediate
  • Write pseudocode for ResolveSpace(player, space) — handle all four space types
  • Sketch a second wireframe showing the Game Over screen — what information does it need to show?
  • Write three specific, measurable test cases for your game (e.g. "Roll 6 on space 20 → player moves to space 2 (board wraps)")
Advanced
  • Design a UIManager.cs class — list all attributes (TMP references), all methods, and write pseudocode for UpdateHUD()
  • Research Unity's Lerp() function — write a pseudocode outline of how you would animate token movement step-by-step between spaces
  • Write a complete test plan: 10 test cases covering normal play, edge cases (bankruptcy, board wrap, Go salary), and UI states
