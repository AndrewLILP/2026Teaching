CT5-COL-01 Social Awareness · Project Management

Collaborative
& Documented
Project Work

"Collaborates and communicates to manage and document group project work"
Collaborates
Working together effectively
Dividing work appropriately, supporting each other's contributions, resolving disagreements constructively — and demonstrating that the group worked as a team rather than in parallel
Documents
Making the process visible
Recording who did what, what decisions were made and why, how disagreements were resolved — the notebook trail that shows collaboration happened rather than just asserting it

What this outcome means

Collaboration is not parallel work

CT5-COL-01 is commonly misunderstood as "working in a group." Simply being in a group and each doing your own part — one student writes the code, another writes the report, both independently — is not collaboration. It is division of labour. Collaboration requires that group members actively contribute to each other's work: reviewing, questioning, helping to fix problems, and making shared decisions.

The key evidence question for COL-01 is: would this work look different if only one person had done it? If the answer is no — if each person's contribution is entirely independent and the pieces were assembled at the end — then collaboration was not genuinely evidenced, regardless of the group size.

COL-01 in small groups: Dice District groups are typically 2–3 students. In a group of 2, collaboration evidence is particularly important because there is a natural tendency toward strict division — one codes, one documents. The outcome requires evidence that both students engaged with both activities. A group of 2 where one student cannot explain the code and the other cannot explain the design decisions has not collaborated — it has divided.

What "documents" requires

Documentation of collaborative work means more than a list of who did what. It means recording the process of collaboration: group decisions and the reasoning behind them, disagreements and how they were resolved, contributions to each other's work (code reviews, testing each other's features, peer editing notebook entries), and reflections on how the group dynamic affected the project outcome.

Genuine collaboration
What COL-01 requires
  • Group decisions are made together, with reasoning documented
  • Each member can explain all parts of the project, not just their assigned section
  • Members review and give feedback on each other's work during the project
  • When one member is stuck, others actively help solve the problem
  • Disagreements are documented — including how they were resolved
  • Peer test feedback is given and received professionally within the group
Division of labour only
Does not fully evidence COL-01
  • Each member works entirely independently on separate tasks
  • Members combine finished pieces at the end without reviewing each other's work
  • One member drives all decisions; others follow without questioning
  • No evidence of members helping each other when stuck
  • Notebook entries are entirely individual with no cross-reference between members
  • One member can't explain what the other built

Group roles in Dice District

Flexible roles, shared ownership

Assigning roles to group members helps distribute work and provides a framework for accountability. But roles in a 2–3 person group should be flexible, not fixed — the test lead should still understand the code they are testing, and the developer should participate in design decisions. Roles describe primary responsibility, not exclusive ownership.

🧭 Project Lead Lead

Coordinates the group's progress against the D&P framework, manages scope decisions, facilitates group discussions, and ensures all members are unblocked.

  • Facilitates scope priority decisions in Lesson 3
  • Monitors progress against the lesson checklist each session
  • Raises blockers with the teacher and coordinates the response
  • Leads the D&P retrospective discussion in Lesson 10
  • Documents group decisions in the notebook with reasoning
Notebook evidence: group decision log entries, scope revision justifications, retrospective facilitation notes
💻 Lead Developer Dev

Primary responsibility for C# implementation — writing, explaining, and pairing on the core scripts. Ensures all group members understand the code, not just the developer who wrote it.

  • Writes and explains starter scripts in Lessons 4–7
  • Pairs with other members when implementing complex methods
  • Reviews and explains any adapted or AI-assisted code before it is merged
  • Documents code decisions and design trade-offs in the notebook
  • Leads the class diagram and pseudocode sessions in Lessons 2–3
Notebook evidence: annotated pseudocode, code decision rationale, pair programming session notes
🧪 Test Lead Test

Primary responsibility for the test log — designing test cases, predicting expected outputs, and documenting results. Also coordinates the peer testing protocol in Lesson 9.

  • Writes expected outputs before tests are run each lesson
  • Coordinates the 28-item peer test checklist in Lesson 9
  • Maintains the bug log and priority fix list
  • Documents edge cases and regression tests after bug fixes
  • Leads the requirements evaluation in Lesson 10
Notebook evidence: test log entries with expected/actual outcomes, bug log with causes, peer test checklist documentation
📓 Documentation Lead Doc

Primary responsibility for the project notebook — ensuring all phases are documented in real time, not retrospectively. Also leads the wireframe, requirements list, and written evaluation.

  • Maintains the notebook chronologically with lesson headings and dates
  • Writes and annotates the wireframe in Lesson 3
  • Leads requirements writing in Lesson 1 and evaluation in Lesson 10
  • Ensures group decisions and reflections are recorded after each lesson
  • Reviews notebook before submission for completeness
Notebook evidence: consistent, dated entries; requirements list; annotated wireframe; written evaluation; D&P retrospective
Role rotation for small groups: In a group of 2, roles should rotate across lessons so both members have experience in each area. A useful rotation: Student A leads development in Lessons 4–5, Student B leads in Lessons 6–7. Both co-author the test log. This rotation is itself COL-01 evidence — note it in the notebook with reasoning.

Task delegation and tracking

Dividing work with accountability

Effective task delegation requires three things: clarity (everyone knows who is responsible for what), trackability (progress can be assessed without asking), and flexibility (tasks can be reassigned when someone is blocked). The contribution log below is the notebook format used in Dice District to document task delegation across lessons.

Contribution log format

A contribution log is a simple table updated at the end of each lesson — not a retrospective summary at the end of the project. Each entry records what was planned, what was completed, who did it, and what changed from the plan.

Lesson Task Planned owner Completed by Notes / changes
L1 Write requirements list (F + NF) Both members Both members Student A drafted F requirements; Student B drafted NF requirements; both reviewed and revised the final list together.
L2 Annotate starter code (analysis activity) Student B (Doc) Student B Completed. Student A reviewed annotations and added 3 additional notes on the GameManager class.
L3 Scope priority planner Both members Both members Disagreement on whether Lerp movement was Must or Nice — resolved by moving to Nice (see decision log L3-D1).
L5 Implement MovePlayer() + test log Student A (Dev) Student A, with Student B writing expected outputs Student A coded; Student B independently predicted expected outputs before running each test. Two failures found and fixed by both members together.
L8 Dice Display + Space Info Panel Student B (rotated to Dev) Student B Role rotation from L8: Student B leads implementation; Student A reviews and tests. Space Info Panel partially complete — carried to L9 fix list.
What distinguishes strong from weak contribution logs: A strong log shows that tasks changed from plan — work was reassigned, timelines slipped, new tasks were identified. A log where every entry says "completed as planned" either reflects a very smooth project or a retrospective document. The notes column is the most important: it should describe what actually happened, including problems and how they were resolved.

Managing disagreement

Conflict is normal — avoidance is not collaboration

Groups that never disagree have either one person dominating all decisions, or a group where no one cares enough about the outcome to advocate for a different approach. Neither is collaboration. Managed disagreement — documented in the notebook, resolved through a defined process, with the outcome and reasoning recorded — is one of the strongest forms of COL-01 evidence available.

Technical disagreement
How to implement it
Two members want to implement the same feature differently — e.g. one wants to use a switch statement, the other an if-else chain for ResolveSpace().
Strategy: each member codes a prototype; both are tested against the same test cases; the group selects the cleaner or more extensible option and documents why. Both prototypes are noted in the notebook.
Scope disagreement
What to include
One member wants to implement the AI opponent (Advanced tier); the other thinks the core game still has bugs that need fixing first.
Strategy: use the priority matrix — Critical bugs before new features. The group agrees that Basic tier must be stable before attempting Advanced, and documents this rule as a group decision in the notebook.
Workload disagreement
Unequal contribution
One member feels they are doing more than their share; the other is stuck on a problem and not contributing as expected.
Strategy: raise it explicitly — not punitively, but problem-solving. Identify specifically what the stuck member needs (explanation, simpler task, teacher help). Document the rebalancing in the contribution log. If unresolved, raise with the teacher.
Design disagreement
What it should look like
Members disagree on the game theme, board layout, colour scheme, or UI structure.
Strategy: refer back to requirements and user needs. The question is not "which do I prefer?" but "which better serves the player?" Evaluate options against the L1 requirements and Shneiderman's principles; document the decision rationale.
Quality disagreement
Is it good enough?
One member wants to move on; the other believes a feature needs more work before it can be called complete.
Strategy: apply the test log. If all test cases pass, it is done. If any fail, it is not. Remove the subjective element by referring to the defined acceptance criteria. Document what criteria were applied and who reviewed them.
Process disagreement
How we're working
One member wants to plan more before coding; the other wants to try things and iterate. Tension between planning and prototyping approaches.
Strategy: the D&P framework resolves this — R&P comes before P&I. Pseudocode should precede code. Agree on a minimum planning threshold (pseudocode for each method) before coding begins, and document this as a group working agreement.

Version control for teams

Protecting and sharing the project

Version control in a group context solves a different problem than in a solo project. Solo: protecting against accidental loss. Group: coordinating changes from multiple contributors so they don't overwrite each other. Both are relevant to COL-01 — the chosen strategy should be documented and applied consistently.

📁Manual sprint backups (recommended for most groups)
Copy the entire Unity project to a shared Google Drive folder at the end of each lesson, named by lesson number. One designated member owns the current "working version" at any time.
Simple — no tool knowledge required
Shared Drive means all members have access
Sprint boundaries provide natural snapshot points
No merge capability — only one person can make changes at a time
Difficult to see what changed between versions
Requires discipline to name versions consistently
🐙GitHub Desktop (extension approach)
A private GitHub repository with GitHub Desktop for commits. Each member works on a branch; merges happen at lesson boundaries with a commit message describing changes.
Full history — every change is traceable
Commit messages are COL-01 evidence of individual contributions
Industry-standard practice
Learning overhead — merge conflicts can be stressful
Unity generates many auto-generated files that need careful .gitignore configuration
Public repos expose code — must use private repo
Regardless of which strategy is used: document it in the notebook. Name the strategy, describe who is responsible for each backup or commit, and record when it was applied. The notebook entry "Sprint backup strategy: Student A copies project to shared Drive at end of each lesson, named DD-L[number]-[date]. Student B verifies the copy opens correctly before next lesson." is stronger COL-01 evidence than simply having backups with no documentation of the shared decision.

Notebook evidence guide

What COL-01 looks like in the notebook

Because collaboration is a process rather than a product, the notebook is the primary evidence vehicle for COL-01. Eight specific evidence types below — each one distinguishes a student who genuinely collaborated from one who worked alongside their group.

All lessons
Group decision log
A dedicated section (or running list) in the notebook recording group decisions: what was decided, who proposed it, who disagreed, and what reasoning led to the final choice. Even simple decisions ("we chose a space station theme because…") count.
Strong evidence: decisions show genuine deliberation — multiple options considered, a reason given for the chosen option. References to other members' views ("Student B argued that… we agreed because…").
L1, L3, L10
Co-authored artefacts
Requirements list, scope planner, and evaluation written collaboratively — each member can speak to the reasoning behind individual items, not just the final list.
Strong evidence: the notebook entry identifies which member contributed which requirement or scope decision, with annotations showing where members challenged or refined each other's ideas.
L4–L8
Pair programming notes
A brief note after any lesson where both members worked on the same code together: who typed, who reviewed, what was discussed, and any disagreements about implementation approach.
Strong evidence: the non-typist's contributions are noted — "Student B suggested extracting the ownership check into IsOwnedByOther() instead of inline in ResolveSpace()" is a concrete collaboration contribution.
L5–L7
Cross-tested work
Evidence that members tested each other's implementations — one member wrote the code, the other wrote the expected test case outputs independently before running. The test log records who predicted each expected outcome.
Strong evidence: expected outputs written before running, identified by which member made the prediction. Disagreements between predictions are noted and investigated.
L3, L8
Documented disagreements
At least one notebook entry recording a genuine disagreement between group members — what the disagreement was about, how each member's position was articulated, and how the group reached resolution.
Strong evidence: the resolution references objective criteria (test results, requirements, Shneiderman's rules) rather than a member just deferring to the other. Both positions are described fairly.
L9
Peer test facilitation
The peer test protocol in Lesson 9 is a formal collaboration activity. The notebook records: how feedback was given and received, how the group responded to negative findings, and how the priority fix list was agreed upon.
Strong evidence: the priority fix list shows that both members' views on severity ratings were considered. At least one fix is explicitly prioritised because of a specific peer tester's finding.
L10
Collaborative retrospective
The D&P retrospective in Lesson 10 is written by both members — not divided into two halves. It reflects on the group process as well as the product: what worked about how the group collaborated, what would change.
Strong evidence: retrospective includes specific moments of collaboration that worked well (e.g. "the pair programming session in L6 caught the ownership bug before it reached the test log") and honest reflection on when collaboration broke down.
All lessons
Role rotation record
A log of which member held which role each lesson, with a note when roles rotated and why. Demonstrates that both members developed broad competency across the project, not just depth in one area.
Strong evidence: rotation decisions are justified ("Student B led development in L6–L7 so both members had experience implementing complex methods") not random. Both members can explain work from all phases.

Exam preparation

Question types and model answers

Distinguish collaboration from division 4 marks
"Two students are working on a game project. Student A writes all the code; Student B writes all the documentation. They combine their work at the end and submit together. Explain why this may not fully satisfy CT5-COL-01, and describe two changes that would improve their collaborative practice."
B-range response
This approach is a division of labour rather than genuine collaboration. COL-01 requires that both students actively contribute to each other's work, not just divide it. Student B may not understand the code they are documenting, and Student A may not understand the design decisions behind the documentation. Two improvements: (1) Both students should review each other's work during the project — Student B should read and question the code before it is finalised. (2) Decisions about scope, design, and implementation should be made together and recorded in a group decision log, so both members contribute to the direction of the project.
A-range response
This arrangement is parallel work, not collaboration. COL-01 asks students to "collaborate and communicate to manage and document group project work" — the emphasis is on the joint management of shared work, not the parallel production of separate deliverables. The test of genuine collaboration is whether the work would look different with only one person: if Student A's code is entirely independent of Student B's documentation decisions, they have not influenced each other's work.

Improvement 1: Cross-review sessions. After Student A implements a method, Student B independently predicts the expected test case outputs before they are run. This requires Student B to understand the code well enough to reason about its behaviour — and creates documented evidence of cross-contribution in the test log (who predicted which expected outcome).

Improvement 2: Joint decision documentation. Before any implementation lesson begins, both students write the pseudocode for the lesson's target method together. The notebook records which member proposed which algorithmic step and any disagreements about sequencing or structure. This ensures that documentation is not a retrospective activity — it is part of the planning that directly influences what Student A implements.
A-range names the specific outcome language ("collaborate and communicate to manage and document") and applies the test of whether work would look different with one person. Both improvements are concrete, testable, and connect to specific COL-01 evidence types — cross-prediction in the test log and joint pseudocode authorship in the notebook. B-range identifies the right problem but proposes improvements that are vague ("review each other's work") rather than specific.
Resolve a group conflict 4 marks
"A group cannot agree on whether to implement the AI opponent feature or spend the time fixing remaining bugs in the core game. Describe how a computing group should approach this disagreement, and explain how the resolution should be documented."
B-range response
The group should have a discussion and list the pros and cons of each option. They should consider how much time is left, how severe the bugs are, and whether the AI is a required feature. They should agree on a decision and write it in their notebook, noting who supported each option and what reasoning led to the final choice. Fixing critical bugs should probably take priority over new features.
A-range response
The group should resolve this disagreement using objective criteria rather than preference. The D&P framework provides a clear answer: the scope priority planner from Lesson 3 classified features as Must / Nice / Cut. If the AI opponent was classified as "Nice" or "Cut" and core game bugs are classified as "Critical" on the severity scale, the resolution is determined by the group's own prior decision — not a fresh preference vote. This prevents the disagreement from becoming personal.

If the scope planner is ambiguous, the group should apply the principle that a stable C-range product is a better outcome than an ambitious but broken B-range attempt. This principle should be stated explicitly, agreed to by both members, and recorded as a group decision in the notebook.

Documentation: The notebook entry should record (1) both positions accurately — what each member argued; (2) the criteria used to resolve it (scope planner, severity table, time estimate); (3) the agreed outcome; (4) any conditions attached (e.g. "if the top three Critical bugs are fixed with 20 minutes remaining, attempt AI opponent stub"). Documented disagreement and resolution is stronger COL-01 evidence than unanimous agreement — it shows the group had genuine investment in different outcomes and navigated it professionally.
A-range anchors the resolution in the group's own prior decisions (scope planner from L3, severity scale) rather than ad hoc preferences — this is the key insight that elevates the answer. The note that documented disagreement is stronger evidence than unanimous agreement is sophisticated: it shows understanding of what COL-01 is actually assessing. The conditions-attached outcome demonstrates practical project management thinking.
Describe version control for groups 3 marks
"Describe an appropriate version management strategy for a two-student team working on a Unity game project. Explain two specific risks the strategy addresses."
B-range response
A good strategy is to copy the project to a shared Google Drive folder at the end of each lesson, labelled with the lesson number and date. This addresses the risk of a computer breaking or files being accidentally deleted. It also means both students can access the project, not just the one whose computer it's saved on, which prevents one student from being blocked if the other is absent.
A-range response
Strategy: at the end of each lesson, one designated member (rotating each session) copies the entire Unity project folder to a shared Google Drive folder named DD-L[number]-[YYYYMMDD]-working. Before any major change in the next lesson, the current working version is also backed up as DD-L[number]-pre-[change description]. Both members verify the backup opens correctly before the lesson ends.

Risk 1 — Hardware failure or accidental deletion: If one student's laptop is damaged or the project file is accidentally overwritten, the team can restore from the most recent Drive backup without losing more than one lesson's work. Storing only on one device means a single point of failure for the whole project.

Risk 2 — Merge conflicts from concurrent editing: Without a version strategy, if both students open and modify the project simultaneously, Unity's binary scene files cannot be merged — one version will overwrite the other. By designating one "working version" owner per session and backing up before merging contributions, the strategy prevents one student's changes from silently destroying the other's. The backup history also provides a recovery path if a merge goes wrong.
A-range gives a specific naming convention (making the strategy auditable), describes the pre-change backup step (showing awareness of when to create checkpoints beyond just end-of-lesson), and identifies two distinct and specific risks — hardware failure and concurrent edit conflicts. The Unity binary scene file detail demonstrates domain-specific knowledge that elevates the answer from generic to technically grounded. B-range correctly identifies the right strategy and one risk, but the second "risk" (absence access) is a convenience rather than a risk.
