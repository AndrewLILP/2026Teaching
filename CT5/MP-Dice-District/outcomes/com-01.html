What this outcome means

Communication is not just writing

CT5-COM-01 covers the full range of digital communication: written documentation, code comments, pseudocode, diagrams, wireframes, presentations, and the game UI itself. All of these are acts of communication — each with an intended audience, a purpose, and an appropriate form. The outcome requires that students select the right tool for each context, not just produce output in whatever format feels familiar.

The key word is appropriate. A technically perfect class diagram communicated to a non-technical audience without any accompanying explanation is not appropriate communication — the format doesn't match the audience. A spoken description of an algorithm without pseudocode is not appropriate for a developer audience — the format doesn't match the precision the audience needs. Selection requires knowing both what the communication needs to say and who it needs to say it to.

COM-01 is the outcome that ties everything else together: pseudocode is technical communication (OPL-01 + COM-01); wireframe annotations are design communication (DES-01 + COM-01); evaluation writing is analytical communication (EVL-01 + COM-01); the project notebook is process communication (DPM-01 + COM-01). A student who communicates well across all formats demonstrates COM-01 throughout the project — not only in a dedicated "communication" activity.

Three dimensions of communication selection

Selecting an appropriate communication tool requires considering three dimensions simultaneously:

  • 1. Audience: Technical (developer, teacher) vs general (student player, parent) — different vocabulary, different assumed knowledge, different level of detail required.
  • 2. Purpose: To instruct (pseudocode, step-by-step guide), to inform (test log, evaluation), to persuade (design justification), or to document (notebook entry, code comment).
  • 3. Context: The medium (on-screen, printed, embedded in code), the time available to the reader, and the consequences of misunderstanding — all shape which format is appropriate.

Communication formats

Six formats used in Dice District

Each format below serves a specific communication purpose. Selecting a format means knowing when a diagram is clearer than text, when code is clearer than a description, and when plain language serves the reader better than technical vocabulary.

📐
NSW Pseudocode
Code-like

Communicates algorithm logic precisely, independent of any programming language syntax. Readable by anyone who knows the conventions — developer and teacher alike.

Appropriate when: communicating a planned algorithm to a technical audience before implementation; explaining logic in a test or exam answer where full C# syntax is unnecessary.
Dice District: game loop pseudocode (L2), MovePlayer() pseudocode (L3), per-method pseudocode in test logs (L5–L7).
🗂
Class Diagram
Visual

Communicates the architecture of an OOP system — components, their attributes and methods, and the relationships between them. More efficient than describing each class in prose.

Appropriate when: communicating a planned or existing OOP architecture to a developer audience; showing how data flows between classes.
Dice District: initial class diagram (L3), updated diagram with delta annotations (L7) showing what changed and why.
🖼
Annotated Wireframe
Visual

Communicates UI intent to both developer and non-developer audiences. The box layout communicates structure; the annotations communicate purpose, behaviour, and UX intent.

Appropriate when: communicating UI design before implementation; justifying layout decisions; bridging the gap between visual designers and developers.
Dice District: L3 wireframe with 5+ annotated elements describing behaviour and communication intent for each zone.
💬
Code Comments
Code-like

Communicates intent, reasoning, and context within the code itself — to future developers (including yourself) reading the code later. Good comments explain why, not just what.

Appropriate when: explaining non-obvious design decisions, documenting known limitations, marking sections for later replacement (stubs), attributing adapted code.
Dice District: // temporary — replaced in L7 on stubs; // clamp prevents index skip after list removal on the index correction.
📝
Written Evaluation
Written

Communicates analytical judgements about the project to an assessor audience. Structured prose with evidence citations — neither a list of features nor a stream of consciousness.

Appropriate when: explaining the reasoning behind design decisions; evaluating the solution against requirements; proposing technically specific improvements.
Dice District: L10 four-question written evaluation — each answer structured as claim → evidence → reasoning → implication.
🎮
The Game Itself (UI)
Visual

The game UI communicates game state to players — whose turn it is, what just happened, how much money each player has. Every interface element is an act of communication to a non-technical audience.

Appropriate when: the audience is the player, not the developer. Communicates through layout, typography, colour, and motion — not text descriptions.
Dice District: message panel (communicates events), HUD (communicates state), space info panel (communicates context), Game Over screen (communicates outcome).

The selection decision

How to choose the right format

The selection in "selects and applies" is demonstrated through documented decisions — not just producing the right format, but explaining why that format was chosen for that audience and purpose. The framework below walks through the five questions that drive format selection.

Question Technical audience answer General audience answer
Who is reading this?
 A developer, teacher, or assessor who understands computing terminology and can read code or diagrams A player, classmate, parent, or peer who doesn't need to know how it works — only what it does
What do they need to know?
 Precisely how the system works — the algorithm, the data structure, the method names, the edge cases What the game does and whether it works as expected — no implementation detail required
What level of detail is appropriate?
 Enough detail to reproduce, verify, or extend the work — specific field names, method signatures, test cases Enough detail to understand the result — feature names, plain outcomes, no jargon
What format communicates most efficiently?
 Pseudocode (algorithms), diagram (architecture), table (test cases), structured prose (evaluation) Plain English prose, labelled screenshot, simple list of features, game demo
What is the risk of misunderstanding?
 High risk if ambiguous — technical communication must be precise. Ambiguity in pseudocode or a class diagram creates implementation errors. High risk if too complex — general communication must be accessible. Jargon in player-facing text breaks the user experience.
The selection is the evidence: A notebook entry that says "I chose a class diagram rather than a prose description because the diagram communicates the relationships between classes more efficiently — a prose description of BoardSpace, Player, and GameManager and their connections would require three paragraphs to say what the diagram shows in one glance" is A-range COM-01 evidence. The format plus the justification is what the outcome asks for.

Code as communication

Code comments that actually communicate

Code documentation is one of the most directly assessable forms of COM-01 in a programming project. Good comments communicate to the future reader of the code — which may be your collaborator in two days, your teacher marking the submission, or yourself returning to a method you wrote three weeks ago. The quality bar is: can a developer understand why this code does what it does, without running it?

✗Weak documentation — C-range

// get index
int idx = activePlayers.IndexOf(p);

// remove player
activePlayers.Remove(p);

// fix index
if (idx <= currentIndex && currentIndex > 0)
    currentIndex--;

Comments restate what the code does — they add no information a reader couldn't get from the code itself. "get index" tells us nothing about why the index must be retrieved before removal, which is the critical insight.
✓Strong documentation — A-range

// Must save index BEFORE Remove() —
// IndexOf() returns -1 after removal
int idx = activePlayers.IndexOf(p);

activePlayers.Remove(p);

// After removal, elements at index > idx
// shift down by 1. Without this correction,
// currentIndex skips the next player.
if (idx <= currentIndex && currentIndex > 0)
    currentIndex--;

Comments explain why — specifically, what breaks if the order changes (IndexOf returns -1) and what breaks if the correction is omitted (next player is skipped). A developer can understand the risk without running the code.

What to comment — and what not to

Not everything needs a comment. Over-commenting creates noise that makes the important comments harder to find. The principle: comment the why, not the what. If the code is readable and the intent is clear, no comment is needed. If the code is non-obvious, surprising, or depends on an ordering constraint, a comment is essential.

Comment quality spectrum — EliminatePlayer() excerpt GameManager.cs 
// ── EliminatePlayer ─────────────────────────────────────────
// Called when ChargeMoney() returns false (player bankrupt).
// Releases properties, removes player from active list,
// corrects turn index, then checks for game end.
public void EliminatePlayer(Player p)
{
    // Release all properties so other players can buy them
    foreach (BoardSpace s in p.ownedProperties)
        s.owner = null;
    p.ownedProperties.Clear();

    // Save index BEFORE Remove() — IndexOf returns -1 after
    int idx = activePlayers.IndexOf(p);
    activePlayers.Remove(p);
    p.token.gameObject.SetActive(false);

    // Correct index: list shift after removal can cause
    // currentIndex to skip the next player
    if (idx <= currentIndex && currentIndex > 0)
        currentIndex--;

    // No comment needed — intent obvious from method name
    if (activePlayers.Count <= 1)
        GameOver();
    else
        NextTurn();
}

The method header comment communicates purpose and entry conditions to a developer reading the file for the first time. The inline comments explain the two non-obvious operations. The final if/else needs no comment — activePlayers.Count <= 1 and GameOver() communicate their own intent.

Communication across the project

Every artefact is a communication act

COM-01 evidence is distributed across the entire project — it is not a separate activity. The table below maps every major Dice District artefact to its communication format, intended audience, and purpose. The selection column explains why this format was chosen for this audience, which is the evidence COM-01 specifically requires.

Artefact Format Audience Why this format is appropriate
Requirements list (L1)
 Written Developer + assessor Numbered list with functional/non-functional labels makes requirements scannable and cross-referenceable in the L10 evaluation. Prose description would obscure individual testable conditions.
Game loop pseudocode (L2)
 Code-like Developer + assessor NSW pseudocode communicates algorithm logic in language-independent, precise steps. More efficient than prose for sequential logic; more accessible than C# for a non-programmer assessor.
Class diagram (L3, updated L7)
 Visual Developer + teacher A diagram communicates three-way class relationships in a single glance that would require three paragraphs in prose. Relationships and data flows are visual, not textual, in nature.
Annotated wireframe (L3)
 Visual Written Developer + designer The box layout communicates structure efficiently; annotations communicate purpose and UX intent that the layout alone cannot convey. Combined format bridges spatial and explanatory communication.
Test log tables (L5–L7)
 Written Developer + assessor A table with Input / Expected / Actual / Result columns communicates test outcomes efficiently and makes Pass/Fail status scannable. Prose test descriptions would obscure individual case results.
Code comments (L4–L8)
 Code-like Developer (future reader) Inline with code so context is never lost. Explains why, not what — non-obvious ordering constraints, stub intentions, and algorithmic reasoning that the code itself cannot communicate.

 Written Developer + tester Structured fields (description, severity, steps to reproduce, cause) communicate exactly what a developer needs to reproduce and fix a bug. Freeform prose would omit critical reproduction details.

Written evaluation quality

Communicating analysis in writing

The written evaluation in Lesson 10 is the most explicitly assessed piece of written communication in the project. It communicates to an assessor audience — which means precision, evidence, and structure matter more than casual voice or length. Two versions of the same evaluation paragraph below illustrate the quality gap.

✗Weak evaluation writing — C-range

"Our game mostly works well. We implemented most of the features we planned. The movement system worked correctly and players could buy properties and pay rent. We had some bugs in the peer test but fixed most of them. If we had more time we would add more features like houses and hotels which would make the game more interesting and fun."

Vague ("mostly works well"), unsupported ("worked correctly" — by which test?), no requirement names, the improvement is described as "more features" without technical detail, and "interesting and fun" is not an evaluative criterion from the requirements list.
✓Strong evaluation writing — A-range

"The game meets 7 of its 8 functional requirements. Requirements F1–F5 (movement, purchase, rent, tax, bankruptcy) were verified through the structured test logs in Lessons 5–7, with all test cases passing after the index correction bug was fixed in L7. Requirement F6 (Chance card system) was partially met — the Fisher-Yates shuffle and draw mechanism function correctly, but only 5 of the 8 planned events were implemented before the peer test deadline."

"If time permitted, I would implement property improvement (houses/hotels) by adding a houseCount field to BoardSpace and replacing the static rentAmount with a CalculateRent() method that returns tiered rent from a lookup array. This would address the strategic depth gap identified in the L10 requirements evaluation."

Specific requirement numbers cited, test evidence referenced by lesson, honest partial-met acknowledgement with specific numbers (5 of 8), and the improvement is technically specific — field names, method signatures, and the connection back to an evaluation finding.

Exam preparation

Question types and model answers

Justify a format selection 4 marks
"A developer wants to communicate how the board wraps from the end back to the beginning to both their project partner and to a player who will read a help screen. Describe the most appropriate format for each audience and justify your choices."
B-range response
For the project partner: pseudocode showing the modulo calculation (position + roll) MOD boardLength, because this gives the exact algorithm their partner needs to implement or test the feature. For the player: plain English on the help screen — something like "when you reach the last space, your token continues from the beginning" — because players don't need to know how the code works, only what the game does.
A-range response
Project partner (technical audience): NSW pseudocode with a trace table showing two examples — a normal move (position=5, roll=4 → newPos=9) and a wrap case (position=21, roll=6 → newPos=3, passedGo=true). This format is appropriate because the partner needs to verify the logic is correct, understand the preconditions (passedGo detection), and know the exact formula — not just that wrapping occurs. A prose description like "the player wraps around" doesn't communicate the modulo operator, the board length constant, or the Go-salary interaction. Pseudocode with examples does all three.

Player (general audience): Plain English on a help screen, possibly with a small animated diagram if the game supports it: "When you roll past the last space, your token laps the board and you collect $300 from the bank." This is appropriate because the player doesn't need the formula — they need to know the outcome (lapping) and the consequence (salary). Technical notation would confuse rather than inform, and violates Shneiderman's Rule 8 (reduce memory load) by requiring the player to interpret notation rather than simply understand the game.
A-range gives a trace table example rather than just naming the format, explains what the pseudocode communicates that prose cannot (the modulo formula, passedGo interaction), and connects the general-audience choice to a named design principle (Shneiderman Rule 8). The distinction between "what the formula is" and "what the outcome is" perfectly captures the audience-awareness that COM-01 requires.
Improve a code comment 3 marks
"The following comment appears in a student's code: // check if player has enough money. Rewrite this comment to meet A-range communication standards, and explain what makes the improved version more effective."
B-range response
Improved comment: // ChargeMoney returns false if the player cannot afford the amount — triggers bankruptcy check.

This is better because it explains what happens when the condition is false, not just what the code checks. The original comment just restates what the code obviously does.
A-range response
Improved comment:
// ChargeMoney(amount) deducts amount and returns false if money drops below 0. // A false return triggers EliminatePlayer() — do not call ChargeMoney() during the purchase prompt (waitingForInput guard prevents this).
This version is more effective for three reasons: (1) it documents the return value contract — what ChargeMoney returns and what that return value means, which the original comment omits entirely; (2) it names the consequence of a false return (EliminatePlayer()), giving the reader a complete picture of the control flow without having to read further; (3) it flags a usage constraint (don't call during purchase prompt) that would cause a bug if violated — this is exactly the kind of non-obvious information a comment should communicate. The original comment tells the reader what they could already see from the method name — it adds no value.
A-range introduces the concept of a "return value contract" (documenting what the return value means to the caller, not just what the method does internally), names three distinct improvements, and specifically calls out the usage constraint as the most important thing to document. The final sentence ("tells the reader what they could already see from the method name") neatly summarises why the original comment violates the "explain why, not what" principle.
Evaluate communication quality 4 marks
"A student submits a project notebook with the following evaluation entry: 'Our game works well and we met all our requirements. If we had more time we would add more features.' Identify two specific weaknesses in this communication and explain how each should be improved."
B-range response
Weakness 1: "works well" is vague — there is no evidence cited. A stronger evaluation would reference specific test cases that confirm the game works, or name specific requirements that were met. Weakness 2: "add more features" is not specific enough. A strong evaluation would name the specific feature, explain how it would be implemented technically, and connect it to a gap identified in the requirements evaluation.
A-range response
Weakness 1: Unsupported claim. "Works well and met all our requirements" is an assertion without evidence. COM-01 requires communication that is appropriate to its audience — an assessor audience expects claims to be supported by evidence, not stated as fact. A stronger version cites specific requirements by name or number, references the test cases that verified them, and acknowledges any requirements that were not fully met. Suppressing negative findings is not appropriate communication to an assessor — it makes the evaluation unconvincing rather than more positive.

Weakness 2: Technically vague improvement. "Add more features" communicates nothing to a technical audience — it doesn't name the feature, explain how it would be implemented, or connect to a specific gap in the current product. Appropriate communication to an assessor requires technical specificity: which class would need a new field, what method would implement the feature, how the new code would integrate with the existing architecture. A vague improvement proposal suggests the student can identify that something is missing but cannot reason technically about how to address it — which is a communication failure as much as a knowledge gap.
A-range frames both weaknesses in terms of what appropriate communication to an assessor audience requires — not just "be more specific" but "assessor audiences expect evidence-backed claims and technically reasoned proposals." The insight that suppressing negative findings makes the evaluation unconvincing (rather than more positive) is particularly strong — it shows understanding of how communication works for an audience who can tell the difference.
