What this outcome means

Data management is not just about databases

CT5-DAT-01 covers how data is selected, organised, stored, and documented within a computing project. In a game development context, this means: the data types used in scripts, the data structures chosen for collections, the way the project itself is organised (folder structure, version control, naming conventions), and how work practices are documented in the notebook.

The outcome has two layers. The technical layer is about data types, structures, and their selection — choosing List<Player> rather than Player[] because the list is dynamic, and being able to explain that choice. The management layer is about how project data (code, assets, backups, documentation) is organised and preserved — folder structures, naming conventions, and version management strategies.

DAT-01 in Dice District: Every data type decision in the three scripts is DAT-01 evidence — int vs float for money, List<Player> vs Player[] for active players, bool for game state flags, string for names. Each choice was made for a reason, and documenting those reasons in the notebook is what distinguishes C-range from A-range work on this outcome.

Data types

Selecting the right type for the data

Data type selection is not arbitrary. Each type has a range, precision, and purpose that makes it suitable for some data and unsuitable for others. Using the wrong type creates bugs — an int storing a probability, a float storing money that accumulates rounding errors, a string storing a value that should be an enum. The selection and the justification are both evidence for DAT-01.

int
Integer — whole numbers

Stores whole numbers — no decimal component. Range: −2,147,483,648 to +2,147,483,647 in C#.

Range(1,7), roll, position, money, rentAmount
Dice District: money is int not float — currency in this game has no fractional values, and float arithmetic can introduce rounding errors that cause unexpected bankruptcies.
float
Floating-point — decimals

Stores decimal numbers. Approximate — not suitable for financial calculations requiring exact values.

moveSpeed, lerp t-value, probability calculations
Dice District: moveSpeed = 0.4f in the Lerp animation — timing values are inherently approximate, making float appropriate here where it would not be for money.
bool
Boolean — true/false

Stores exactly two states. More readable and type-safe than using int (0/1) for binary state.

isActive, gameOver, waitingForInput, passedGo
Dice District: waitingForInput gates the Roll button — a boolean is the exact right type because the state is precisely binary: either waiting for a purchase decision or not.
string
Text — character sequences

Stores text of any length. In Unity: player names, message panel content, space names.

playerName, spaceName, groupColour (hex)
Dice District: spaceName is string — space names are theme-specific text set in the Inspector per project. An enum would require recompilation for each new theme.
enum
Enumeration — named constants

A set of named integer constants. Prevents invalid values at compile time — a string-based type check would require runtime validation.

SpaceType.Property, SpaceType.Tax, SpaceType.Go
Dice District: SpaceType enum ensures only valid space types are assigned — the compiler prevents typos like "proprety" that a string comparison would miss until runtime.
Color
Struct — composite value

A Unity struct storing RGBA values. Used for visual identification of groups in BoardSpace.

groupColour — identifies property colour group
Dice District: groupColour is Color not string — it drives the material tint directly in Start() without a string-to-Color conversion step.

Data structures

Choosing the right collection

A data structure organises multiple values. Choosing between an array and a list — or deciding which data belongs in which structure — requires understanding what operations the data will need to support. The key questions: Is the size fixed or dynamic? Do elements need to be added or removed at runtime? Is order important? Is random access by index required?

Structure Characteristics When to use Dice District use
Array T[]
 Fixed size, index access, contiguous memory. Cannot add or remove elements after creation. When the number of elements is known at design time and never changes. boardSpaces[] — the board has exactly 24 spaces; this never changes during play. Array is appropriate because random access by index (position) is the primary operation.
List List<T>
 Dynamic size, index access, Add() and Remove() methods. Backed by an array but resizes automatically. When elements will be added or removed at runtime, but order and index access are still needed. activePlayers — players are eliminated during the game, shrinking the list. ownedProperties on each Player — properties are added during play.
Dictionary Dictionary<K,V>
 Key-value pairs, O(1) lookup by key, unordered. When data needs to be looked up by a meaningful key rather than a numeric index. Not used in core Dice District — but would be appropriate for a Chance card lookup table: Dictionary<string, ChanceCard> keyed by card ID.
Queue Queue<T>
 First-in, first-out (FIFO). Enqueue() adds to back, Dequeue() removes from front. When items must be processed in the order they arrive — e.g. a card draw pile. An alternative to List for the Chance card deck — Queue<ChanceCard> makes DrawCard() semantically clearer than RemoveAt(0) on a List.
The selection justification is the evidence: "I used List<Player> for activePlayers because players are eliminated during the game — the list shrinks dynamically. I used BoardSpace[] for boardSpaces because the board size never changes and I need O(1) index access by position number" is DAT-01 evidence. "I used a list" is not.

Project data management

Organising the project itself

Beyond the data in scripts, DAT-01 also covers how the project's assets, files, and documentation are organised. A well-managed Unity project has a consistent folder structure, meaningful naming conventions, and a version management strategy. These are the "manages and documents work practices" aspects of the outcome.

Unity folder structure
  • Assets/Scripts/ — all C# scripts
  • Assets/Scenes/ — Unity scene files
  • Assets/Models/ — 3D assets and prefabs
  • Assets/Audio/ — sound effect clips
  • Assets/Materials/ — player/space materials
  • Assets/Fonts/ — TextMeshPro font assets
  • Documenting the chosen structure in the notebook establishes a standard all group members follow — deviation from the standard is itself a management decision worth documenting.
Naming conventions
  • Scripts: PascalCase — GameManager.cs, BoardSpace.cs
  • Public fields: camelCase — playerName, goSalary
  • Methods: PascalCase — RollDice(), NextTurn()
  • Constants: ALL_CAPS or descriptive camelCase — depends on team agreement
  • Scene objects: descriptive — Space_00_Go, Player1Token
  • Backup folders: DD-L05-20250510 — lesson number + date

Exam preparation

Question types and model answers

Justify a data type choice 4 marks
"A developer is storing the list of active players in a board game. Explain why List<Player> is a more appropriate data structure than Player[], and describe one situation where the array would be more appropriate."
B-range response
List<Player> is better because players can be eliminated during the game, which means the number of active players changes. A list allows elements to be removed with Remove() without breaking the structure. A Player[] has a fixed size, so removing a player would require shifting all subsequent elements manually or leaving null gaps. An array would be more appropriate for storing the board spaces, because the number of spaces never changes and random access by index is required.
A-range response
List<Player> is appropriate because the active player count is dynamic — players are eliminated with activePlayers.Remove(p) during the game, and List<T> handles resizing automatically. With a fixed-size Player[], removing a player would require either shifting all subsequent elements manually (O(n) operation) or leaving a null at the removed index, which would then require null-checking in every method that iterates the array. The List also provides Count and IndexOf() that are used in NextTurn() and EliminatePlayer() — these are not natively available on arrays without LINQ.

An array is more appropriate for boardSpaces: the board has a fixed 24 spaces that never change at runtime. Array index access by position (boardSpaces[player.position]) is O(1) and the fixed size prevents accidental resizing. Using a List here would suggest the board can be extended at runtime, which is not the design intent.
A-range names the specific methods used (Remove(), Count, IndexOf()), explains the operational cost of the alternative (O(n) shifting or null gaps), and justifies the array choice for boardSpaces in terms of design intent (fixed structure) rather than just "it doesn't change." The mention of design intent is the key insight — data structure choice communicates something about the intended behaviour of the data.
Explain a type substitution 3 marks
"A student stores the space type as a string in their BoardSpace class and checks it with: if (space.spaceType == "property"). Explain why an enum would be more appropriate and identify one specific bug this change would prevent."
B-range response
An enum is better because it is type-safe — you can only assign values that are defined in the enum, whereas a string can be any text including misspellings. For example, if a student types "proprety" instead of "property", the string comparison would silently fail at runtime. With an enum, the compiler would catch SpaceType.Proprety as an error immediately.
A-range response
An enum is more appropriate for three reasons. First, compile-time type safety: SpaceType.Property is validated by the compiler — a typo like SpaceType.Proprety causes an immediate compile error, whereas "proprety" is a valid string that silently fails the comparison at runtime. Second, switch statement exhaustiveness: a switch on an enum can be checked by the IDE for missing cases — if a new space type is added to the enum, the IDE warns that ResolveSpace() doesn't handle it. With strings, adding a new type produces no warning. Third, refactoring safety: renaming an enum value updates all references automatically; renaming a string literal requires manual find-and-replace across all files.

Specific bug prevented: if the Inspector sets spaceType = "Property" (capital P) but the comparison is == "property" (lowercase p), the condition is always false and no properties can ever be purchased. With an enum, case is irrelevant — SpaceType.Property is always the same value regardless of how it is displayed.
A-range gives three distinct reasons rather than one, includes the switch exhaustiveness checking insight (which is a less obvious but highly sophisticated point), and the specific bug is precisely described — capital vs lowercase mismatch causing silent failure. B-range correctly identifies the typo problem but treats it as the only reason, missing the refactoring and switch-exhaustiveness dimensions.
