CT5-DAT-02 Technical · Project Management

Data Analysis
& Representation

"Acquires, stores, analyses and visualises data to create information"
Acquires
Collecting
Gathering data from a source — a game, a survey, a sensor, a file — with a defined purpose
Stores
Organising
Keeping data in a structured format that can be retrieved, filtered, and processed reliably
Analyses
Finding patterns
Applying operations — sorting, filtering, calculating statistics — to find patterns, outliers, and insights in data
Visualises
Communicating
Representing data in a visual form that communicates findings to an audience more efficiently than raw numbers

What this outcome means

From raw data to information

CT5-DAT-02 is about the full data analysis pipeline: collecting data, storing it appropriately, processing it to reveal meaning, and presenting those findings visually. The critical distinction is between data (raw facts, numbers, records) and information (data that has been processed and interpreted to answer a question).

In a game development context, DAT-02 is most directly evidenced through the test logs and peer test findings — these are structured data sets that can be analysed to answer specific questions: which test cases failed most often? which space types were most frequently reported as confusing? what is the average game length? The analysis of these findings, and their presentation in the evaluation, is the core DAT-02 evidence in Dice District.

Data vs information — the key distinction: A test log with 28 results is data. "67% of test cases passed in Lesson 5; after fixing the index correction bug in Lesson 7, the pass rate increased to 95%" is information — the data has been processed to reveal a trend. DAT-02 requires demonstrating the transformation, not just collecting the data.

The data pipeline

Four stages — applied to Dice District

Every data analysis project follows the same four-stage pipeline. Understanding how each stage applies in the Dice District context makes DAT-02 evidence visible across the whole project rather than only in the analysis section.

Stage 1
Acquire
Collecting data from a defined source with a specific purpose. Data without a stated purpose cannot be analysed meaningfully.
Dice District: test log tables (L5–L7), peer test checklist (L9), bug log severity counts, game playthrough timing data.
Stage 2
Store
Organising data in a structured format. Unstructured data (freeform notes) cannot be filtered, sorted, or aggregated efficiently.
Dice District: test log structured as a table (ID / Input / Expected / Actual / Result). Peer test checklist as Pass/Fail/N/A per item. Bug log with severity field enabling counting by category.
Stage 3
Analyse
Applying operations to find patterns: counts, percentages, trends across time, comparisons between groups.
Dice District: pass rate per lesson (trend analysis), fail rate by space type (group comparison), critical vs minor bug ratios (distribution analysis), requirements met/partial/not-met counts.
Stage 4
Visualise
Presenting findings in a visual format that communicates the insight more efficiently than raw numbers.
Dice District: bar chart of pass/fail per lesson; pie chart of bug severity distribution; requirements evaluation summary table with colour-coded ratings.

Visualisation

Choosing the right chart type

A visualisation is only appropriate if the chart type matches the data's structure and the question being answered. Using a pie chart to show a trend over time, or a line chart to compare discrete categories, is a communication failure — the chart type should make the pattern in the data immediately visible.

📊
Bar Chart
Compare discrete categories
Best for comparing counts or values across named categories — space types, severity levels, lesson test passes. Each bar represents one category.
Dice District: test pass count per lesson (L5: 4/6, L6: 5/7, L7: 7/7) — grouped bar shows improvement over time by lesson.
🥧
Pie / Doughnut Chart
Show proportions of a whole
Best for showing how a total is divided among categories — most meaningful when there are 2–5 slices. Avoid when differences between slices are small.
Dice District: bug severity distribution (Critical 2, Major 4, Minor 5, Cosmetic 3) — shows the proportion of serious vs minor issues at a glance.
📈
Line Chart
Show trends over time
Best for continuous data over an ordered sequence — time, lessons, versions. The slope communicates rate of change visually.
Dice District: cumulative pass rate across L5–L9 (as tests were added and bugs fixed). The upward trend communicates improving reliability.
📋
Summary Table
Communicate exact values
When exact numbers matter more than pattern — a table allows readers to look up specific values. Often paired with a chart for context.
Dice District: requirements evaluation table (Met/Partial/Not Met with justification per requirement) — the exact rating and evidence matter more than a visual pattern.
🔢
Frequency Count
Count occurrences in a category
The simplest form of analysis — counting how many times each value occurs. Forms the basis for percentages and more complex analysis.
Dice District: counting Pass/Fail totals in the peer test checklist — 19 Pass, 6 Fail, 3 N/A. The frequency count is the foundation of the pass rate calculation.
📉
Comparison / Heatmap
Compare values across two dimensions
A matrix format showing values at intersections of two categorical dimensions — e.g. test result (Pass/Fail) by lesson and by space type.
Dice District: a checklist section vs Pass/Fail matrix reveals which sections of the game had the most peer test failures, guiding the priority fix list.

Applied analysis

Four worked analyses from Dice District data

The four examples below demonstrate the full Acquire → Store → Analyse → Visualise pipeline applied to real Dice District project data. Each starts with a research question, describes the data collection method, explains the analysis, and identifies the appropriate visualisation.

How does test pass rate change across the build lessons?
Trend analysis
Data + analysis
Acquire: Test log tables from L5, L6, L7 — each records Pass/Fail per test case.
Store: Count total cases and passes per lesson: L5 = 4/6 (67%), L6 = 5/7 (71%), L7 = 7/7 (100%).
Analyse: Pass rate trend — increasing from 67% to 100% across lessons. The L5→L6 improvement is smaller (67→71%) than L6→L7 (71→100%) — the index correction bug fixed in L7 resolved multiple test failures simultaneously.
Visualisation + insight
Chart type: Line chart (trend over ordered sequence of lessons).
Information created: The steep improvement at L7 is visible in the chart slope — it identifies L7's bug fix as the single most impactful quality improvement event in the project. This finding supports the evaluation claim that the index correction bug was a Major severity issue, not a Minor one.
Which sections of the game had the most peer test failures?
Category comparison
Data + analysis
Acquire: Peer test checklist (L9) — 28 items across 6 sections (Game Start, Turn & Movement, Space Actions, UI, Win Condition, Theme).
Store: Count Fail results per section: Game Start 0/5, Movement 1/6, Space Actions 2/7, UI 2/4, Win Condition 1/7, Theme 0/3.
Analyse: UI section has the highest fail rate (50%) despite having the fewest items. Space Actions has the most absolute failures (2).
Visualisation + insight
Chart type: Grouped bar chart (fail count and fail rate per section).
Information created: The UI fail rate (50%) reveals that usability is the primary problem area — more failures per item than any other section. This directly justifies prioritising UI fixes in the L9 priority fix list over additional feature implementation.
What is the distribution of bug severity in the project?
Distribution analysis
Data + analysis
Acquire: Bug log from L8–L9 — each entry has a severity field (Critical / Major / Minor / Cosmetic / Design).
Store: Count by severity: Critical 2, Major 4, Minor 5, Cosmetic 3, Design 1. Total: 15 bugs.
Analyse: 40% of bugs are Critical or Major — these required immediate attention. 87% of Critical/Major bugs were fixed before submission; 60% of Minor bugs remained open.
Visualisation + insight
Chart type: Pie chart (proportion of total by severity) + separate fix-rate bar chart.
Information created: The fix rate analysis (87% of serious bugs fixed vs 60% of minor) shows that the priority fix list was applied correctly — high-severity issues were addressed first. This is both DAT-02 evidence and an EVL-01 refinement justification.
Does the game meet the non-functional duration requirement?
Threshold analysis
Data + analysis
Acquire: Time three complete playthroughs (stop-watch or system clock). Record: 11 min, 14 min, 9 min.
Store: Simple table: Playthrough 1 = 11 min, P2 = 14 min, P3 = 9 min.
Analyse: Average = 11.3 min. Min = 9 min, Max = 14 min. NF-03 requirement: "game duration 10–20 minutes." One playthrough (9 min) falls slightly below the lower bound.
Visualisation + insight
Chart type: Bar chart with a horizontal reference line at the 10-minute lower bound.
Information created: The reference line makes the threshold violation visually immediate — P3 falls below it. The information created is: "NF-03 is partially met — average duration is within range but minimum duration falls below the threshold in fast games." This converts raw timing data into an evaluated requirement rating.

Data integrity

Trustworthy data produces trustworthy analysis

Data analysis is only as reliable as the data it is based on. Data integrity means that the data collected is accurate, complete, consistent, and free from errors introduced during collection or storage. In the Dice District context, several practices protect data integrity.

Threats to data integrity
  • ✗ Recording actual outputs before running the test (confirmation bias)
  • ✗ Omitting Fail results from the test log to improve the apparent pass rate
  • ✗ Using a non-representative sample (only playtesting with the developer)
  • ✗ Changing data after analysis to fit a desired conclusion
  • ✗ Collecting too few data points to draw meaningful conclusions
Data integrity practices
  • ✓ Expected outputs defined before running each test case
  • ✓ All test results recorded — Fail entries not omitted or corrected to Pass
  • ✓ Peer testing uses a different group — independent of the developer
  • ✓ Multiple playthroughs to generate timing data (not a single result)
  • ✓ Requirements Not Met acknowledged honestly in the evaluation
Data integrity and SAF-01: Falsifying or selectively reporting test data is both a data integrity failure and an ethical practice failure — it connects DAT-02 to SAF-01. A student who omits Fail results to make their test log look better is not just producing bad data analysis; they are being dishonest in their project documentation. Acknowledging failures honestly is evidence of both outcomes simultaneously.

Exam preparation

Question types and model answers

Distinguish data from information 3 marks
"A student records the following from their game testing: Lesson 5 — 4 pass, 2 fail. Lesson 6 — 5 pass, 2 fail. Lesson 7 — 7 pass, 0 fail. Explain whether this is data or information, and describe how it could be transformed into information."
B-range response
This is data — it is raw counts that haven't been interpreted yet. To transform it into information, you could calculate the pass rate for each lesson (L5: 67%, L6: 71%, L7: 100%) and note that the pass rate improved across lessons. You could then use a line chart to show this trend visually, which would communicate the improvement more clearly than the raw numbers.
A-range response
This is data — it is a structured set of raw counts that have not yet been processed to answer a specific question. Numbers alone are not information until they are interpreted in context.

To transform it into information, three steps: (1) Calculate pass rates: L5 = 4/6 = 67%, L6 = 5/7 = 71%, L7 = 7/7 = 100%. (2) Identify the trend: pass rate increased steadily across lessons, with the largest single improvement between L6 and L7 (+29 percentage points vs +4 between L5 and L6). (3) Interpret the cause: the L6→L7 jump suggests that a specific fix in L7 resolved multiple failures simultaneously — cross-referencing with the debugging log reveals this was the index correction in EliminatePlayer().

Information: "Test reliability improved from 67% to 100% across Lessons 5–7. The majority of this improvement (29 of 33 percentage points) occurred between L6 and L7, attributable to the EliminatePlayer() bug fix. This supports the assessment that the bug was Major severity, not Minor." This answers a specific question (did quality improve, and when?) using the raw data as evidence.
A-range performs the transformation explicitly (raw counts → percentages → trend → interpretation → information statement), cross-references the debugging log to explain the cause, and produces a sentence that directly answers a meaningful question. The key move is the final "information" sentence — it is a claim supported by evidence, which is the definition of information in this context. B-range correctly identifies the distinction and performs the calculation but doesn't carry the interpretation to a conclusion.
Select an appropriate visualisation 3 marks
"A developer wants to show the proportion of bugs in their project that are Critical, Major, Minor, and Cosmetic. Identify the most appropriate chart type, explain why it is appropriate, and describe one limitation of this visualisation."
B-range response
A pie chart is most appropriate because it shows how a total is divided into parts. Each slice represents one severity category, and the size of the slice shows what proportion of bugs fall into that category. A limitation is that if the slices are similar in size, it can be hard to tell them apart — in that case, a bar chart would be clearer because the heights can be compared more easily.
A-range response
A pie chart or doughnut chart is most appropriate because the question asks about proportion of a total — how the bug count is distributed across severity categories. A pie chart makes it immediately visible whether, say, Critical and Major bugs constitute a majority or minority of the total. Each slice's area is proportional to its count, enabling the reader to compare proportions at a glance.

It is appropriate here specifically because the categories are exhaustive and mutually exclusive — each bug belongs to exactly one severity level, and together they sum to 100% of all bugs. This is the precondition for a pie chart to be meaningful.

Limitation: a pie chart does not communicate exact values — a reader can see that "Critical" is a larger slice than "Cosmetic" but cannot determine the precise count or percentage without labels. If the developer needs the reader to compare exact numbers (e.g. 2 Critical vs 4 Major), a bar chart would be more appropriate because bar heights are more precisely comparable than arc lengths. Additionally, if there are many categories with similar counts, the slices become hard to distinguish even with colour coding.
A-range states the precondition for a pie chart to be meaningful (exhaustive, mutually exclusive categories that sum to 100%), explains the specific limitation in terms of reader perception (arc lengths are less precisely comparable than bar heights), and offers a concrete alternative with reasoning. B-range identifies the right chart and limitation but doesn't explain the precondition or the perceptual mechanism behind the limitation.
