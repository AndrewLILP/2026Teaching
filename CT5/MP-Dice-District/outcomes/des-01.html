What this outcome means

UI and UX are not the same thing

The outcome asks for both — user interface and user experience. These are related but distinct, and understanding the difference is the first mark of DES-01 sophistication. Many students build a functional UI but never consider the experience it creates; others describe an intended experience without building anything that delivers it.

User Experience (UX)
How it feels to use

The overall quality of a person's interaction with a system — their emotions, perceptions, and sense of control. UX is not visible in the interface; it is felt through it.

  • Does the player always know whose turn it is?
  • Is it clear what just happened and why?
  • Does the interface communicate game state without the player having to guess?
  • Can a new player understand the interface without being told how to use it?
  • Does the feedback from actions feel immediate and proportionate?
User Interface (UI)
What the player sees and touches

The visual and interactive elements a user encounters — buttons, labels, layouts, colours, and typography. UI is what UX is delivered through.

  • The Roll Dice button and its placement on screen
  • The HUD showing player names and money
  • The message panel text after each action
  • The space info panel on the right
  • The Buy / Skip buttons that appear contextually
  • The Game Over screen with winner information
The key insight: A UI can be technically complete and still create a poor user experience. The HUD could show player money — but if it only shows the current player's money, not all players simultaneously, it creates an experience where players must remember each other's balances. The decision to show all players at once is a UX decision, not just a UI implementation detail.

Both verbs must be evidenced separately

Like CT5-OPL-01, DES-01 has two verbs — designs and creates. A student who builds a good UI without a wireframe can only demonstrate from C upward for this outcome. A student who produces a detailed wireframe but a minimal UI has evidence of design thinking but limited implementation. Both are required — and the relationship between the wireframe and the final UI is itself evidence: it shows whether design informed creation or was added retrospectively.

The design progression

From sketch to screen

UI design in a project context moves through three stages. Each produces evidence for DES-01, and each is visible in the Dice District lesson sequence.

Stage 1
Low-fidelity Wireframe

Boxes and labels — no colour, no typography, no detail. The goal is information architecture: what needs to be on screen, where, and why. Rough is appropriate here; polish is a distraction from structure.

Tool: Paper / notebook
Dice District: Lesson 3 — the hand-drawn wireframe with 5+ labelled elements and annotations explaining what data each element communicates
Stage 2
UI Implementation

Building the interface in the target environment — Unity Canvas, TextMeshPro, Buttons. The wireframe is the specification; the Canvas is the implementation. Differences from the wireframe are design decisions and should be noted.

Tool: Unity Canvas, TextMeshPro
Dice District: Lessons 5–7 — HUD added L5, Buy/Skip added L6, Game Over panel added L7. Each UI element wired to GameManager methods.
Stage 3
Iterative Refinement

Revising the UI based on testing feedback. Changes should be documented: what was changed, what was wrong with the original, and what the change achieves. Refinement without documentation is invisible.

Tool: Peer testing checklist (L9)
Dice District: Lesson 9 — peer testers evaluate UI readability. Their observations drive changes in the priority fix list and inform the Lesson 10 evaluation.

Interface design principles

Eight principles — applied to Dice District

Shneiderman's Eight Golden Rules of Interface Design provide a framework for evaluating and justifying UI decisions. These are the principles that distinguish a considered design from an assembled one — and they give students the vocabulary to explain their decisions in evaluations and exam answers.

Rule 1
Strive for Consistency
Similar actions should look and behave the same way. Consistent terminology, colour, and layout reduce the cognitive load of learning the interface.
Dice District: the Buy and Skip buttons always appear together, always in the same position, always in the same colours. The message panel always explains the same type of information in the same format.
Rule 2
Seek Universal Usability
Design for diverse users — different experience levels, abilities, and contexts. Shortcuts for experts shouldn't break the experience for novices.
Dice District: text sizes large enough to read from 1 metre away (non-functional requirement from L1). Colour not the only differentiator — space type also shown in text, not just tile colour.
Rule 3
Offer Informative Feedback
Every user action should produce a visible system response. Users should never have to wonder whether something happened.
Dice District: every game event writes to the message panel. ShowMessage() is called in every branch of ResolveSpace() — no action is silent. The Console logs further confirm actions during development.
Rule 4
Design Dialogs to Yield Closure
Sequences of actions should have a clear beginning, middle, and end. Users should feel a sense of completion when a task finishes.
Dice District: each turn follows a predictable sequence — Roll → Move → Message → (Buy/Skip or auto-resolve) → HUD update → Next player's name shown. The closure signal is the "X's turn" message that appears after every action.
Rule 5
Prevent Errors
Design so that users cannot make serious errors. If errors occur, they should be easy to recover from.
Dice District: waitingForInput = true disables the Roll button during a purchase decision — preventing double-rolls mid-action. Buy is blocked if the player can't afford it, rather than putting them into debt silently.
Rule 6
Permit Easy Reversal
Actions should be reversible where possible. This reduces anxiety and encourages exploration.
Dice District: the Skip button on a purchase prompt is the reversal option — a player who sees the price can choose not to commit. The game doesn't allow undoing a roll, which mirrors the physical Monopoly convention.
Rule 7
Support Internal Locus of Control
Users should feel they are in control of the interface, not controlled by it. The interface responds to users; it doesn't surprise them or act autonomously without warning.
Dice District: the Roll button only advances the game when the current player chooses to press it. Nothing moves without player input. The message panel explains each system action — keeping the player informed and in control.
Rule 8
Reduce Short-Term Memory Load
Interfaces should display the information users need rather than requiring them to remember it from a previous screen or action.
Dice District: the full-player HUD (L8, Basic feature) shows all players' balances simultaneously — players don't need to remember each other's money. The space info panel shows the landed space's name and rent so players don't need to refer to an external card.
Exam and evaluation tip: When asked to justify a UI design decision, naming the relevant Shneiderman rule and explaining how the decision applies it immediately demonstrates A-range design thinking. "I put the message panel at the bottom of the screen so players can see the board (Rule 8 — reduce memory load)" is more sophisticated than "I put it there because it looked good."

Accessibility considerations

Designing for all players

Accessibility in game UI is not just about compliance — it is a direct expression of Rule 2 (universal usability). The considerations below are relevant to Dice District and should inform both the design and the evaluation.

🎨
Colour contrast
Text and important UI elements should have sufficient contrast against their background. Dark text on light backgrounds or light text on dark backgrounds — never grey on grey.
Dice District: TextMeshPro text in cream/white on dark panel backgrounds. Property colour groups use both colour and text labels — colour alone is not the only differentiator.
📏
Text size & legibility
Text must be readable in the context where the game is played. Minimum 18–24pt for body text in a game viewed from 1 metre; UI labels should never be decorative-only.
Dice District: the L1 non-functional requirement "text readable from 1 metre away" is an accessibility requirement — testing it is part of the peer test checklist in L9.
🖱
Target size & spacing
Interactive elements (buttons) should be large enough to click accurately. Buttons too small or too close together cause mis-clicks — frustrating on a touchscreen and imprecise with a mouse.
Dice District: Roll Dice button minimum 160×50 pixels; Buy and Skip buttons at least 80×50 with visible spacing between them so hot-seat players don't accidentally skip a purchase.
🔔
Multiple feedback channels
Don't rely solely on colour or sound to communicate state. Combine visual feedback (colour change), text feedback (message panel), and — where implemented — audio feedback (sound effects).
Dice District: the active player chip changes background colour AND updates text showing "current" — two feedback channels for the same information. Players who are colour-blind can still identify whose turn it is.
⏱
Timing & animation
Animations should aid understanding, not delay it. Token movement that takes 3+ seconds between turns slows the game unnecessarily. Give users control where possible.
Dice District: the recommended Lerp duration of 0.4 seconds is fast enough to be satisfying but short enough not to frustrate. Step-by-step movement (L8 Advanced) is visually clear but adds ~3 seconds per turn — a UX trade-off worth documenting.
📖
Language clarity
Message panel text should use plain, unambiguous language. Avoid jargon. State what happened and what the player can do next — not just one of the two.
Dice District: "Harbour St is available! Buy for $220? You have $1200." is better than "Property: purchasable." The second tells the player what; the first gives them the information they need to decide.

Content dot points

What the syllabus requires

The dot points for DES-01 span the full design-to-implementation process. Each is mapped to its Dice District evidence below.

Dot point What it requires Dice District evidence
Needs analysis
 Identifying user needs and functional/non-functional requirements that the UI must satisfy L1: non-functional requirements ("text readable at 1m", "UI visually consistent"). The game primitive analysis includes identifying what information players need visible at all times.
Wireframing
 Low-fidelity layout sketches showing information hierarchy and element placement before implementation L3: hand-drawn wireframe with minimum 5 labelled elements. The Lesson 3 page includes an annotated reference wireframe showing six UI zones with purpose annotations.
Interface design principles
 Applying recognised principles (e.g. Shneiderman's rules) to justify design decisions L10 evaluation: Q2 asks students to justify a design decision — naming a principle is the A-range response. L3 wireframe annotations demonstrate principles being applied at the planning stage.
Prototyping
 Building a version of the interface to test before the final implementation L3: the wireframe is a paper prototype. L5: the initial HUD with only message text is a working prototype — it is extended in L6 and L7 as needs are confirmed through testing.
Implementation
 Building the UI using appropriate tools — layout, typography, colour, and interactivity L5–L7: Unity Canvas + TextMeshPro implementation across multiple lessons. Button wiring via OnClick events. Panel show/hide with SetActive(). Conditional display (Buy/Skip hidden by default).
User testing
 Testing the interface with real users and documenting findings L9: the peer test checklist includes four dedicated UI/UX items (HUD readability, message clarity, button visibility, Game Over screen). Tester findings directly inform the priority fix list.
Evaluation
 Assessing the interface against original requirements and design principles L10: the requirements evaluation table includes non-functional UI requirements. Q2 of the written evaluation asks specifically about a design decision — UI decisions are the most common choice.
Accessibility
 Considering how the design accommodates diverse users — contrast, text size, colour independence L3 wireframe annotations; L9 peer test checklist item on readability; L10 evaluation of whether accessibility requirements were met.

Exam preparation

Question types and model answers

Justify a design decision 4 marks
"A developer decides to show all players' names and money simultaneously in the HUD, rather than only showing the current player's information. Justify this design decision using UX principles."
B-range response
Showing all players at once means players don't have to remember each other's money between turns. It is important in a competitive game because knowing your opponents' balance affects your decisions — such as whether to buy a property they can't afford to pay rent on. This reduces the mental effort required to play, making the experience better for everyone at the table.
A-range response
This decision applies Shneiderman's Rule 8 — Reduce Short-Term Memory Load. In a hot-seat game, players take turns and may wait several minutes between their own turns. Requiring them to remember each opponent's balance across multiple turns creates unnecessary cognitive load and increases the likelihood of disputes. Showing all balances simultaneously means each player can make informed decisions about whether to buy, skip, or risk bankruptcy without asking others. It also applies Rule 3 — Offer Informative Feedback — because every money transfer is immediately visible to all players, providing transparent confirmation that transactions executed correctly.
A-range names two specific Shneiderman rules with their numbers and explains how each applies. The hot-seat gaming context (turns separated by time) is a UX-specific insight that goes beyond generic "it's easier to use." B-range gives sound reasoning with one implicit principle but doesn't name or frame it within a recognised design framework.
Evaluate a UI element 5 marks
"A game's Buy button remains visible on screen at all times, even when a player is not on an unowned property. Identify the UX problem this creates and propose a solution, using design principles to justify your answer."
B-range response
The problem is that players may try to click Buy when it doesn't do anything, which is confusing. The button should be hidden when there is no property to buy. This can be done by calling buyButton.gameObject.SetActive(false) by default and only showing it when a player lands on an available property. This makes the interface clearer and prevents player frustration.
A-range response
The problem violates Rule 5 — Prevent Errors. A permanently visible Buy button invites players to click it at inappropriate moments, causing confusion when nothing happens (or worse, an unintended action). It also violates Rule 7 — Support Internal Locus of Control — because the interface offers an option that is meaningless in the current context, making players feel uncertain about what the system can and cannot do.

Solution: implement contextual visibility using buyButton.gameObject.SetActive(false) as the default in Start(), and SetActive(true) only inside OfferPurchase(). HidePurchaseButtons() restores the default. This follows the principle of progressive disclosure — showing only the options relevant to the current context, which reduces cognitive load and prevents mistaken actions. The peer test checklist item "Buttons appear and disappear at the right times" verifies this behaviour.
A-range identifies two violated principles by name and number, introduces the concept of progressive disclosure as an additional design framework, and references the specific Dice District implementation (OfferPurchase() / HidePurchaseButtons()) as evidence of how the solution was implemented. The peer test checklist reference connects design evaluation to testing practice — a sophisticated cross-outcome connection.
Annotate a wireframe element 3 marks
"Write an annotation for the message panel in the Dice District wireframe that demonstrates A-range design thinking. Your annotation must explain purpose, content, and behaviour."
B-range response
Message Panel: Shows what just happened in the game. Updates after every action. Located at the bottom of the screen where players naturally look after pressing a button. Text is large enough to read quickly during play.
A-range response
Message Panel (bottom-centre, full width): Communicates the outcome of every game event in plain language — not just what happened, but enough context for the player to understand why and what happens next (e.g. "Nomad paid $30 rent to Strider for Harbour St. Strider's turn — roll the dice!"). Driven by ShowMessage() called in every branch of ResolveSpace(). Updates synchronously with HUD money changes so both panels always reflect the same game state. Applies Rule 3 (informative feedback) — no action in the game loop is silent. Font: TextMeshPro, minimum 24pt, high-contrast cream on dark background.
A-range annotation includes a worked example of message content, identifies the driving method, explains the synchronisation with HUD, names the applicable rule, and specifies typography. It treats the annotation as a specification that another developer could implement without further guidance — that level of precision is A-range design documentation.
