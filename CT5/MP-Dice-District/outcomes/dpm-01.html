What this outcome means

Process is the product

CT5-DPM-01 is the only outcome that is entirely about how a solution was built — not what it does. A game that works perfectly but was coded in a single session from a blank file, with no planning, no testing, and no documented iterations, cannot demonstrate this outcome beyond a C. Conversely, a game with some unresolved bugs but a well-evidenced iterative process — requirements, pseudocode, test logs, scope decisions, and honest reflections — can demonstrate B-range process management.

The key word is applies — not "knows about" or "describes." Evidence requires that iterative processes were actually used, in sequence, with the notebook showing each phase's artefacts before the next phase began.

The retrospective documentation trap: The most common DPM-01 failure is students completing the project first, then filling in the notebook at the end. Retrospective documentation looks different from genuine process documentation: dates are consistent, planning artefacts show no revision history, and there are no "dead ends" or changed decisions. Genuine iterative process leaves a messy trail — crossed-out scope items, updated class diagrams with delta annotations, test logs that record failures. That mess is the evidence.

What "iterative" actually means

Iterative does not mean doing the same thing twice. It means each cycle informs the next — building a little, learning something, adjusting the plan, building the next part. In Dice District, each lesson is a cycle: the pseudocode from Lesson 2 informs the code in Lesson 4; the peer test findings from Lesson 9 inform the evaluation in Lesson 10. The connections between lessons are the iteration.

✗ Not iterative
Plan everything upfront, then build without revisiting the plan
Build everything, then test at the end
Write the notebook after the project is finished
Scope never changes — everything planned in L1 is still in the final game
No stubs — either full implementation or nothing
Peer feedback noted but nothing changed as a result
✓ Genuinely iterative
Requirements written in L1, then revisited and rated in L10 — changes documented
Test log updated after each lesson's build, not all at once at the end
Scope planner moved features between Must/Cut based on progress
Temporary stubs replaced with full implementations across multiple lessons
Class diagram updated in L7 with delta annotations showing what changed and why
At least one design decision changed directly because of peer test feedback

The Design & Production Framework

Four phases — one project

The NSW D&P framework structures the project into four phases. Each phase has a guiding question, a set of artefacts, and a set of lessons in Dice District where it is active. Phases overlap — in a real iterative project, you are often researching while producing, and evaluating while building. What matters is that each phase's artefacts are produced before the next phase is fully underway.

Phase 1
Identifying & Defining
"What does the solution need to do, and for whom?"
Dice District: Lesson 1
  • Requirements list (F + NF)
  • Game primitive analysis
  • Board sketch
  • Theme selection with justification
  • Initial scope definition
Phase 2
Researching & Planning
"How will it be built, and what do we need to know first?"
Dice District: Lessons 2–3
  • Starter code analysis
  • Game loop pseudocode
  • Class diagram
  • UI wireframe
  • Scope priority planner
  • MovePlayer() pseudocode
Phase 3
Producing & Implementing
"How do we build it, and what do we learn as we build?"
Dice District: Lessons 4–8
  • Starter scripts configured
  • Board + player setup
  • Dice + movement implemented
  • Space logic implemented
  • Turn + win condition implemented
  • Extension features (L8)
  • Test logs per lesson
Phase 4
Testing & Evaluating
"Does it work? Does it meet requirements? What would we change?"
Dice District: Lessons 9–10
  • Peer test checklist
  • Bug log (tester)
  • Priority fix list
  • Requirements evaluation table
  • Written evaluation (4 questions)
  • D&P retrospective

The iterative cycle within each phase

Each phase contains its own micro-iteration cycle. Within the Producing phase (Lessons 4–8), each lesson follows: pseudocode or plan → implement → test → reflect → next lesson's plan. The test log at the end of each lesson connects the end of one cycle to the beginning of the next.

The micro-iteration cycle — repeated within each lesson
📐

Pseudocode, class diagram, scope decision
→
🔧

→
🧪

📓

The reflection at the end of each lesson is the bridge between cycles — it converts what was learned into what is planned next. Without it, cycles are disconnected.

The project notebook as evidence

What the notebook needs to show

The project notebook is the primary evidence vehicle for DPM-01. It is not a polished report — it is a record of thinking in progress. Rough entries with crossed-out decisions and honest debugging notes are better evidence than a clean retrospective document that shows no revision.

Lesson Phase Required artefact What makes it strong DPM evidence
L1 I&D Requirements list + board sketch Requirements are testable (not vague). The sketch shows at least one design revision — a crossed-out space name or repositioned element shows genuine planning, not a first-draft final copy.
L2 R&P Game loop pseudocode + class cards Pseudocode precedes the corresponding code. Class diagram written before L4 — dated entries or lesson headings confirm this. Annotation shows reasoning, not just description.
L3 R&P Wireframe + scope priority list Scope planner shows features moved between Must/Nice/Cut with justification. Wireframe annotated with behaviour intent, not just element names.
L4–L7 P&I Screenshots + test log per lesson Test logs show expected AND actual outcomes — not just "it worked." Reflections note what broke, what was tried, and what changed before the next lesson. Screenshots dated/sequenced.
L7 P&I Updated class diagram The delta annotations ("REPLACED", "NEW") show the plan changed during production. A class diagram identical to the L3 version suggests either no iteration occurred or the artefact was created retrospectively.
L8 P&I Feature build log + self-assessment Self-assessment is honest — acknowledges features not yet working. Feature log shows testing happened immediately after building each feature, not at the end of the lesson.
L9 T&E Bug log + priority fix list Priority fix list shows at least one change made during the lesson — demonstrating the test → fix cycle, not just test → list. Bug log entries include cause, not just symptom.
L10 T&E Requirements evaluation + D&P retrospective Requirements explicitly rated Met / Partial / Not Met — "Not Met" acknowledged honestly. Retrospective connects back to L1–L3 decisions, showing the loop closed. Changes from L9 peer test mentioned.
The strongest DPM-01 notebooks have one thing in common: they show that the student genuinely didn't know how everything was going to turn out when they started. Scope items cut under time pressure, bugs that took multiple attempts to fix, design decisions that changed after testing — these are marks of authentic process, not weakness.

Selecting appropriate processes

Right tool, right time

The word appropriate in the outcome is significant. DPM-01 is not just about using iterative processes — it is about selecting the right process for each phase. Using pseudocode at the production stage instead of the planning stage is not appropriate. Building before wireframing is not appropriate. The table below maps each process to when it is appropriate in the D&P framework.

Process / tool Appropriate phase(s) Why it belongs there Dice District instance
Requirements list I&D Defines what the solution must do before any design begins — the target all later phases are measured against L1: 6F + 2NF requirements; used as the evaluation checklist in L10
NSW Pseudocode R&P Expresses the algorithm before code — forces logical structure independent of syntax. Written before the code it describes. L2: game loop pseudocode. L3: MovePlayer() pseudocode. L5–L7: method-level pseudocode before each implementation lesson
Class diagram R&P P&I Planned in R&P to define architecture before coding; updated in P&I as understanding develops — the delta is the iteration evidence L3: initial class diagram. L7: updated diagram with delta annotations
Wireframe R&P Defines UI layout and information architecture before any Canvas elements are created — the specification the implementation is compared to L3: hand-drawn wireframe with 5+ annotated elements
Scope planner R&P P&I Sets realistic boundaries at planning stage; revised at L8 when actual production time is known — revision is explicit iteration evidence L3: Must / Nice / Cut planner. L8: self-assessment table revisits scope realism
Implementation stubs P&I Placeholder methods that compile and run but don't do their final job — allow testing of surrounding code before full implementation is ready L5: temporary NextTurn() stub. L6: CheckBankruptcy() stub — both deliberately replaced in later lessons
Test log P&I T&E Records expected vs actual outcomes — appropriate during P&I (unit testing per lesson) and T&E (system testing with peers) L5–L7: lesson test tables. L9: peer test checklist + bug log
Peer testing T&E External testers find issues the developer cannot — particularly usability and edge-case problems. Appropriate after P&I is largely complete. L9: structured 28-item checklist + bug log from an external group
Requirements evaluation T&E Returns to the I&D phase requirements and measures the completed solution against them — closing the iterative loop L10: requirements evaluation table rating each requirement Met / Partial / Not Met

Grade indicators for CT5-DPM-01

What evidence looks like at each grade

Grade Process (planning artefacts) Iteration (change evidence) Evaluation (loop closure)
A Skilfully applies — all four phases evidenced with dated/sequenced artefacts. Appropriate processes selected for each phase: pseudocode before code, wireframe before UI, test cases predicted before running. Multiple documented iterations: scope changed with reasoning; class diagram updated with delta; at least one design decision changed because of peer feedback. Stubs documented as deliberate prototypes. L10 evaluation returns to L1 requirements by name; "Not Met" acknowledged honestly; D&P retrospective addresses all four phases; identifies what would be done differently with specific technical reasoning.
B Applies appropriate — all four phases present. Most planning artefacts precede their corresponding production work. Test logs present across P&I lessons. Evidence of at least two iterations: class diagram updated; scope revision documented; or at least one design change driven by testing. Requirements evaluation table complete with justifications. D&P retrospective addresses most phases. Some connection between L9 findings and L10 evaluation visible.
C Applies iterative processes — all phases present but some planning artefacts may have been produced after the corresponding production work. Test logs present for most lessons. Some iteration evidence — class diagram may not be updated, but scope decisions or test fixes show iteration occurred. Requirements evaluation attempted; written evaluation present. Some connection to original requirements visible.
D Uses processes — some planning artefacts present. Phases partially evidenced. Process may appear mostly linear rather than iterative. Limited iteration evidence. Changes may have occurred but are not documented or connected across lessons. Some evaluation present but may not connect back to original requirements. D&P framework partially applied.
E Identifies processes — names the phases or processes but limited evidence of applying them. Notebook entries primarily descriptive rather than process-evidencing. Minimal iteration evidence — may describe what was built without connecting it to what was planned. Limited evaluation; may identify that requirements exist without comparing the solution to them.

Exam preparation

Question types and model answers

Explain iterative process 4 marks
"Explain how a software development team would apply an iterative process to develop a board game application. In your answer, refer to at least two phases of the Design & Production framework."
B-range response
An iterative process means building the project in cycles rather than all at once. In the Identifying & Defining phase, the team would write requirements for what the game needs to do. In the Producing phase, they would build part of the game and test it. If the test shows something is wrong, they go back and fix it before continuing. This means the final product is better because problems are found and fixed during development rather than only at the end.
A-range response
An iterative process applies cycles of plan → build → test → refine across the whole project. In the Researching & Planning phase, the team would write pseudocode for the game loop and create a class diagram, then use the scope priority planner to decide which features are essential versus optional — these decisions constrain the first build cycle. In the Producing & Implementing phase, rather than building everything at once, they would implement stub methods (e.g. a temporary NextTurn() that simply advances the index), test that movement works correctly, then replace the stub with the full implementation in a subsequent cycle. Each cycle's test log informs what the next cycle prioritises. The iteration is closed in the Testing & Evaluating phase when peer test findings are fed back into the production cycle — specific bugs identified in L9 drive the priority fix list, and the requirements evaluation in L10 returns to the L1 requirements to measure what was actually achieved.
A-range names specific phases, describes the stub pattern as a concrete iteration mechanism, explains how test logs connect cycles, and closes the loop by connecting T&E back to I&D requirements. B-range understands the concept but describes it abstractly — "if the test shows something is wrong they go back and fix it" is the right idea but lacks the specific mechanism and evidence language.
Evaluate process appropriateness 3 marks
"A student wrote their project notebook entirely after the project was finished, filling in all planning documents retrospectively. Identify two problems this creates for demonstrating CT5-DPM-01, and explain why."
B-range response
The first problem is that there is no evidence that planning happened before coding — the pseudocode and class diagram may have been copied from the finished code rather than designed independently. The second problem is that if planning and building happened at the same time (retrospectively documented), there is no evidence of iteration — the notebook will show a perfect path from start to finish with no revisions or dead ends, which doesn't look like a genuine iterative process.
A-range response
Problem 1: Artefact sequence cannot be verified. DPM-01 requires appropriate processes — pseudocode is only appropriate if it preceded the code it describes. Retrospective pseudocode is a description of finished code, not a design tool. The notebook cannot demonstrate that planning informed production because there is no temporal evidence (dates, staged entries, revision marks) showing which came first.

Problem 2: Iteration evidence disappears. A genuine iterative process leaves a messy record: scope items moved between categories, class diagrams with delta annotations, test logs recording failures before fixes. Retrospective documentation produces a clean, linear narrative with no dead ends, no revised decisions, and no evidence of learning from testing. DPM-01 specifically requires evidence that the process was iterative — a notebook that shows no uncertainty or revision suggests a linear (non-iterative) process was used, regardless of whether iteration actually occurred.
A-range identifies the two distinct problems — artefact sequencing and iteration evidence — and explains each in terms of what the outcome specifically requires (appropriate process + iterative evidence). The "messy record" insight is particularly strong: it shows understanding that authentic process documentation looks different from retrospective documentation, and that assessors can tell the difference.
Identify the appropriate process 3 marks
"A developer wants to implement a Chance card system but is unsure whether it will work with the existing game loop. Describe the most appropriate process to use before writing the full implementation, and explain why it is more appropriate than coding the full system immediately."
B-range response
The most appropriate process is to write pseudocode for the Chance card system first. This lets the developer work out the logic — how cards are drawn, shuffled, and applied — without worrying about C# syntax. It is more appropriate than coding immediately because if the logic is wrong, it is much faster to fix pseudocode than compiled code.
A-range response
Two processes are appropriate here, in sequence. First, pseudocode for the algorithm: writing BEGIN DrawChanceCard(player) → card ← chanceDeck[0] → chanceDeck.RemoveAt(0) → ApplyEffect(player, card) → IF chanceDeck.count = 0 THEN Shuffle() END IF → END lets the developer verify the draw-and-reshuffle logic before any code is written, and check that it integrates cleanly with the existing ResolveSpace() switch statement. Second, an implementation stub: replacing the existing Chance stub (ShowMessage("Chance: nothing happens yet")) with a version that draws from a one-card test deck before the full 8-card system is built. This lets the developer test that the Chance space correctly triggers the new method and that the game continues after the card is resolved, without needing all 8 cards defined first. Coding the full system immediately risks implementing 8 card effects and their edge cases before knowing whether the core draw mechanism integrates correctly — making debugging harder because more things can go wrong at once.
A-range proposes two sequenced processes and explains why the stub specifically allows integration testing before full implementation — this is a sophisticated understanding of the stub pattern as an iterative process tool, not just a shortcut. The explanation of why immediate full implementation is riskier (more things can go wrong at once) demonstrates genuine iterative process thinking.
