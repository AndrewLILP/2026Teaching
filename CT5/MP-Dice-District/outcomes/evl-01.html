Testing,
Evaluation
& Refinement

"Tests, evaluates and refines computing solutions against requirements"
Tests
Verifying behaviour
Structured test cases with defined inputs, expected outputs, and recorded actual outcomes — not just "I ran it and it worked"
Evaluates
Judging quality
Measuring the solution against requirements, design principles, and user needs — forming a reasoned judgement about what works and what doesn't
Refines
Improving it
Changing the solution based on test findings and evaluation — and documenting what was changed, why, and what the result was

What this outcome means

Testing and evaluating are not the same activity

CT5-EVL-01 bundles three activities — testing, evaluating, and refining — that are commonly conflated or under-evidenced. Testing is an empirical activity: run a specific scenario, record what happened. Evaluating is an analytical activity: judge whether what happened is good enough, and why. Refining is a responsive activity: change something based on what testing and evaluation revealed.

All three must be evidenced — and the relationship between them matters. A test log with no evaluation says "here is what I found." An evaluation with no test evidence says "here is my opinion." Refinement without both is random change. The outcome requires all three working as a cycle.

Testing
Empirical — what actually happens
  • A specific input is applied and the output is recorded
  • Expected output is defined before the test is run
  • Results are Pass or Fail — not opinion
  • Failures are recorded without omission — even embarrassing ones
  • Edge cases are tested deliberately, not just typical paths
  • The test stub technique isolates individual components
Evaluation
Analytical — what it means
  • Returns to the original requirements and measures the solution against each one
  • References test evidence to support judgements — not just assertion
  • Addresses both functional (does it work?) and non-functional (does it work well?) requirements
  • Acknowledges what was not met honestly — "Not Met" with a reason is better than silence
  • Proposes specific, technically reasoned improvements — not vague "add more features"
The most common EVL-01 gap: Students run the program and confirm it works, then write "the game meets all requirements" in the evaluation. This is not evaluation — it is a claim without evidence. Evaluation requires citing specific test case results, naming requirements by their original text, acknowledging failures honestly, and proposing technically specific improvements. A single honest "Not Met" entry with a cause and a proposed fix is worth more than ten uncritical "Met" ticks.

Test case anatomy

What a test case contains

A test case is a structured record of one specific scenario. It has five components — all five must be present for a test case to be useful as evidence. Without the expected output defined in advance, there is no basis for judging whether the actual output is correct.

Example test case — Board wrap detection (Lesson 5) ✓ Pass
Test ID
TC-05-03 — Board wrap + Go salary
Input / setup
Player position = 21 (space 21, "Corner"). Dice result forced to 6 using test stub: return 6; in RollDice(). Player money = $1200. boardSpaces.Length = 24.
Expected output
newPosition = (21 + 6) % 24 = 3. passedGo = true. Player money = $1200 + $200 = $1400. Token moves to boardSpaces[3]. Message panel shows "passed Go! Collected $200".
Actual output
newPosition = 3 ✓. Money = $1400 ✓. Token at space 3 ✓. Message: "Nomad passed Go! Collected $200." ✓
Result
PASS — all expected outputs matched.
Notes
Test stub restored to Random.Range(1,7) + Random.Range(1,7) after test. Test also verified that passing Go does NOT trigger when newPosition > oldPosition (normal mid-board move).

Weak vs strong test cases

The quality gap between a C and an A test log is almost entirely about specificity. Compare the two versions of the same test below.

✗Weak — C-range
Test: Board wrap
What I did: Moved near end of board
Result: Player wrapped around ✓
Notes: Go salary worked too
No starting position. No dice value. No expected output defined before running. "Worked" is subjective — what should have happened? The Go salary note is vague — did $200 arrive?
✓Strong — A-range
Test: Board wrap + Go salary
Setup: Pos=21, roll=6 (stub), money=$1200
Expected: newPos=3, money=$1400, message contains "passed Go"
Actual: newPos=3 ✓, money=$1400 ✓, message correct ✓
Result: PASS. Stub restored after test.
Specific starting state. Exact dice value via stub. All expected outputs defined before running. Each actual output verified independently. Test restoration noted. Reproducible by anyone.

The test stub technique

A test stub is a temporary replacement for a non-deterministic method that returns a fixed, known value for testing purposes. In Dice District, replacing RollDice() with return 6; means any board-wrap test can be run reliably — the developer doesn't wait for a random 6 to appear. The stub must be documented in the test log and restored afterward.

Why the expected output must be defined before testing: If you run the program first and then write down what happened as the "expected" output, you are not testing — you are documenting. The expected output is derived from the algorithm (by tracing the pseudocode manually), not from the running program. A test log where expected = actual for every single case, with no failures, is a sign that expected outputs were written retrospectively.

Types of testing

Four testing approaches across the project

Different testing approaches are appropriate at different stages. EVL-01 is best evidenced by using multiple approaches across the project — not just one type applied at the end.

Unit Testing
During P&I — each lesson
Testing one method or component in isolation. Uses test stubs to control inputs. Verifies that a single piece of logic does exactly what its pseudocode specifies.
Dice District: the 6-case test tables at the end of L5 (movement), L6 (space resolution), and L7 (elimination). Each tests one method with defined inputs and expected outputs.
Integration Testing
During P&I — after connecting components
Testing that multiple components work together correctly. Reveals problems that don't appear when components are tested in isolation — particularly data flow issues across class boundaries.
Dice District: testing that Player.AddProperty() correctly sets BoardSpace.owner, and that GameManager.ResolveSpace() then reads that field correctly — a three-script data flow.
System Testing
Late P&I — full game playthrough
Running the complete system end-to-end to verify the whole game works as a unit. Tests the game loop from start to finish, including win condition, elimination, and all space types.
Dice District: L7 final test — "Play a complete game to its natural end (may take 5–15 min)." The full game playthrough is the system test; it passes only when all components integrate correctly.
User/Peer Testing
T&E phase — Lesson 9
Testing by someone who didn't build the solution. Reveals usability issues, edge cases the developer didn't think of, and UI clarity problems invisible to someone who knows how the game works.
Dice District: L9 structured peer test using the 28-item observation checklist. External testers find bugs the developer cannot — particularly in UX clarity and unexpected interaction sequences.
Edge Case Testing
During and after P&I
Deliberately testing the boundaries and extremes of the system — the cases that are unlikely but possible. Distinguishes B-range from A-range test logs.
Dice District edge cases: player money exactly $0 (not bankrupt — bankrupt is <0); landing exactly on Go (not just passing); eliminating the player at index 0; eliminating the last player in the list; rolling during a purchase prompt (should be blocked).
Regression Testing
After any change
Re-running previous test cases after making a change, to verify that the fix didn't break something that was working before. Particularly important after bug fixes.
Dice District: after fixing the "Roll Dice active during purchase" bug in L9, re-running L6 test cases TC-06-01 through TC-06-05 to verify that the Buy/Skip flow still works correctly after the waitingForInput change.

Three distinct activities

The difference between tests, evaluation, and refinement

The three verbs target three different questions. Understanding the distinction helps students produce evidence that clearly demonstrates each activity — rather than conflating them into a single undifferentiated "testing" section.

Testing
What did it do?
"Did the program behave as specified for this specific input?"

An objective, empirical activity. The result is binary — Pass or Fail. The tester's opinion is not relevant; what matters is whether actual output matched expected output.

Produces: test logs, bug logs, Console output screenshots.

Dice District example: "Roll=6 from position 21 → newPosition=3, money=$1400. PASS."
Evaluation
Is it good enough?
"Does the solution meet the requirements it was designed to meet, and to what extent?"

An analytical activity. Uses test evidence to support judgements — not just assertion. Evaluates against the original requirements list (from L1), design principles (Shneiderman), and user feedback (L9 checklist).

Produces: requirements evaluation table, written evaluation (4 questions), D&P retrospective.

Dice District example: "Requirement F4 (detect bankruptcy) was met. Test TC-07-01 confirmed that money <0 triggers elimination. Requirement F6 (Chance cards) was partially met — draw and apply works, but only 5 of 8 events were implemented."
Refinement
What was changed?
"What was changed in response to testing and evaluation — and what was the result of the change?"

A responsive activity. Every refinement should be traceable to a test finding or evaluation judgement. The change, the reason, and the outcome of the change must all be documented.

Produces: priority fix list, updated code, debugging entries in notebook.

Dice District example: "Bug from L9 peer test: Roll button remained active during purchase prompt. Fix: confirmed waitingForInput = true set in OfferPurchase() and StartTurn() guard checks it. Re-tested TC-06-07: PASS."

Evaluating against requirements

Closing the loop — L1 to L10

The most important evaluation activity is also the simplest to describe and the hardest to do honestly: return to the requirements list written in Lesson 1 and measure the completed game against each one. This loop closure is the definitive evidence of EVL-01 — it shows the project had a defined target and was assessed against it.

The table below shows a worked example of the requirements evaluation format used in Lesson 10, with three example requirements at different fulfilment levels. The key is specificity — each rating must be supported by evidence.

Req. Type Original requirement Rating Evidence and justification
F1 F The game must allow 2–4 players to take turns rolling a dice and moving their token ✓ Met Test TC-05-06 (turn cycling) confirmed all players take turns in order. TC-05-01 to TC-05-05 confirmed movement across normal, wrap, and Go cases. Three-player configuration tested in L7 full playthrough.
F6 F Chance spaces must trigger a random event from a shuffled deck of at least 8 events ◑ Partial The shuffle and draw mechanism works correctly (tested in L8). However, only 5 of the 8 planned events were implemented before the L9 peer test. The remaining 3 events were in scope but cut due to time. The mechanism is correct; the content is incomplete.
NF2 NF The UI text must be readable from 1 metre away ✗ Not met L9 peer tester (Group 3) noted that the space info panel text was too small to read during play. The panel uses TextMeshPro at 14pt — below the minimum 18pt established in the L3 wireframe non-functional requirements. This requirement was not enforced during implementation. Fix: increase font size to 20pt and widen the panel to prevent text truncation.
What the "Not Met" entry demonstrates: Acknowledging a requirement was not met — honestly, with a specific cause and a technically precise fix — is stronger EVL-01 evidence than claiming everything was met. The Not Met entry above names the specific font size, cites the original wireframe specification, references the peer tester who found it, and proposes a concrete fix. That chain of evidence-to-judgement-to-recommendation is the structure of high-quality evaluation.

Debugging as refinement evidence

The debugging chain

Debugging is the refinement activity most visible in project notebooks. A debugging entry that shows a chain of reasoning — symptom observed → hypothesis formed → change made → result recorded → next step if unsuccessful — is A-range refinement evidence. A debugging entry that says "it was broken and I fixed it" is not.

Debugging entry — A-range example (Lesson 7)
Bug symptom
After Player 2 (index 1) is eliminated, Player 3 (originally index 2) is skipped and it immediately becomes Player 1's turn again.
Error type
Logic error — no compile error, game runs but behaviour is wrong.
Hypothesis 1
EliminatePlayer() might be calling NextTurn() before removing the player from activePlayers, so the count is still 3 when the modulo runs.
Test 1
Added Debug.Log("Count before remove: " + activePlayers.Count) and "Count after: " + activePlayers.Count in EliminatePlayer(). Console showed count went from 3→2 correctly before NextTurn() was called. Hypothesis 1 incorrect.
Hypothesis 2
When Player 2 (index 1) is removed, Player 3 shifts from index 2 to index 1. currentIndex was 1 (Player 2's turn). After removal, index 1 now points to Player 3. NextTurn() does (1+1)%2 = 0 — back to Player 1. Player 3 never gets called as index 0.
Fix
Saved the eliminated player's index before removing: int idx = activePlayers.IndexOf(p); then after Remove(p): if (idx <= currentIndex && currentIndex > 0) currentIndex--;
Result
FIXED. Retested with 3 players, eliminated P2, confirmed P3 takes next turn. Re-ran all L7 test cases — all pass.

Note what makes this A-range: two hypotheses formed and tested in sequence, each with evidence from Debug.Log rather than guesswork; a root cause explanation that names the specific mechanism (index shift); and regression testing confirming no regressions. The entry is reproducible — someone else could follow this reasoning without asking for clarification.

Grade indicators for CT5-EVL-01

What evidence looks like at each grade

Grade Testing Evaluation Refinement
A Technically concise — multiple test types used (unit, integration, peer). Edge cases tested deliberately. Test stubs documented and restored. Expected outputs defined before running via pseudocode trace. Requirements evaluated by name; "Not Met" entries present with honest causes. References specific test cases as evidence. Evaluation addresses both functional and non-functional requirements. Proposed improvements are technically specific. Debugging entries show hypothesis-test-result chains. Regression tests run after fixes. At least one design change traceable directly to peer test feedback, with the original and changed version both documented.
B Develops and tests — structured test tables with expected and actual outputs across all major functions. At least one edge case tested. Peer test checklist complete. Requirements evaluation table complete with justifications. Written evaluation references test outcomes. At least one requirement rated less than "Met" with a reason. Priority fix list used to drive L9 changes. At least one fix with cause and result documented. Some regression testing present.
C Develops common test cases — test tables present for most functions. Expected and actual outputs recorded, though expected may not always be defined first. Requirements evaluation attempted. Some requirements rated; evaluation may not always cite test evidence. Written evaluation present. Some fixes applied after testing. Debugging notes may describe what was changed without full cause reasoning.
D Basic test cases — some structured testing present. Results may be descriptive ("it worked") rather than structured. Some evaluation present. May claim requirements are met without evidence. Limited analysis of what doesn't work. Some fixes applied. Changes not consistently documented with cause or result.
E Follows/identifies — testing attempted. May describe running the program without structured test cases. Limited evaluation. May identify features rather than evaluate them against requirements. Some changes made in response to problems. Not documented in a way that demonstrates the test-evaluate-refine cycle.

Exam preparation

Question types and model answers

Design a test case 4 marks
"Design a test case to verify that a player is correctly eliminated when their money drops below zero. Include all five components of a structured test case."
B-range response
Test: Player bankruptcy
Setup: Player 1 money = $10. Player 2 owns a property with $30 rent.
Expected: Player 1 is removed from activePlayers. Token disappears. Their properties become unowned.
Actual: Player 1 eliminated. Token hidden. Properties released.
Result: PASS
A-range response
Test ID: TC-07-01 — Bankruptcy detection and elimination
Setup: 3 active players. Player 2 (index 1) money = $10, position = 0. Player 1 (index 0) owns Space_05 with rentAmount = $30. Force Player 2 to land on Space_05 by setting position = 4 and using roll stub return 1;.
Expected: ChargeMoney(30) returns false (money = $10−$30 = −$20 < 0). CheckBankruptcy() called. Player 2 removed from activePlayers (count: 3→2). Player 2 token SetActive(false). All Player 2 properties: owner = null. GameOver() not called (2 players remain). Next turn = Player 3.
Actual: Record after running.
Notes: Restore roll stub. Re-test with player at index 2 eliminated to verify index correction in EliminatePlayer().
A-range defines all expected outputs precisely before running (including the count change, SetActive call, and owner reset), names the methods involved, specifies the stub needed, and flags a follow-up edge case (eliminating the last-index player). B-range has the right structure but expected outputs are vague — "properties become unowned" doesn't specify how this is verified or what field is checked.
Distinguish testing from evaluation 4 marks
"A student says: 'I tested my game by playing through it and it worked, so I evaluated it and it meets all requirements.' Identify two problems with this statement and explain what genuine testing and evaluation would look like instead."
B-range response
Problem 1: Playing through the game is not structured testing. Structured testing requires specific inputs, expected outputs defined before running, and results recorded. "It worked" is not a result — it doesn't say what was checked or how.

Problem 2: Claiming all requirements are met without evidence is not evaluation. Evaluation requires comparing the game to each requirement individually, with evidence from test cases to support each rating. Genuine evaluation would include a requirements table with pass/fail for each requirement, supported by specific test cases.
A-range response
Problem 1: No structured testing occurred. "Playing through it" is exploratory interaction, not testing. Structured testing requires: a defined test case ID, specific inputs (e.g. Player position=21, roll stub=6), expected outputs derived from the algorithm by tracing pseudocode (e.g. newPos=3, money+=$200), actual outputs recorded after running, and a Pass/Fail determination. Without expected outputs defined before running, there is no basis for judging correctness — the tester is just observing, not testing.

Problem 2: Evaluation requires evidence and honesty, not assertion. "It meets all requirements" requires returning to each requirement from the original list, citing specific test evidence for each, and acknowledging partial or unmet requirements. A project that claims to meet every requirement with no failures has either not tested edge cases, has not been honestly evaluated, or has not evidenced its claims. Genuine evaluation includes a requirements table with Met / Partial / Not Met ratings, where "Not Met" entries name the cause and propose a technically specific fix — demonstrating that the student understands what would need to change and why.
A-range names both problems precisely, explains what structured testing actually requires (the five components), and specifically identifies that "expected outputs before running" is the missing element — not just "more testing." The evaluation problem is framed in terms of what the outcome requires (evidence, honesty, technically specific proposals), not just "do it more carefully."
Explain refinement 3 marks
"Explain why refinement is considered part of the testing and evaluating outcome rather than the producing outcome. Use an example from a software project to support your answer."
B-range response
Refinement is part of testing and evaluating because it is a response to what testing and evaluation found — not something done independently. If you change code without testing it first, you're not refining, you're just building. In a board game project, if a peer tester finds that the Roll Dice button stays active during a purchase, fixing that bug is refinement because it directly responds to the test finding.
A-range response
Refinement belongs to the T&E phase because it is a response to evaluation evidence, not independent development. In the producing phase, code is written to implement a planned design. Refinement changes existing code because testing revealed that the implementation doesn't match the specification or doesn't meet requirements — the motivation is evaluative, not additive. Without the test-evaluate sequence, there is no basis for knowing what to refine or why. In Dice District, fixing the index correction bug in EliminatePlayer() is refinement — it was driven by a specific test failure (Player 3 was skipped after Player 2's elimination), a reasoned hypothesis about the cause (list index shift after Remove()), and a verification that the fix resolved the specific symptom without breaking other tests (regression testing). That full chain — find, diagnose, fix, verify — is what makes it refinement rather than just coding.
A-range makes the conceptual point that refinement is evidence-driven (not additive development), then applies it to a specific Dice District example that includes all three refinement elements: the test finding, the diagnostic process, and the regression test. The distinction between "implementing a design" (producing) and "correcting an implementation" (refining) is the key insight.
