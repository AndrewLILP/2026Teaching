The Four Computing Domains

How outcomes are organised

The NSW Computing Technology syllabus organises learning across four domains. Every outcome belongs to at least one domain — understanding which domain an outcome lives in helps explain why it matters and how it connects to the others. In Dice District, all four domains are active simultaneously.

Course Performance Descriptors

What achievement looks like across the course

The CPDs describe typical achievement at each grade level for the whole course — not just one outcome. Each outcome page will drill into what these descriptors mean specifically for that outcome. The keywords in each grade distinguish one level from the next.

A
Extensive
Skilfully applies appropriate iterative processes
Technically concise algorithms in a general-purpose / OO language
Creativity & innovation in UI design
Selects relevant data, media and processes
B
Thorough
Applies appropriate iterative processes
Functional algorithms implemented
Creativity in UI design
Selects relevant data to communicate appropriately
C
Sound
Applies iterative processes
Common algorithms developed and implemented
Designs and implements user interfaces
Selects appropriate data to communicate
D
Basic
Uses processes to produce solutions
Basic algorithms developed and/or implemented
Basic elements of UI design implemented
Uses data to communicate basic information
E
Limited
Identifies processes that may produce a solution
Follows algorithms and/or partially implements
Identifies elements of UI contributing to UX
Uses data in a very limited way

Source: NSW NESA Course Performance Descriptors — Computing Technology 7–10 (2022), Stage 5. Each outcome page on this site translates these whole-course descriptors into outcome-specific indicators.

How to use this site

Two entry points, one site

This site is designed to be useful for both the people building the project and the people assessing it. The content is the same — the difference is in how each audience uses it.

🔒 For teachers
  • Use the outcome pages to plan which lessons address which dot points — cross-reference with the Dice District lesson site
  • Use the A–E evidence indicators on each outcome page to build marking rubrics and moderate assessments
  • Use the CPD strip on this page to discuss grade descriptors with students at the start of each project phase
  • The annotated code examples on OPL-01 can be projected during lessons to make the outcome-to-code connection explicit
  • Use the dot point clusters to identify gaps in coverage and plan supplementary activities
🎮 For students
  • Use each outcome page to understand why the Dice District tasks are structured the way they are — every lesson connects to outcomes here
  • Use the A–E evidence tables to understand what your notebook needs to contain at each grade level
  • Before an exam, use the "Exam preparation" section on OPL-01 — it shows exactly the question types you will face and what an A-range answer looks like
  • Use the pseudocode reference to check NSW conventions when writing algorithm questions
  • Use the Dice District examples to understand how code you've written connects to the formal syllabus language
