What this outcome means

All three verbs must be present

CT5-OPL-01 is not satisfied by writing working code alone. The outcome requires evidence of all three activities — designing (pseudocode or flowchart before coding), producing (implemented code that functions), and evaluating (test cases with recorded outcomes). A student who submits clean code with no planning or testing evidence cannot demonstrate above a C for this outcome.

The most common gap: Students who code confidently often skip the design phase — they open their editor before writing pseudocode. The evaluation phase is equally skipped: running the program and seeing it work is not evaluation. Evaluation means recording specific test cases, noting expected and actual outcomes, and documenting what was changed when results differed.

Why pseudocode comes first

NSW pseudocode conventions serve a specific purpose in this outcome: they force the algorithm to be expressed in language that is independent of any particular syntax. A pseudocode loop written before coding commits the student to a logical structure — the code then implements that structure rather than discovering it by trial and error. The notebook entry showing pseudocode before the corresponding code is the evidence marker that distinguishes design from production.

What "object-oriented" requires

Implementing in an OO language does not just mean writing code in C# or Python. It means using the features of the OO paradigm — classes, objects, encapsulation, inheritance, polymorphism. A single script with no class structure is general-purpose programming, not OOP. The Dice District project explicitly builds OOP evidence through its three-script architecture (BoardSpace, Player, GameManager) and the methods added across Lessons 4–7.

Use NSW pseudocode conventions to represent algorithms, including sequence, selection, and iteration

2

Trace through an algorithm manually to predict its output for a given input
The modulo visualiser in Lesson 5 demonstrates manual tracing: given position=21 and roll=6, trace (21+6) % 24 = 3. Test log tables require students to predict then verify outputs.
4
Decompose a problem into sub-problems and represent each as a separate algorithm or function
The game loop decomposes into RollDice(), MovePlayer(), ResolveSpace(), and NextTurn() — each a separate method with a single responsibility, designed independently before integration.
5
Identify and describe the purpose of variables, constants, and parameters within an algorithm
goSalary is a constant (set in Inspector, not changed at runtime); currentIndex is a state variable; roll is a parameter passed to MovePlayer().
🔧 Programming Constructs 6 dot points ▶
1
Implement selection using if / else if / else and switch statements
ResolveSpace() uses a switch on SpaceType. Inside the Property case, nested if / else if handles the three ownership sub-branches.
2
Implement iteration using for, foreach, and while loops
EliminatePlayer() uses foreach (BoardSpace space in p.ownedProperties) to release all properties. The Chance card shuffle uses a for loop with index manipulation.
3
Select and use appropriate data types: int, float, string, bool, and arrays / lists
money and position are int; isActive is bool; playerName is string; ownedProperties is List<BoardSpace>; boardSpaces is BoardSpace[].
4
Use arithmetic, relational, and logical operators correctly
Modulo (%) for board wrapping; < for bankruptcy detection; && for compound conditions in EliminatePlayer(); != for ownership checks.
5
Design and implement modular programs with functions / methods that accept parameters and return values
RollDice() returns int; ChargeMoney(int amount) returns bool (whether the player could afford it); IsAvailable() and IsOwnedByOther(Player p) are helper methods on BoardSpace that keep decision logic out of GameManager.
6
Structure code to facilitate computation — selecting appropriate data types, understanding limitations, and structuring code systematically
Using List<Player> for activePlayers (dynamic, shrinks on elimination) vs BoardSpace[] for boardSpaces (fixed-size — board never changes). The choice between these demonstrates understanding of data structure characteristics.
🏗 Object-Oriented Programming 6 dot points ▶
1
Define classes with attributes (fields) and behaviours (methods)
Player.cs defines attributes (playerName, money, position) and behaviours (AddMoney(), ChargeMoney(), AddProperty()) in a single class — the textbook class definition structure.
2
Instantiate objects from classes and access their members
Each Player and BoardSpace in the scene is an instance (object) of its class. activePlayers[currentIndex].playerName accesses the attribute of a specific Player object.
3
Apply encapsulation — using access modifiers to control what is publicly accessible
currentIndex is private — only GameManager advances the turn. gameOver is private — external scripts cannot accidentally set it. playerName is public — visible in the Inspector and readable by other scripts.
4
Use collections (arrays and lists) to manage multiple objects
activePlayers (List) holds all current players; boardSpaces (array) holds all board positions in order. Operations like activePlayers.Remove(p) and iteration with foreach demonstrate collection management.
5
Explore inheritance, polymorphism, and interfaces as features of OOP languages
All three scripts inherit from MonoBehaviour — Unity's base class. The SpaceType enum demonstrates type safety. Extension students implement IInteractable-style interfaces or a ChanceCard inheritance hierarchy.
6
Understand how predefined algorithms and libraries extend what a program can do
Random.Range() from UnityEngine; List<T> from System.Collections.Generic; Vector3.Lerp() for smooth movement; Debug.Log() for output — all predefined. The project relies heavily on Unity's predefined algorithms for rendering, physics, and UI.
🧪 Implementation, Testing & Debugging 4 dot points ▶
1
Develop and apply test cases — specifying inputs, expected outputs, and recording actual outcomes
Lessons 5, 6, and 7 each include a structured test table with Input, Expected, Actual, and Pass/Fail columns. The test stub technique (replacing RollDice() with a fixed return) is documented as a testing strategy.
2
Identify and correct syntax, logic, and runtime errors using debugging tools
Unity's Console distinguishes red (compile-time/runtime) from yellow (warnings). The GameManager.Start() validation errors are intentional — they surface misconfiguration immediately. The notebook bug log from Lesson 9 records error type, cause, and fix.
3
Document code using comments that explain purpose and logic, not just syntax
The Dice District starter scripts use [Header("")] groups in the Inspector, XML-style // ── Section ─ dividers, and inline comments explaining why (e.g. // temporary — replaced in Lesson 7) not just what.
4
Use version management strategies to preserve working states of a project
Manual sprint-based backups (copy the entire project folder at the end of each lesson) are appropriate for most Year 10 students. GitHub Desktop provides real VCS. Unity Version Control is reserved for extension students.

NSW Pseudocode — Full game loop
BEGIN GameLoop
  WHILE activePlayers.count > 1
    roll ← RollDice()
    MovePlayer(currentPlayer, roll)
    ResolveSpace(currentPlayer,
      boardSpaces[currentPlayer.position])
    IF currentPlayer.money < 0 THEN
      EliminatePlayer(currentPlayer)
    END IF
    IF activePlayers.count = 1 THEN
      GameOver()
    ELSE
      currentIndex ←
        (currentIndex + 1)
        MOD activePlayers.count
    END IF
  END WHILE
END GameLoop
C# — Distributed across GameManager methods
// StartTurn() → entry point
public void StartTurn()
{
  if (waitingForInput||gameOver)
    return;
  int roll = RollDice();
  MovePlayer(activePlayers
    [currentIndex], roll);
}
// MovePlayer() calls ResolveSpace()
// ResolveSpace() calls NextTurn()
// NextTurn() may call GameOver()
// → the loop is event-driven,
//   not a literal WHILE loop —
//   the button triggers each turn

Connection to CT5-THI-01

The thinking behind the algorithms

CT5-OPL-01 and CT5-THI-01 are interdependent — the thinking skills outcome describes the cognitive process that produces the artefacts OPL-01 requires. A student cannot demonstrate OPL-01 well without applying the thinking skills in THI-01, and vice versa.

🔬 Computational Thinking
Decomposition
Breaking the game into sub-problems: movement is separate from space resolution, which is separate from turn management. Each sub-problem maps to a method.
→ OPL-01 evidence: the four methods in GameManager each solve exactly one sub-problem
🔬 Computational Thinking
Abstraction
Hiding complexity behind clean interfaces. IsAvailable() hides the null-check logic. ChargeMoney() hides the update-and-return-bool pattern. Callers don't need to know how these work.
→ OPL-01 evidence: helper methods on BoardSpace and Player; public/private distinction
🔬 Computational Thinking
Pattern Recognition
The same pattern — "do something, update the HUD, advance the turn" — repeats in every space branch. Recognising this pattern leads to the ShowMessage() → UpdateHUD() → NextTurn() convention.
→ OPL-01 evidence: consistent structure across all ResolveSpace() cases
🔬 Computational Thinking
Algorithmic Thinking
Expressing solutions as precise, ordered steps that can be followed without ambiguity. The modulo formula, the elimination index correction, the Fisher-Yates shuffle — each requires precise step-ordering.
→ OPL-01 evidence: pseudocode written before code; trace tables in test logs
🎨 Design Thinking
Prototyping & Iteration
The temporary NextTurn() stub in Lesson 5 is a prototype — it makes the system testable before the full implementation. Replacing it in Lesson 7 is a deliberate iterative refinement, not an error correction.
→ OPL-01 evidence: the staged replacement of stubs across Lessons 5–7
⚙ Systems Thinking
Interconnection
The three scripts form a system where a change in one affects others. BoardSpace.owner is set by Player.AddProperty() and read by GameManager.ResolveSpace() — understanding this data flow is systems thinking.
→ OPL-01 evidence: class diagram showing data flows between classes

Dice District — Annotated Examples

OPL-01 in the code

The four examples below are drawn directly from the Dice District project. Each is annotated to show which dot points it evidences and how the code demonstrates the outcome.

RollDice() — Algorithm + randomness + return value
Sequence Functions/methods Return values Predefined libraries
GameManager.cs
public int RollDice()          // L1
{
  int d1 = Random.Range(1, 7);  // L3
  int d2 = Random.Range(1, 7);  // L3
  int total = d1 + d2;           // L5
  Debug.Log(                     // L6
    $"Rolled: {d1}+{d2}={total}");
  return total;                   // L7
}
L1
Method signature: public (accessible from UI button); int return type — the caller needs the value to pass to MovePlayer()
L3
Predefined library: Random.Range(1,7) from UnityEngine — the upper bound is exclusive, so this gives 1–6 inclusive (matching a physical die)
L5
Two dice, not one: Two separate calls produce a non-uniform distribution — rolling 7 is most probable. A single Range(2,13) would give uniform distribution, which doesn't match Monopoly's probability model
L6
Testing support: Debug.Log writes to the Console — essential during development. This line is what allows test logs to record actual dice values
L7
Return value: the total is passed back to StartTurn(), which passes it to MovePlayer() — clean separation of responsibilities
OPL-01 pseudocode for this method: BEGIN RollDice · die1 ← RANDOM(1,6) · die2 ← RANDOM(1,6) · total ← die1 + die2 · RETURN total · END RollDice
MovePlayer() — Modulo, state update, Go detection
Arithmetic operators Selection Variable assignment Data type int
GameManager.cs
int oldPos = p.position;        // L1
int newPos =                    // L2
  (p.position + roll)
  % boardSpaces.Length;

bool passedGo =                 // L3
  (newPos <= oldPos) && (roll > 0);

if (passedGo)                   // L4
  p.AddMoney(goSalary);

p.position = newPos;            // L5
L1
Saving state: oldPos is stored before the move — needed to detect Go crossing. Without this, the comparison on L3 would always use the already-updated position
L2
Modulo operator %: if position=21 and roll=6, result = 27 % 24 = 3. This is the core algorithm that makes the board loop — without it, the array index would go out of range
L3
Logical operator &&: both conditions must be true. newPos <= oldPos detects wrapping; roll > 0 guards against a degenerate case (a roll of 0 would incorrectly trigger Go)
L4
Conditional method call: AddMoney() on the Player object — not direct field access. This preserves encapsulation: GameManager doesn't manipulate p.money directly
L5
Order matters: position is updated after the Go check, not before. This is an example of algorithmic sequencing — swapping lines 3 and 5 would break the logic
This algorithm demonstrates the critical thinking skill of ordering — each line depends on the state left by the lines before it. Test case: position=21, roll=6 → expected newPos=3, passedGo=true.
ResolveSpace() — Multi-branch selection, OOP method calls
Switch/selection Nested if-else OOP — method calls on objects Encapsulation
GameManager.cs
switch (space.spaceType)        // L1
{
  case SpaceType.Property:       // L2
    if (space.IsAvailable())     // L3
      OfferPurchase(p, space);
    else if(space.IsOwnedByOther(p))
      ChargeRent(p, space);
    else
    { ShowMessage("Own property");
      NextTurn(); }
    break;

  case SpaceType.Tax:            // L4
    p.ChargeMoney(space.taxAmount);
    UpdateHUD();
    NextTurn();
    break;
}
L1
Switch vs if-else: a switch on an enum is more readable and less error-prone than a chain of if (space.spaceType == SpaceType.X). Each case is exhaustive — a new space type would require a new case
L2
Enum type safety: SpaceType.Property — the compiler prevents typos. Without the enum, "property" and "Property" would be different values with no compile error
L3
Abstraction via helper methods: IsAvailable() and IsOwnedByOther(p) are defined on BoardSpace, not inline here. This is encapsulation — the decision logic about a space lives on the space class, not the manager
L4
Consistent pattern: every branch ends with ShowMessage() → UpdateHUD() → NextTurn() (or delegates to a method that does). Pattern recognition → consistent structure
The flowchart for this method (drawn in Lesson 6 before coding) directly maps each branch to a diagram path — an example of pseudocode/flowchart preceding implementation.
EliminatePlayer() — Collection manipulation, index correction, order of operations
foreach loop List operations Selection Algorithmic sequencing
GameManager.cs
foreach (BoardSpace s           // L1
         in p.ownedProperties)
  s.owner = null;
p.ownedProperties.Clear();

int idx = activePlayers         // L2
  .IndexOf(p);
activePlayers.Remove(p);        // L3

if (idx <= currentIndex        // L4
    && currentIndex > 0)
  currentIndex--;

if (activePlayers.Count <= 1)  // L5
  GameOver();
else
  NextTurn();
L1
foreach releases properties: sets owner = null on every property this player owned — making them available for others to buy. Clear() then empties the list on the player object
L2
Save index before removal: IndexOf(p) must be called before Remove(p) — after removal, the element no longer exists in the list to find
L3
Dynamic list shrinks: after Remove(), every element after the removed index shifts down by 1. This is the source of the index-skip bug if L4 is omitted
L4
Index correction: if the eliminated player had an index ≤ currentIndex, the list shift means currentIndex now points one too far. Decrementing compensates — without this, a player gets skipped
L5
Win check after list update: the count is checked after removal so the final player remaining is found correctly. Checking before removal would read the pre-removal count
This method demonstrates that order of operations is not just a syntax concern — it is a logical requirement. The game-state simulator in Lesson 7 lets students see the index-skip bug by interactively eliminating players in different orders.

Exam preparation

Question types and model answers

CT5-OPL-01 is assessed in the Stage 5 exam (Assessment 2) through a predictable set of question types. The examples below show a B-range and A-range response to each type, with commentary on what distinguishes them.

Trace the algorithm 4 marks
"A player is on space 20 of a 24-space board and rolls a 7. Show the values of oldPos, the raw sum, newPos, and passedGo after MovePlayer() executes."
B-range response
oldPos = 20, roll = 7
raw = 20 + 7 = 27
newPos = 27 % 24 = 3
passedGo = true (3 < 20)
A-range response
oldPos = 20, roll = 7
raw = 27; newPos = 27 % 24 = 3
passedGo = (3 <= 20) && (7 > 0) = true
→ AddMoney(goSalary) is called
→ player.position updated to 3
→ ResolveSpace(p, boardSpaces[3]) called next
A-range answer traces the full condition (both parts of the &&), names the salary call, and identifies what happens next — demonstrating understanding of the sequence, not just arithmetic.
Write pseudocode 6 marks
"Write NSW pseudocode for an algorithm that searches a list of player objects and returns the player with the highest money value."
B-range response
BEGIN FindRichest(players)
  best ← players[0]
  FOR EACH p IN players
    IF p.money > best.money THEN
      best ← p
    END IF
  END FOR EACH
  RETURN best
END FindRichest
A-range response
BEGIN FindRichest(players)
  // Guard: handle empty list
  IF players.count = 0 THEN
    RETURN NULL
  END IF
  best ← players[0]
  FOR EACH p IN players
    IF p.money > best.money THEN
      best ← p
    END IF
  END FOR EACH
  RETURN best
END FindRichest
A-range adds the empty-list guard — handling the edge case where no players are passed in. This demonstrates awareness that algorithms must handle unexpected inputs, not just the typical case. Both use correct NSW FOR EACH and RETURN conventions.
Identify the error 3 marks
"The following code is meant to advance to the next player after one is eliminated. Identify the error and explain how to fix it."

activePlayers.Remove(p);
int idx = activePlayers.IndexOf(p);
if (idx <= currentIndex) currentIndex--;
B-range response
The error is that IndexOf(p) is called after Remove(p). Once a player is removed from the list, IndexOf returns -1 because the element no longer exists. The fix is to swap the two lines — call IndexOf before Remove.
A-range response
This is a logic error — the code compiles and runs but produces the wrong result. Remove(p) on line 1 deletes the element, so IndexOf(p) on line 2 returns -1. The condition -1 <= currentIndex is always true (currentIndex is never negative), so currentIndex is always decremented — every elimination incorrectly moves the turn back regardless of which player was eliminated. Fix: move IndexOf(p) before Remove(p).
A-range identifies the error type (logic, not syntax), explains exactly why -1 causes the wrong behaviour, and describes the precise consequence — not just "it doesn't work." The fix is identical in both answers, but the reasoning is what earns marks.
