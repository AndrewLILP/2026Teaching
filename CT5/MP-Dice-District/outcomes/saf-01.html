CT5-SAF-01 Social Awareness

Safe, Secure
& Ethical
Practices

"Selects and applies safe, secure and ethical practices in the use of data"
Safe
Personal wellbeing + data
Protecting yourself and others from harm when using, sharing, or storing data — including personal information, passwords, and digital identity
Secure
Technical protection
Implementing technical measures to protect data from unauthorised access, modification, or loss — encryption, access control, version management
Ethical
Fair and responsible use
Using data and computing tools in ways that are honest, fair, and respectful of others' rights — including copyright, privacy, consent, and the impacts of computing on society

What this outcome means

Three pillars — one outcome

CT5-SAF-01 is deceptively broad. The three words — safe, secure, ethical — are often treated as synonyms but target distinctly different concerns. Safe is about personal wellbeing and responsible behaviour online. Secure is about technical measures that protect data integrity and access. Ethical is about the values and responsibilities that govern how computing power is used. A student who discusses online safety without addressing data security or ethical decision-making, or who focuses only on copyright without addressing cybersecurity, is only partially evidencing this outcome.

The key words are selects and applies. Simply knowing what privacy means is not sufficient — the outcome requires visible decisions: choosing to attribute an asset, choosing not to store player data, choosing to review the ethical implications of a feature. The project notebook is where these decisions are documented.

SAF-01 in a game development context: At first glance, a local Unity game might seem to have little SAF-01 relevance — no server, no user accounts, no sensitive data collection. But safe, secure, and ethical practices are present throughout Dice District: attribution of Synty asset packs (ethical/copyright), version management protecting work from loss (secure), peer testing protocols that respect student feedback honestly (ethical), and the AI ethics thread if extended into web applications (ethical). SAF-01 is about the habits of mind a computing practitioner develops, not just the technical features of a specific project.

Why "selects" matters

The outcome doesn't say "follows safe practices" — it says selects them. Selection implies judgement: recognising which practice is appropriate for a given situation, choosing between options, and being able to explain why the chosen practice is appropriate. This is what separates A-range evidence (a documented decision with reasoning) from C-range evidence (compliance with a rule without reflection).

The three pillars in depth

Safe, Secure, Ethical — defined and distinguished

🛡
Pillar 1
Safe — protecting people
Key concepts
  • ◈
    Personal Data Protection
    Not collecting, storing, or sharing personal information beyond what is necessary. Data minimisation: only collect what you need.
  • ◈
    Digital Footprint Awareness
    Understanding that interactions with digital systems leave traces — in logs, in cloud storage, in analytics. Computing practitioners make these traces intentionally, not accidentally.
  • ◈
    User Wellbeing
    Designing systems that don't manipulate, mislead, or harm users. Dark patterns — UX design techniques that trick users into unintended actions — are a safety concern as much as a design concern.
  • ◈
    Safe Communication Online
    When computing work involves online platforms (GitHub, itch.io, forums), practising safe communication — not sharing private information, understanding community standards.
Dice District + project context
All lessons — Data minimisation
Dice District stores no personal player data. Player names are entered per session and never persisted between runs. This is a deliberate design decision — the notebook entry documenting that no localStorage or file I/O was implemented (and why) is SAF-01 evidence.
→ Notebook evidence: a note in the design phase explaining the decision not to store persistent player data
Lesson 9 — Peer testing safety
The peer testing protocol requires honest, constructive feedback — not identifying or shaming individual students' work. The observation checklist is structured to describe observations ("the message panel text was unclear") rather than judgements about the developer.
→ Notebook evidence: reflection on peer testing experience, including how feedback was given and received
Extension — itch.io publishing
Publishing a WebGL build on itch.io raises safe practices questions: what is displayed on the profile page? Are player names collected (they shouldn't be)? What age rating is applied? Safe publishing decisions are documented and justified.
→ Notebook evidence: itch.io publishing decisions with reasoning
🔐
Pillar 2
Secure — protecting data technically
Key concepts
  • ◈
    Access Control
    Restricting who can access what data or system. In code: private vs public access modifiers. In projects: not sharing Unity project passwords or GitHub credentials.
  • ◈
    Version Management
    Preserving working states of a project so that changes can be reviewed and reverted. Manual sprint backups, GitHub Desktop, or Unity VCS — all protect against accidental data loss.
  • ◈
    Data Integrity
    Ensuring that data is not corrupted or altered unintentionally. In the context of game development: saving the Unity scene before making changes; not overwriting a working build before testing a new version.
  • ◈
    Threat Awareness
    Understanding common threats to computing systems: malware, phishing, unauthorised access. Recognising that publishing code online (GitHub) makes it publicly visible — API keys and credentials should never be committed.
Dice District + project context
All lessons — Version management
The sprint backup strategy — copying the entire Unity project folder at the end of each lesson — is a secure practice. It creates a restoration point before each new development cycle. The notebook records which backup corresponds to which lesson.
→ Notebook evidence: backup log with lesson reference and description of working state at each backup point
Lesson 4 — Access modifiers as security
The deliberate use of private for currentIndex, gameOver, and other state variables is not just encapsulation — it is access control. Only GameManager can change the turn index. Explaining this reasoning in the notebook connects a coding decision to a security principle.
→ Notebook evidence: annotation explaining why specific fields are private — "only GameManager should change the turn order"
Extension — GitHub
Students publishing to GitHub should understand that public repositories are visible to anyone. The repository should not contain Unity .meta files with user-specific paths, credentials, or sensitive personal data in comments.
→ Notebook evidence: a .gitignore decision entry explaining what was excluded and why
⚖
Pillar 3
Ethical — responsible and fair use
Key concepts
  • ◈
    Intellectual Property & Copyright
    Respecting ownership of creative works — code, images, music, fonts, 3D assets. Understanding the difference between proprietary licences, Creative Commons, and open source. Attributing correctly.
  • ◈
    Privacy & Consent
    Only collecting data that users have consented to provide. Not using data for purposes beyond what it was collected for. Being transparent about what data is collected and how it is used.
  • ◈
    Bias & Fairness
    Recognising that computing systems can embed and amplify human biases — in algorithms, in training data, in design decisions. A game that only represents certain characters or cultures is making an ethical choice, whether consciously or not.
  • ◈
    Social Impact
    Understanding the broader consequences of computing on employment, access, environment, and society. AI, automation, and data collection all have societal impacts that a responsible practitioner considers.
  • ◈
    Academic Integrity
    In a school context: using AI tools transparently, attributing code assistance, not submitting others' work as your own. Understanding where the line is between legitimate learning support and academic dishonesty.
Dice District + project context
All lessons — Asset attribution
The Synty low-poly asset pack used in Dice District is a commercial licence. Students must understand: the licence permits use in student projects but not resale; modifications are permitted; the publisher must be credited in any public release. The notebook records the licence type and attribution decision.
→ Notebook evidence: asset list with source, licence type, and attribution plan for each
All lessons — Code attribution
Code from the Dice District starter scripts, Stack Overflow, or AI assistance must be attributed. The distinction between "I adapted this" and "I wrote this" is an ethical one. Comments in code like // Adapted from Dice District L5 starter — modified modulo formula are evidence of ethical practice.
→ Notebook evidence: a note in the design phase listing which code was written from scratch vs adapted, and from where
Lesson 1 — Representation in game design
The theme selection in Lesson 1 involves choosing names, spaces, and visual style. A game set in a real city with only certain cultural references is making a representation choice. A brief reflection on who the game is designed for and whether the design is inclusive is an ethical consideration in DES-01 and SAF-01.
→ Notebook evidence: theme selection reflection noting whether the design is culturally inclusive

Applied scenarios

Twelve situations — which pillar applies?

The table below presents twelve realistic computing scenarios. For each, the relevant pillar(s) are identified and the appropriate practice is described. These scenarios are representative of the type of scenario-based questions that appear in Stage 5 exams and assessments.

Scenario Pillar(s) Appropriate practice
Using a free music track found on YouTube for a game
 Ethical Check the licence — not all YouTube content is free to use. Search for Creative Commons licensed music (e.g. Free Music Archive). Attribute the creator in the game's credits regardless of licence type.
A student copies a sorting algorithm from Stack Overflow into their project
 Ethical Add a comment attributing the source: // Fisher-Yates shuffle — adapted from stackoverflow.com/q/1150917. In the notebook, distinguish this from self-written code. Understand the licence (Stack Overflow uses CC BY-SA).
The Unity project folder is only on one student's laptop
 Secure Create a shared backup location (shared Google Drive, USB, or private GitHub repo) so all group members have access. Document the backup strategy in the notebook. Don't rely on a single point of failure.
A web app collects the user's name and stores it in localStorage
 Safe Ethical Only collect what is necessary. Inform the user that their name is stored and why. Provide a way to clear the data. Never share localStorage data with third parties or use it for unintended purposes.
A student uses GitHub Copilot to generate several methods for their project
 Ethical Disclose AI tool use in the notebook. Attribute AI-generated code clearly in comments. Verify the generated code works correctly and understand what it does — submitting code you cannot explain is academically dishonest regardless of source.
A peer tester finds a serious bug and tells the teacher without telling the developer
 Ethical Peer testing protocols require honest, direct feedback to the developer. Bypassing the developer to report to the teacher is not respectful of the collaborative process. The structured checklist and bug log are the appropriate channels.
A student publishes their Unity game to GitHub with their school email in the code comments
 Safe Secure Remove personal identifying information before publishing. A public repository is accessible by anyone — school email addresses in code comments can attract spam. Use a .gitignore to exclude generated files containing user-specific paths.
An AI system is trained on biased data and recommends unfair loan rates
 Ethical Recognise this as an algorithmic bias issue — the training data embedded historical discrimination, and the system amplified it. Responsible AI development requires auditing training data, testing for bias, and designing human oversight into consequential decisions.
A student reuses last year's project with minor changes and submits it as new work
 Ethical Academic integrity requires that submitted work represents current, original effort. Reusing prior work without disclosure is self-plagiarism. If prior work is built upon, the nature and extent of that reuse must be disclosed.
A game includes a leaderboard that stores player scores and usernames online
 Safe Secure Ethical Inform players their data is stored and how. Implement access control — only the owner can delete their entry. Consider data retention — how long are scores kept? Ensure the username field can't be used to share harmful content.
A student uses a free Synty asset in their game and publishes to itch.io
 Ethical Check whether the free Synty pack permits publishing. Most Synty packs require a paid licence for commercial or public publication — the free education licence may be for classroom use only. Credit Synty in the game's credits/readme regardless.
Automation replaces 30% of workers in a logistics company
 Ethical The social impact pillar: computing practitioners bear some responsibility for the societal consequences of the systems they build. This doesn't mean automation is wrong — but it means the decision should be made with awareness of its human consequences, not purely as an efficiency calculation.

The AI ethics thread

Artificial intelligence and ethical practice

AI ethics is increasingly central to SAF-01 in contemporary computing contexts. The scenarios below are most directly relevant to the Web Applications and Enterprise Information Systems units, but they apply wherever automated decision-making, AI tools, or data-driven systems are used — including in game development.

🤖
Five core AI ethics issues
Relevant to SAF-01 · Social Awareness domain
1
Bias and Fairness
AI systems trained on historically biased data reproduce and amplify that bias at scale. A facial recognition system trained predominantly on lighter-skinned faces performs worse on darker-skinned faces — not through malice, but through unexamined data selection. Ethical practice requires actively auditing data and testing systems across diverse groups, not just assuming they work because the overall accuracy is high.
SAF-01 connection: selecting and applying ethical practices means questioning whether a system produces fair outcomes for all users, not just the majority.
2
Transparency and Explainability
A system that makes consequential decisions (loan approval, medical diagnosis, criminal sentencing) without being able to explain its reasoning is ethically problematic. Users affected by automated decisions have a right to understand why. "The algorithm said so" is not an acceptable explanation when livelihoods or freedoms are at stake.
SAF-01 connection: an ethical practitioner designs systems with explainability in mind, and ensures that automated decisions are subject to human review.
3
Privacy and Surveillance
AI systems can process vast amounts of personal data to infer things people haven't disclosed — location patterns, political views, health status, emotional state. Data collected for one purpose (navigation) can be used for another (targeted advertising or law enforcement). The ethical question is not just "is this legal?" but "is this what the user agreed to, and is it what they expected?"
SAF-01 connection: data minimisation, transparency about data use, and respecting the boundaries of consent are core ethical practices.
4
Accountability and Responsibility
When an AI system causes harm — a self-driving car injures a pedestrian, a recommendation algorithm radicalises a user — who is responsible? The developer? The organisation deploying it? The user? Ethical practice requires thinking about accountability before deploying systems, not after harm occurs. This includes building in kill switches, monitoring, and clear escalation paths.
SAF-01 connection: selecting and applying ethical practices includes planning for system failure, not just designing for success.
5
Automation and Employment
Automation powered by AI is displacing jobs at a scale and pace that outstrips the creation of new roles. This is not an argument against automation — efficiency and safety gains are real — but ethical practice requires that the consequences for affected workers be considered, not externalised. A computing practitioner who designs an automation system without considering its social consequences is not practicing ethically.
SAF-01 connection: the social impact of computing decisions is explicitly within the scope of this outcome — "the use of data" extends to the use of computing power to affect people's lives.
Using AI tools in your project: GitHub Copilot, ChatGPT, and similar tools are AI assistance — using them is not unethical, but using them without disclosure is. The ethical practice is: (1) note in your notebook that AI assistance was used; (2) attribute AI-generated code with a comment; (3) verify and understand the code before submitting it; (4) be able to explain what it does in your own words. Code you cannot explain is code you don't own.

Exam preparation

Question types and model answers

Distinguish safe / secure / ethical 3 marks
"A student is developing a web application that collects user names and game scores. Describe one safe, one secure, and one ethical practice the student should apply, and explain why each is necessary."
B-range response
Safe: Tell users that their name and score will be stored and how it will be used, so they can make an informed decision about whether to enter their information.

Secure: Use a backup system for the database and restrict who can delete or edit entries, so the data is protected from accidental loss or unauthorised changes.

Ethical: Only collect the data needed for the leaderboard — don't also collect email addresses or other personal details that aren't necessary for the game to function.
A-range response
Safe: Inform users clearly, before they enter their name, that it will be publicly visible on the leaderboard and stored between sessions. This is necessary because users have a right to know what will happen to their data — posting a name publicly without consent is a safety and privacy violation even if the data seems harmless.

Secure: Implement access control so that only the score submission endpoint can write to the leaderboard database — not arbitrary requests. This prevents malicious users from submitting fake scores or deleting legitimate entries. Validate all input server-side to prevent injection attacks.

Ethical: Apply data minimisation — collect only the username and score, not any other identifiable information. Under Australian privacy principles, data should not be collected beyond its stated purpose. Storing an IP address "in case we need it later" is not ethical if it wasn't disclosed to users as part of data collection.
A-range gives a specific reason why each practice is necessary — not just what to do, but what harm would result without it. The safe answer names the specific violation (posting publicly without consent). The secure answer names the specific threat (injection attacks, malicious writes). The ethical answer names the relevant principle (Australian privacy principles, data minimisation) and gives a concrete example of a violation (storing IP addresses without disclosure). B-range has the right structure but reasons are generic rather than principle-anchored.
Evaluate an ethical issue 5 marks
"A game company uses player data collected from millions of users to train an AI that predicts which players are likely to make in-app purchases. The AI then serves these players more frequent purchase prompts. Identify the ethical issues raised and evaluate whether this practice is acceptable."
B-range response
There are several ethical issues. First, users probably didn't consent to having their data used to train an AI system — they consented to playing a game, not to being analysed. Second, targeting players who are more likely to spend money with more prompts is manipulative — it exploits vulnerable users, possibly including people with gambling tendencies or children. Third, there is a transparency problem — players can't see that they are being profiled and treated differently. This practice is not acceptable because it uses data beyond the purpose users consented to and is designed to exploit rather than serve users.
A-range response
This scenario raises three distinct ethical issues, each from a different pillar of SAF-01.

Privacy (Safe/Ethical): Users consented to play a game — not to have their behavioural data mined to build a predictive model. Using data collected for one purpose (gameplay) for another (commercial targeting) violates the principle of purpose limitation, a core requirement of privacy frameworks including the Australian Privacy Act.

Manipulation and User Wellbeing (Ethical): Serving more frequent purchase prompts to users identified as more likely to spend is a dark pattern — an interface design technique that exploits psychological vulnerability rather than serving user interests. This is particularly concerning if the model identifies players with problem gambling behaviours or minors, who may lack the capacity for fully informed consent.

Transparency and Accountability (Ethical): Players are unaware they are being profiled and that their experience is personalised to extract money. Responsible AI practice requires transparency about automated decision-making that affects users.

Evaluation: This practice is not acceptable under an ethical framework even if it is currently legal in many jurisdictions. The test of ethical practice is not "is it legal?" but "does it respect user autonomy, apply data only as consented to, and avoid harming users?" This system fails all three tests. An ethical alternative: use aggregate data to improve the game experience for all users, with transparent disclosure of how data is used.
A-range structures the answer across three named pillars, names specific principles (purpose limitation, Australian Privacy Act, dark patterns), identifies the specific vulnerable group (problem gamblers, minors), evaluates using a three-part ethical test (autonomy / consent / harm), and proposes an ethical alternative. B-range identifies the same issues but without the framework, the named principles, or the evaluative structure. The final "this is not acceptable because" in B-range is correct but not well-evidenced — A-range earns marks for the reasoning chain, not just the conclusion.
Apply to a project context 3 marks
"Describe two safe, secure, or ethical practices you applied in your computing project. For each, explain the decision you made and why it was the appropriate choice."
B-range response
Practice 1 (Secure): I made a backup of our Unity project at the end of every lesson. This meant that if something went wrong in a later lesson, we could restore to a working version. It prevented data loss and made the team feel safer making changes.

Practice 2 (Ethical): I attributed the Synty asset pack in our game's readme file. The licence allows student use but requires attribution if published. We noted the licence type in our notebook so we knew we were using the assets legally.
A-range response
Practice 1 (Secure — Version Management): We created a sprint backup at the end of each lesson by copying the entire Unity project folder to a shared Google Drive folder named by lesson number and date. I chose this practice because a single corruption event, accidental overwrite, or hardware failure would have destroyed all our work without an external copy. The decision to use Google Drive rather than just a local USB was specifically to protect against hardware loss — a single-location backup is not genuinely secure. I documented this in the notebook with a backup log listing which lesson each backup corresponds to.

Practice 2 (Ethical — AI Tool Attribution): I used GitHub Copilot to suggest the Fisher-Yates shuffle algorithm for the Chance card system. I disclosed this in my notebook and added a comment in the code: // Fisher-Yates shuffle — structure suggested by GitHub Copilot, verified and adapted by me. I chose this practice because submitting code I did not write without attribution would be academically dishonest, even if AI assistance is permitted. The attribution also shows I reviewed and understood the code rather than blindly accepting the suggestion.
A-range gives a specific decision rationale for each practice — not just "I did X" but "I chose X over Y because Z". The backup answer explains why Google Drive was chosen over a USB (single-location risk), and the AI attribution answer engages directly with academic integrity rather than just describing what was done. Both answers name the notebook as the evidence location, which connects the practice to the assessment evidence trail.
