What this outcome means

Thinking is not the same as doing

CT5-THI-01 is the only outcome that is entirely about cognitive process — not about what was built, but about how it was thought through. It is also the most commonly underassessed outcome, because students who think well but don't document their thinking leave no evidence.

The key word is applies. Simply knowing what decomposition or abstraction means is not sufficient — the outcome requires these thinking skills to be visibly applied during the development of a solution. The project notebook is the primary evidence vehicle: annotations, justifications, diagrams, and reflections that show thinking, not just doing.

The documentation problem: A student who thinks in sophisticated ways but writes "I wrote the code and it worked" in their notebook cannot demonstrate this outcome above a C. Conversely, a student whose code is unpolished but whose notebook shows careful decomposition, an annotated class diagram, and honest reflection on what didn't work can demonstrate B-range thinking skills. The quality of the thinking, not the quality of the product, drives this outcome.

Why three different thinking modes?

The three modes target different aspects of solution development. Computational thinking drives the algorithmic structure — how the code is organised and why. Design thinking drives the user-facing decisions — who the game is for and how they experience it. Systems thinking drives the architectural decisions — how the three scripts interact and what happens when one part changes.

In Dice District, all three modes are active simultaneously — and a sophisticated notebook makes all three visible.

Relationship to CT5-OPL-01

THI-01 and OPL-01 are deeply interdependent. The thinking skills in THI-01 are the process that produces the artefacts OPL-01 requires. Pseudocode is the product of algorithmic thinking. A class diagram is the product of decomposition and abstraction. A test log is the product of evaluative thinking. Neither outcome can be strongly evidenced without the other — they are assessed together through the same notebook and code.

The three thinking modes

Each mode — what it is and what it produces

🔬
Mode 1
Computational Thinking
The four sub-skills
◈
Decomposition
Breaking a complex problem into smaller, manageable sub-problems. Each sub-problem can be solved independently then integrated. In code, this maps directly to methods and classes.
◈
Abstraction
Hiding complexity behind a clean interface. Identifying what details matter for a given purpose and ignoring the rest. In code: methods, encapsulation, helper functions.
◈
Pattern Recognition
Identifying similarities across problems or within a solution — then exploiting them. Repeated structures become reusable methods; repeated sequences become refactored code.
◈
Algorithmic Thinking
Expressing solutions as precise, unambiguous, ordered steps. Writing pseudocode, tracing through algorithms, and predicting outputs before running code.
Dice District examples
Lesson 2 — Decomposition
The game loop is decomposed into four independent sub-problems: RollDice(), MovePlayer(), ResolveSpace(), NextTurn(). Each method was designed separately in pseudocode before any code was written.
→ Notebook evidence: game loop diagram with labelled responsibilities per method
Lesson 4 — Abstraction
IsAvailable() and IsOwnedByOther() abstract the ownership logic onto BoardSpace. GameManager.ResolveSpace() reads clean conditions without knowing how they work internally.
→ Notebook evidence: annotation explaining why helper methods were placed on BoardSpace rather than GameManager
Lessons 6–7 — Pattern Recognition
Every branch of ResolveSpace() ends with ShowMessage() → UpdateHUD() → NextTurn(). Recognising this pattern produces a consistent, predictable structure across all four space types.
→ Notebook evidence: reflection noting the repeated structure and explaining why it was kept consistent
Lesson 5 — Algorithmic Thinking
The modulo board-wrap formula: (position + roll) % boardLength. Written as pseudocode, traced for the case position=21/roll=6 before implementation. The test log records predicted vs actual output.
→ Notebook evidence: pseudocode + trace table for at least 3 test cases
🎨
Mode 2
Design Thinking
The five stages (in a project context)
◆
Empathise
Understanding who will use the solution and what they need — before defining requirements. Analysing Monopoly in Lesson 1 is empathy: what does a board game player need to have a satisfying experience?
◆
Define
Translating observations into a precise problem statement and requirements list. Lesson 1's functional and non-functional requirements are the design thinking "Define" stage.
◆
Ideate
Generating multiple possible solutions before committing to one. The theme selection in Lesson 1, the scope planner in Lesson 3, and the UI wireframe alternatives are all ideation activities.
◆
Prototype
Building a version quickly to test an idea — accepting that it will be replaced. The temporary NextTurn() stub in Lesson 5 is a deliberate prototype: it makes the system testable before the full implementation exists.
◆
Test & Iterate
Testing with real users, then feeding findings back into the design. The Lesson 9 peer testing protocol and the priority fix list are the design thinking test stage.
Dice District examples
Lesson 1 — Empathise + Define
The requirements list (6 functional + 2 non-functional) is the output of empathising with the players and defining what the game must do. The board sketch is the first concrete design artefact.
→ Notebook evidence: requirements justified with "because the player needs to…" reasoning, not just listed
Lesson 3 — Ideate + Prototype
The scope priority planner (Must / Nice / Cut) is ideation under constraint. The UI wireframe is an early prototype — it tests information layout before any code is written.
→ Notebook evidence: scope decisions justified ("we moved Lerp movement to Nice because…")
Lesson 8 — Iterate
The self-assessment table in Lesson 8 is design thinking applied to the process itself — evaluating where the current state sits against the target state and deciding what to improve.
→ Notebook evidence: specific features changed after self-assessment, with reasoning
Lesson 9 — Test with users
The peer testing checklist gives external users a structured way to test the game. The bugs found are fed directly into the priority fix list — a genuine design iteration cycle.
→ Notebook evidence: at least one design decision changed specifically because of peer feedback
⚙
Mode 3
Systems Thinking
Key concepts
⬡
Components & Responsibilities
Every system has identifiable parts, each with a clear role. In Dice District: BoardSpace holds space data; Player holds player state; GameManager orchestrates the loop.
⬡
Relationships & Data Flows
How components pass data between each other. BoardSpace.owner is written by Player.AddProperty() and read by GameManager.ResolveSpace() — a data flow that crosses all three scripts.
⬡
Feedback Loops
How outputs become inputs. Money flows between players; bankrupting one player changes the active player list, which changes the turn cycle — a feedback loop that drives the game to its end state.
⬡
Emergent Behaviour
Outcomes that arise from component interactions — not planned directly. When Player A owns all blue properties, the combined rent effect on others creates a winning momentum not explicitly coded as a feature.
⬡
Cascading Effects
A change in one component affects others. Understanding that removing a player from the list requires clamping the current index — seeing the ripple — is systems thinking applied to debugging.
Dice District examples
Lesson 2 — Components
The class diagram is the primary systems thinking artefact — it maps components, their attributes, methods, and the relationships between them before any code exists.
→ Notebook evidence: class diagram with arrows showing data flows between classes
Lesson 6 — Data flows
Tracing what happens when a player buys a property: GameManager calls Player.ChargeMoney() and Player.AddProperty(); AddProperty() sets BoardSpace.owner; later ResolveSpace() reads that field. Understanding this chain is systems thinking.
→ Notebook evidence: a data flow trace annotating which method sets which field and where it is read
Lesson 7 — Feedback loops
The elimination simulator in Lesson 7 makes the feedback loop visible: eliminating a player changes activePlayers.Count, which changes the modulo formula in NextTurn(), which determines who plays next. Understanding the loop requires systems thinking.
→ Notebook evidence: explanation of why the index must be clamped after removal (the cascading effect)
Lesson 10 — System evaluation
Returning to requirements in Lesson 10 is systems thinking: comparing the designed system to the intended system, identifying gaps, and articulating what changed and why — not just listing features.
→ Notebook evidence: requirements not met explained in terms of what the system currently does instead

Notebook evidence guide

What counts — and what doesn't

THI-01 is the outcome most dependent on notebook quality. Because the thinking happens in the mind, the notebook is the only window into it. The cards below distinguish evidence that demonstrates thinking from responses that merely describe activity.

🔬
Computational thinking — evidence that counts
  • Pseudocode written before the corresponding code, with at least one annotation per step explaining its purpose
  • A decomposition diagram showing which sub-problem maps to which method — drawn at the planning stage, not reconstructed after
  • A trace table showing predicted outputs for at least 3 test cases before running the program
  • An explanation of why a helper method was placed on a specific class rather than another
  • A reflection identifying a repeated pattern in the code and explaining how it was exploited
Does not count
  • Pseudocode written after the code is finished (retrospective documentation)
  • "I used decomposition" without naming what was decomposed into what
  • A diagram that was drawn after the code was built, showing the code rather than the plan
🎨
Design thinking — evidence that counts
  • Requirements justified with user-centred reasoning — not just listed
  • Scope decisions that name what was cut and explain why, not just what was kept
  • A UI wireframe with annotations explaining what information each element communicates to the player
  • At least one design decision that was changed specifically because of peer feedback — described with what the original decision was
  • A reflection addressing what would be different if you designed it again — not just what went wrong
Does not count
  • A wireframe with no annotations (just boxes and labels)
  • "We didn't have time" as the only explanation for scope cuts
  • Reflections that only describe what was built, without evaluating the design decisions
⚙
Systems thinking — evidence that counts
  • A class diagram that shows relationships between classes (not just lists of fields)
  • A data flow annotation tracing where a specific value originates, where it is passed, and where it is used
  • A debugging reflection that traces a bug from symptom back to root cause across multiple methods
  • An explanation of why a change in one part of the code required a corresponding change elsewhere
  • The updated class diagram in Lesson 7 with a "delta" annotation explaining what changed from the original plan and why
Does not count
  • A class diagram that only lists attributes (no methods, no relationships)
  • Bug log entries that say "it was broken" without a cause
  • Describing the system after it was built without evidence of planning how it fit together
📓
Reflection — the most underused evidence type
  • Reflections that explain why a decision was made, not just what was done
  • Honest documentation of what didn't work and what was tried — including dead ends
  • A "what I would do differently" entry that is technically specific (names a method, a class, a data structure)
  • Connections between lessons — a reflection that refers back to an earlier decision and evaluates it with new understanding
Does not count
  • "This lesson was good/hard/interesting" with no substantive content
  • Reflections that only describe what the lesson was about, not what the student thought about it
  • "I would add more features" without explaining how, in technical terms

Exam preparation

Question types and model answers

THI-01 is assessed in the exam through questions that ask students to apply — not define — the thinking skills. The most common formats are scenario-based questions where a partial solution is given and students must demonstrate thinking to extend or evaluate it.

Apply decomposition 5 marks
"A student is designing a digital card game. The game needs players to draw cards, play cards to affect opponents, track health points for each player, and end when one player reaches zero health. Using decomposition, identify the sub-problems and describe a class or method responsible for each."
B-range response
Sub-problems:
1. Drawing cards — Deck.DrawCard()
2. Playing cards — GameManager.PlayCard()
3. Tracking health — a Player class with a health attribute
4. Ending the game — GameManager.CheckWin()

Each sub-problem is handled by a separate method or class so they can be developed and tested independently.
A-range response
Decomposed sub-problems:
1. Card management — a Deck class with cards[], DrawCard() returning a Card object, and Shuffle(). Separating deck logic from game logic means the deck can be tested in isolation.
2. Card effects — a Card class with an effect attribute and ApplyEffect(Player target). Placing effect logic on the card (not the manager) means adding new card types doesn't require changing GameManager.
3. Player state — Player class with health, TakeDamage(int amount) returning bool (survived), and hand (List<Card>).
4. Win condition — GameManager.CheckWin() iterates players and calls GameOver() when health ≤ 0.
A-range justifies each decomposition decision — explaining why the logic lives in that class (e.g. effect logic on Card so GameManager doesn't need to change for new card types). B-range correctly identifies the sub-problems but doesn't explain the reasoning behind the structure.
Identify and explain 4 marks
"A developer notices that every time a player lands on a space, the same three actions occur: show a message, update the HUD, then advance the turn. They create a helper method EndSpaceAction(string msg) that does all three. Name the computational thinking skill used and explain how it improves the code."
B-range response
The thinking skill is pattern recognition. The developer saw that the same three steps appeared in multiple places and combined them into one method. This means if the pattern needs to change, it only needs to be updated in one place.
A-range response
The thinking skill is pattern recognition, combined with abstraction. The developer recognised that ShowMessage + UpdateHUD + NextTurn was a recurring sequence across all four space type branches. By abstracting this into EndSpaceAction(), two improvements result: (1) the code is DRY (Don't Repeat Yourself) — a future change to the end-of-turn sequence only requires modifying one method instead of four; (2) the branches in ResolveSpace() become more readable — the reader can see the intent (end the space action) rather than three separate implementation steps. This also reduces the risk of a bug where one branch updates the HUD but another forgets to.
A-range correctly names both skills (pattern recognition + abstraction), explains two concrete benefits (DRY principle, readability), and adds a risk-reduction argument (one branch forgetting a step). B-range names the correct skill and gives one benefit — solid B-range reasoning.
Apply systems thinking 4 marks
"A developer changes the money field in the Player class from public to private. Describe two other parts of the system that would need to change as a result, and explain why systems thinking is needed to make this change safely."
B-range response
If money is private, GameManager can no longer access p.money directly. A public getter method like GetMoney() would need to be added to Player. Also, any UI code that reads player.money to display the balance would need to use this getter instead. Systems thinking is needed to find all the places that use money before making the change.
A-range response
Making money private creates a cascading change through the system. Two parts that must change: (1) GameManager.UpdateHUD() reads p.money to display the balance — this would need either a public getter GetMoney() or a public property public int Money { get; private set; }. (2) Any debug or test code that sets player.money = -1 directly (as used in Lesson 7 testing) would break — a setter method or a test-only method would be needed. Systems thinking is essential here because a single field change produces compile errors in other classes — the developer needs to map all consumers of money before making the change, not just the class that declares it. Without systems thinking, the change appears simple but breaks silently in unexpected places.
A-range identifies a specific second consumer (test/debug code), proposes two solution options (getter method vs C# property), and explains why systems thinking is essential in terms of compile errors and silent failures. B-range correctly identifies GameManager and UI but doesn't explore the testing implication or explain the compile-time vs runtime failure modes.
