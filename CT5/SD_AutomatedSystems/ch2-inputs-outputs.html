Inputs, processes & outputs

Every automated system follows the same fundamental pattern — sense something, decide what to do, act on that decision. This chapter maps that pattern to Unity so you can build real systems in code.

2.1 — The IPO model

Sense — decide — act

Whether you're looking at a dishwasher or a self-driving car, every automated system follows the same three-stage pattern. Engineers call it the IPO model: Input → Process → Output. Understanding this model lets you analyse any system and, more importantly, design your own.

I Input Data collected from the environment via sensors — temperature, distance, motion, time
→
P Process The program logic that reads inputs, compares them to rules, and decides what action to take
→
O Output Physical or digital responses — motors moving, lights switching, displays updating, sounds playing

In Unity, this same model maps directly onto the game loop. The engine calls Update() every frame — that single method is where all three stages happen: read inputs, run logic, drive outputs. This is why Unity is such a powerful tool for simulating automated systems: the architecture already matches how real systems work.

UNITY NOTE

Unity has two main update loops. Update() runs every rendered frame — use this for reading input and driving UI. FixedUpdate() runs at a fixed physics timestep (default 50 times per second) — use this for any logic that applies forces to Rigidbodies. For most automated system simulations, Update() is the right choice unless you're applying physics forces.

2.2 — Inputs (sensors → Unity)

How sensors become code

Real mechatronic systems use physical sensors to detect the world. In Unity we simulate those sensors using the physics engine and scripting API. The table below maps the most common real-world sensors to their Unity equivalent — use this as a reference when designing your own system.

Real-world sensor Unity equivalent How it works in Unity
PIR / proximity sensor — detects presence within a zone OnTriggerEnter(Collider c) Attach a Trigger Collider to the detection zone. When a tagged object enters, the method fires automatically.
Distance / IR sensor — measures exact distance to an object Physics.Raycast() Fire a ray from a point. If it hits something, hit.distance gives the exact distance in world units.
Camera / vision — detects objects within a field of view Raycast + angle check Calculate the angle between the sensor's forward vector and the direction to the target using Vector3.Angle().
Collision / pressure sensor — detects physical contact OnCollisionEnter(Collision c) Fires when a Rigidbody collides with another object. collision.impulse.magnitude gives the force of impact.
Temperature / level / moisture sensor — reads a continuous value float variable (drifting over time) Use a float that changes each frame via Time.deltaTime to simulate a value rising or falling naturally.
Timer / clock — tracks elapsed time or schedule Time.time / Time.deltaTime Time.time is seconds since play began. Time.deltaTime is time since the last frame — essential for frame-rate independent logic.
Velocity / accelerometer — measures speed and direction Rigidbody.velocity A Vector3 giving speed in all three axes. rb.velocity.magnitude gives total speed regardless of direction.
Button / switch — human-triggered input Input.GetKeyDown(KeyCode.E) Returns true on the single frame the key is pressed. Use for player-initiated events like pedestrian crossing buttons.
DESIGN TIP

When choosing between OnTriggerEnter and Physics.Raycast, ask: does my real sensor define a zone (trigger collider) or a beam (raycast)? A PIR motion light has a cone-shaped zone — trigger collider. A distance sensor on a robot arm points in one direction — raycast. Matching the right Unity tool to the real sensor makes your simulation more accurate and your code more readable.

2.3 — Data types for automation

Choosing the right type for sensor data

Every sensor reading has to be stored as a variable. The data type you choose affects what operations you can do with the value, how much memory it uses, and how clearly your code communicates its intent. For automated systems, there are five data types you'll use constantly.

float A decimal number. Use for any continuous measurement — distance, speed, temperature, time elapsed, fill level. float distance = hit.distance;
float temperature = 21.4f;
float tankLevel = 0.75f;
bool True or false only. Use for on/off states — is the sensor triggered? Is the door open? Is the alarm active? bool isTriggered = false;
bool doorOpen = true;
bool alarmActive;
int A whole number. Use for counts — items on a conveyor, cycles completed, number of zones triggered. int itemCount = 0;
int cyclesRun = 3;
int zonesActive = 2;
Vector3 Three floats (x, y, z). Use for position, velocity direction, or the direction a sensor is pointing. Vector3 velocity =
rb.velocity;
Vector3 sensorDir =
transform.forward;
enum A named set of states. Use to track which phase your system is in — far cleaner than using magic numbers. enum DoorState {
Closed, Opening,
Open, Closing
}
DoorState state;
string Text. Rarely used in automation logic itself — mainly for UI labels, log messages, and tag comparisons. string tag =
other.gameObject.tag;
Debug.Log("Triggered!");
STAGE 6 FLAG

In Stage 6 Software Engineering you'll encounter structs — a way to group related sensor readings into a single named type (e.g. a SensorReading struct with a float value, float timestamp, and string sensorId). For Stage 5 this isn't required, but it's worth knowing the concept exists for when you reach it.

2.4 — Outputs (actuators → Unity)

How actuators become code

Once the process logic has made a decision, it needs to act on it. In the real world that means motors, lights, valves, and speakers. In Unity we drive those same outputs using transforms, physics, the animation system, and the UI toolkit.

Motor / linear actuator — pushes or pulls
↳
Transform.Lerp() / MoveTowards()
Smoothly moves an object from one position to another over time. Control speed with the t parameter.
Servo motor — rotates to an angle
↳
Transform.Rotate() / Quaternion.Lerp()
Rotates an object around an axis. Quaternion.Lerp gives smooth rotation to a target angle.
DC motor — applies continuous force
↳
Rigidbody.AddForce()
Applies a physics force in a direction. Use ForceMode.Acceleration for consistent speed regardless of mass.
Animated mechanism (arm, door, gate)
↳
Animator.SetBool() / SetTrigger()
Drive a pre-built animation from code. SetBool("isOpen", true) transitions the Animator state machine.
Light — switches on/off or changes intensity
↳
Light.enabled / Light.intensity
Toggle a Light component or ramp its intensity value to simulate dimming or strobe effects.
Display panel / indicator
↳
TextMeshPro.text / Image.color
Update a world-space TMP text component or change a UI Image colour to red/green to show system state.
Buzzer / speaker / alarm
↳
AudioSource.Play() / PlayOneShot()
Play() loops a sound continuously. PlayOneShot(clip) plays once without interrupting other audio.
Wheeled / mobile robot
↳
NavMeshAgent.SetDestination()
Pass a world-space Vector3 destination. The agent pathfinds automatically, avoiding obstacles on the NavMesh.
