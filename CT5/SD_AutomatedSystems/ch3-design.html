3.2 — Functional & non-functional requirements

Specifying what your system must do

Before writing code, engineers write down exactly what a system is supposed to do. These are called requirements, and they come in two types. Getting requirements right is what separates a system that works from one that merely runs.

Functional requirements

These describe what the system does — specific behaviours that can be verified with a test case. Each one should be precise enough that you could write a test to check it.

  • The door must open within 0.5 seconds of the player entering the trigger zone
  • The door must close automatically 2 seconds after the player leaves
  • The light must activate when any tagged "Player" object enters the zone
  • The pump must deactivate when the tank level exceeds 90%

Non-functional requirements

These describe how well the system performs — constraints on quality, safety, performance, and reliability that apply across the whole system.

  • The simulation must run at a minimum of 30 frames per second
  • All sensor logic must be frame-rate independent (use Time.deltaTime)
  • Any system failure state must display a visible error message to the user
  • The system must handle edge cases without crashing (e.g. two objects in zone at once)

Worked example — automatic sliding door

R01 Door opens when a Player object enters the trigger zone Functional Walk character into zone — door animation plays within 0.5s

3.3 — Writing use cases

Describing how your system gets used

A use case describes one specific scenario in which your system is triggered and used. It captures the actor (who or what starts it), the steps that follow, and the state the system is in when it's done. Use cases are written from the outside — what does an observer see happening? — not from the code's point of view.

Select a system below to see a fully worked use case you can use as a model for your own.

Use case
UC-01: Player passes through automatic doors
Actor
Player character (human-controlled)
Precondition
Door is in closed state. Player is outside the trigger zone.
Trigger
Player character enters the trigger collider zone in front of the door.
Main flow
  1. System detects Player tag in trigger zone via OnTriggerEnter
  2. System sets isOpen = true and resets the close timer
  3. Animator plays the door-open animation
  4. Player moves through the doorway
  5. Player exits the trigger zone — OnTriggerExit fires
  6. Close timer begins counting down (2 seconds)
  7. Timer reaches zero — Animator plays door-close animation
  8. System returns to closed state
Alternate flow
If the Player re-enters the zone while the door is closing, the timer resets and the door reopens (R03).
Postcondition
Door is closed. Player is on the opposite side of the doorway.
Use case
UC-01: Security floodlight activates on motion detection
Actor
Player character (intruder scenario)
Precondition
Floodlight is off. Player is outside the detection zone. Hold timer is at zero.
Trigger
Player enters the PIR detection trigger zone.
Main flow
  1. System detects Player in zone via OnTriggerEnter
  2. Hold timer is set to 5 seconds
  3. Light component is enabled — floodlight turns on
  4. Each frame: timer decrements by Time.deltaTime
  5. Player leaves the detection zone
  6. Timer continues counting down (no reset on exit)
  7. Timer reaches zero — light component is disabled
Alternate flow
If Player re-enters while light is on, timer resets to 5 seconds — light stays on for the full duration after each detection.
Postcondition
Floodlight is off. Hold timer is zero. System is ready to detect again.
Use case
UC-01: Defective item is detected and diverted to reject lane
Actor
Conveyor belt system (fully autonomous)
Precondition
Belt is running. Sensor gate is in neutral position. Item is on the belt.
Trigger
An item with tag "Defective" enters the sensor trigger zone.
Main flow
  1. Sensor fires OnTriggerEnter as item arrives
  2. System reads the item's ItemData.itemType component
  3. Type is "Defective" — reject gate Animator plays "Open" trigger
  4. Item travels along belt into reject lane
  5. Reject counter increments; UI label updates
  6. After 1 second, reject gate Animator plays "Close" trigger
Alternate flow
If item type is "Accept", the accept gate opens instead and the accept counter increments.
Postcondition
Item is in the reject bin. Gate is closed. System is ready for the next item.
