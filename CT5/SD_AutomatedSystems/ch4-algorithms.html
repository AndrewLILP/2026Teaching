4.1 — The three control patterns

Most automated systems use one of three patterns

Once you can recognise which pattern a system uses, you know exactly how to design its algorithm and which C# structures to reach for. More complex systems — like the Grade A patrol drone — combine all three.

Pattern 1 Threshold / on-off A measured value is compared to one or more thresholds. When the value crosses a threshold, an output switches on or off. The logic runs every frame. if / else · float comparison
Water tank, floodlight, greenhouse
Pattern 2 Sequence A fixed series of steps executes in order, each waiting a set duration before the next begins. The sequence runs once per trigger event, not every frame. IEnumerator · WaitForSeconds
Car wash, airlock, morning routine
Pattern 3 Finite state machine The system exists in exactly one named state at a time. Events or conditions trigger transitions between states. Each state has its own behaviour. enum + switch statement
Traffic lights, patrol drone, robot arm
KEY IDEA

Grade E–D projects usually use Pattern 1 only. Grade C adds a second threshold or combines Pattern 1 with a timer. Grade B introduces Pattern 3. Grade A systems often use all three together — an FSM whose states each contain threshold logic, with coroutines handling timed transitions.

4.2 — Flowcharts

Drawing the algorithm before writing code

A flowchart lets you trace the logic of your algorithm visually before committing to code. For automated systems, the most important part is the decision diamond — it's where your system "thinks." Every diamond must have exactly two exits: yes and no.

UNITY NOTE

The "Wait for next frame ↺" terminal represents Unity calling Update() again automatically. You never write the loop itself — Unity handles it. Your job is to write only what happens inside one frame, and Unity takes care of repeating it 30–120 times per second.

Drawing rules for flowcharts

Every flowchart you submit must follow these conventions. Your teacher will check for them when assessing your project notebook.

Terminals are oval/pill shaped (rounded rectangle) — start and end only
Process boxes are plain rectangles — one action per box
Decision diamonds have exactly two exits — label each YES and NO
Arrows show direction of flow — never leave an arrow unlabelled at a branch
Merge points (where paths rejoin) are shown with a dot or convergence arrow
Loop-back arrows are drawn on the left side of the flowchart, going upward
4.3 — NSW pseudocode conventions

Writing algorithms in structured English

NSW Computing Technology uses a specific pseudocode style. It sits between plain English and real code — precise enough to be unambiguous, but readable without knowing C#. All keywords are uppercase. Use this style in your project notebook and in assessment tasks.

Keyword reference

BEGIN / END — program boundary
IF condition THEN … ELSE … END IF
WHILE condition DO … END WHILE
REPEAT … UNTIL condition
FOR var ← start TO end DO … END FOR
PROCEDURE name() … END PROCEDURE
Assignment uses ← (not = or :=)
Comparison uses = (not == like in C#)
Logic: AND OR NOT (not && || !)
Comments start with //
DISPLAY text — output to screen
WAIT n seconds — pause execution
4.4 — Pattern explorer

Pseudocode to C# — side by side

Select a pattern to see the pseudocode and C# implementation side by side. Study how each pseudocode keyword maps to a specific C# structure — this is the translation skill your algorithm design tasks are assessing.

Security floodlight · Grade D–C · A single measured value (timer) compared to a threshold each frame. Simple, predictable, and the foundation of almost all automation.

NSW Pseudocode
BEGIN SecurityFloodlight
  timer    ← 0
  holdTime ← 5      // seconds light stays on

  PROCEDURE Update()
    // INPUT — read sensor state
    IF playerInZone = TRUE THEN
      timer ← holdTime   // reset countdown
    ELSE IF timer > 0 THEN
      timer ← timer - deltaTime
    END IF

    // OUTPUT — drive the light
    IF timer > 0 THEN
      light.enabled ← TRUE
    ELSE
      light.enabled ← FALSE
    END IF
  END PROCEDURE
END SecurityFloodlight
C# (Unity)
public float holdTime = 5f;
public Light  floodlight;
private float timer = 0f;
private bool  playerInZone;

void Update() {
    // INPUT — trigger zone sets playerInZone
    if (playerInZone) {
        timer = holdTime;
    } else if (timer > 0f) {
        timer -= Time.deltaTime;
    }

    // OUTPUT — drive the light
    floodlight.enabled = timer > 0f;
}

void OnTriggerEnter(Collider c) {
    if (c.CompareTag("Player")) playerInZone = true;
}
void OnTriggerExit(Collider c) {
    if (c.CompareTag("Player")) playerInZone = false;
}

IF … THEN … ELSE … END IF → C# if () { } else { } · timer ← holdTime → assignment timer = holdTime; · timer > 0 → C# uses the same operator. The entire pseudocode IF timer > 0 THEN light ← TRUE ELSE light ← FALSE END IF collapses to a single C# line: floodlight.enabled = timer > 0f; — C# evaluates the comparison as a bool directly.

Car wash sequence · Grade B · A fixed series of steps runs in order, each waiting a set duration. The key insight: this does not use Update(). Instead, a Coroutine pauses execution mid-method using yield return.

NSW Pseudocode
BEGIN CarWashSystem

  PROCEDURE StartWash()
    boomGate.Close()
    DISPLAY "Soap phase"
    WAIT 3 seconds
    DISPLAY "Rinse phase"
    WAIT 3 seconds
    DISPLAY "Dry phase"
    WAIT 2 seconds
    boomGate.Open()
    DISPLAY "Wash complete"
  END PROCEDURE

  PROCEDURE OnVehicleDetected()
    CALL StartWash()
  END PROCEDURE

END CarWashSystem
C# (Unity)
public Animator boomGate;
public TextMeshPro statusText;

// CALL StartWash() → StartCoroutine()
void OnTriggerEnter(Collider c) {
    if (c.CompareTag("Vehicle"))
        StartCoroutine(StartWash());
}

// PROCEDURE StartWash → IEnumerator
IEnumerator StartWash() {
    boomGate.SetTrigger("Close");

    statusText.text = "Soap phase";
    yield return new WaitForSeconds(3f);

    statusText.text = "Rinse phase";
    yield return new WaitForSeconds(3f);

    statusText.text = "Dry phase";
    yield return new WaitForSeconds(2f);

    boomGate.SetTrigger("Open");
    statusText.text = "Wash complete";
}

WAIT n seconds → yield return new WaitForSeconds(n); — this pauses the Coroutine without blocking the rest of the game. PROCEDURE → IEnumerator method type (required for any method that uses yield return). CALL StartWash() → StartCoroutine(StartWash()) — regular method calls don't run Coroutines; you must use StartCoroutine(). Everything between two yield return lines runs in a single frame.

Traffic light intersection · Grade B · The system is always in exactly one named state. A timer triggers transitions. The enum gives each state a readable name; the switch runs the right behaviour for the current state.

NSW Pseudocode
BEGIN TrafficLightFSM
  state ← RED
  timer ← 0

  PROCEDURE Update()
    timer ← timer + deltaTime

    IF state = RED AND timer >= 30 THEN
      state ← GREEN
      timer ← 0
    ELSE IF state = GREEN AND timer >= 25 THEN
      state ← AMBER
      timer ← 0
    ELSE IF state = AMBER AND timer >= 5 THEN
      state ← RED
      timer ← 0
    END IF

    UPDATE light colours // based on state
  END PROCEDURE

END TrafficLightFSM
C# (Unity)
enum LightState { Red, Green, Amber }

private LightState state = LightState.Red;
private float timer = 0f;
public  Light redLight, greenLight, amberLight;

void Update() {
    timer += Time.deltaTime;

    switch (state) {
        case LightState.Red:
            if (timer >= 30f) Transition(LightState.Green);
            break;
        case LightState.Green:
            if (timer >= 25f) Transition(LightState.Amber);
            break;
        case LightState.Amber:
            if (timer >= 5f)  Transition(LightState.Red);
            break;
    }
    UpdateLights();
}

void Transition(LightState next) {
    state = next;
    timer = 0f;
}

void UpdateLights() {
    redLight.enabled   = state == LightState.Red;
    greenLight.enabled = state == LightState.Green;
    amberLight.enabled = state == LightState.Amber;
}

state ← RED → enum LightState { Red, Green, Amber } then LightState state = LightState.Red;. The enum is the direct C# equivalent of naming states in pseudocode. IF state = RED AND timer >= 30 → the switch statement handles the state check; the if (timer >= 30f) handles the condition inside it. Extracting Transition() and UpdateLights() as separate methods keeps each method focused on one responsibility — this is what makes it a Grade A implementation rather than Grade B.

4.5 — Desk checking

Tracing an algorithm by hand

Desk checking means manually tracing through your algorithm step by step, recording every variable's value at each stage, before running it in Unity. It proves your logic is correct and is one of the fastest ways to find bugs before they become Unity errors.

The tool you use is a trace table — one column per variable, one row per step. Work through the algorithm with a specific input scenario and fill in every cell.

Scenario: Security floodlight. The player enters the zone on frame 1, then leaves. Trace the next 5 frames. Assume deltaTime = 1 second (simplified for tracing — real deltaTime is ~0.016s).

WHY BOTHER?

Tracing a 5-frame scenario on paper takes 2 minutes. Debugging the same logic error in Unity can take 20. The syllabus requires you to "validate algorithms with desk checking" — include a completed trace table in your project notebook for your final algorithm before you build it.
