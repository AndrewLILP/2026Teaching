SINGLE RESPONSIBILITY

The single most important rule for keeping your Unity project manageable: one script, one job. A script that detects the player should not also be moving a door and updating a UI label. When each script has one clear responsibility, bugs are easier to find and fixes don't break other things. The four-script pattern below shows how to split the work.

5.2 — Modular code structure

Four scripts, one system

Grade B and above projects should use this pattern. Each script has exactly one responsibility and communicates with the others through public method calls or UnityEvents. Even for simpler systems, naming your script clearly (e.g. DoorSensor.cs not Door.cs) signals that you understand the principle.

Detect SensorReader.cs Reads inputs from trigger colliders, raycasts, float variables, or Input. Notifies the controller when a condition changes. Knows nothing about what to do with the information.
Decide SystemController.cs Holds the FSM or threshold logic. Receives data from sensors, makes the decision, then calls methods on the actuator. Never directly touches Unity components like Animator or Light.
Act ActuatorDriver.cs Receives commands from the controller and drives Unity components — Animator, Light, NavMeshAgent, AudioSource. Knows nothing about why it was told to act.
Display UIManager.cs Updates TextMeshPro labels, sliders, and image colours to reflect system state. Only reads data — never changes the state itself. Called by the controller each frame or on state change.
GRADE A SIGNAL

Assessors look for evidence of design thinking — not just code that works. A Grade A project notebook explains why you split the responsibilities the way you did, what you would change if requirements changed, and how the architecture made debugging easier. The code is the evidence; the notebook is the argument.
