Sunday Drive: Outback Explorer

A depth study mapping my Unity driving game to the NSW Stage 5 Computing Technology syllabus.

Focus area: Software development — creating games and simulations · Engine: Unity (C#) · ~30 custom scripts + integrated third-party packages

1. Project overview

Sunday Drive: Outback Explorer is a relaxing exploration driving game I am building in Unity. The player begins in the rural Australian countryside in a small starter car (a PolyStang), drives to a city where they can buy a different vehicle through a customisation system (RealisticCarControllerV4, abbreviated RCC), and then returns to the countryside in their chosen car. The world develops as they drive — buildings appear, animals can be discovered, and conversations can be had with characters along the way.

The purpose of the system is calm, low-pressure entertainment. There is no time limit, no enemies and no fail state. Instead, the design encourages the player to slow down, look at the landscape, and notice the way rural and remote Australia is represented — including Country, weather, fauna and the architecture of small Australian towns. This purpose drives almost every design decision in the rest of this depth study.

2. Identifying and defining

Functional requirements (what the system must do)

  • The player can drive a starter car using keyboard or controller input.
  • The player can exit the car and walk around as a character on foot.
  • The player can re-enter their car when they are within 3 metres of it.
  • The world tracks how many times the player has visited each named zone.
  • Buildings activate (fade or scale into view) once a zone has been visited a set number of times.
  • The player can interact with NPCs and signs to view dialogue.
  • The player can discover Australian animals, which are recorded in a discovery list.
  • The game saves automatically every 2 minutes and on quit, and reloads on launch.

Non-functional requirements (qualities the system must have)

  • Performance — must hold 60 fps on a mid-range laptop; large terrain must use Level-of-Detail (LOD).
  • Reliability — saves must not corrupt if the player force-quits.
  • Usability — controls must be discoverable in under 30 seconds without reading a manual.
  • Tone — never use stress mechanics: no timers, no chases, no health bar.
  • Privacy — all save data stays on the local computer; nothing is transmitted online.

Use case — "Discover an animal"

The player drives slowly past a kangaroo. AnimalController checks the distance to the player every 0.5 seconds. When the player comes within range, AnimalDiscoveryManager.DiscoverAnimal() is called. The HUD pulses the new animal icon and shows an "Animal Discovered!" panel. The discovery is added to the save file.

3. Researching and planning

Before writing code, I looked at how other games solve the same problems my game has. Forza Horizon influenced the open-world driving feel; Euro Truck Simulator 2 influenced the calm pacing; Untitled Goose Game influenced the absence of fail states. From Microsoft Flight Simulator I learned how AI and predefined algorithms can control huge numbers of background objects (other aircraft, weather) without overloading the CPU. These products also showed me how simulations are used to solve real-world problems — flight simulators train pilots, driving simulators are used in driving instruction, and city-building games like Cities: Skylines are used by urban planners to model traffic.

Generating alternative designs

I considered three alternatives before settling on the current design:

  • Option A — Single open world with no city. Simpler, but the player would never feel the contrast between country and city, and there would be nowhere to "spend" progress.
  • Option B — Mission-based driving with delivery quests. More structured, but contradicts the "relaxing, no pressure" purpose.
  • Option C (chosen) — Country → city → country with a vehicle swap. Best matches the purpose: it gives the player a reason to drive, but the journey itself is the reward.

Data privacy & cybersecurity (planning)

I decided up-front that no personal data leaves the player's computer. Save files are stored as plain JSON in Application.persistentDataPath. The player owns the file. The trade-off is that a player could open and edit it, but I judged that risk acceptable because there is no competitive multiplayer and no leaderboard.

4. Object-oriented design — the State Pattern

The most important class structure in my project is the player state machine. The player can be in one of two states: Driving or OnFoot. Each state behaves completely differently — different physics, different camera, different controls — but I wanted to switch between them without one giant if-statement that handled every case. The classic object-oriented solution to this is the State Pattern: each state is its own class, and they all inherit from a common abstract base class.

The base class declares the contract every state must obey:

Assets/Scripts/Player/PlayerState.cs
public abstract class PlayerState
{
    protected PlayerStateManager stateManager;

    public PlayerState(PlayerStateManager manager)
    {
        stateManager = manager;
    }

    // Called once when entering this state.
    public abstract void Enter();
    // Called every frame while in this state.
    public abstract void Update();
    // Called once when exiting this state.
    public abstract void Exit();

    public virtual void FixedUpdate() { }
}

The PlayerStateManager holds a reference to the current state and simply forwards each frame to it. Switching state is a single, clean function call:

Assets/Scripts/Player/PlayerStateManager.cs (excerpt)
public void ChangeState(PlayerState newState)
{
    currentState?.Exit();           // 1. tell old state we're leaving
    currentState = newState;        // 2. swap reference
    currentState?.Enter();          // 3. let new state set up
}

private void Update()  { currentState?.Update(); }
private void FixedUpdate() { currentState?.FixedUpdate(); }
Why this matters for the syllabus: this is a textbook example of the features of an object-oriented programming language — a base class, abstract methods, two concrete subclasses that override those methods, and polymorphism (the manager calls Update() without caring which state it actually is).

5. Logic — branching, iteration, operators

The clearest place to see branching, iteration and relational operators in my game is the HandleMovement function in OnFootState. It runs every frame and converts WASD input into smooth, camera-relative movement.

Assets/Scripts/Player/OnFootState.cs (excerpt)
private void HandleMovement()
{
    float horizontal = Input.GetAxis("Horizontal");
    float vertical   = Input.GetAxis("Vertical");

    Vector3 cameraForward = mainCamera.transform.forward;
    Vector3 cameraRight   = mainCamera.transform.right;
    cameraForward.y = 0f;  cameraRight.y = 0f;
    cameraForward.Normalize();  cameraRight.Normalize();

    Vector3 moveDirection = (cameraForward * vertical
                          + cameraRight   * horizontal).normalized;

    if (moveDirection.magnitude >= 0.1f) {                   // branching + relational
        Quaternion target = Quaternion.LookRotation(moveDirection);
        stateManager.PlayerCharacter.transform.rotation =
            Quaternion.Slerp(stateManager.PlayerCharacter.transform.rotation,
                             target, turnSpeed * Time.deltaTime);
        characterController.Move(moveDirection * moveSpeed * Time.deltaTime);
    }

    velocity.y += gravity * Time.deltaTime;                       // iteration: gravity
    characterController.Move(velocity * Time.deltaTime);          // accumulates each frame

    if (characterController.isGrounded && velocity.y < 0)        // logical AND
        velocity.y = -2f;
}

This single function demonstrates several syllabus dot points at once:

  • Sequence — the lines run top-to-bottom, in the same order, every frame.
  • Branching — if (moveDirection.magnitude >= 0.1f) only rotates and moves the player when the joystick is actually pushed; the small dead-zone prevents jitter.
  • Iteration — Unity calls Update() roughly 60 times per second, which makes the gravity line a loop in disguise: velocity.y is increased every frame.
  • Relational operators — >=, <.
  • Logical operators — && in the grounded check.

6. Event-driven programming

Unity already runs an event loop for me — the engine calls Awake, Start, OnEnable, Update, FixedUpdate, OnTriggerEnter, OnDestroy at appropriate moments. On top of that I built a smaller event system of my own using C# event and Action delegates. The DialogueManager is a clean example: it knows nothing about UI; it just announces what is happening, and any UI can subscribe.

Assets/Scripts/Core/DialogueManager.cs (excerpt)
// Events for UI to observe (Observer Pattern)
public event Action<DialogueData> OnDialogueStarted;
public event Action<string, string> OnDialogueLineChanged; // (speaker, text)
public event Action OnDialogueEnded;

public void StartDialogue(DialogueData data)
{
    if (data == null || data.LineCount == 0) return;
    currentDialogue = data;
    currentLineIndex = 0;
    isDialogueActive = true;
    OnDialogueStarted?.Invoke(data);     // fire event — UI listens
    DisplayCurrentLine();
}

The UI subscribes in OnEnable and unsubscribes in OnDisable — this is critical so that destroyed UI objects do not get called by stale events (a common run-time bug):

Assets/Scripts/UI/DialogueUIController.cs (pattern)
private void OnEnable() {
    DialogueManager.Instance.OnDialogueLineChanged += HandleLine;
}
private void OnDisable() {
    DialogueManager.Instance.OnDialogueLineChanged -= HandleLine;
}
Why this matters: the syllabus asks me to "implement an event loop in a game using event-driven programming". Unity's lifecycle is the event loop, and the += / -= publisher–subscriber pattern is the way idiomatic C# game code communicates between systems. The dialogue, UI, save system and animal HUD all communicate this way, with no system holding a hard reference to another system's UI.

7. Data — types, structures, storage and transmission

The save system is where I worked hardest to think about data. Three managers (visits, animals, buildings) each hold runtime state. GameSaveData is a serialisable container that flattens that state into JSON-friendly types.

Assets/Scripts/Core/GameSaveData.cs (excerpt)
[System.Serializable]
public class GameSaveData
{
    // JsonUtility cannot serialise Dictionary, so we use parallel lists.
    public List<string> zoneIDs        = new List<string>();
    public List<int>    visitCounts    = new List<int>();

    public List<string> discoveredAnimals    = new List<string>();
    public List<string> activatedBuildingIDs = new List<string>();

    public string saveTimestamp;
    public int    saveVersion = 1;
}

I use a different data structure in each manager based on what it needs to do most often:

Dictionary<string,int>

VisitManager — fast lookup of "how many times have I visited this zone?"

HashSet<string>

AnimalDiscoveryManager — O(1) duplicate-prevention so the same animal isn't recorded twice.

List<string>

GameSaveData — JSON-friendly. Order-preserving and Unity can serialise it.

Parallel lists

zoneIDs & visitCounts — same index = same zone. A workaround because JsonUtility won't serialise Dictionaries.

The save flow below shows how data moves from in-memory managers, through serialisation, to a JSON file on disk:

Assets/Scripts/Core/GameSaveManager.cs (excerpt)
public void Save()
{
    if (VisitManager.Instance == null) return;
    if (AnimalDiscoveryManager.Instance == null) return;

    try {
        GameSaveData data = new GameSaveData();
        CollectSaveData(data);                          // gather
        string json = JsonUtility.ToJson(data, true); // transform
        File.WriteAllText(saveFilePath, json);            // transmit to disk
    } catch (System.Exception e) {
        Debug.LogError($"Save failed: {e.Message}");
    }
}
Cybersecurity reflection: the JSON file is plain text, which means a player could open it and edit their visit count. I judged that acceptable because the game has no online ranking and no real-money items. If I added a leaderboard later I would need to either move the save online with authentication, or sign the file with a hash. Either change would change the data-flow diagram above significantly.

8. Interfaces & modular design

To keep my code modular and easy to extend, I defined two interfaces. Any class can implement them, and the rest of the game can talk to them through the interface without knowing what concrete class is hiding behind it.

Assets/Scripts/World/IInteractable.cs
public interface IInteractable
{
    void   Interact();
    string GetInteractionPrompt();
    bool   CanInteract();
}

This means I can add a new interactive object — a sign, a fuel pump, an NPC — by writing one new class that implements the three methods. PlayerInteractionDetector doesn't need to change at all. This is the practical pay-off of the syllabus dot point "design and implement modular programs (code) with functions for games".

9. Coroutines & animation — the world develops as you drive

The "developing world" effect — buildings appearing in the landscape as the player explores — is implemented in ActivatableObject using Unity coroutines. A coroutine is a method that can pause itself and resume on the next frame, which is perfect for animating something over time without freezing the game.

Assets/Scripts/World/ActivatableObject.cs (excerpt)
private IEnumerator FadeInCoroutine()
{
    isActivating = true;
    SetObjectVisibility(true);
    SetObjectAlpha(0f);

    float elapsed = 0f;
    while (elapsed < activationDuration)            // iteration
    {
        elapsed += Time.deltaTime;
        float alpha = Mathf.Clamp01(elapsed / activationDuration);
        SetObjectAlpha(alpha);
        yield return null;                         // pause until next frame
    }
    SetObjectAlpha(1f);
    isActive = true;
    isActivating = false;
}

The activation type is selected by an enum, and a switch statement chooses which coroutine to run — branching at its clearest:

switch (activationType) {
    case ActivationType.Instant:      ActivateInstant();                         break;
    case ActivationType.FadeIn:       StartCoroutine(FadeInCoroutine());       break;
    case ActivationType.ScaleUp:      StartCoroutine(ScaleUpCoroutine());      break;
    case ActivationType.FadeAndScale: StartCoroutine(FadeAndScaleCoroutine()); break;
}
Connection to the syllabus: this maps to "simulate a 2D or 3D physical world using fixed time increments". The fixed increment is Time.deltaTime; the iteration is the while loop; and the smooth-step easing in ScaleUpCoroutine (t = t*t*(3-2t)) is a small piece of mathematics that makes the appearance feel natural rather than mechanical.

10. UI / UX — the design of feeling calm

The user interface uses Unity's UI Toolkit (UXML + USS, similar to HTML and CSS). The HUD shows speed and animal-discovery progress; the dialogue UI shows speaker name and text; the pause menu suspends time. Every UI piece is event-driven — it reacts to events, never polls every frame for changes.

Key UX choices that follow from the "relaxing" purpose:

  • No HUD spam. The HUD shows only speed and discovery progress. There is no minimap and no quest tracker.
  • Single key prompt. Every interaction is "Press E". The InteractionPromptUI only ever shows one prompt at a time.
  • ESC pauses softly. PauseMenuController uses Time.timeScale = 0 so the simulation freezes but the UI stays responsive.
  • Animal-discovery feedback is gentle. A pulsing icon and a one-line panel — not a full-screen pop-up.

Where the design could be more inclusive: I am planning controller remapping (for players with disability), high-contrast text mode, and subtitles with adjustable text size. The syllabus explicitly asks me to consider players "with disability" and "of different ages", so this is a planned improvement, not a finished feature.

11. Simulating a physical world

I did not write a car physics engine from scratch. Instead, I am interpreting and extending two existing third-party packages — PolyStang for the country starter car, and RealisticCarControllerV4 (RCC) for the city vehicle. The syllabus explicitly rewards "interpret and modify existing programs (code) for games". My contribution is the integration layer:

  • Garage swap — when the player buys a vehicle in the city, that vehicle becomes the active car when they return to the country scene. The garage-swap logic is mine; the physics is RCC's.
  • State-machine handover — my PlayerStateManager freezes the car's Rigidbody when on foot (isKinematic = true) and unfreezes it when re-entering. That keeps RCC and my OnFootState from fighting each other.
  • Save persistence — the chosen car survives a save / quit / reload because the car ID is part of GameSaveData (a planned extension; current build saves visits, animals and buildings).

The on-foot side of the world is simulated by my code. The gravity loop in OnFootState (Section 5) is a textbook example of "simulate a 3D physical world using fixed time increments": gravity accelerates the player downward each frame, and a ground-check stops the velocity from growing forever.

  • Run-time — null-reference on quit when VisitManager was destroyed before GameSaveManager. Fixed with an isQuitting static flag and null guards.

13. Social, ethical & cybersecurity considerations

Diverse perspectives

The game is set on Aboriginal and Torres Strait Islander Country. I am committed to not placing sacred imagery, ceremony or cultural designs as decorations without proper consultation. Where the game references Country, I plan to credit the relevant Traditional Owners and to consult before adding any cultural representations rather than relying on internet image searches. Roads, fauna and town buildings are general Australian rural elements rather than culturally specific markers.

Other diverse-perspective considerations:

  • Older players — large, readable HUD text; option to slow the camera response.
  • Players with disability — full controller remapping, subtitles, no flashing strobe effects, no time-pressure tasks.
  • EAL/D players — minimal text in the world; icons used instead of words for prompts where possible.

Ethical & legal

I credit the asset packs I use (PolyStang, RCC, Pup Up Productions Rural Australia Pack, UBS) in the project's documentation. Their licences allow the use I am making. I do not include any copyrighted music or real brands.

Cybersecurity

Three principles drive my approach:

  • What data do I collect? Only zone visits, animal discoveries, and activated building IDs. No personal data, no telemetry.
  • Who owns it? The player. The save file is on their machine.
  • How is it protected? File-system permissions only. There is no encryption because there is no sensitive data.

14. Where this could go next — gaps the syllabus identifies

Some syllabus dot points my project does not yet cover. I treat each one as a research opportunity and a future build.

VR, AR and MR

Driving simulators are one of the strongest VR success stories — products like Gran Turismo Sophy, iRacing on Vive, and Volvo's hazard-perception VR training rig show how immersive driving experiences both entertain and train. A VR mode of Sunday Drive would let the player look out the car window with head movement and see the kangaroo before the system tells them about it. AR could overlay an animal-discovery field-guide onto the windscreen on a real device. MR (mixed reality) could place a tabletop diorama of my country world for a non-VR friend to watch the driver.

2D cell-based simulation

Updating a cell based on its neighbour — Conway's Game of Life is the canonical example. In my world this could be used for a fire-spread minigame on agricultural land, or for grass growth/regeneration after the car drives over it. Both would teach me grid-based iteration and would map directly to the "simulate a 2D environment by updating the state of a cell based on its neighbour" dot point.

AI-driven NPCs

Predefined algorithms already steer RCC's AI cars along waypoints. A natural extension is a simple finite-state machine for animals (idle / wander / flee) and pathfinding (A*) for NPC drivers reacting to the player. Dot point: "explore how predefined algorithms and AI control computer components."

Tutorial

The current build assumes the player figures out controls. A short context-aware tutorial — "Press W to drive forward", "Press E to exit the car" — would make the game accessible to younger and older players. The tutorial would itself be event-driven, listening to the same player-state events I already publish.

Online leaderboards (and the security cost)

If I added a "longest distance driven" leaderboard, my data flow would change: saves would have to be transmitted to a server, which means authentication, HTTPS and protection against tampering. The syllabus warning about "investigate data collection... privacy and cybersecurity" becomes immediately concrete here.

16. Reflection & careers

Building Sunday Drive showed me that game development is far more than physics and graphics. Most of the interesting decisions are about how to keep code maintainable when many systems need to talk to each other. The state pattern, interfaces and event system make my game easier to extend (a new interactable, a new animal, a new save field) without rewriting existing code.

The careers I have looked at while making this project include game programmer, technical artist, simulation engineer (driving and flight schools, surgical training), tools developer (the people who build editors like the one in BuildingIDHelper.cs), UX designer, and accessibility specialist. The skills I am practising here — OOP, event-driven design, data persistence, testing, working with third-party code — are common to all of them.

Most of all, working on a calm, exploration-focused game has reminded me that software does not have to be stressful or aggressive to be interesting. A relaxing drive through rural Australia is a perfectly valid design goal, and pursuing it carefully has taught me as much as a more action-packed project would have.
