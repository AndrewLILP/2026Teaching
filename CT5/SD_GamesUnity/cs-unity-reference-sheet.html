C# Language Fundamentals

Variables · Data types · Operators · Comments
Core CT5-OPL-01 CT5-DAT-01

Every C# script in Unity is built on variables, types and operators. Choosing the right data type is the first step toward efficient code — the syllabus requires you to understand data type limitations, not just syntax.

int

int score = 0;

Whole numbers. Use for counts, scores, lives. Range: −2,147,483,648 to 2,147,483,647.

float

float speed = 5.5f;

Decimal numbers. Always write the f suffix. Use for position, speed, time.

bool

bool isAlive = true;

True or false only. Use for flags, states, conditions.

string

string name = "Player";

Text in double quotes. Use for labels, dialogue, file names.

const

const int MAX = 10;

Value set once, never changes. Prevents accidental modification.

Debug.Log

Debug.Log(score);

Prints to the Unity Console. Essential for tracing variable values during testing.

Operators quick-referenceC#
// Arithmetic
int total = score + bonus;   // +  −  *  /  %
int remain = 10 % 3;         // modulus → 1

// Relational (always return bool)
score == 100   // equal to
score != 0     // not equal
score >= 50    // greater than or equal
score <  0     // less than

// Logical
isAlive && hasAmmo  // AND — both true
isAlive || hasShield // OR  — at least one true
!isAlive            // NOT — flips the bool

// Assignment shortcuts
score += 10;  // same as score = score + 10
score++;      // increment by 1
Grade A link: Selecting the most appropriate data type (e.g., float for movement, int for lives) and understanding overflow limitations demonstrates the computational thinking the Grade A descriptor calls "technically concise algorithms."

Control Flow — Sequence, Branching, Iteration

if · switch · for · while · foreach
Core CT5-OPL-01 CT5-THI-01

The syllabus explicitly names sequences, branching and iteration with logical and relational operators. All three must appear in your flowcharts, pseudocode and implemented code.

BranchingC#
if (lives > 0)
{
    // player still alive
}
else if (lives == 0)
{
    GameOver();
}
else
{
    // shouldn't happen — defensive code
}

// Switch on enum state
switch (currentState)
{
    case GameState.Playing:
        UpdateGame(); break;
    case GameState.Paused:
        ShowMenu();  break;
    default:
        break;
}
IterationC#
// For loop — known count
for (int i = 0; i < enemies.Length; i++)
{
    enemies[i].TakeDamage(10);
}

// While — unknown count
while (coins > 0)
{
    coins--;
    score += 10;
}

// Foreach — clean list traversal
foreach (var item in inventory)
{
    item.Use();
}

// break exits loop early
// continue skips this iteration
Syllabus link: "Investigate the representation of logic when designing games and simulations, including sequences, branching and iteration, including logical and relational operators." Every control structure you write should first be designed as a flowchart or pseudocode — then coded.

Methods and Functions

Parameters · Return types · Scope · Single responsibility
Core CT5-OPL-01 CT5-DES-01

The syllabus requires you to "design and implement modular programs with functions." A well-designed method does one thing — this is the single-responsibility principle, and it makes code easier to test, debug and extend.

Method anatomy & IPO patternC#
// INPUT → PROCESS → OUTPUT

// void method — no return value
private void TakeDamage(int amount)         // INPUT: amount
{
    health -= amount;                          // PROCESS
    UpdateHealthUI();                          // OUTPUT: updates display
}

// Typed method — returns a value
private bool IsGameOver()                    // INPUT: none
{
    return health <= 0 && lives == 0;         // OUTPUT: bool result
}

// Calling methods
TakeDamage(25);
if (IsGameOver()) EndGame();

Scope

private int _hp;

Field: declared in the class, available to all methods.
Local: declared inside a method — invisible outside it.

Access modifiers

public / private

public — visible everywhere.
private — only inside this class. Prefer private by default (encapsulation).

Method overloading

Heal() / Heal(int)

Same name, different parameters. Unity uses this (e.g. Instantiate variants). Extension

Grade A link: A student at Grade A demonstrates "technically concise algorithms." Short, well-named single-purpose methods — rather than one giant Update() block — are what "technically concise" looks like in code.

Object-Oriented Programming (OOP)

Classes · Inheritance · Polymorphism · Interfaces
Core Extension: interfaces, abstract CT5-OPL-01 CT5-THI-01

The syllabus requires you to "describe the features of an object-oriented programming language" and "interpret and extend or implement an object-oriented program." The four pillars — encapsulation, inheritance, polymorphism, abstraction — must all be visible in your project.

Encapsulation & Class basicsC#
public class Enemy : MonoBehaviour
{
    // Fields — private by default preference
    [SerializeField] private int maxHealth = 100;
    private int _currentHealth;

    // Property — controlled public access
    public int Health => _currentHealth;

    // Constructor-equivalent in Unity
    private void Awake()
    {
        _currentHealth = maxHealth;
    }

    public void TakeDamage(int dmg)
    {
        _currentHealth -= dmg;
        if (_currentHealth <= 0) Die();
    }

    protected virtual void Die()
    {
        Destroy(gameObject);
    }
}
Inheritance & PolymorphismC#
// BossEnemy INHERITS from Enemy
public class BossEnemy : Enemy
{
    [SerializeField] private int phase = 1;

    // OVERRIDES the parent's Die()
    protected override void Die()
    {
        if (phase < 2)
        {
            phase++;
            // enter phase 2 instead of dying
            EnterPhaseTwo();
        }
        else
        {
            base.Die();  // call parent's Die()
        }
    }
}

// Interface — Extension / Grade A
public interface IInteractable
{
    void Interact();
    string GetPrompt();
}
Extension / Stage 6 flag: abstract classes and interface patterns appear here. At Stage 5 you need to read and modify OOP code. Creating your own interface hierarchy is Stage 6 Software Engineering depth.

Data Structures

Arrays · List · Dictionary · Enum
Core CT5-OPL-01 CT5-DAT-01

The syllabus requires selecting "appropriate data types and structures" and understanding their limitations. The key question: does your data have a fixed or variable count, and do you need to look it up by key?

Array — fixed size

int[] scores = new int[5];

Size set at creation. Fast index access. Use when count is known (e.g., 5 high scores).

⚠ Out-of-bounds = runtime error.

List<T> — dynamic

List<Enemy> enemies = new();

Grows/shrinks at runtime. Use when count changes (e.g., enemies spawned).

Dictionary<K,V>

Dictionary<string, int> inv;

Key → value lookup. Use for inventory, settings, item databases. Extension

Enum — named constants

enum State { Playing, Paused }

Converts magic numbers into readable names. Essential for game state machines.

Modifying an array — observe effects of changing variablesC#
string[] players = { "Alice", "Bob", "Cara" };

// Iterate and print — foreach is cleaner than for when index isn't needed
foreach (string p in players)
    Debug.Log(p);

// Modify element by index
players[1] = "Beatrice";   // changes "Bob" → "Beatrice"

// List — dynamic example
List<int> coins = new List<int>();
coins.Add(5);
coins.Add(10);
coins.Remove(5);
Debug.Log(coins.Count);    // → 1

Unity Fundamentals

Lifecycle · MonoBehaviour · Transform · Time
Core CT5-OPL-01 CT5-DES-01

Unity calls your script methods in a precise order — the lifecycle. Understanding this order is essential: code in the wrong method causes bugs that are hard to trace.

Unity lifecycle — execution orderC#
public class MyScript : MonoBehaviour
{
    void Awake()  { /* 1. First, always. Initialise references. */ }
    void Start()  { /* 2. Once, before first frame. Use after Awake. */ }

    void Update()
    {
        /* 3. Every frame (~60×/sec). Input, movement, logic. */
        transform.Translate(Vector3.forward * speed * Time.deltaTime);
    }

    void FixedUpdate()
    {
        /* 4. Fixed timestep. ALWAYS use for physics. */
        rb.AddForce(Vector3.up * jumpForce);
    }

    void OnDestroy(){ /* 5. Cleanup when object is destroyed. */ }
}

Time.deltaTime

speed * Time.deltaTime

Seconds since last frame. Multiply movement by this to keep speed consistent at any frame rate.

Time.timeScale

Time.timeScale = 0f;

0 = paused, 1 = normal, 0.5 = slow-motion. FixedUpdate also pauses at 0.

GetComponent

GetComponent<Rigidbody>()

Finds a component on the same GameObject. Cache in Awake — never call in Update (expensive).

Instantiate / Destroy

Instantiate(prefab);

Creates a copy of a prefab at runtime. Destroy(gameObject) removes it.

Event-Driven Programming

Input · Triggers · C# events · Observer pattern
Core Extension: delegates & C# events CT5-OPL-01 CT5-DES-01

The syllabus requires you to "implement an event loop in a game using event-driven programming." Unity's lifecycle is the event loop — the engine calls your methods at the right moments. Your job is to respond to those events cleanly.

Input & collision eventsC#
void Update()
{
    // Keyboard input
    if (Input.GetKeyDown(KeyCode.Space))
        Jump();

    // Axis input (WASD / joystick)
    float h = Input.GetAxis("Horizontal");
    float v = Input.GetAxis("Vertical");
}

// Called by the physics engine
void OnTriggerEnter(Collider other)
{
    if (other.CompareTag("Coin"))
    {
        score++;
        Destroy(other.gameObject);
    }
}

void OnCollisionEnter(Collision col)
{
    if (col.gameObject.CompareTag("Enemy"))
        TakeDamage(10);
}
Observer pattern — C# events (Extension)C#
// Publisher — announces events
public class GameManager : MonoBehaviour
{
    public static event Action<int> OnScoreChanged;

    public void AddScore(int pts)
    {
        score += pts;
        OnScoreChanged?.Invoke(score);  // fire!
    }
}

// Subscriber — UIManager listens
public class UIManager : MonoBehaviour
{
    void OnEnable()
    {
        GameManager.OnScoreChanged += UpdateUI;
    }
    void OnDisable()
    {
        GameManager.OnScoreChanged -= UpdateUI;
    }
    void UpdateUI(int newScore) { /* ... */ }
}
Syllabus link: "Implement an event loop in a game or simulation using event-driven programming." The Observer pattern above shows the data-flow chain: Coin → PlayerCoinCollector → GameManager → UIManager. Each class knows only its immediate neighbour — this is clean modular design.

Game State Management

Enum states · GameManager · Scene loading
Core CT5-OPL-01 CT5-DES-01

The syllabus requires "implementing common game features including the game state." Game state controls what the player can do at any moment — playing, paused, or game over.

GameManager with state machineC#
public enum GameState { Playing, Paused, GameOver, Win }

public class GameManager : MonoBehaviour
{
    // Singleton pattern — one GameManager accessible anywhere
    public static GameManager Instance { get; private set; }
    public GameState State { get; private set; }

    void Awake()
    {
        if (Instance != null) { Destroy(gameObject); return; }
        Instance = this;
    }

    public void ChangeState(GameState newState)
    {
        State = newState;
        switch (newState)
        {
            case GameState.Paused:
                Time.timeScale = 0f; break;
            case GameState.Playing:
                Time.timeScale = 1f; break;
            case GameState.GameOver:
                LoadScene("GameOver");  break;
        }
    }

    void LoadScene(string name) =>
        UnityEngine.SceneManagement.SceneManager.LoadScene(name);
}
Design thinking: Before coding, draw a state diagram — a circle for each state (Playing, Paused, GameOver, Win) with arrows showing what triggers each transition. This is the "designing algorithms" the syllabus requires, evidenced in your project notebook.

Physics and World Simulation

Rigidbody · FixedUpdate · Raycasting · Cell automaton
Core CT5-OPL-01 CT5-THI-01

The syllabus requires two distinct simulation approaches: "simulate a 2D or 3D physical world using fixed time increments" (Unity physics) and "simulate a 2D environment by updating the state of a cell based on its neighbour" (cellular automaton).

3D Physics — FixedUpdateC#
public class PlayerMovement : MonoBehaviour
{
    [SerializeField] float speed  = 5f;
    [SerializeField] float jump   = 7f;
    private Rigidbody _rb;
    private bool _grounded;

    void Awake() => _rb = GetComponent<Rigidbody>();

    void FixedUpdate()          // fixed timestep!
    {
        float h = Input.GetAxis("Horizontal");
        float v = Input.GetAxis("Vertical");
        Vector3 dir = new Vector3(h, 0, v);
        _rb.MovePosition(transform.position
                         + dir * speed * Time.fixedDeltaTime);
    }

    void Update()               // input check in Update
    {
        if (Input.GetButtonDown("Jump") && _grounded)
            _rb.AddForce(Vector3.up * jump, ForceMode.Impulse);
    }
}
2D Cell Automaton (Conway-style)C#
bool[,] grid    = new bool[rows, cols];
bool[,] next    = new bool[rows, cols];

void Step()
{
    for (int r = 0; r < rows; r++)
    for (int c = 0; c < cols; c++)
    {
        int n = CountNeighbours(r, c);
        if (grid[r,c])
            // alive: survives with 2 or 3 neighbours
            next[r,c] = (n == 2 || n == 3);
        else
            // dead: born with exactly 3 neighbours
            next[r,c] = (n == 3);
    }
    grid = next;  // swap buffers
}

int CountNeighbours(int r, int c)
{
    int count = 0;
    for (int dr=-1; dr<=1; dr++)
    for (int dc=-1; dc<=1; dc++)
    {
        if (dr==0 && dc==0) continue;
        count += grid[r+dr, c+dc] ? 1 : 0;
    }
    return count;
}
Key rule: Always use FixedUpdate for physics forces and movement via Rigidbody. Use Update only for detecting input. Mixing them causes jittery movement at different frame rates.

UI and UX Design

Canvas · TextMeshPro · Design principles · HUD
Core CT5-DES-01 CT5-OPL-01

The syllabus requires "developing the UI and UX of a game" and "demonstrating creativity and innovation in the design and implementation of user interfaces." Design must be intentional — every element earns its place.

Updating UI from a scriptC#
using TMPro;
using UnityEngine.UI;

public class UIManager : MonoBehaviour
{
    [SerializeField] private TextMeshProUGUI scoreText;
    [SerializeField] private TextMeshProUGUI livesText;
    [SerializeField] private Slider healthBar;
    [SerializeField] private GameObject gameOverPanel;

    public void UpdateScore(int score)
        => scoreText.text = $"Score: {score}";

    public void UpdateHealth(float current, float max)
        => healthBar.value = current / max;       // 0.0 → 1.0

    public void ShowGameOver()
        => gameOverPanel.SetActive(true);
}

UI Design Principles

  • Clarity — the player always knows their current state (health, score, lives).
  • Feedback — every action produces a visible or audible response.
  • Affordance — buttons look clickable; interactive elements signal their function.
  • Consistency — same colour always means the same thing.
  • Accessibility — sufficient contrast, readable fonts (no size below 14px in-game).
  • Minimal clutter — show only what the player needs right now (HUD vs menus).

HUD vs Menus

  • HUD (Heads-Up Display) — always visible during play: health bar, score, minimap.
  • Pause menu — activated on demand, freezes game (timeScale = 0).
  • Start / Game Over screens — full-screen panels activated via SetActive(true/false).
  • Diegetic UI — information embedded in the game world (e.g. health on a character's back). Grade A territory.
Grade A link: "Demonstrates creativity and innovation in the design and implementation of user interfaces to create engaging user experiences." Novelty alone isn't enough — engagement means the UI serves the player. Document your design decisions and iterations in your project notebook.

Coroutines and Timing

IEnumerator · yield return · WaitForSeconds
Core CT5-OPL-01

A coroutine is a method that can pause itself and resume on a later frame — without freezing the game. It maps directly to the syllabus requirement to "simulate a 2D or 3D physical world using fixed time increments" for timed effects.

Coroutine basicsC#
// Return type MUST be IEnumerator
private IEnumerator FadeOut(float duration)
{
    float elapsed = 0f;

    while (elapsed < duration)    // iteration
    {
        elapsed += Time.deltaTime;
        float alpha = 1f - (elapsed / duration);
        SetAlpha(alpha);
        yield return null;          // pause, resume next frame
    }
    SetAlpha(0f);
}

// Start and stop
StartCoroutine(FadeOut(1.5f));
StopAllCoroutines();
Common yield patternsC#
// Pause for N seconds (real time)
yield return new WaitForSeconds(2f);

// Pause for N seconds (ignores timeScale)
yield return new WaitForSecondsRealtime(2f);

// Wait until next frame
yield return null;

// Wait until a condition is true
yield return new WaitUntil(() => isLoaded);

// Spawn wave then wait
private IEnumerator SpawnWave()
{
    for (int i = 0; i < 5; i++)
    {
        Instantiate(enemyPrefab, SpawnPoint(), Quaternion.identity);
        yield return new WaitForSeconds(0.5f);
    }
}

Algorithms and Pseudocode

NSW pseudocode · Flowcharts · Desk checking · Trace tables
Core CT5-OPL-01 CT5-THI-01

Algorithms must be designed before coding — the syllabus requires you to represent, validate and desk-check them. Pseudocode is the bridge between your design thinking and your C# implementation.

Pseudocode ↔ C# Side-by-Side

NSW Pseudocode — score systemPseudocode
BEGIN ScoreSystem
  SET score ← 0
  SET lives ← 3

  WHILE lives > 0
    IF coinCollected THEN
      SET score ← score + 10
    END IF

    IF enemyHit THEN
      SET lives ← lives - 1
    END IF

    IF score >= 100 THEN
      OUTPUT "You Win!"
      SET lives ← 0   // exit loop
    END IF
  END WHILE

  OUTPUT "Final Score: ", score
END ScoreSystem
C# ImplementationC#
public class ScoreSystem : MonoBehaviour
{
    private int score = 0;
    private int lives = 3;

    public void CollectCoin()
    {
        score += 10;
        if (score >= 100)
        {
            Debug.Log("You Win!");
            GameManager.Instance.ChangeState(GameState.Win);
        }
    }

    public void HitByEnemy()
    {
        lives--;
        if (lives <= 0)
            GameManager.Instance.ChangeState(GameState.GameOver);
    }
}

Data, Storage and Ethics

PlayerPrefs · JSON · Privacy · Cybersecurity · Social impacts
Core CT5-DAT-01 CT5-DAT-02 CT5-SAF-01

The syllabus requires you to "select and apply safe, secure and ethical practices in the use of data" and "investigate data collection… adhering to privacy and cybersecurity principles." In games: what data are you collecting, who owns it, and how is it protected?

PlayerPrefs — simple local storageC#
// Save data (locally on device)
PlayerPrefs.SetInt("HighScore", score);
PlayerPrefs.SetFloat("Volume", 0.8f);
PlayerPrefs.Save();

// Load data (with default if key missing)
int hi  = PlayerPrefs.GetInt("HighScore", 0);
float vol = PlayerPrefs.GetFloat("Volume", 1f);

// JSON — structured save file
[System.Serializable]
public class SaveData { public int score; public int level; }

SaveData save = new() { score = 250, level = 3 };
string json = JsonUtility.ToJson(save);
PlayerPrefs.SetString("Save", json);

SaveData loaded = JsonUtility.FromJson<SaveData>(json);

Ethical & Legal Responsibilities

  • What data is collected? Name it explicitly — don't collect what you don't need.
  • Who owns it? Player data belongs to the player. Don't sell it.
  • How is it protected? Encrypt sensitive data; don't store passwords in PlayerPrefs.
  • Copyright: All assets (music, sprites, fonts) must be licensed for use. Acknowledge sources.
  • Age ratings: Know the IARC/Australian Classification system. Design content appropriately.
  • Representation: Games shape culture. Avoid harmful stereotypes; include diverse characters.
  • Addiction design: Loot boxes, pay-to-win, endless loops — evaluate the ethics of your monetisation choices.
  • Online safety: Leaderboards and chat introduce HTTPS, authentication, and anti-cheat responsibilities.
Key question for your notebook: For every piece of data your game touches — ask: What? Why? Who? How protected? How long kept?
Grade A link: "Selects and applies safe, secure and ethical practices in the use of data." At Grade A, students don't just use data safely — they can justify every decision with reference to privacy principles, legal obligations and social impact. Your project notebook is where this reasoning lives.
