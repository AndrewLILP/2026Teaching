Part A

Concepts — 30 minutes

What is a shader?

Every object you see in a Unity scene goes through a small program before it appears on screen. That program — the shader — runs directly on the GPU and answers two questions for every pixel it touches:

  1. Where does each vertex go? (the vertex stage)
  2. What colour is each pixel? (the fragment stage)
Analogy: Think of the GPU as a massive printing press. The CPU hands it a template (the mesh) and a colour recipe (the shader). The GPU uses the recipe to colour millions of tiny dots simultaneously — far faster than any single CPU core could manage.

In Unity 6 with URP, shaders are written in a language called ShaderLab, which wraps standard GPU code (HLSL) in Unity-friendly syntax. You don't need to understand every HLSL detail today — you need to understand the structure so you can read and modify shaders confidently.

Why code instead of Shader Graph? Unity's Shader Graph is a valid visual tool. But if you understand the code underneath, you can read error messages, diagnose bugs, and create effects Shader Graph can't express. The principle here: analyse an existing structure first, then write your own.

The render pipeline (simplified)

When Unity draws a mesh, your shader is called at two specific points in the GPU pipeline:

CPUsends mesh data
→
Vertex Shadertransforms positions
→
Rasteriserfills triangles
→
Fragment Shadercolours pixels
→
Screenfinal output

For a beginner unlit shader, the vertex shader simply passes positions through unchanged — it's boilerplate we copy every time. All of the interesting work happens in the fragment shader, where you decide what colour each pixel is.

StageRuns once per…What we do there today
Vertexvertex (corner of a triangle)Convert 3D object position to 2D screen position — unchanged in all four steps
Fragmentpixel covered by the meshOutput an RGBA colour — this is where all our changes happen

ShaderLab anatomy

Read this entire block before writing anything. Identify each block's purpose and annotate it in your notebook before moving on.

skeleton — read and annotate only, don't type this yet ShaderLab 
// 1. The name shown in the Material Inspector dropdown.
Shader "Custom/MyShaderName"
{
    // 2. Properties block — variables displayed in the Inspector.
    Properties
    {
        // e.g.  _Color ("Tint", Color) = (1,1,1,1)
    }

    // 3. SubShader — the actual render instructions.
    SubShader
    {
        // 4. Tags tell Unity HOW and WHEN to render this object.
        Tags { "RenderType"="Opaque" }

        // 5. One Pass = one draw call sent to the GPU.
        Pass
        {
            HLSLPROGRAM          // start of GPU code

            #pragma vertex   vert   // which function is the vertex shader
            #pragma fragment frag   // which function is the fragment shader

            // ... structs, variables, and functions go here ...

            ENDHLSL              // end of GPU code
        }
    }
}
Trace task: Before typing Step 1, copy this skeleton into your notebook and next to each comment write in your own words what that block controls. Which block would you change to make an object semi-transparent? Where would a colour variable live?

Properties — connecting the Inspector to the GPU

A Property is a variable declared in the Properties block that Unity shows in the Material Inspector. You connect it to your HLSL code by declaring a variable with the exact same name, including the underscore prefix.

Inspector typeShaderLab syntaxHLSL variable type
Colour picker_Color ("Label", Color) = (1,1,1,1)half4 _Color;
Float slider_Alpha ("Label", Range(0,1)) = 1float _Alpha;
Texture_MainTex ("Label", 2D) = "white" {}sampler2D _MainTex;
Most common beginner mistake: If the name in Properties and the HLSL variable don't match exactly — including the underscore — the variable silently becomes zero and your object appears black or invisible. Always check spelling first.

Part B

Code Steps — 30 minutes

Project setup

  1. Open your Unity 6 project (URP template).
  2. In the Project panel, right-click → Create → Shader → Unlit Shader. Name it UnlitColour.
  3. Right-click the new shader → Create → Material. Name it Mat_UnlitColour.
  4. In the scene, add 3D Object → Cube. Drag Mat_UnlitColour onto it.
  5. Double-click UnlitColour.shader to open it in your code editor.
  6. Read the entire file Unity generated before deleting anything. Find the Properties block, the Tags, and the HLSLPROGRAM section.
  7. Select all and delete — you will type each step from scratch.
Don't copy-paste Step 1. Type it by hand. Typing slowly forces you to read every token. You will make a typo — that's deliberate. Finding and fixing it teaches you more than a clean paste.

Step 1 — Hardcoded colour

Goal: Get the absolute minimum working shader on screen — a solid colour with no Inspector controls whatsoever. Every line is essential; nothing is decoration. The cube should turn red-ish after you save and Unity recompiles (~2 seconds).
UnlitColour.shader — Step 1 (complete file) ShaderLab / HLSL 
// The name shown in the Material Inspector dropdown.
Shader "Custom/UnlitColour"
{
    // No properties yet — the colour is hardcoded directly in the GPU code below.
    Properties { }

    SubShader
    {
        // "Opaque" = render as a solid surface.
        // "UniversalPipeline" = only use this shader with URP (not Built-In).
        Tags
        {
            "RenderType"     = "Opaque"
            "RenderPipeline" = "UniversalPipeline"
        }

        Pass
        {
            HLSLPROGRAM

            // Tell the compiler which function is the vertex shader
            // and which function is the fragment shader.
            #pragma vertex   vert
            #pragma fragment frag

            // This include gives us URP helper functions (e.g. TransformObjectToHClip).
            #include "Packages/com.unity.render-pipelines.universal/ShaderLibrary/Core.hlsl"

            // ── INPUT STRUCT ─────────────────────────────────────────────────────
            // Attributes = data the CPU sends to the vertex shader for each corner.
            // POSITION is a "semantic" — it tells the GPU "this float4 is a position".
            struct Attributes
            {
                float4 positionOS : POSITION;  // OS = Object Space (local to the mesh)
            };

            // ── OUTPUT STRUCT ────────────────────────────────────────────────────
            // Varyings = data passed FROM the vertex shader TO the fragment shader.
            // SV_POSITION tells the GPU this float4 is the final clip-space position.
            struct Varyings
            {
                float4 positionHCS : SV_POSITION;  // HCS = Homogeneous Clip Space
            };

            // ── VERTEX SHADER ────────────────────────────────────────────────────
            // Runs once for every corner (vertex) of the mesh.
            // Its only job here is to convert the 3D local position to 2D screen space.
            // TransformObjectToHClip does the maths — we will not need to change this.
            Varyings vert(Attributes input)
            {
                Varyings output;
                output.positionHCS = TransformObjectToHClip(input.positionOS.xyz);
                return output;
            }

            // ── FRAGMENT SHADER ──────────────────────────────────────────────────
            // Runs once for every pixel the mesh covers on screen.
            // SV_Target means "write this colour to the render target (the screen)".
            // half4 is a colour with four channels: Red, Green, Blue, Alpha.
            // Each channel is a float from 0.0 (none) to 1.0 (full).
            half4 frag(Varyings input) : SV_Target
            {
                // Return a fixed colour: R=1.0, G=0.2, B=0.2, A=1.0
                // → red-ish, fully opaque.
                // Predict: what would you change to make it bright green?
                // Write your prediction in your notebook before testing it.
                return half4(1.0, 0.2, 0.2, 1.0);
            }

            ENDHLSL
        }
    }
}
Trace before you change anything: Find the line that sets the colour. The four numbers are R, G, B, A. What values would produce bright green? Pure white? Pure black? Write your predictions in your notebook, then test each one before moving to Step 2.

Step 2 — Expose colour in the Inspector

Goal: Add a colour picker to the Material Inspector so the colour can be changed without editing the shader. Three things change from Step 1 — they are marked with a green highlight and a NEW STEP 2 comment.
↳ What's new in Step 2: _Color property added to Properties block  ·  CBUFFER block added to wire it to HLSL  ·  hardcoded half4(…) in frag replaced with _Color
UnlitColour.shader — Step 2 (complete file) ShaderLab / HLSL 
Shader "Custom/UnlitColour"
{
    Properties
    {
        // ── NEW IN STEP 2 ─────────────────────────────────────────────────
        // _Color  = the HLSL variable name (must start with underscore).
        // "Tint Colour" = the label shown in the Material Inspector.
        // Color   = a colour picker widget.
        // (1,1,1,1) = the default value (white, fully opaque).
        _Color ("Tint Colour", Color) = (1, 1, 1, 1)
    }

    SubShader
    {
        Tags
        {
            "RenderType"     = "Opaque"
            "RenderPipeline" = "UniversalPipeline"
        }

        Pass
        {
            HLSLPROGRAM
            #pragma vertex   vert
            #pragma fragment frag
            #include "Packages/com.unity.render-pipelines.universal/ShaderLibrary/Core.hlsl"

            // ── NEW IN STEP 2 ─────────────────────────────────────────────────
            // CBUFFER groups material properties so the GPU can batch draw calls.
            // Without CBUFFER_START / CBUFFER_END you get a warning and
            // GPU instancing breaks — even though the shader still compiles.
            // The variable name MUST match the property name exactly (including _).
            CBUFFER_START(UnityPerMaterial)
                half4 _Color;   // matches Properties entry above
            CBUFFER_END

            struct Attributes
            {
                float4 positionOS : POSITION;
            };

            struct Varyings
            {
                float4 positionHCS : SV_POSITION;
            };

            // Vertex shader is identical to Step 1 — no changes needed here.
            Varyings vert(Attributes input)
            {
                Varyings output;
                output.positionHCS = TransformObjectToHClip(input.positionOS.xyz);
                return output;
            }

            half4 frag(Varyings input) : SV_Target
            {
                // ── NEW IN STEP 2 ─────────────────────────────────────────────
                // Return _Color instead of a hardcoded value.
                // Now changing the colour in the Inspector updates the cube in real time.
                return _Color;
            }

            ENDHLSL
        }
    }
}
Deliberate error exercise: Delete just the CBUFFER_START(UnityPerMaterial) and CBUFFER_END lines but keep half4 _Color; on its own. Save and check the Console. Record the warning message in your notebook. Then restore the two lines and explain in your own words why the wrapper is required.

Step 3 — Add transparency

Goal: Add an alpha slider so the object fades from fully opaque to fully invisible. Transparency requires three coordinated changes: a new property, updated render tags, and a blend command. If any one of the three is missing, the effect fails in a different way. New lines are marked green.
↳ What's new in Step 3: _Alpha property  ·  Tags changed to Transparent queue  ·  Blend + ZWrite commands added  ·  CBUFFER gains float _Alpha  ·  frag applies alpha to the colour
UnlitColour.shader — Step 3 (complete file) ShaderLab / HLSL 
Shader "Custom/UnlitColour"
{
    Properties
    {
        _Color ("Tint Colour", Color) = (1, 1, 1, 1)

        // ── NEW IN STEP 3 ─────────────────────────────────────────────────
        // Range(0,1) creates a slider in the Inspector.
        // 0 = fully invisible.  1 = fully opaque.  Default = 1.
        _Alpha ("Transparency", Range(0, 1)) = 1
    }

    SubShader
    {
        Tags
        {
            // ── NEW IN STEP 3 ─────────────────────────────────────────────────
            // Changed from "Opaque" to "Transparent" so Unity draws this object
            // AFTER all opaque objects, back-to-front.  Without this, objects
            // behind won't show through correctly.
            "RenderType"     = "Transparent"
            "Queue"          = "Transparent"
            "RenderPipeline" = "UniversalPipeline"
        }

        Pass
        {
            // ── NEW IN STEP 3 ─────────────────────────────────────────────────
            // Blend tells the GPU HOW to mix this object's colour with whatever
            // is already drawn behind it.
            // SrcAlpha OneMinusSrcAlpha = standard "alpha blending":
            // finalColour = (objectColour × alpha) + (background × (1 − alpha))
            // At alpha=1 you see only this object.  At alpha=0 you see only background.
            Blend SrcAlpha OneMinusSrcAlpha

            // ZWrite Off = don't write to the depth buffer for transparent objects.
            // If ZWrite is On, this transparent object incorrectly blocks objects behind it.
            ZWrite Off

            HLSLPROGRAM
            #pragma vertex   vert
            #pragma fragment frag
            #include "Packages/com.unity.render-pipelines.universal/ShaderLibrary/Core.hlsl"

            CBUFFER_START(UnityPerMaterial)
                half4 _Color;
                // ── NEW IN STEP 3 ─────────────────────────────────────────────
                // _Alpha must match the property name above exactly.
                float _Alpha;
            CBUFFER_END

            struct Attributes
            {
                float4 positionOS : POSITION;
            };

            struct Varyings
            {
                float4 positionHCS : SV_POSITION;
            };

            // Vertex shader unchanged from Step 2.
            Varyings vert(Attributes input)
            {
                Varyings output;
                output.positionHCS = TransformObjectToHClip(input.positionOS.xyz);
                return output;
            }

            half4 frag(Varyings input) : SV_Target
            {
                // ── NEW IN STEP 3 ─────────────────────────────────────────────
                // Copy _Color into a local variable so we can modify just its alpha.
                // col.r = red channel.  col.g = green.  col.b = blue.  col.a = alpha.
                half4 col = _Color;

                // Override the alpha channel with the Inspector slider value.
                col.a = _Alpha;

                return col;
            }

            ENDHLSL
        }
    }
}
Why three changes for one effect? The GPU doesn't automatically know an object is transparent — you must tell it at every level: (1) the Queue tag controls draw order, (2) the Blend command controls how colours mix on screen, (3) the HLSL alpha value controls how transparent this specific pixel is. Miss any one and the effect breaks in a different way — try it deliberately and record what you observe.

Add a second cube to the scene, place it behind the first, then drag the Transparency slider. The back cube should become visible as alpha decreases.

Step 4 — Pulse animation with _Time Extension · Grade B–A

Goal: Make the alpha pulse automatically using Unity's built-in _Time variable — no C# required. A new _PulseSpeed property controls the rate. Only the Properties block, CBUFFER, and frag function change. New lines are marked green.
↳ What's new in Step 4: _PulseSpeed property added  ·  CBUFFER gains float _PulseSpeed  ·  frag calculates a pulsing alpha using sin(_Time.y)
UnlitColour.shader — Step 4 (complete file) ShaderLab / HLSL 
Shader "Custom/UnlitColour"
{
    Properties
    {
        _Color ("Tint Colour", Color) = (1, 1, 1, 1)
        _Alpha ("Max Transparency", Range(0, 1)) = 1

        // ── NEW IN STEP 4 ─────────────────────────────────────────────────
        // Controls how many full pulses occur per second.
        // 1 = one pulse per second.  10 = ten pulses per second (very fast).
        _PulseSpeed ("Pulse Speed", Range(0.5, 10)) = 2
    }

    SubShader
    {
        Tags
        {
            "RenderType"     = "Transparent"
            "Queue"          = "Transparent"
            "RenderPipeline" = "UniversalPipeline"
        }

        Pass
        {
            Blend SrcAlpha OneMinusSrcAlpha
            ZWrite Off

            HLSLPROGRAM
            #pragma vertex   vert
            #pragma fragment frag
            #include "Packages/com.unity.render-pipelines.universal/ShaderLibrary/Core.hlsl"

            CBUFFER_START(UnityPerMaterial)
                half4 _Color;
                float _Alpha;
                // ── NEW IN STEP 4 ─────────────────────────────────────────────
                float _PulseSpeed;
            CBUFFER_END

            struct Attributes
            {
                float4 positionOS : POSITION;
            };

            struct Varyings
            {
                float4 positionHCS : SV_POSITION;
            };

            // Vertex shader unchanged from Step 3.
            Varyings vert(Attributes input)
            {
                Varyings output;
                output.positionHCS = TransformObjectToHClip(input.positionOS.xyz);
                return output;
            }

            half4 frag(Varyings input) : SV_Target
            {
                half4 col = _Color;

                // ── NEW IN STEP 4 ─────────────────────────────────────────────
                // _Time is a built-in Unity variable uploaded automatically every frame.
                // _Time.y = elapsed seconds since the scene started.
                // (_Time.x = time/20,  _Time.z = time×2,  _Time.w = time×3)
                //
                // sin() returns a value that oscillates between -1.0 and +1.0.
                // Multiplying by 0.5 and adding 0.5 remaps that range to 0.0–1.0,
                // so 'pulse' smoothly rises 0 → 1 → 0 → 1 … on every cycle.
                float pulse = sin(_Time.y * _PulseSpeed) * 0.5 + 0.5;

                // Multiply _Alpha by pulse so the slider still acts as a ceiling.
                // If _Alpha = 0.5, the object oscillates between alpha 0.0 and 0.5.
                col.a = _Alpha * pulse;

                return col;
            }

            ENDHLSL
        }
    }
}
Why drive this on the GPU instead of C# Update()? This fragment shader runs on every pixel, every frame, entirely on the GPU with no MonoBehaviour overhead. A C# approach would call material.SetFloat("_Alpha", …) each frame — a CPU-to-GPU data transfer per object. For one cube the difference is negligible. For hundreds of objects, it matters. Add this comparison to your notebook for the Task 3 evaluation.
