UV Coordinates

In Lesson 01 the fragment shader always returned a fixed colour. Now it will look up a colour from an image — and you will understand exactly how that lookup works at the GPU level.

Duration: 60 min (two 30-min parts)
Engine: Unity 6 (URP)
Language: ShaderLab + HLSL
Requires Lesson 01 completed — you should already understand the ShaderLab skeleton, the Properties block, CBUFFER_START, and the vertex/fragment function pair.
Part A

Concepts — 30 minutes

What is a UV coordinate?

Every vertex in a 3D mesh carries a hidden extra piece of data: a 2D coordinate that describes where on a flat image that vertex maps to. These coordinates are called UV coordinates — U runs horizontally, V runs vertically, and both range from 0.0 to 1.0 across the texture.

UV Space

The texture fills exactly the 0–1 square. Every point on the mesh surface maps to a point in this square.

UV Unwrap

A 3D mesh is "unfolded" flat like a cardboard box. Each face gets a region in the 0–1 UV square.

Analogy: UV unwrapping is like peeling the label off a tin can and laying it flat. The flat label is the texture image; the label's position on the flat sheet is the UV coordinate. When Unity renders the can, it re-applies the label by following the UV map.

When the GPU is filling in a pixel, it knows the UV coordinate at that point (interpolated from the surrounding vertices). It uses that (U, V) pair to look up a colour in the texture — that colour becomes the pixel's output.

The texture lookup

In HLSL, a texture is stored as a TEXTURE2D object and sampled using a SAMPLER. The lookup itself is done with the SAMPLE_TEXTURE2D macro, which returns a half4 colour — just like the hardcoded value we returned in Lesson 01.

URP vs Built-In pipeline: Older Unity tutorials use tex2D(_MainTex, uv). That works in the Built-In pipeline but causes errors in URP. Always use SAMPLE_TEXTURE2D(_MainTex, sampler_MainTex, uv) in URP shaders. If you find a tutorial using tex2D, that's your signal it's written for the old pipeline.

Passing data through Varyings

In Lesson 01, the Varyings struct held only positionHCS and the vertex shader was pure boilerplate. Now it needs to carry UV data too — and the vertex shader must actually do something: copy the UV from Attributes into Varyings so the fragment shader can use it.

Why does the vertex shader pass UVs at all? The GPU runs the vertex shader on each corner of a triangle, then interpolates all the Varyings values smoothly across every pixel in between. So a UV of (0,0) at one corner and (1,0) at another corner produces all the intermediate values — (0.1,0), (0.2,0), … — automatically. The fragment shader receives the already-interpolated UV for its specific pixel.

Tiling & offset — the _MainTex_ST convention

Every texture property in Unity automatically generates a companion variable named _YourTextureName_ST. The ST stands for Scale and Translation — it stores the Tiling (XY) and Offset (ZW) values you set in the Material Inspector as a single float4.

To apply tiling and offset to your raw UVs, use the TRANSFORM_TEX macro:

UV transformation — concept snippet HLSL 
// Raw UV from the mesh (0–1 across the surface)
float2 rawUV = input.uv;

// TRANSFORM_TEX applies the Inspector's Tiling and Offset values.
// _MainTex_ST is automatically provided by Unity — you just declare it in CBUFFER.
// Without this line, changing Tiling/Offset in the Inspector has no effect.
float2 tiledUV = TRANSFORM_TEX(rawUV, _MainTex);
Deliberate error (do this now): In Step 2 you will add TRANSFORM_TEX. Before doing so, set the Tiling to (3, 3) in the Inspector and observe that nothing changes. Then add TRANSFORM_TEX and watch the texture tile correctly. This demonstrates exactly what the macro does.

Code Steps — 30 minutes

Project setup

  1. In the Project panel, right-click → Create → Shader → Unlit Shader. Name it UnlitTexture.
  2. Right-click the new shader → Create → Material. Name it Mat_UnlitTexture.
  3. Import a texture image into your project — drag any .png or .jpg into the Project panel. A simple flat-colour grid or brick image works well for seeing tiling clearly.
  4. In the scene, add 3D Object → Cube. Drag Mat_UnlitTexture onto it.
  5. Open UnlitTexture.shader in your code editor. Read the whole file, then select all and delete — type each step from scratch.

Step 1 — Sample a texture

Goal: Get a texture onto the object. Three things are new compared to Lesson 01: UVs flow through the Attributes and Varyings structs, the vertex shader copies them across, and the fragment shader samples the texture using them. The cube should display your imported texture after saving.
UnlitTexture.shader — Step 1 (complete file) ShaderLab / HLSL 
// A new shader name for this lesson — distinct from Lesson 01's UnlitColour.
Shader "Custom/UnlitTexture"
{
    Properties
    {
        // 2D = a texture slot.  "white" = the default if nothing is assigned.
        // The empty braces {} are required syntax for texture properties.
        _MainTex ("Texture", 2D) = "white" {}
    }

    SubShader
    {
        Tags
        {
            "RenderType"     = "Opaque"
            "RenderPipeline" = "UniversalPipeline"
        }

        Pass
        {
            HLSLPROGRAM
            #pragma vertex   vert
            #pragma fragment frag
            #include "Packages/com.unity.render-pipelines.universal/ShaderLibrary/Core.hlsl"

            // ── TEXTURE OBJECTS ──────────────────────────────────────────────────
            // TEXTURE2D and SAMPLER are declared OUTSIDE the CBUFFER.
            // Textures are GPU resources, not simple float values — they cannot
            // live inside a constant buffer.
            // The sampler name must be "sampler_" + the texture name exactly.
            TEXTURE2D(_MainTex);
            SAMPLER(sampler_MainTex);

            CBUFFER_START(UnityPerMaterial)
                // _MainTex_ST stores the Tiling (xy) and Offset (zw) values.
                // We are not using it yet — but it MUST be declared here
                // whenever you have a texture property or Unity will warn you.
                float4 _MainTex_ST;
            CBUFFER_END

            // ── INPUT STRUCT ─────────────────────────────────────────────────────
            // NEW: float2 uv carries the texture coordinate for each vertex.
            // TEXCOORD0 is the semantic — "first UV channel on this mesh".
            struct Attributes
            {
                float4 positionOS : POSITION;
                float2 uv         : TEXCOORD0;  // NEW: UV from the mesh
            };

            // ── OUTPUT STRUCT ────────────────────────────────────────────────────
            // NEW: float2 uv is passed from the vertex shader to the fragment shader.
            // The GPU automatically interpolates this across each triangle.
            struct Varyings
            {
                float4 positionHCS : SV_POSITION;
                float2 uv          : TEXCOORD0;  // NEW: UV passed to fragment shader
            };

            // ── VERTEX SHADER ────────────────────────────────────────────────────
            // NEW: the vertex shader now does two things instead of one.
            // It still converts the position, AND it copies the UV across.
            Varyings vert(Attributes input)
            {
                Varyings output;
                output.positionHCS = TransformObjectToHClip(input.positionOS.xyz);
                output.uv = input.uv;  // NEW: copy raw UV to output
                return output;
            }

            // ── FRAGMENT SHADER ──────────────────────────────────────────────────
            // NEW: instead of returning a hardcoded colour, we look up the texture.
            // SAMPLE_TEXTURE2D takes three arguments:
            //   1. the TEXTURE2D object  (_MainTex)
            //   2. the SAMPLER object    (sampler_MainTex)
            //   3. the UV coordinate     (input.uv)
            // It returns a half4 colour — the texel at that UV position.
            half4 frag(Varyings input) : SV_Target
            {
                return SAMPLE_TEXTURE2D(_MainTex, sampler_MainTex, input.uv);
            }

            ENDHLSL
        }
    }
}
Trace task: Find the three places where uv appears in this file. Draw a flow diagram in your notebook showing how the UV value travels from the mesh vertex data → Attributes → vert() → Varyings → frag() → SAMPLE_TEXTURE2D. Where does the GPU do the interpolation step?

Step 2 — Tiling & offset

Goal: Make the Inspector's Tiling and Offset fields actually work. Right now in Step 1, changing Tiling in the Inspector has no effect — the raw UV is used as-is. One line in the vertex shader fixes this. New lines are marked green.
↳ What's new in Step 2: TRANSFORM_TEX applied to the UV in vert() — raw UV replaced with tiled/offset UV
UnlitTexture.shader — Step 2 (complete file) ShaderLab / HLSL 
Shader "Custom/UnlitTexture"
{
    Properties
    {
        _MainTex ("Texture", 2D) = "white" {}
    }

    SubShader
    {
        Tags
        {
            "RenderType"     = "Opaque"
            "RenderPipeline" = "UniversalPipeline"
        }

        Pass
        {
            HLSLPROGRAM
            #pragma vertex   vert
            #pragma fragment frag
            #include "Packages/com.unity.render-pipelines.universal/ShaderLibrary/Core.hlsl"

            TEXTURE2D(_MainTex);
            SAMPLER(sampler_MainTex);

            CBUFFER_START(UnityPerMaterial)
                float4 _MainTex_ST;
            CBUFFER_END

            struct Attributes
            {
                float4 positionOS : POSITION;
                float2 uv         : TEXCOORD0;
            };

            struct Varyings
            {
                float4 positionHCS : SV_POSITION;
                float2 uv          : TEXCOORD0;
            };

            Varyings vert(Attributes input)
            {
                Varyings output;
                output.positionHCS = TransformObjectToHClip(input.positionOS.xyz);
                // ── NEW IN STEP 2 ─────────────────────────────────────────────
                // TRANSFORM_TEX applies the _MainTex_ST scale and offset values.
                // Equivalent to:  output.uv = input.uv * _MainTex_ST.xy + _MainTex_ST.zw
                // Without this, changing Tiling/Offset in the Inspector does nothing.
                output.uv = TRANSFORM_TEX(input.uv, _MainTex);
                return output;
            }

            // Fragment shader unchanged from Step 1.
            half4 frag(Varyings input) : SV_Target
            {
                return SAMPLE_TEXTURE2D(_MainTex, sampler_MainTex, input.uv);
            }

            ENDHLSL
        }
    }
}
What TRANSFORM_TEX actually does: It expands to uv * _MainTex_ST.xy + _MainTex_ST.zw. The XY components are the Tiling scale, the ZW components are the Offset translation. Multiplying the UV by 3 makes the texture repeat three times across the surface — the UV range effectively becomes 0–3 instead of 0–1. Most texture samplers wrap at the boundary, so value 1.1 is treated the same as 0.1 — producing the tiling repeat.

In the Inspector, set Tiling to (3, 3). The texture should now tile three times across each axis. Try (1, 3) for horizontal stretching. Record what you observe and why it happens.

Step 3 — Colour tint

Goal: Add a colour tint that multiplies with the texture sample — darkening, colouring, or brightening the texture without replacing it. This revisits the _Color pattern from Lesson 01 and combines it with the texture sample. New lines are marked green.
↳ What's new in Step 3: _Color property added; CBUFFER gains half4 _Color; frag multiplies texture sample by _Color
UnlitTexture.shader — Step 3 (complete file) ShaderLab / HLSL 
Shader "Custom/UnlitTexture"
{
    Properties
    {
        _MainTex ("Texture", 2D) = "white" {}
        // ── NEW IN STEP 3 ─────────────────────────────────────────────────
        // A colour tint multiplied on top of the texture.
        // White (1,1,1,1) = no tint, texture appears as-is.
        // Red (1,0,0,1)   = texture appears red-tinted.
        // Grey (0.5,0.5,0.5,1) = texture appears half as bright.
        _Color ("Tint Colour", Color) = (1, 1, 1, 1)
    }

    SubShader
    {
        Tags
        {
            "RenderType"     = "Opaque"
            "RenderPipeline" = "UniversalPipeline"
        }

        Pass
        {
            HLSLPROGRAM
            #pragma vertex   vert
            #pragma fragment frag
            #include "Packages/com.unity.render-pipelines.universal/ShaderLibrary/Core.hlsl"

            TEXTURE2D(_MainTex);
            SAMPLER(sampler_MainTex);

            CBUFFER_START(UnityPerMaterial)
                float4 _MainTex_ST;
                // ── NEW IN STEP 3 ─────────────────────────────────────────────
                half4  _Color;
            CBUFFER_END

            struct Attributes
            {
                float4 positionOS : POSITION;
                float2 uv         : TEXCOORD0;
            };

            struct Varyings
            {
                float4 positionHCS : SV_POSITION;
                float2 uv          : TEXCOORD0;
            };

            // Vertex shader unchanged from Step 2.
            Varyings vert(Attributes input)
            {
                Varyings output;
                output.positionHCS = TransformObjectToHClip(input.positionOS.xyz);
                output.uv = TRANSFORM_TEX(input.uv, _MainTex);
                return output;
            }

            half4 frag(Varyings input) : SV_Target
            {
                // ── NEW IN STEP 3 ─────────────────────────────────────────────
                // Sample the texture into a local variable first.
                half4 texColour = SAMPLE_TEXTURE2D(_MainTex, sampler_MainTex, input.uv);

                // Multiply the texture colour by the tint colour channel-by-channel.
                // White tint (1,1,1,1) leaves the texture unchanged: texColour × 1 = texColour.
                // Any other colour shifts the texture toward that hue.
                return texColour * _Color;
            }

            ENDHLSL
        }
    }
}
Why multiply instead of add? Multiplying keeps the texture's dark areas dark and bright areas bright — it's a filter. Adding would shift everything brighter, blowing out highlights. Multiplying by white (1,1,1) is a no-op; multiplying by black (0,0,0) makes everything black. This is the standard technique used in virtually every game texture shader. Test both operations in your notebook.

Step 4 — Scroll UVs over time Extension · Grade B–A

Goal: Make the texture scroll automatically using _Time.y — no C# required. A _ScrollSpeed property controls the rate. The UV offset is added in the frag function after TRANSFORM_TEX so tiling still works correctly. New lines are marked green.
↳ What's new in Step 4: _ScrollSpeedX and _ScrollSpeedY properties added; CBUFFER gains both floats; frag offsets the UV using _Time.y before sampling
UnlitTexture.shader — Step 4 (complete file) ShaderLab / HLSL 
Shader "Custom/UnlitTexture"
{
    Properties
    {
        _MainTex      ("Texture",      2D)         = "white" {}
        _Color        ("Tint Colour",  Color)      = (1, 1, 1, 1)
        // ── NEW IN STEP 4 ─────────────────────────────────────────────────
        // Separate horizontal and vertical scroll speeds.
        // Positive = scroll right/up.  Negative = scroll left/down.  0 = no scroll.
        _ScrollSpeedX ("Scroll Speed X", Range(-2, 2)) = 0.5
        _ScrollSpeedY ("Scroll Speed Y", Range(-2, 2)) = 0
    }

    SubShader
    {
        Tags
        {
            "RenderType"     = "Opaque"
            "RenderPipeline" = "UniversalPipeline"
        }

        Pass
        {
            HLSLPROGRAM
            #pragma vertex   vert
            #pragma fragment frag
            #include "Packages/com.unity.render-pipelines.universal/ShaderLibrary/Core.hlsl"

            TEXTURE2D(_MainTex);
            SAMPLER(sampler_MainTex);

            CBUFFER_START(UnityPerMaterial)
                float4 _MainTex_ST;
                half4  _Color;
                // ── NEW IN STEP 4 ─────────────────────────────────────────────
                float  _ScrollSpeedX;
                float  _ScrollSpeedY;
            CBUFFER_END

            struct Attributes
            {
                float4 positionOS : POSITION;
                float2 uv         : TEXCOORD0;
            };

            struct Varyings
            {
                float4 positionHCS : SV_POSITION;
                float2 uv          : TEXCOORD0;
            };

            // Vertex shader unchanged from Step 3.
            Varyings vert(Attributes input)
            {
                Varyings output;
                output.positionHCS = TransformObjectToHClip(input.positionOS.xyz);
                output.uv = TRANSFORM_TEX(input.uv, _MainTex);
                return output;
            }

            half4 frag(Varyings input) : SV_Target
            {
                // ── NEW IN STEP 4 ─────────────────────────────────────────────
                // Add a time-based offset to the UV before sampling.
                // _Time.y increases by 1 every second, so multiplying by _ScrollSpeedX
                // shifts the U coordinate at that many units per second.
                // Because the texture wraps at 0 and 1, the image appears to scroll.
                float2 scrolledUV = input.uv + float2(_ScrollSpeedX, _ScrollSpeedY) * _Time.y;

                // Sample using the scrolled UV instead of the original.
                half4 texColour = SAMPLE_TEXTURE2D(_MainTex, sampler_MainTex, scrolledUV);

                return texColour * _Color;
            }

            ENDHLSL
        }
    }
}
Why scroll in the fragment shader rather than offsetting the Material in C#? Updating material.SetTextureOffset() in C# fires a CPU-to-GPU data transfer every frame for every object using this material. The GPU approach — adding _Time.y in the fragment shader — requires no CPU involvement at all once the material is set up; the GPU reads the built-in time value independently. For a single scrolling lava plane it barely matters; for a scene with water, animated clouds, and scrolling UI elements, it adds up.
