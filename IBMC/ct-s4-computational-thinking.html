CT2.1What is Computational Thinking? CT5-THI-01

Computational thinking is not about thinking like a computer — it is a problem-solving approach that allows humans to express solutions in a way that can be executed by a computer. It is useful far beyond programming: scientists, engineers, doctors, and designers all apply computational thinking whether or not they ever write code.

Inquiry Question

When you follow a recipe, navigate using a map app, or play a strategy game — which elements of computational thinking are you using? Is computational thinking uniquely human, or can machines "think" computationally?

The Four Pillars

🔍

Decomposition

Breaking a complex problem into smaller, more manageable sub-problems that are easier to understand, solve, and verify independently.

🔎

Pattern Recognition

Identifying similarities, trends, or regularities within or across problems. Patterns allow us to apply known solutions to new but similar situations.

🎛️

Abstraction

Focusing on essential information and hiding unnecessary detail. A map is an abstraction of the physical world — it shows what matters for navigation and omits everything else.

📋

Algorithm Design

Creating a precise, ordered sequence of steps — an algorithm — that solves the problem. The algorithm must be unambiguous enough for a computer to execute without additional interpretation.

Worked Example CT2.1a — Applying All Four Pillars

Problem: Design a system that recommends a movie to a user based on their preferences.

1
Decomposition: Break into sub-problems — collect user preferences, access a movie database, match preferences to movies, rank results, display recommendation.
2
Pattern recognition: Users who like action movies also often like science fiction. Users who rate movies above 8/10 tend to re-watch them. These patterns can improve recommendations.
3
Abstraction: We don't need to know a movie's budget, cinematographer, or distributor — only its genre, rating, duration, and audience score. We abstract away irrelevant data.
4
Algorithm design: Get user's preferred genre → Filter database by genre → Sort by rating (descending) → Display top 5 results → Ask user to select or request different genre.

CT2.2Algorithms — Pseudocode and Flowcharts CT5-OPL-01

An algorithm is a finite, ordered sequence of unambiguous steps that solves a problem or accomplishes a task. Before writing code in any programming language, we design algorithms in language-neutral notations: pseudocode and flowcharts. This separates the logic of a solution from the syntax of a specific language.

Inquiry Question

Recipes, instructions for flat-pack furniture, and GPS navigation directions are all algorithms. What properties make some algorithms clearer and more reliable than others?

Properties of a Good Algorithm

Essential Algorithm Properties
PropertyMeaningViolation example
FinitenessThe algorithm must terminate after a finite number of stepsAn infinite loop with no exit condition
DefinitenessEach step must be precisely and unambiguously defined"Sort the list somehow" — what order? what method?
InputZero or more inputs are accepted from an external sourceA calculation with no way to accept values
OutputOne or more results are producedA process that runs but produces nothing observable
EffectivenessEach step must be executable — not theoretical or impossible"Solve this NP-hard problem in one step"

Pseudocode is structured English — halfway between a natural language description and actual code. NESA uses a consistent set of conventions so that pseudocode can be read and written unambiguously. Keywords are written in ALL CAPS. Actions start with an Uppercase letter. Variables are written in lowercase.

Worked Example CT2.2a — Sequential Algorithm in Pseudocode

Write a pseudocode algorithm that asks the user for a temperature in Celsius and displays the equivalent in Fahrenheit. Formula: \(F = \frac{9C}{5} + 32\)

BEGIN // Convert Celsius to Fahrenheit Display "Enter temperature in Celsius: " Get celsius Set fahrenheit = (9 / 5) * celsius + 32 Display "Temperature in Fahrenheit: " + fahrenheit END

This is a sequence — steps are executed one after another, top to bottom, with no branching or repetition.

Calculate and display the sum of all integers from 1 to n, where n is entered by the user.

Mathematics Connection — Algebra and Algorithm Design

The formula for the sum of 1 to n is \(\dfrac{n(n+1)}{2}\) — from Chapter 1 algebra. The loop-based algorithm above produces the same answer by a different method. For n = 100: the loop runs 100 iterations; the formula gives the answer in one calculation. This is the efficiency trade-off in algorithm design — a well-chosen formula (abstracted from patterns) can outperform a brute-force loop. In computing, this is measured as time complexity: the loop is O(n); the formula is O(1).

Worked Example CT2.4a — Grade Classifier (Multi-branch selection)

Accept a percentage mark and display the corresponding grade: A (≥85), B (≥70), C (≥55), D (≥40), E (<40).

Pseudocode
BEGIN Get mark IF mark >= 85 THEN Display "Grade A" ELSE IF mark >= 70 THEN Display "Grade B" ELSE IF mark >= 55 THEN Display "Grade C" ELSE IF mark >= 40 THEN Display "Grade D" ELSE Display "Grade E" ENDIF END
Python
mark = int(input("Enter mark: "))

if mark >= 85:
    print("Grade A")
elif mark >= 70:
    print("Grade B")
elif mark >= 55:
    print("Grade C")
elif mark >= 40:
    print("Grade D")
else:
    print("Grade E")

Important: Python uses indentation (4 spaces) to define code blocks — there are no curly braces {} or BEGIN/END keywords. Indentation errors are syntax errors.

Iteration — for loops

A for loop iterates over a sequence — a range of numbers, a list, or a string. Use it when you know in advance how many iterations are needed.

Worked Example CT2.4b — For Loop with List

Given a list of exam scores, calculate and display the mean, the highest score, and the number of scores above the mean.

scores = [72, 88, 61, 95, 54, 79, 83]

# Calculate mean
total = 0
for score in scores:
    total += score
mean = total / len(scores)

# Count above mean
above = 0
for score in scores:
    if score > mean:
        above += 1

print(f"Mean:          {mean:.1f}")
print(f"Highest score: {max(scores)}")
print(f"Above mean:    {above} of {len(scores)}")

Output: Mean: 76.0  |  Highest score: 95  |  Above mean: 4 of 7

Iteration — while loops

A while loop repeats as long as a condition is True. Use it when the number of iterations is not known in advance — for example, input validation loops.

Worked Example CT2.4c — While Loop for Input Validation

Keep asking for a username until it is at least 6 characters and contains no spaces. Display the number of attempts.

attempts = 0
valid    = False

while not valid:
    username = input("Choose a username: ")
    attempts += 1
    if len(username) >= 6 and " " not in username:
        valid = True
    else:
        print("Username must be ≥6 chars with no spaces.")

print(f"Username '{username}' accepted after {attempts} attempt(s).")
Infinite Loops — A Common Error

A while loop whose condition never becomes False runs forever, crashing the program. Always ensure:

  • The loop condition can become False
  • Something inside the loop changes the condition variable
  • There is always an exit path (even if it's break)

Example of an infinite loop: while True: print("help") — the condition is always True and never changes.

Technology 7–8 Connection — Block-Based to Text-Based Programming

In Technology 7–8, digital technology projects often use block-based environments like Scratch, micro:bit's MakeCode, or MIT App Inventor. These tools represent the same control structures — sequence, selection, iteration — using visual puzzle-piece blocks. When you move from block-based to text-based Python in Computing Technology, the concepts do not change; only the notation does. An if block in Scratch is exactly the same as if … elif … else: in Python. This chapter makes that transition explicit.

CT2.5Functions and Modular Design CT5-OPL-01 · CT5-THI-01

As programs grow, writing all logic in one block becomes unmanageable. Functions (called procedures when they return no value) allow us to break a program into named, reusable components — each solving one specific sub-problem. This is modular design, and it directly implements the decomposition pillar of computational thinking.

Inquiry Question

Every large software system — a bank, a hospital, a game — is built from thousands of small functions. What makes a well-designed function? How does modular design affect reliability, teamwork, and maintenance?

Defining and Calling Functions in Python

Function Anatomy
def function_name(parameter1, parameter2):
    """Docstring: what this function does."""
    # Function body — the logic
    return result   # optional — omit for procedures
  • def — keyword that starts a function definition
  • Parameters — variables that receive values when the function is called (inside the parentheses)
  • Docstring — a string literal describing the function's purpose (best practice)
  • return — sends a value back to the caller; a function without return returns None
  • Arguments — the actual values passed when calling the function
Worked Example CT2.5a — Functions with Parameters and Return Values

Refactor the pay calculator from CT2.3b into modular functions.

def calculate_gross(rate, hours):
    """Return gross pay for given rate and hours."""
    return rate * hours

def calculate_tax(gross, rate=0.19):
    """Return tax amount. Default rate is 19%."""
    return gross * rate

def display_payslip(name, gross, tax):
    """Print a formatted payslip."""
    net = gross - tax
    print(f"\n--- PAYSLIP: {name} ---")
    print(f"Gross pay: ${gross:.2f}")
    print(f"Tax:       ${tax:.2f}")
    print(f"Net pay:   ${net:.2f}")

# --- Main program ---
emp_name  = input("Employee name: ")
hourly    = float(input("Hourly rate: $"))
hrs       = float(input("Hours worked: "))

g = calculate_gross(hourly, hrs)
t = calculate_tax(g)
display_payslip(emp_name, g, t)

Why modular? If the tax rate changes, you only edit calculate_tax() — not every place tax appears. If a payroll team has three developers, each can work on a separate function independently. Testing is also easier — each function can be tested in isolation.

Scope — Local and Global Variables

A variable's scope determines where in the program it can be accessed. Variables defined inside a function are local — they exist only while the function runs and cannot be accessed from outside. Variables defined at the top level of the program are global.

Scope — A Common Pitfall
total = 0   # global

def add_item(price):
    total = total + price   # ERROR: Python treats 'total' as local here
    return total

Fix: Pass total as a parameter and return the updated value — or use global total inside the function (less preferred). Best practice: functions should receive all they need via parameters and return results, minimising reliance on global variables.

Worked Example CT2.5b — Recursive Function (Extension)

A recursive function calls itself. Calculate the factorial of a number (\(n! = n \times (n-1) \times \ldots \times 1\), with \(0! = 1\)).

def factorial(n):
    """Return n! using recursion. n must be a non-negative integer."""
    if n == 0:               # base case — stops recursion
        return 1
    return n * factorial(n - 1)   # recursive case

print(factorial(5))   # 120
print(factorial(0))   # 1

Call stack for factorial(4): 4 × factorial(3) → 4 × 3 × factorial(2) → 4 × 3 × 2 × factorial(1) → 4 × 3 × 2 × 1 × factorial(0) → 4 × 3 × 2 × 1 × 1 = 24

Mathematics Connection — Functions and Algebra

A mathematical function \(f(x) = x^2 + 3x\) maps an input \(x\) to a single output. A Python function does exactly the same thing — it maps inputs (parameters) to outputs (return values). The notation f(5) in mathematics and calculate_gross(15.50, 38) in Python are conceptually identical: give it inputs, get back an output. Recursion also connects to mathematics: the recursive definition of factorial is exactly the one used in combinatorics and probability (Chapter 3). The concept of a base case in recursion mirrors the initial condition in mathematical induction.

CT2.6Testing, Debugging, and the Iterative Development Process CT5-DPM-01 · CT5-OPL-01

Writing code that runs without crashing is only the first requirement — it must also produce the correct results for all valid inputs, and handle invalid inputs gracefully. Testing verifies that software works as intended. Debugging is the process of finding and fixing the errors (bugs) revealed by testing.

Inquiry Question

In 1962 the Mariner 1 spacecraft was destroyed because a single missing hyphen in Fortran code caused it to veer off course. In 1996 the Ariane 5 rocket exploded because of an integer overflow bug. What does this tell us about the relationship between software correctness and real-world consequences?

Types of Errors

The Three Error Types
Error TypeWhen detectedDescriptionPython example
Syntax errorBefore running (at parse time)Code violates the grammar rules of the language — Python can't understand it at allprint("Hello" — missing closing parenthesis
Runtime errorWhile runningCode is grammatically valid but an impossible operation is attempted during executionx = int("hello") — can't convert "hello" to int

Worked Example CT2.6a — Identifying and Fixing Errors

The following program is meant to calculate the average of a list of numbers but has three bugs. Find and fix each one.

# BUGGY VERSION
numbers = [10, 20, 30, 40, 50]
total = 0
for num in numbers
    total = total + num
average = total / len(numbers) + 1
print(f"The average is {Average:.1f}")
1
Syntax error: for num in numbers is missing a colon : at the end → for num in numbers:
2
Logic error: total / len(numbers) + 1 adds 1 after dividing instead of dividing by len(numbers) alone → total / len(numbers)
3
Runtime error: Average (capital A) does not match the variable average (lowercase a) — Python is case-sensitive → {average:.1f}
# FIXED VERSION
numbers = [10, 20, 30, 40, 50]
total = 0
for num in numbers:
    total = total + num
average = total / len(numbers)
print(f"The average is {average:.1f}")
# Output: The average is 30.0

Worked Example CT2.7a — Automated Plant Watering System

Problem: Design a system that automatically waters a plant when the soil moisture drops below a threshold.

1
Identifying and Defining: The problem is that plants die when owners forget to water them. The system must sense soil moisture, decide if watering is needed, and activate a water pump accordingly. Users: households and small offices. Constraint: must run on battery; no internet required.
2
Researching and Planning: Components needed — soil moisture sensor, submersible mini pump, relay module, Arduino Uno, 9V battery. Algorithm (pseudocode):
BEGIN REPEAT Set moisture = ReadSensor(MOISTURE_PIN) IF moisture < THRESHOLD THEN ActivatePump(5) // water for 5 seconds Display "Watering..." ELSE Display "Soil is moist. OK." ENDIF Wait(3600) // check every hour UNTIL power_off END
3
Testing and Evaluating: Test cases — dry soil (sensor reading < threshold): pump activates ✓; wet soil (reading ≥ threshold): pump stays off ✓; sensor disconnected: program handles gracefully with error display. Evaluation — does it water for too long? Is threshold calibrated for this plant species?

Worked Example CT2.7b — Number Guessing Game (Complete, Testable Program)

A complete, well-structured number guessing game implementing all CT2 concepts: functions, selection, iteration, input validation, and a test plan.

import random

def generate_secret(low=1, high=100):
    """Return a random integer between low and high inclusive."""
    return random.randint(low, high)

def get_valid_guess(low, high):
    """Prompt until user enters an integer within [low, high]."""
    while True:
        try:
            guess = int(input(f"Guess ({low}–{high}): "))
            if low <= guess <= high:
                return guess
            print(f"Out of range. Enter {low}–{high}.")
        except ValueError:
            print("Please enter a whole number.")

def give_hint(guess, secret):
    """Print a hint based on guess vs secret."""
    if guess < secret:
        print("Too low! Try higher. ↑")
    elif guess > secret:
        print("Too high! Try lower. ↓")

def play_game():
    """Run one game session. Return number of attempts."""
    secret   = generate_secret()
    attempts = 0
    found    = False
    print("\n🎯 Guess the number between 1 and 100!")
    while not found:
        guess     = get_valid_guess(1, 100)
        attempts += 1
        if guess == secret:
            found = True
            print(f"✅ Correct! You guessed it in {attempts} attempt(s).")
        else:
            give_hint(guess, secret)
    return attempts

# --- Entry point ---
play_game()

Worked Example CT2.7c — Simple HTML/CSS/JavaScript App

A minimal BMI calculator — client-side only (no server needed). Demonstrates HTML structure, CSS styling, and JavaScript logic working together.

<!-- bmi.html -->
<form>
  <label>Weight (kg): <input type="number" id="weight"></label>
  <label>Height (m):  <input type="number" id="height" step="0.01"></label>
  <button onclick="calcBMI()">Calculate</button>
</form>
<p id="result"></p>

<script>
function calcBMI() {
  const w = parseFloat(document.getElementById('weight').value);
  const h = parseFloat(document.getElementById('height').value);
  if (isNaN(w) || isNaN(h) || h <= 0) {
    document.getElementById('result').textContent = "Please enter valid values.";
    return;
  }
  const bmi = (w / (h * h)).toFixed(1);
  document.getElementById('result').textContent = `BMI: ${bmi}`;
}
</script>

This implements the same formula \(\text{BMI} = w/h^2\) from Maths Ch. 2 (measurement and health contexts), expressed in JavaScript. Input validation mirrors the test-case thinking from CT2.6.

Systems Thinking Connection — CT1 ↔ CT2 ↔ CT3

Every application built in this chapter operates within a larger system studied in CT1. A web app runs on a server, accessed via HTTP over the internet. A mechatronic system connects physical sensors to code running on a microcontroller which may report to a cloud-based enterprise system. A game running on a phone sends analytics to a server for product improvement. Computational Thinking (CT2) is the craft of building components; Systems Thinking (CT1) gives us the architecture those components live in. Design Thinking (CT3 — coming next) bridges the two by ensuring the components we build actually serve the people who use them.
