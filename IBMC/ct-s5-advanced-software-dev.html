CT5.1Object-Oriented Programming (OOP) CT5-OPL-01

Stage 4 CT2 taught procedural programming — sequences of functions operating on data. Stage 5 introduces Object-Oriented Programming (OOP), the dominant paradigm in enterprise software, games, and application development. OOP organises code around objects — bundles of related data and behaviour — making large programs easier to design, extend, and maintain.

Inquiry Question

Every app on your phone is built using objects. A button is an object. A user account is an object. A database connection is an object. What does it mean to "think in objects," and why do professional programmers prefer this approach for large systems?

The Four Pillars of OOP

OOP Core Concepts
ConceptDefinitionReal-world analogy
ClassA blueprint or template defining the attributes (data) and methods (behaviour) that objects of this type will haveAn architectural blueprint — it describes a house but is not itself a house
Object (instance)A specific, concrete occurrence created from a class. Many objects can be created from one class.An actual house built from the blueprint

InheritanceA class (subclass) can inherit attributes and methods from another class (superclass), extending or overriding as neededA SportsCar inherits from Car — it has everything a Car has, plus additional features

Defining Classes in Python

Worked Example CT5.1a — A Student Class
class Student:
    """Represents a student in the school system."""

    # Class variable — shared by ALL instances
    school_name = "Riverside High School"

    def __init__(self, student_id, first_name, last_name, year_group):
        """Initialise a new Student instance."""
        # Instance variables — unique to each object
        self.student_id  = student_id
        self.first_name  = first_name
        self.last_name   = last_name
        self.year_group  = year_group
        self._grades     = {}    # underscore = "private" by convention

    def full_name(self):
        """Return the student's full name."""
        return f"{self.first_name} {self.last_name}"

    def add_grade(self, subject, grade):
        """Add or update a grade for a subject."""
        if not 0 <= grade <= 100:
            raise ValueError(f"Grade must be 0–100, got {grade}")
        self._grades[subject] = grade

    def average_grade(self):
        """Return average grade across all subjects, or None if no grades."""
        if not self._grades:
            return None
        return sum(self._grades.values()) / len(self._grades)

    def __str__(self):
        """String representation — used by print()."""
        avg = self.average_grade()
        avg_str = f"{avg:.1f}%" if avg else "No grades"
        return f"{self.full_name()} (Year {self.year_group}) — Avg: {avg_str}"

# --- Using the class ---
s1 = Student(1042, "Mei", "Chen", 9)
s1.add_grade("Maths", 88)
s1.add_grade("English", 74)
print(s1)   # Mei Chen (Year 9) — Avg: 81.0%
print(Student.school_name)  # Riverside High School

Key points: __init__ is the constructor — called when creating a new instance. self refers to the specific object calling the method. __str__ defines how the object appears when printed. Leading underscore on _grades signals it should not be accessed directly from outside the class.

class Person:
    def __init__(self, first_name, last_name, email):
        self.first_name = first_name
        self.last_name  = last_name
        self.email      = email

    def full_name(self):
        return f"{self.first_name} {self.last_name}"

class Teacher(Person):   # Teacher inherits from Person
    def __init__(self, first_name, last_name, email, teacher_id, faculty):
        super().__init__(first_name, last_name, email)  # call Person's __init__
        self.teacher_id = teacher_id
        self.faculty    = faculty
        self.subjects   = []

    def add_subject(self, subject_name):
        self.subjects.append(subject_name)

    def __str__(self):
        return f"Teacher: {self.full_name()} | {self.faculty} | {len(self.subjects)} subject(s)"

t = Teacher("Maya", "Patel", "mpatel@school.edu", 201, "Science")
t.add_subject("Chemistry")
print(t)  # Teacher: Maya Patel | Science | 1 subject(s)

super() calls the parent class's method — avoiding duplicating the initialisation code from Person.

CT2 Extension — From Functions to Objects

In CT2.5 you learned to decompose programs into functions. OOP takes this further: a class groups related functions (methods) with the data they operate on (attributes). The Student class bundles add_grade() and average_grade() with the student's _grades dictionary — they belong together conceptually and practically. OOP is decomposition at a higher level of abstraction. Every design principle from CT2 still applies: single responsibility, meaningful names, testability — they just apply to classes rather than isolated functions.

CT5.2File I/O and Data Persistence CT5-OPL-01 · CT5-DAT-01

Programs that lose all their data when they close are called volatile. Real applications must persist data — save it to a file or database so it survives between sessions. File I/O (Input/Output) in Python gives programs this capability.

Text Files and CSV

Worked Example CT5.2a — Reading and Writing Text Files
# Writing — always use 'with' for automatic file closing
with open('students.txt', 'w') as f:
    f.write("1042,Mei,Chen,9\n")
    f.write("1019,Amir,Hassan,10\n")

# Reading line by line
with open('students.txt', 'r') as f:
    for line in f:
        parts = line.strip().split(',')
        sid, fname, lname, year = parts
        print(f"{fname} {lname} — Year {year}")

File modes: 'r' = read, 'w' = write (overwrites), 'a' = append (adds to end), 'r+' = read and write.

Worked Example CT5.2b — CSV with the csv Module
import csv

# Write CSV with headers
with open('students.csv', 'w', newline='') as f:
    writer = csv.DictWriter(f, fieldnames=['id', 'name', 'year'])
    writer.writeheader()
    writer.writerow({'id': 1042, 'name': 'Mei Chen', 'year': 9})
    writer.writerow({'id': 1019, 'name': 'Amir Hassan', 'year': 10})

# Read CSV as list of dictionaries
with open('students.csv', 'r') as f:
    reader = csv.DictReader(f)
    for row in reader:
        print(f"{row['name']} is in Year {row['year']}")

JSON — Structured Data Exchange

JSON (JavaScript Object Notation) is the web's standard data interchange format. Python's json module converts between Python dictionaries/lists and JSON strings, enabling persistent storage of structured data and communication with web APIs.

Worked Example CT5.2c — Saving and Loading Objects as JSON
import json

# Save student objects to JSON
students_data = [
    {"id": 1042, "name": "Mei Chen", "year": 9, "grades": {"Maths": 88, "English": 74}},
    {"id": 1019, "name": "Amir Hassan", "year": 10, "grades": {"Science": 91}}
]
with open('students.json', 'w') as f:
    json.dump(students_data, f, indent=2)

# Load back from JSON
with open('students.json', 'r') as f:
    loaded = json.load(f)
    for s in loaded:
        avg = sum(s['grades'].values()) / len(s['grades'])
        print(f"{s['name']}: avg {avg:.1f}%")
CT4 Connection — File I/O vs Databases

CSV and JSON files are appropriate for small datasets, simple tools, and data exchange. For large, structured, multi-user datasets — the enterprise systems studied in CT4 — a relational database (SQLite, PostgreSQL) is far superior: it handles concurrent access, enforces data integrity, and executes complex queries efficiently. In professional Python development, you often start with JSON for rapid prototyping, then migrate to a database when the system scales up. Understanding when to use each is a key professional judgement.

CT5.3Advanced Algorithms — Searching, Sorting, and Complexity CT5-OPL-01 · CT5-THI-01

Stage 4 introduced basic iteration and sequential algorithms. Stage 5 extends this to algorithm analysis — not just does the algorithm work, but how efficiently? This distinction matters enormously at scale: an algorithm that takes 1 second for 1,000 records may take 1,000,000 seconds (11.5 days) for 1,000,000 records if it is poorly designed.

Searching Algorithms

Linear Search vs Binary Search
Linear SearchBinary Search
MethodCheck each element one by one from the startRepeatedly halve the search space by checking the middle element
RequirementWorks on any list (sorted or unsorted)List must be sorted first
Best caseTarget is first element — 1 comparisonTarget is the middle — 1 comparison
Worst case (n items)n comparisonslog₂(n) comparisons
Example (1,000,000 items)Up to 1,000,000 stepsAt most 20 steps (log₂(1,000,000) ≈ 20)
Complexity notationO(n)O(log n)
Worked Example CT5.3a — Binary Search
def binary_search(sorted_list, target):
    """Return index of target in sorted_list, or -1 if not found."""
    low, high = 0, len(sorted_list) - 1
    while low <= high:
        mid = (low + high) // 2
        if sorted_list[mid] == target:
            return mid          # found!
        elif sorted_list[mid] < target:
            low = mid + 1       # target must be in right half
        else:
            high = mid - 1      # target must be in left half
    return -1               # not found

scores = [42, 55, 61, 74, 78, 83, 91, 95, 98]
print(binary_search(scores, 83))   # 5 (index)
print(binary_search(scores, 70))   # -1 (not found)

Trace for target = 83: List has 9 items. Mid = 4 (value 78) → 83 > 78, so low = 5. Mid = 6 (value 91) → 83 < 91, so high = 5. Mid = 5 (value 83) → found! Only 3 comparisons.

Sorting Algorithms

Worked Example CT5.3b — Bubble Sort vs Python's Built-in Sort
def bubble_sort(data):
    """Sort list in-place using bubble sort. O(n²)."""
    n = len(data)
    for i in range(n - 1):
        for j in range(n - 1 - i):
            if data[j] > data[j + 1]:
                data[j], data[j + 1] = data[j + 1], data[j]  # swap
    return data

scores = [78, 55, 91, 42, 83]
print(bubble_sort(scores))      # [42, 55, 78, 83, 91]

# Python's built-in — O(n log n) — always prefer this in practice
scores2 = [78, 55, 91, 42, 83]
scores2.sort()                  # in-place
scores3 = sorted(scores2, reverse=True)  # returns new list, descending
Mathematics Connection — Logarithms and Algorithm Complexity

Big-O notation uses functions you recognise from mathematics: O(1) = constant, O(n) = linear, O(n²) = quadratic, O(log n) = logarithmic, O(n log n) = linearithmic. The logarithm base 2 appears in binary search because each step halves the problem: \(\log_2(1{,}000{,}000) \approx 20\). This connects to the index laws in Maths Chapter 1 — \(2^{20} = 1{,}048{,}576\). Binary search is efficient because exponents grow far faster than logarithms: to search 10× more data, binary search needs only \(\log_2(10) \approx 3.3\) extra steps, not 10× more steps.

Advanced Control — PID Controllers

A PID controller (Proportional–Integral–Derivative) is the most widely used control algorithm in industrial automation. It automatically adjusts an output (e.g. motor speed, heater power) to maintain a desired setpoint despite disturbances, by continuously measuring the error between the setpoint and actual value.

PID Control Terms
TermWhat it corrects forEffect of too much
Proportional (P)Current error — output proportional to how far off we are nowOvershoot; oscillation; never reaches exact setpoint
Integral (I)Accumulated past error — corrects steady-state offsetSlow response; wind-up; instability
Derivative (D)Rate of change of error — predicts future error and dampens responseNoise amplification; instability
Worked Example CT5.4a — Simplified PID in MicroPython (Temperature Controller)
# Simplified PID controller — runs on a Raspberry Pi Pico
class PIDController:
    def __init__(self, kp, ki, kd, setpoint):
        self.kp = kp          # proportional gain
        self.ki = ki          # integral gain
        self.kd = kd          # derivative gain
        self.setpoint  = setpoint
        self._integral = 0
        self._prev_err = 0

    def compute(self, measured_value):
        """Return control output for current measurement."""
        error = self.setpoint - measured_value
        self._integral  += error
        derivative      = error - self._prev_err
        output          = (self.kp * error +
                           self.ki * self._integral +
                           self.kd * derivative)
        self._prev_err  = error
        return max(0, min(100, output))   # clamp 0–100%

# Simulation: maintain 37°C body temperature in an incubator
controller = PIDController(kp=2.0, ki=0.1, kd=0.5, setpoint=37.0)
temps = [35.0, 35.8, 36.3, 36.8, 37.1, 37.0]
for t in temps:
    power = controller.compute(t)
    print(f"Temp: {t}°C → Heater power: {power:.1f}%")

OOP Game Architecture with Pygame

Worked Example CT5.5a — OOP Pygame: Player and Enemy Classes
import pygame

class Player(pygame.sprite.Sprite):
    """Player character — moves with arrow keys, fires with spacebar."""
    def __init__(self, x, y):
        super().__init__()
        self.image = pygame.Surface((40, 40))
        self.image.fill((0, 180, 80))      # green square placeholder
        self.rect   = self.image.get_rect(center=(x, y))
        self.speed  = 4
        self.health = 100
        self.score  = 0

    def update(self):
        keys = pygame.key.get_pressed()
        if keys[pygame.K_LEFT]:  self.rect.x -= self.speed
        if keys[pygame.K_RIGHT]: self.rect.x += self.speed
        if keys[pygame.K_UP]:    self.rect.y -= self.speed
        if keys[pygame.K_DOWN]:  self.rect.y += self.speed
        # Keep player on screen
        self.rect.clamp_ip(pygame.display.get_surface().get_rect())

    def take_damage(self, amount):
        self.health = max(0, self.health - amount)

    def is_alive(self):
        return self.health > 0

class Enemy(pygame.sprite.Sprite):
    """Enemy that moves toward the player."""
    def __init__(self, x, y, speed=2):
        super().__init__()
        self.image = pygame.Surface((30, 30))
        self.image.fill((220, 50, 50))       # red square
        self.rect  = self.image.get_rect(topleft=(x, y))
        self.speed = speed

    def update(self, player_rect):
        # Move toward player (simplified)
        if self.rect.x < player_rect.x: self.rect.x += self.speed
        elif self.rect.x > player_rect.x: self.rect.x -= self.speed
        if self.rect.y < player_rect.y: self.rect.y += self.speed
        elif self.rect.y > player_rect.y: self.rect.y -= self.speed

Finite State Machines

A finite state machine (FSM) models an object that can exist in a fixed set of states and transitions between them based on events. FSMs are essential in game development (enemy AI, game menus) and control systems (traffic lights, vending machines).

Worked Example CT5.5b — Enemy AI with a State Machine
class EnemyAI:
    """Enemy with patrol, chase, attack, and flee states."""
    PATROL = 'patrol'; CHASE = 'chase'
    ATTACK = 'attack'; FLEE  = 'flee'

    def __init__(self):
        self.state  = self.PATROL
        self.health = 100

    def update(self, player_distance, player_health):
        """Transition states based on game conditions."""
        if self.state == self.PATROL:
            if player_distance < 200:
                self.state = self.CHASE
        elif self.state == self.CHASE:
            if player_distance < 50:
                self.state = self.ATTACK
            elif player_distance > 300:
                self.state = self.PATROL
        elif self.state == self.ATTACK:
            if self.health < 30:
                self.state = self.FLEE
            elif player_distance > 50:
                self.state = self.CHASE
        elif self.state == self.FLEE:
            if player_distance > 400:
                self.state = self.PATROL

    def act(self):
        actions = {
            self.PATROL: "Patrolling...",
            self.CHASE:  "Chasing player!",
            self.ATTACK: "ATTACKING!",
            self.FLEE:   "Fleeing..."
        }
        return actions[self.state]

APIs — Consuming External Data

An API (Application Programming Interface) is a defined way for one software system to communicate with another. Web APIs typically communicate over HTTP using JSON. They allow your application to access real-world data: weather, maps, sports results, exchange rates, social media.

Worked Example CT5.6a — Consuming a REST API with Python
import requests, json

# Open-Meteo: free weather API, no API key required
# Get current temperature in Sydney (lat=-33.87, lon=151.21)
url = ("https://api.open-meteo.com/v1/forecast"
       "?latitude=-33.87&longitude=151.21"
       "¤t=temperature_2m,weathercode"
       "&timezone=Australia/Sydney")

response = requests.get(url)

if response.status_code == 200:
    data = response.json()
    current = data['current']
    temp    = current['temperature_2m']
    code    = current['weathercode']

else:

HTTP Status codes: 200 = success, 404 = not found, 403 = forbidden, 500 = server error. Always check the status code before processing the response.

Building a Flask Web App with a Database

Worked Example CT5.6b — Flask App Connected to SQLite
from flask import Flask, jsonify, request
import sqlite3

app = Flask(__name__)
DB  = 'school.db'

def get_db():
    conn = sqlite3.connect(DB)
    conn.row_factory = sqlite3.Row   # rows as dict-like objects
    return conn

# Route: GET all students as JSON
@app.route('/api/students', methods=['GET'])
def get_students():
    year = request.args.get('year')  # optional query param: /api/students?year=9
    conn = get_db()
    if year:
        rows = conn.execute(
            "SELECT * FROM Students WHERE year_group = ?", (year,)
        ).fetchall()
    else:
        rows = conn.execute("SELECT * FROM Students").fetchall()
    conn.close()
    return jsonify([dict(r) for r in rows])

# Route: POST a new student
@app.route('/api/students', methods=['POST'])
def add_student():
    data = request.get_json()
    conn = get_db()
    conn.execute(
        "INSERT INTO Students VALUES (?,?,?,?,?,?)",
        (data['student_id'], data['first_name'], data['last_name'],
         data['date_of_birth'], data['year_group'], data['email'])
    )
    conn.commit(); conn.close()
    return jsonify({'message': 'Student added'}), 201

if __name__ == '__main__':
    app.run(debug=True)

Note the use of parameterised queries (? placeholders) — this is critical for preventing SQL injection attacks (covered in CT1.5). Never concatenate user input directly into SQL strings.

SQL Injection — The Most Critical Web Security Vulnerability

If you build a query like: f"SELECT * FROM Students WHERE name = '{user_input}'" and a user enters ' OR '1'='1, the query becomes: SELECT * FROM Students WHERE name = '' OR '1'='1' — which returns every student. Always use parameterised queries with ? placeholders. The database driver handles escaping, making injection impossible.

CT5.7Project Management and Collaboration CT5-DPM-01 · CT5-COL-01 · CT5-COM-01

Real software is built by teams. No professional developer works entirely alone. Stage 5 formalises the practices that make collaborative software development possible — version control, agile planning, code review, and documentation.

Version Control with Git

Git is the universal version control system. It tracks every change to every file in a project, allowing teams to work simultaneously without overwriting each other, and to revert to any previous state if something breaks.

Essential Git Commands
CommandWhat it does
git initInitialise a new Git repository in the current directory
git clone <url>Copy an existing repository from a remote source (e.g. GitHub)
git add <file>Stage a file — mark it as ready to be committed
git commit -m "message"Save a snapshot of all staged changes with a descriptive message
git pushUpload local commits to the remote repository
git pullDownload and merge changes from the remote repository
git branch <name>Create a new branch for developing a feature without affecting main
git merge <branch>Merge changes from a branch back into the current branch
git logView the commit history — who changed what, and when

Agile Development

Agile is an iterative software development methodology. Instead of one long waterfall cycle (plan → build → test → ship), Agile works in short sprints (1–4 week cycles), each delivering a working increment of the product.

Agile vs Waterfall
WaterfallAgile
PlanningComplete upfront — all requirements defined before any codeProgressive — requirements evolve as the team learns
DeliveryOne delivery at the end of the projectWorking software delivered every sprint
ChangeDifficult and expensive to change once startedChange is expected and welcomed
Best forWell-understood, stable requirements (e.g. building a bridge)Innovative products where user needs evolve

NESA's four content groups — Identifying and defining · Researching and planning · Producing and implementing · Testing and evaluating — map to Agile sprint cycles, not to waterfall phases.
