CT4.1Data, Databases, and Information Systems CT5-DAT-01

Every enterprise information system is, at its core, a data management system. The decisions a business makes, the services a hospital provides, the performance a school tracks — all of it ultimately lives in a database: an organised, structured collection of related data. Understanding how databases work is one of the most transferable skills in all of computing.

Inquiry Question

How does Spotify know what to recommend? How does Medicare know which treatments you've claimed? How does Amazon know to warn you when your order might be late? All of these require structured data at massive scale — how does that data get organised?

Data vs Information vs Knowledge

The Data–Information–Knowledge Hierarchy
LevelDefinitionExample
DataRaw, unprocessed facts and figures with no inherent meaning on their own78, 2024-03-15, Sydney
InformationData that has been processed, organised, or structured to provide context and meaning"A student scored 78% on the 15 March 2024 Maths test in Sydney"
KnowledgeInformation that has been analysed and interpreted to support decisions"Students who scored below 80% in March tend to struggle with the June exam — targeted revision is needed"

Relational Databases

A relational database stores data in tables (also called relations). Each table has fields (columns, representing attributes) and records (rows, representing individual instances). Tables are linked through keys.

Core Database Terminology
TermDefinitionExample
Table (relation)A structured collection of related data organised in rows and columnsA Students table in a school database
Field (attribute/column)A single characteristic stored for each recordstudent_id, first_name, year_group
Record (tuple/row)A single entry in a table — one complete set of related fieldsThe data for one specific student
Primary Key (PK)A field (or combination of fields) that uniquely identifies each record in a table. Must be unique and never NULL.student_id — no two students share the same ID
Foreign Key (FK)A field in one table that references the Primary Key of another table, creating a relationship between themclass_id in the Enrolments table references class_id in the Classes table
Data typeThe kind of data a field stores — affects storage, validation, and operations allowedINTEGER, TEXT, REAL, DATE, BOOLEAN
NULLThe absence of a value — distinct from zero or an empty stringA student's middle_name field if they don't have one
Mathematics Connection — Data Classification

Database field types map directly to the data classification from Maths Chapter 3 (Statistics and Probability, Section 3.1). INTEGER fields store numerical discrete data (e.g. count of goals). REAL fields store numerical continuous data (e.g. a temperature reading). TEXT fields store categorical nominal data (e.g. suburb name). BOOLEAN fields store binary categorical data (True/False). Choosing the wrong data type is a form of data misclassification — with real consequences for what analyses are possible.

CT4.2Database Design and Entity-Relationship Diagrams CT5-DAT-01 · CT5-DPM-01

Before writing a single line of SQL, a database must be designed. Poor design leads to redundant data, update anomalies, and queries that become impossibly complex. An Entity-Relationship Diagram (ERD) is the standard tool for visualising and communicating a database's structure before it is built.

Inquiry Question

If you store a student's class teacher's name in the student's record, what happens when that teacher changes? How does separating data into related tables (normalisation) protect against this kind of problem?

Relationships Between Tables

Types of Relationships
RelationshipDescriptionExampleHow implemented
One-to-One (1:1)Each record in Table A relates to exactly one record in Table B, and vice versaEach student has exactly one student ID card; each ID card belongs to one studentFK in either table (or merge into one table)
One-to-Many (1:N)Each record in Table A can relate to many records in Table B, but each record in B relates to only one in AOne teacher teaches many classes; each class has one teacherFK in the "many" table pointing to the PK of the "one" table
Many-to-Many (M:N)Each record in Table A can relate to many in B, and vice versaStudents enrol in many subjects; each subject has many studentsA third "junction" or "bridge" table is created with FKs to both tables

Worked Example CT4.2a — Reading an ERD

Using the ERD above, answer:

1
What is the relationship between Students and Subjects?
Many-to-many — a student enrols in many subjects, and a subject has many students. Resolved through the Enrolments junction table.
2
What foreign keys are in the Enrolments table?
student_id (references Students.student_id) and subject_id (references Subjects.subject_id).
3
Why is a teacher's name NOT stored in the Subjects table?
If a teacher's name changed, you'd have to update it in every subject they teach. Storing only teacher_id in Subjects and all teacher details in Teachers means you only update one record — this is normalisation.
4
What is the relationship between Teachers and Subjects?
One-to-many — one teacher can teach many subjects; each subject has one assigned teacher. The FK teacher_id sits in the Subjects table (the "many" side).

CT4.3SQL — Querying Data CT5-DAT-02

SQL (Structured Query Language) is the standard language for communicating with relational databases. It is used by virtually every enterprise system in the world — from hospital records to banking to social media. SQL is declarative: you describe what you want, not how to find it.

NESA SQL Syntax Reference — SELECT Statement
SELECT
The field(s) or calculated values to display. Use * for all fields. Use AS to rename a field for display.
FROM
The table(s) to retrieve data from.
WHERE
Filter condition(s) — only records matching the condition are returned. Uses relational operators: = != < > <= >= and logical operators: AND OR NOT.
ORDER BY
Sort results by one or more fields. ASC = ascending (A→Z, 0→9); DESC = descending (Z→A, 9→0).
LIMIT
Return only the first n records. Useful for "top 5" style queries.

Using the school database from CT4.2, with sample data in the tables.

Find all Year 9 students whose last name starts with a letter between A and M (alphabetically before N), sorted by first name.

SELECT first_name, last_name, email
FROM Students
WHERE year_group = 9
  AND last_name < 'N'

Find the three highest-achieving students in the Enrolments table (highest grade — assume grade is stored as a numeric percentage).

Worked Example CT4.3d — Renaming Fields with AS

Display student names as a single "Full Name" column and their year group as "Year".

SELECT first_name || ' ' || last_name AS 'Full Name',
       year_group AS 'Year'
FROM Students
ORDER BY last_name ASC;

|| is the string concatenation operator in SQLite. In MySQL/PostgreSQL use CONCAT(first_name, ' ', last_name).

Computational Thinking Connection — SQL as Declarative Programming

In CT2 you wrote imperative code in Python — telling the computer exactly how to solve a problem step by step. SQL is declarative — you describe what you want and the database engine decides how to retrieve it. This is a fundamentally different programming paradigm. Writing SELECT * FROM Students WHERE year_group = 9 is equivalent to writing a Python loop that checks every record, but the database handles the searching algorithm internally, often using indexes that make it dramatically faster than a Python loop over millions of rows.

CT4.4SQL — Aggregate Functions and Multi-Table Queries CT5-DAT-02

Single-table queries answer simple questions. Real enterprise analysis requires combining data from multiple tables and computing summaries. Aggregate functions and JOIN operations are the tools that make complex analysis possible in a single SQL statement.

Aggregate Functions

NESA Aggregate Functions
FunctionReturnsExample
COUNT(field)Number of non-NULL values in the fieldCOUNT(student_id) → number of students
SUM(field)Total of all valuesSUM(grade) → total of all grades
AVG(field)Mean (arithmetic average)AVG(grade) → class average
MAX(field)Largest valueMAX(grade) → highest score
MIN(field)Smallest valueMIN(grade) → lowest score

Aggregate functions collapse many rows into one result. Use GROUP BY to compute aggregates for sub-groups.

Worked Example CT4.4a — Aggregate Functions with GROUP BY

For each subject in the 2024 Enrolments, calculate the number of students enrolled, the average grade, and the highest grade. Show only subjects with more than 5 enrolments, sorted by average grade descending.

SELECT subject_id,
       COUNT(student_id) AS 'Enrolments',
       AVG(grade)        AS 'Average Grade',
       MAX(grade)        AS 'Top Grade'
FROM Enrolments
WHERE year = 2024
GROUP BY subject_id
HAVING COUNT(student_id) > 5
ORDER BY AVG(grade) DESC;

Key distinction: WHERE filters individual records before grouping. HAVING filters groups after aggregation — it is the aggregate equivalent of WHERE.

Joining Tables

A JOIN combines rows from two or more tables based on a related field. The most common join type, INNER JOIN (or simply specifying multiple tables in FROM), returns only records that have matching values in both tables.

Worked Example CT4.4b — Multi-Table Query (Implicit JOIN)

Display student names alongside the subjects they are enrolled in and their grades. Only show Year 9 students. Sort by student last name.

-- NESA-style: multiple tables in FROM, conditions in WHERE
SELECT Students.first_name,
       Students.last_name,
       Subjects.name        AS 'Subject',
       Enrolments.grade
FROM Students, Enrolments, Subjects
WHERE Students.student_id  = Enrolments.student_id
  AND Enrolments.subject_id = Subjects.subject_id
  AND Students.year_group   = 9
ORDER BY Students.last_name ASC;

When querying multiple tables, always qualify field names with TableName.field_name to avoid ambiguity when two tables share a field name.

Worked Example CT4.4c — Joining Three Tables for a Report

Produce a report showing each teacher's name, the subjects they teach, and the number of students enrolled in each subject.

SELECT Teachers.first_name || ' ' || Teachers.last_name AS 'Teacher',
       Subjects.name                                       AS 'Subject',
       COUNT(Enrolments.student_id)                        AS 'Students'
FROM Teachers, Subjects, Enrolments
WHERE Teachers.teacher_id  = Subjects.teacher_id
  AND Subjects.subject_id   = Enrolments.subject_id
GROUP BY Subjects.subject_id
ORDER BY Teachers.last_name ASC, COUNT(Enrolments.student_id) DESC;
Statistics Connection — SQL Aggregates = Statistical Measures

AVG() computes the mean — the same measure from Maths Chapter 3, Section 3.3. COUNT() gives you the sample size n. MAX() and MIN() together give you the range. SQL doesn't natively compute median or standard deviation (without additional functions), which is why real data analysts export results and use visualisation tools for full statistical analysis. The aggregate functions here are the SQL implementation of descriptive statistics.

CT4.5Data Manipulation and Validation CT5-DAT-02 · CT5-SAF-01

Querying data is read-only. To create and maintain a database, we also need to insert, update, and delete records. These Data Manipulation Language (DML) operations must be used carefully — especially DELETE and UPDATE, which can cause irreversible data loss if WHERE conditions are omitted.

INSERT, UPDATE, DELETE

DML Operations
OperationSyntax PatternNotes
INSERTINSERT INTO table (fields) VALUES (values)Adds a new record. Field order must match values order.
UPDATEUPDATE table SET field = value WHERE conditionAlways include WHERE — without it, every record is updated.
DELETEDELETE FROM table WHERE conditionAlways include WHERE — without it, the entire table is deleted.
Worked Example CT4.5a — INSERT, UPDATE, DELETE
-- INSERT: add a new student
INSERT INTO Students (student_id, first_name, last_name, date_of_birth, year_group, email)
VALUES (1050, 'Zara', 'Ali', '2009-04-12', 9, 'zali@school.nsw.edu.au');

-- UPDATE: correct a grade error for one enrolment
UPDATE Enrolments
SET grade = 87
WHERE student_id = 1042 AND subject_id = 3 AND year = 2024;

-- DELETE: remove a student who has left the school
DELETE FROM Students
WHERE student_id = 1031;
The Most Dangerous SQL Statements

Running UPDATE Enrolments SET grade = 0; (missing WHERE) sets every student's grade to zero. Running DELETE FROM Students; (missing WHERE) deletes every student. Always:

  1. Write the WHERE clause first, before the UPDATE/DELETE keyword
  2. Test the WHERE condition with a SELECT statement to see which records it matches
  3. Back up the database before any bulk operation
  4. Use transactions (BEGIN; ... COMMIT; or ROLLBACK;) to undo mistakes

Data Validation

Data validation ensures that data meets required quality standards before it is stored. Invalid data — wrong types, out-of-range values, inconsistent formats — causes incorrect query results, broken calculations, and unreliable decisions. Validation happens at multiple levels:

Layers of Data Validation
LayerValidation typeExample
Database constraintsType checking — only correct data type can be storedyear_group INTEGER — string "Year 9" is rejected
Database constraintsNOT NULL — field must have a valueEvery student must have a student_id
Database constraintsUNIQUE — no duplicate values allowedTwo students cannot share the same email
Database constraintsCHECK — value must satisfy a conditionCHECK (grade BETWEEN 0 AND 100)
Application layerRange checking — value within acceptable boundsDate of birth must be between 1990 and today
Application layerFormat checking — value follows expected patternEmail must contain @ and a domain suffix
Application layerPresence checking — required fields are not blankName, DOB, year group are all mandatory
User interfaceInput masks, dropdowns, date pickersYear group is a dropdown 7–12, not a free text field
Worked Example CT4.5b — Impact of Invalid Data

A school assistant accidentally enters a student's year group as 99 instead of 9. Show the cascade of errors this causes.

1
Year group report: A report grouping students by year now shows a "Year 99" group with 1 student — confusing administrators.
2
Average calculation: AVG(year_group) across all students is skewed upward — any statistics on year groups are now wrong.
3
Filtering failure: WHERE year_group = 9 queries exclude this student — they appear invisible in all Year 9 reports.
4
Prevention: A CHECK constraint CHECK(year_group BETWEEN 7 AND 12) would reject 99 at insertion. A UI dropdown would prevent typing non-integer values entirely.

CT4.6Data Visualisation CT5-DAT-02 · CT5-COM-01

Raw query results, no matter how accurate, rarely communicate insights effectively to non-technical stakeholders. Data visualisation transforms tabular data into charts, maps, and dashboards that make patterns, trends, and outliers immediately visible. Choosing the wrong chart type, however, can obscure or actively mislead.

Inquiry Question

If a bar chart and a line chart can display the same data, what does the choice between them communicate to the reader? When is a chart honest, and when does it mislead — even without altering the underlying numbers?

Choosing the Right Chart

Chart Type Selection Guide
Chart TypeBest forData requirementsCommon misuse
Bar chartComparing discrete categoriesCategorical x-axis, numerical y-axisUsing to show trend over time (use line chart instead)
Line chartShowing trends and change over timeContinuous/ordered x-axis (time), numerical y-axisUsing for unordered categories — implies a connection between points that doesn't exist
Pie / donut chartShowing proportions of a whole (parts that sum to 100%)Categorical data, values sum to 100%Too many slices (>5) become unreadable; using when categories don't sum to whole
Scatter plotShowing relationship (correlation) between two numerical variablesTwo numerical axesImplying causation from correlation
HistogramShowing distribution of a single numerical variableOne numerical variable grouped into intervalsConfused with bar chart — histograms have no gaps between bars
HeatmapShowing patterns across a two-dimensional gridTwo categorical axes, numerical values represented by colourPoor colour choice makes it inaccessible to colour-blind users
Worked Example CT4.6a — Python Data Visualisation with matplotlib

Using the SQL query results from CT4.4 (average grade per subject), create a horizontal bar chart showing class performance. The data has been retrieved from the database and stored in Python lists.

# Visualising school database query results
import matplotlib.pyplot as plt

# Data from SQL query: subject names and their average grades
subjects = ['Mathematics', 'English', 'Science', 'History', 'Computing']
averages = [72.4, 78.1, 69.3, 74.8, 81.5]
colours  = ['#06b6d4' if avg >= 75 else '#e2e8f0' for avg in averages]

fig, ax = plt.subplots(figsize=(8, 4))
bars = ax.barh(subjects, averages, color=colours, edgecolor='white')

# Add value labels to each bar
for bar, avg in zip(bars, averages):
    ax.text(bar.get_width() - 1, bar.get_y() + bar.get_height()/2,
            f'{avg:.1f}%', va='center', ha='right', color='white', fontweight='bold')

ax.axvline(75, color='#be185d', linestyle='--', label='School target (75%)')
ax.set_xlabel('Average Grade (%)')
ax.set_title('2024 Average Grade by Subject')
ax.legend(); ax.set_xlim(0, 100)
plt.tight_layout(); plt.savefig('subject_averages.png', dpi=150)

Key features: colour-coding bars above/below the target threshold; a reference line marking the target; value labels on bars; descriptive title and axis label. Each of these is a deliberate communication decision — not just cosmetic.

Worked Example CT4.6b — Identifying Trends, Patterns, and Outliers

A scatter plot of student study hours vs exam grade shows:

  • Trend: Generally, more study hours correlate with higher grades (positive correlation)
  • Pattern: Students studying 4–6 hours cluster around 65–75%, suggesting diminishing returns at higher study times
  • Outlier: One student studied 12 hours but scored only 43% — investigating this outlier might reveal external factors (illness, test anxiety) that aggregate data masks

Outliers are not errors to be deleted — they are often the most informative data points. Always investigate before removing.

Statistics Connection — Visualisation and Distribution Shape

In Maths Chapter 3 (Section 3.4) you learned to describe distribution shape — symmetric, right-skewed, left-skewed, bimodal. The same analytical vocabulary applies here: when you visualise a histogram of exam scores, you identify whether the distribution is symmetric (class performed consistently) or right-skewed (a few outliers pulled the mean above the median). A scatter plot reveals correlation — the statistical relationship between two variables — which you began studying in Ch. 3 and develop fully in Stage 5 Maths (bivariate data, lines of best fit). SQL gives you the data; visualisation reveals what the data means.

CT4.7Evaluating Data Solutions CT5-DPM-01 · CT5-EVL-01 · CT5-SAF-01

A professional data solution must also be evaluated against its requirements, tested for correctness and performance, and assessed for its social, ethical, and legal implications. This section completes the iterative development cycle for data-focused projects.

  4. Testing and evaluating: Complete an evaluation table (as in CT4.7a) for all requirements. Write a 300-word evaluation covering: what works, what limitations remain, ethical considerations, and one proposed improvement.
