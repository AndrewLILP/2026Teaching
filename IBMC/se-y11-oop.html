SE2.1Comparing Programming Paradigms — Procedural vs OOP SE-11-01 SE-11-03

A programming paradigm is a fundamental style or approach to building programs. The NESA syllabus requires you to understand four paradigms: imperative/procedural, object-oriented, logic, and functional. This chapter focuses on the contrast between procedural and OOP, which is the most practically significant shift in how most software is written.

Inquiry Question

The Unity coin collector game from the depth study had four separate C# script files. Each one was a class. Could you have written the entire game as one 500-line function? Technically yes — so why didn't anyone design it that way?

The Four Paradigms

Programming Paradigms — NESA Reference
ParadigmCore ideaKey featuresExample languages

Object-Oriented (OOP)Data and behaviour bundled into objects; objects communicate by sending messagesClasses, inheritance, encapsulation, polymorphism, message passingPython, Java, C#, C++, Swift

Procedural vs OOP — A Direct Comparison

Worked Example SE2.1a — Same Problem, Two Paradigms

Model a coin with a value, a rotation, and the ability to be collected. Compare procedural and OOP approaches in Python.

Procedural — data and logic separated
# Data stored in a plain dictionary
coin = {
    "value": 5,
    "rotation": 0.0,
    "collected": False
}

# Logic lives in separate functions
def rotate_coin(coin, speed, dt):
    coin["rotation"] += speed * dt

def collect_coin(coin, player):
    if not coin["collected"]:
        player["score"] += coin["value"]
        coin["collected"] = True

# Nothing stops code from doing this:
coin["value"] = -999  # oops — no protection
OOP — data and logic bundled together
class Coin:
    def __init__(self, value: int):
        if value <= 0:
            raise ValueError("Value must be positive")
        self.__value = value   # private
        self.rotation  = 0.0
        self._collected = False

    def rotate(self, speed, dt):
        self.rotation += speed * dt

    def collect(self, player):
        if not self._collected:
            player.score += self.__value
            self._collected = True

    def get_value(self):
        return self.__value

Key differences: In the OOP version, the coin's value is private — external code can read it via get_value() but cannot arbitrarily set it to −999. The coin's behaviour (rotate, collect) travels with its data as a single unit. Add a new coin type? Create a new class. No global functions to hunt down and modify.

When to Choose OOP vs Procedural
OOP is better when…Procedural is better when…
ScaleLarge system with many interacting entities (game with 50 enemy types, 200 item types)Small utility scripts, quick automation tasks, single-purpose tools
ReuseCode needs to be reused across many contexts via inheritance or compositionCode is specific to one task and unlikely to be reused
TeamsMultiple developers work on the same codebase — classes provide clear ownership boundariesSolo developer, short-lived scripts
Data protectionData must be protected from accidental corruption (e.g. game score should never go negative)Data access is not a concern
ModellingThe problem naturally maps to real-world entities (Player, Enemy, Weapon, Level)The problem is primarily a calculation or data transformation pipeline

SE2.2Classes, Objects, Attributes and Methods SE-11-02 SE-11-08

A class is a blueprint or template. An object is a specific instance created from that blueprint. Every object has attributes (its data, also called instance variables or properties) and methods (its behaviour — functions that belong to the class). When a method is called on an object, that is called message passing.

Core OOP Vocabulary
TermDefinitionGame example
ClassA blueprint defining the attributes and methods all objects of this type will haveclass Coin — defines what every coin is and can do
Object / InstanceA specific realisation of a class; occupies memory with its own attribute valuesgold_coin = Coin(value=5) — one specific coin in the scene
AttributeA variable that belongs to an object; stores its current stateself.value, self.rotation, self._collected
MethodA function that belongs to a class; defines what objects of this type can dorotate(), collect(), get_value()
ConstructorA special method called when an object is created; initialises attributes__init__(self, value) — runs when you write Coin(5)
Message passingOne object calling a method on another object — the mechanism by which objects interactgold_coin.collect(player) — the coin receives a "collect" message
Class variableA variable shared by all instances of the class, not specific to one objectCoin.total_spawned — counts how many coins have been created in total
selfReference to the current object instance; used inside methods to access the object's own attributesself.value inside a Coin method refers to this specific coin's value
Worked Example SE2.2a — A Complete Python Class: Player
class Player:
    """Represents the player character in the coin collector game."""

    # Class variable — shared across ALL Player instances
    MAX_SCORE = 9999

    def __init__(self, name: str, start_health: int = 100):
        """Create a new player with a name and starting health."""
        self.name          = name           # public attribute
        self._health       = start_health   # protected (convention)
        self.__score       = 0              # private (name mangled)
        self.coins_held    = 0
        self.is_alive      = True

    # ── Methods ──
    def add_coins(self, amount: int) -> None:
        """Add collected coin value to score (capped at MAX_SCORE)."""
        self.coins_held += 1
        self.__score = min(self.__score + amount, Player.MAX_SCORE)

    def take_damage(self, damage: int) -> None:
        """Reduce health; mark player as dead if health reaches zero."""
        self._health = max(0, self._health - damage)
        if self._health == 0:
            self.is_alive = False

    def get_score(self) -> int:
        """Return current score (private attribute — read-only access)."""
        return self.__score

    def __str__(self) -> str:
        return (f"{self.name} | Score: {self.__score} | "
                f"Health: {self._health} | Coins: {self.coins_held}")

# ── Creating objects and sending messages ──
p1 = Player("Mei")          # constructor called
p2 = Player("Amir", 150)    # different start_health

p1.add_coins(5)              # message passing: p1 receives "add_coins"
p1.take_damage(20)
print(p1)                    # Mei | Score: 5 | Health: 80 | Coins: 1
print(p1.get_score())        # 5
print(Player.MAX_SCORE)       # 9999 — class variable, same for all
Unity / C# — Classes Are Everywhere

In Unity, every script is a class. The four files from the depth study are all classes inheriting from MonoBehaviour. The key difference from Python: C# is statically typed — you must declare the type of every attribute upfront.

// C# equivalent of the Python Player class above
public class Player : MonoBehaviour
{
    // Class constant (static readonly)
    public static readonly int MAX_SCORE = 9999;

    // Attributes — declared with explicit types
    public string  name;                    // public
    protected int _health     = 100;      // protected
    private int   __score     = 0;        // private
    public int    coinsHeld   = 0;
    public bool   isAlive     = true;

    // Methods
    public void AddCoins(int amount) {
        coinsHeld++;
        __score = Mathf.Min(__score + amount, MAX_SCORE);
    }
    public int GetScore() => __score;  // C# expression-bodied method
}

SE2.3Encapsulation and Abstraction SE-11-02 SE-11-08

Encapsulation bundles data and the methods that operate on that data into a single unit (a class), and controls access to the data from outside that unit. Abstraction hides implementation complexity behind a simple interface — you use the interface without needing to know how it works internally.

Access Modifiers — Three Levels of Visibility
LevelPython conventionC# keywordAccessible fromUML symbol
PublicNo underscore: self.namepublicAnywhere — inside or outside the class+
ProtectedSingle underscore: self._healthprotectedInside the class and its subclasses#
PrivateDouble underscore: self.__scoreprivateOnly inside the class itself-

Python doesn't enforce access modifiers like C# does — _ and __ are conventions that signal intent to other developers. C# enforces them at compile time.

Worked Example SE2.3a — Properties: Controlled Access

A property (Python: @property; C#: get/set accessors) is a method that looks like an attribute — providing controlled, validated access to private data. This is encapsulation in its most refined form.

class Player:
    def __init__(self, name):
        self.name      = name
        self.__score   = 0
        self.__health  = 100

    # Getter — read access to private __health
    @property
    def health(self) -> int:
        return self.__health

    # Setter — write access WITH validation
    @health.setter
    def health(self, value: int):
        if not 0 <= value <= 100:
            raise ValueError(f"Health must be 0–100, got {value}")
        self.__health = value
        if self.__health == 0:
            print(f"{self.name} has died!")

# Using the property — looks like an attribute, runs a method
p = Player("Mei")
p.health = 75    # calls the setter — validates first
print(p.health)  # calls the getter → 75
p.health = -5   # → ValueError: Health must be 0–100
Worked Example SE2.3b — Abstraction: Using Without Understanding

Abstraction means providing a simple interface that hides complex implementation. The user of a class doesn't need to know how it works internally — only what it does.

class AudioManager:
    """Plays game audio. Internal implementation hidden."""

    def __init__(self):
        self.__audio_buffer   = {}   # hidden implementation detail
        self.__sample_rate    = 44100
        self.__active_channels = []

    def play(self, sound_name: str) -> None:
        """Play a sound. Caller doesn't need to know HOW."""
        self.__load_if_needed(sound_name)  # complex, but hidden
        self.__mix_and_output(sound_name)  # complex, but hidden

    def stop_all(self) -> None:
        """Stop all sounds. Simple interface, complex implementation."""
        self.__active_channels.clear()

    def __load_if_needed(self, name): pass  # private - hidden
    def __mix_and_output(self, name): pass  # private - hidden

# The Coin class USES AudioManager — it only needs the simple interface
audio = AudioManager()
audio.play("coin_pickup")  # No idea how this works — don't need to know

This is exactly how AudioSource.PlayClipAtPoint() works in Unity — you call one method; Unity's audio subsystem handles buffering, mixing, spatial 3D audio, and channel management internally.

SE2.4Inheritance and Generalisation SE-11-02 SE-11-08

Inheritance allows one class (the subclass or child class) to acquire all attributes and methods of another class (the superclass or parent class), and then extend or override them. Generalisation is the process of identifying what multiple classes have in common and extracting that into a parent class.

Inheritance Terminology

Worked Example SE2.4a — Game Enemy Hierarchy

All enemies share: health, speed, take_damage(), is_alive(). Specific enemy types add unique behaviour. Generalise the common parts into a parent Enemy class.

from abc import ABC, abstractmethod

class Enemy(ABC):   # Abstract base class — cannot instantiate directly
    """Base class for all enemy types."""

    def __init__(self, name: str, health: int, speed: float):
        self.name     = name
        self._health  = health
        self.speed    = speed

    def take_damage(self, amount: int) -> None:
        """Reduce health. Inherited by all enemy subclasses."""
        self._health = max(0, self._health - amount)

    def is_alive(self) -> bool:
        return self._health > 0

    @abstractmethod
    def attack(self, player) -> None:
        """Each enemy type MUST implement its own attack."""
        pass

    def __str__(self) -> str:
        return f"{self.name} [HP:{self._health} SPD:{self.speed}]"


class Goblin(Enemy):  # Inherits from Enemy
    def __init__(self):
        super().__init__("Goblin", health=30, speed=3.5)  # call parent __init__
        self.steal_chance = 0.25   # Goblin-specific attribute

    def attack(self, player) -> None:  # Override abstract method
        player.take_damage(8)
        print(f"{self.name} slashes for 8 damage!")


class Troll(Enemy):   # Different subclass
    def __init__(self):
        super().__init__("Troll", health=120, speed=1.0)
        self.regen_rate = 5    # Trolls regenerate health

    def attack(self, player) -> None:
        player.take_damage(25)
        print(f"{self.name} smashes for 25 damage!")

    def regenerate(self) -> None:  # Troll-only method — extends parent
        self._health = min(120, self._health + self.regen_rate)


class BossGoblin(Goblin):  # Multi-level: Goblin → BossGoblin
    def __init__(self):
        super().__init__()          # calls Goblin's __init__
        self.name       = "Boss Goblin"
        self._health    = 200       # override parent defaults
        self.steal_chance = 0.5    # better at stealing

    def attack(self, player) -> None:  # Override Goblin's attack
        super().attack(player)    # do Goblin's attack first
        player.take_damage(12)  # then deal bonus damage
        print("... and strikes again for 12 bonus damage!")
Unity / C# — MonoBehaviour Is a Superclass

When you write public class coin : MonoBehaviour in Unity, you are inheriting from Unity's MonoBehaviour superclass. This gives your script access to Update(), Start(), OnTriggerEnter(), Destroy() and all the other Unity lifecycle methods — without you having to write them. The same inheritance hierarchy you built in Python above is the exact pattern Unity uses internally for its own component system.

// The exact same enemy hierarchy in C# / Unity
public abstract class Enemy : MonoBehaviour
{
    public string    enemyName;
    protected int    health;
    public float     speed;

    public void TakeDamage(int amount) =>
        health = Mathf.Max(0, health - amount);

    public abstract void Attack(Player player);  // must override
}

public class Goblin : Enemy
{
    public override void Attack(Player player) {
        player.TakeDamage(8);
    }
}

SE2.5Polymorphism and Message Passing SE-11-02 SE-11-08

Polymorphism (from Greek: "many forms") means that the same message can be sent to objects of different types, and each object responds in its own appropriate way. It is what allows a function to work with a Goblin, a Troll, or a BossGoblin without needing to know or care which specific type it is — as long as they all respond to the same interface.

Worked Example SE2.5a — Polymorphism in the Enemy Wave System
def process_enemy_wave(enemies: list, player) -> None:
    """Attack the player with every alive enemy.
    Works for ANY enemy type — no type checking required."""
    for enemy in enemies:
        if enemy.is_alive():
            enemy.attack(player)  # polymorphism: each calls its OWN attack

# A mixed wave of different enemy types
wave = [Goblin(), Troll(), BossGoblin(), Goblin()]
p = Player("Mei")

process_enemy_wave(wave, p)
# Goblin slashes for 8 damage!
# Troll smashes for 25 damage!
# Boss Goblin slashes for 8... and strikes again for 12!
# Goblin slashes for 8 damage!

process_enemy_wave() does not contain a single if isinstance(enemy, Goblin) check — it doesn't need to. Each enemy knows how to attack. Adding a new Dragon enemy type requires zero changes to this function — just create the Dragon class with its own attack(). This is the power of polymorphism.

Worked Example SE2.5b — Duck Typing: Python's Implicit Polymorphism

Python uses duck typing — "if it walks like a duck and quacks like a duck, it is a duck." You don't need a common superclass for polymorphism to work; you just need objects that respond to the same method name.

class ParticleSystem:
    """Spawns visual effects."""
    def play(self):
        print("✨ Particle burst!")

class AudioClip:
    """Plays a sound effect."""
    def play(self):
        print("🔊 Pickup sound!")

class UIAnimation:
    """Animates a UI element."""
    def play(self):
        print("🎬 Score popup animates!")

# All have play() — none share a superclass — still polymorphic
effects = [ParticleSystem(), AudioClip(), UIAnimation()]
for effect in effects:
    effect.play()  # each responds in its own way

SE2.7Design Patterns in Games SE-11-01 SE-11-03

Design patterns are reusable, proven solutions to common software design problems. They are not libraries or frameworks — they are templates for how to structure code. The GoF (Gang of Four) catalogue defines 23 patterns; four appear explicitly in the coin collector scripts and are essential knowledge for game development.

Creational

Singleton

Ensures exactly one instance of a class exists, with a global access point. Used in GameManager and UIManager — there can only be one game manager.

Behavioural

Observer

Objects subscribe to events; when the event fires, all subscribers are notified automatically. Unity's UnityEvent and C# event/delegate implement this. Decouples the event source from its handlers.

Behavioural

State

An object changes its behaviour when its internal state changes — as if it changed its class. Used for game states (Menu/Playing/Paused) and enemy AI (Patrol/Chase/Attack/Flee).

Creational

Object Pool

Pre-create a pool of reusable objects instead of instantiate/destroy. Bullets, particles, and enemies are pooled in AAA games to avoid garbage collection stutters at runtime.

Worked Example SE2.7a — Singleton Pattern in Python

Implement the exact pattern used by GameManager.cs and UIManager.cs in Python.

class GameManager:
    """Singleton: only one GameManager can exist."""
    _instance: "GameManager | None" = None  # class variable

    def __new__(cls):
        """Called before __init__. Return existing instance if present."""
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance._total_coins = 0
        return cls._instance  # always returns THE SAME object

    def update_coin_count(self, n: int) -> None:
        self._total_coins = n

    def get_coin_count(self) -> int:
        return self._total_coins

# Proof: both variables point to the SAME object
gm1 = GameManager()
gm2 = GameManager()
gm1.update_coin_count(42)
print(gm2.get_coin_count())   # 42 — same instance!
print(gm1 is gm2)              # True
Worked Example SE2.7b — Observer Pattern: Event System
from typing import Callable, List

class EventSystem:
    """Simple observer: publish/subscribe event bus."""
    def __init__(self):
        self._listeners: dict[str, List[Callable]] = {}

    def subscribe(self, event: str, callback: Callable) -> None:
        self._listeners.setdefault(event, []).append(callback)

    def publish(self, event: str, **data) -> None:
        for cb in self._listeners.get(event, []):
            cb(**data)

# Wire up: UIManager subscribes to coin events
events = EventSystem()
events.subscribe("coin_collected", lambda total: print(f"UI: Coins: {total}"))
events.subscribe("coin_collected", lambda total: print(f"Achievements: {total} coins!"))

# When a coin is collected, ALL subscribers notified automatically
events.publish("coin_collected", total=5)
# UI: Coins: 5
# Achievements: 5 coins!

Now adding a new feature (e.g. a sound effect on coin collection) requires zero changes to Coin or GameManager — just subscribe a new listener. This is the Observer pattern's key benefit: loose coupling.

SE2.8Testing OOP Systems SE-11-06 SE-11-07

Testing OOP systems requires new strategies compared to procedural testing. You test individual units (classes and methods) in isolation, then test how they integrate, then test the whole system from a user's perspective. The principle of encapsulation actually makes OOP code easier to test — each class is a self-contained, independently testable unit.
