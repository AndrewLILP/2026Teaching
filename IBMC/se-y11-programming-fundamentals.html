SE1.1Software Development Lifecycle SE-11-01 SE-11-03

Before writing a single line of code, professional software engineers follow a structured process. The Software Development Lifecycle (SDLC) is the sequence of phases that takes a vague idea and transforms it into reliable, maintainable software. Understanding each phase — and the documents it produces — is foundational to every course you will encounter in Software Engineering.

Inquiry Question

Why do large software projects fail? Studies consistently show that unclear requirements, poor planning, and inadequate testing — not technical difficulty — are the primary causes of failure. How does the SDLC address each of these?

The Seven Phases

1

Requirements Definition

Identify what the software must do. Gather stakeholder needs through interviews, observation, and surveys. Produce a Requirements Specification Document. Requirements are functional (what the system does) or non-functional (how well it does it — speed, reliability, security).

🎮 Game equivalent: the Game Design Document (GDD) — player personas, game mechanics, win/lose conditions, platform constraints. "The player must collect 20 coins within 60 seconds" is a functional requirement.
2

Determining Specifications

Translate requirements into precise, measurable technical specifications. Data types, data structures, interfaces, performance targets. Produce a Technical Specification Document. This is where vague needs become unambiguous engineering constraints.

🎮 Game equivalent: frame rate target (60 FPS), data schema for coinValue: int, target platform (PC, iOS), Unity version, memory budget per scene.
3

Design

Determine how the software will meet the specifications. Produce algorithm designs (pseudocode, flowcharts), data dictionaries, data flow diagrams, and system architecture diagrams. This phase is language-independent — design exists before any code is written.

🎮 Game equivalent: Unity scene hierarchy design, class diagrams for Coin / GameManager / UIManager, level layout blueprint, pseudocode for the coin collection algorithm.
4

Development

Translate designs into working code in a chosen programming language. Follows coding standards, comments code appropriately, uses version control (Git). In Agile, this phase cycles with testing repeatedly in short sprints.

🎮 Game equivalent: writing coin.cs, GameManager.cs, building Unity scenes, importing assets, configuring the physics engine.
5

Integration

Combine individually developed components into a working whole. Often reveals interface mismatches — one module assumes data arrives in a format another module doesn't produce. Integration testing verifies that components work together.

🎮 Game equivalent: connecting PlayerCoinCollector.AddCoins() to GameManager.UpdateCoinCount() to UIManager.UpdateCoinUI() — the chain must work end-to-end.
6

Testing and Debugging

Systematically verify that the software meets its requirements under all expected inputs, including edge cases and invalid data. Document test cases, expected outputs, actual outputs, and pass/fail results. Fix defects (bugs) and re-test.

🎮 Game equivalent: a playtesting session — "What happens when the player collects the last coin? What if they fall off the map? What if coinValue is 0?"
7

Installation and Maintenance

Deploy to end users and support the software over its lifetime. Bug fixes, security patches, performance improvements, and new features are ongoing maintenance activities. Software is rarely "finished" — it evolves.

🎮 Game equivalent: shipping the build on Steam/itch.io; post-launch patches; the Year 12 project extension that adds features based on player feedback.

SE1.2Designing Algorithms SE-11-01 SE-11-02

An algorithm is a finite, unambiguous sequence of steps that solves a problem. Before writing code in any language, engineers design algorithms in language-neutral notations — pseudocode and flowcharts. This separates the logic of the solution from the syntax of a particular programming language.

NESA Pseudocode Conventions

The HSC examination uses a standardised pseudocode format. You must be able to both write and interpret pseudocode in this format. Key rules:

NESA Pseudocode Rules
  • Keywords are written in ALL CAPITALS: BEGIN, END, IF, ELSE, ENDIF, WHILE, ENDWHILE, FOR, TO, NEXT, REPEAT, UNTIL, CASEWHERE, OTHERWISE, ENDCASE, AND, OR, NOT
  • Structural elements come in pairs: every BEGIN has an END; every IF has an ENDIF; every WHILE has an ENDWHILE
  • Indentation shows the body of each control structure
  • Subroutines are declared as BEGIN subroutineName(parameters) ... END subroutineName
  • Comments begin with //
  • Complex algorithms start with a concise mainline that calls named subroutines — detail goes in the subroutine, not the mainline

Algorithmic Design Strategies

Divide and conquer breaks a problem into smaller independent sub-problems, solves each, then combines results. Merge sort and binary search are classic examples. In game development, rendering a frame divides the scene into independent objects, renders each, then composites the result.

Backtracking explores a path and, if it leads to a dead end, "backtracks" to try an alternative. Maze-solving algorithms and Sudoku solvers use backtracking. In games, AI pathfinding that hits an obstacle backs up and tries another route.

SE1.3Data for Software Engineering SE-11-02 SE-11-04

Computers only understand one thing at the hardware level: sequences of 0s and 1s. Every data type — text, images, numbers, audio — is ultimately encoded in binary. Understanding how data is represented and typed is essential for writing correct programs and designing appropriate data storage.

Number Systems

Binary, Decimal, and Hexadecimal
SystemBaseDigitsExample: 42Used for
Binary20, 100101010CPU instructions, memory addresses, bit flags in game engines
Decimal100–942Human-readable numbers; most input/output
Hexadecimal160–9, A–F0x2AColour values (#FF5733), memory addresses, binary file headers

Converting binary to decimal: \(00101010_2 = 0(128) + 0(64) + 1(32) + 0(16) + 1(8) + 0(4) + 1(2) + 0(1) = 32 + 8 + 2 = 42_{10}\)

Converting decimal to hexadecimal: \(42 \div 16 = 2\text{ rem }10 \Rightarrow 42_{10} = 2A_{16}\)

Unity / Game Design Connection — Hex Colours and Bit Flags

In Unity, colours are expressed as RGBA values: Color.red = #FF0000FF = each channel 0–255 in hex. Bit flags use binary's ability to pack multiple boolean states into one integer — Unity's physics layer mask is a 32-bit integer where each bit represents one layer: layerMask = 1 << 6 sets bit 6 (layer 6) to 1. Understanding binary is not abstract — it is how Unity's internal systems work.

Two's Complement

Two's complement is the standard way computers represent negative integers in binary. To negate a number: flip all bits (one's complement), then add 1.

Worked Example SE1.3a — Two's Complement

Represent −42 as an 8-bit two's complement number.

1

Start with +42: 00101010

2

Flip all bits (one's complement): 11010101

3

Add 1: 11010101 + 00000001 = 11010110

4

So 11010110 = −42 in 8-bit two's complement. An 8-bit signed integer can represent −128 to +127.

SE1.4Data Structures SE-11-02 SE-11-08

A data structure organises a collection of data items so they can be stored, accessed, and modified efficiently. Choosing the right structure for the problem is one of the most consequential decisions in software design.

Arrays and Lists

An array is a fixed-size, ordered collection where every element has the same type. Arrays are stored in contiguous memory, making indexed access \(O(1)\). A list (Python's list) is a dynamic array — it resizes automatically and can hold mixed types.

Python — Arrays and Lists in a Game Context
# 1D list — enemies in current wave
enemies = ["Goblin", "Orc", "Troll", "Dragon"]

print(enemies[0])        # "Goblin" — indexed from 0
print(enemies[-1])       # "Dragon" — negative index from end
print(len(enemies))       # 4
enemies.append("Skeleton") # add to end
enemies.remove("Orc")     # remove by value

# 2D list — game grid (3×3 tile map)
grid = [
    [1, 1, 0],   # 1 = wall, 0 = open
    [0, 0, 0],
    [0, 1, 1]
]
print(grid[1][2])          # 0 — row 1, col 2 (open)

Dictionaries (Hash Tables)

A dictionary stores key-value pairs. Keys must be unique. Lookup by key is \(O(1)\) average, because Python uses a hash table internally. Ideal for: player inventories, game settings, high-score records.

Python — Dictionaries for Game State
# Player inventory — item name: quantity
inventory = {"gold_coin": 12, "health_potion": 3, "key": 1}

print(inventory["gold_coin"])     # 12
inventory["gold_coin"] += 5       # add 5 coins
inventory["sword"] = 1            # add new item
del inventory["key"]              # remove item

print("health_potion" in inventory) # True

# Iterate all items
for item, qty in inventory.items():
    print(f"{item}: {qty}")

Trees and Records

A tree is a hierarchical structure with a root node, branch nodes, and leaf nodes. Each node has at most one parent but any number of children. A record is a fixed structure grouping related fields of different types — the equivalent of a Python namedtuple or dataclass, or a C struct.

def validate_password(correct):
    attempts = 0
    while True:         # REPEAT
        pwd = input("Password: ")
        attempts += 1
        if pwd != correct:
            print("Incorrect.")
        if pwd == correct:  # UNTIL
            break
    print(f"Granted. Attempts: {attempts}")

SE1.6Subprograms — Functions and Procedures SE-11-02 SE-11-08

The NESA syllabus uses the terms subroutine, module, subprogram, and procedure interchangeably to mean a named block of code that performs a specific task. A function is a subprogram that returns a value. A procedure performs an action but does not return a value.

Parameter Passing

A parameter is a variable in the subroutine definition that receives a value when the subroutine is called. An argument is the actual value passed when calling. There are two passing mechanisms:

  • Pass by value — a copy of the argument is passed. Changes inside the subroutine do not affect the original variable. Python passes immutable types (int, str, bool) this way.
  • Pass by reference — a reference to the original object is passed. Changes inside the subroutine do affect the original. Python passes mutable types (list, dict) this way.
Worked Example SE1.6a — Subprograms from the Coin Collector Scripts

The provided C# scripts already demonstrate good subprogram design. Here they are as Python equivalents, showing parameter passing, return values, and the role of each subprogram.

# --- Equivalent to PlayerCoinCollector.cs ---
def add_coins(current_coins: int, amount: int) -> int:
    """Add amount to current_coins and return the new total.
    Pass by value — original variable unchanged outside."""
    return current_coins + amount

# --- Equivalent to UIManager.cs ---
def update_coin_ui(total: int) -> None:
    """Update the display. Returns nothing — this is a procedure."""
    print(f"Coins: {total}")

# --- Equivalent to GameManager.cs ---
def update_coin_count(state: dict, new_amount: int) -> None:
    """Mutates the state dict (pass by reference — dict is mutable)."""
    state["total_coins"] = new_amount  # affects original dict
    update_coin_ui(state["total_coins"])

# --- Main program ---
game_state = {"total_coins": 0}
coins_held = 0

coins_held = add_coins(coins_held, 5)   # collect a coin worth 5
update_coin_count(game_state, coins_held) # → prints "Coins: 5"

Python — Implementation with Error Handling
def collect_coin(current_coins: int, coin_value: int) -> int:
    """Add coin_value to current_coins. Returns new total capped at 9999.
    Raises ValueError for negative inputs, TypeError for non-integers."""
    if not isinstance(current_coins, int) or not isinstance(coin_value, int):
        raise TypeError("Both arguments must be integers")
    if current_coins < 0 or coin_value < 0:
        raise ValueError("Coin values must be non-negative")
    return min(current_coins + coin_value, 9999)

# Test cases

except ValueError as e:
    print(f"Caught: {e}")

SE1.8Large-Scale Software Development SE-11-01 SE-11-03 SE-11-09

A single developer writing a small script is very different from a 200-person studio building an AAA game. Large-scale software requires systems for managing change, maintaining code quality, coordinating concurrent work, and keeping documentation aligned with the codebase. These practices are the difference between software that grows and software that collapses under its own weight.

Version Control with Git

Git is the universal version control system. Every professional development team uses it. In a game studio, Git (or Unity Version Control) tracks every change to every file — enabling teams to work simultaneously without overwriting each other, and to revert to any previous state if a change breaks the build.

Unity + Git — What Goes in .gitignore

A Unity project generates many gigabytes of files that should not be version-controlled — they are either too large, auto-generated, or platform-specific. The .gitignore file tells Git to ignore them.

# Standard Unity .gitignore entries
[Ll]ibrary/          # Unity's asset import cache — auto-regenerated
[Tt]emp/             # Temporary build files
[Bb]uild/            # Build output (distribute separately)
*.csproj             # IDE project files — auto-generated
*.meta               # DO track these — they store Unity asset GUIDs
UserSettings/        # Personal IDE preferences

What you MUST commit: Assets/, ProjectSettings/, Packages/, and all .meta files. Without .meta files, Unity loses track of which scripts are attached to which GameObjects.

Python — Well-Documented Game Module
"""
coin_system.py — Coin collection logic for the NSW Stage Textbook game.

Follows PEP 8 style guide. All functions validated with unit tests
in test_coin_system.py. See README.md for usage.

Author: Year 11 Software Engineering Student
Last updated: 2026-04-27
"""

MAX_SCORE = 9999   # Named constant — no magic numbers
COIN_VALUES = {"bronze": 1, "silver": 5, "gold": 10}

def collect_coin(current_score: int, coin_type: str) -> int:
    """Collect a coin of the given type and return the new score.

    Args:
        current_score: The player's current score (0–MAX_SCORE).
        coin_type: One of "bronze", "silver", or "gold".

    Returns:
        New score after collection, capped at MAX_SCORE.

    Raises:
        ValueError: If coin_type is not a valid type.
    """
    if coin_type not in COIN_VALUES:
        raise ValueError(f"Unknown coin type: {coin_type!r}")
    return min(current_score + COIN_VALUES[coin_type], MAX_SCORE)
