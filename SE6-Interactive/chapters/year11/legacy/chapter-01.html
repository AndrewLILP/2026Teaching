Chapter 01 · Programming Fundamentals

Thinking Like a Software Engineer

Before writing a single line of code, great software engineers learn to think systematically. This chapter covers the mindset, planning tools, and processes that distinguish professional engineers from hobbyist coders — and shows how these apply directly to game development.

Focus Area: Programming Fundamentals
Outcomes: SE-11-01 · SE-11-02 · SE-11-06 · SE-11-08
Estimated time: 6 hours
Sections: 8 + 2 activities
Section 1.1

What is Software Engineering?

Software engineering is the systematic application of engineering principles to the design, development, testing, and maintenance of software. It is not just programming — it is a discipline that treats software development as a rigorous engineering process, where quality, reliability, and maintainability matter as much as whether the program runs.

Definition — IEEE
Software engineering is "the application of a systematic, disciplined, quantifiable approach to the development, operation, and maintenance of software." — IEEE Standard 610.12

Consider the difference between a student writing a quick Python script and a team of engineers shipping a game like Minecraft. Both involve code — but only one involves requirements documents, design specifications, version control, automated testing, deployment pipelines, and post-release maintenance. That difference is software engineering.

🎮
Game Development Connection

A modern game engine like Unity is itself a product of software engineering at its most demanding. Millions of lines of C++ and C# code, maintained by hundreds of engineers, serving millions of developers — all requiring the same disciplined processes you will learn in this course. When you use Unity, you are using software engineering.

Software engineers must be able to communicate their ideas before they write any code. In this chapter you will learn the two most important planning tools in the engineer's toolkit: pseudocode and flowcharts. You will also learn how different software development approaches structure the work from start to finish.

Section 1.2

Software Development Steps

Every software project — from a student assignment to a commercial game — moves through a common set of steps. Understanding these steps is the first requirement of outcome SE-11-01: describing methods used to plan, develop and engineer software solutions.

The standard software development steps are:

The 8 Software Development Steps
1. Requirements definition — What does the software need to do? Who will use it?
2. Determining specifications — Detailed functional and performance requirements.
3. Design — Architecture, data structures, algorithms, UI mockups.
4. Development — Writing and documenting the actual code.
5. Integration — Combining components; ensuring they work together.
6. Testing and debugging — Finding and fixing faults; verifying requirements.
7. Installation — Deploying to the target environment or distributing to users.
8. Maintenance — Ongoing bug fixes, updates, and improvements.

These steps are most visible in the Waterfall development approach, where each step is completed before the next begins. In practice, many projects revisit earlier steps as new information emerges. However, understanding the full sequence is essential — you must know where you are in the process at all times.

Section 1.3

Development Approaches

There is no single way to work through the software development steps. Different development approaches organise those steps differently, depending on project size, client needs, and how well requirements are understood upfront. The NSW syllabus requires you to understand Waterfall, Agile, and the WAgile hybrid.

Waterfall
Sequential · Structured
Each phase is completed fully before the next begins — like water falling down a cliff. Requirements are locked in early. Changes late in the process are expensive. Best for well-understood, stable projects with fixed requirements.
When: Large-scale, government or safety-critical systems where all requirements are known.
Agile
Iterative · Flexible
Work is broken into short sprints (1–4 weeks), each delivering a working increment. Requirements evolve with client feedback. Teams collaborate daily. The most widely used approach in professional software and game development.
When: Products with evolving requirements — games, apps, startups — where speed and adaptability matter.
WAgile
Hybrid · Pragmatic
A hybrid of Waterfall and Agile. The project begins with Waterfall-style planning and requirements definition, then switches to Agile sprints for development. Gives structured foundations with flexible execution.
When: Large projects requiring upfront planning but flexible delivery — enterprise software upgrades.
🕹️
How Game Studios Work

Most professional game studios (Valve, CD Projekt Red, Riot Games) use Agile or a custom variant. Work is broken into sprints with a playable build delivered at the end of each cycle — so the team always has something to show and test. Your Year 12 Software Engineering Project will use the same sprint-based workflow.

Section 1.4

Online Code Collaboration Tools

Modern software is rarely written alone. Professional engineers use online code collaboration tools to manage code across teams, track changes, and work simultaneously on the same project without overwriting each other's work.

Key Tool — Git & GitHub
Git is a distributed version control system — it tracks every change made to a codebase, who made it, and when. GitHub is the most popular platform for hosting Git repositories online. You will use Git for all projects in this course, including your Unity game.

Key concepts: commit (save a snapshot), push (upload to server), pull (download updates), branch (independent line of work), merge (combine branches).

When researching collaboration tools, consider: How does it support simultaneous editing? How are conflicts resolved? Is it monitored or administered by a third party? How does it contribute to team efficiency? These are the kinds of questions the syllabus expects you to investigate and document.

Other important collaboration tools include Jira (project management and sprint tracking), Confluence (documentation), and Slack (team communication) — all widely used in the game industry.

Section 1.5

Designing Algorithms

An algorithm is a precise, finite sequence of instructions designed to solve a problem. Every program you will ever write is built on algorithms. Before writing code, engineers design algorithms to think through the logic without being distracted by programming syntax.

The key algorithmic design strategies in this course are:

Algorithmic Design Strategies
Computational thinking — Breaking a problem into logical, solvable steps.
Decomposition — Splitting a complex problem into smaller subproblems.
Divide and conquer — Recursively dividing a problem until it becomes trivial to solve.
Backtracking — Exploring possible solutions and undoing choices that don't work.
Pattern recognition — Identifying recurring structures that can be reused.

Algorithms contain three fundamental building blocks — known as control structures. Every algorithm, no matter how complex, is built from combinations of these three:

The 3 Control Structures
Sequence — Steps executed one after another, in order.
Selection — A decision point: IF a condition is true THEN do one thing, ELSE do another.
Repetition (Iteration) — Repeating a set of steps: WHILE a condition is true, or REPEAT UNTIL a condition is met.
Section 1.6

Pseudocode

Pseudocode is a structured, English-like way of describing an algorithm. It uses capitalised keywords and indentation to show the logic clearly — without the strict syntax rules of a programming language. Pseudocode is assessed in the HSC exam.

HSC Pseudocode Rules
• All keywords are written in CAPITALS: BEGIN, END, IF, THEN, ELSE, ENDIF, WHILE, ENDWHILE, REPEAT, UNTIL, FOR, ENDFOR
• Structural elements always come in pairs: every BEGIN has an END, every IF has an ENDIF
• Indentation shows the structure of control statements
• Variable assignment uses the arrow: ← (e.g. score ← 0)
• Complex algorithms start with a clear mainline that calls subroutines

Here is a complete pseudocode algorithm for a game loop — the core cycle that runs every real-time game, including every Unity game you will build:

Pseudocode — Unity-style Game Loop
// Mainline algorithm BEGIN GameLoop InitialiseGame() playerLives ← 3 score ← 0 gameRunning ← TRUE WHILE gameRunning = TRUE ProcessInput() UpdateGameState() RenderFrame() IF playerLives = 0 THEN gameRunning ← FALSE ENDIF ENDWHILE DisplayGameOver(score) END GameLoop // Subroutine BEGIN UpdateGameState MoveEnemies() CheckCollisions() IF collision = TRUE THEN playerLives ← playerLives - 1 PlaySound("hit") ENDIF END UpdateGameState

Section 1.7

Flowcharts

A flowchart is a diagram that visually represents an algorithm, read from top to bottom and left to right. Where pseudocode describes logic in words, flowcharts make the flow of control immediately visible — especially useful for communicating algorithms to non-programmers or clients.

Flowchart Symbol Reference
Terminator (start/end)
Process
Decision (diamond)
Input/Output
Subroutine / subprogram

When reading or drawing flowcharts, always label the branches leaving a decision diamond with True/False or Yes/No to remove ambiguity. Flowcharts should only use the standard symbols — avoid decorative or non-standard shapes in HSC work.

Section 1.8

Programming Paradigms

A programming paradigm is a fundamental style or approach to programming. Different paradigms reflect different ways of thinking about computation. You are required to be aware of four paradigms for the HSC — but the course primarily uses the imperative and object-oriented paradigms.

Object-Oriented
Classes · Objects · Messages
Models the world as interacting objects with state (attributes) and behaviour (methods). Python and C# (Unity) are both OOP languages. Chapters 5–7 focus on this paradigm.
Know how to: define classes, use inheritance, polymorphism, encapsulation.
Imperative
Commands · Sequence · State
Programs are sequences of commands that change program state. This is the foundation of procedural programming — variables, assignments, loops, and conditionals.
Know how to: use control structures, variables, assignment, expressions, subroutines.
Functional
Functions · Immutable · Pure
Programs are built from functions that avoid shared state and mutable data. Functions are first-class values — they can be passed as arguments or returned.
Know how to: call functions, use recursion, use functions as first-class objects.
Logic
Facts · Rules · Inference
Programs declare facts and rules; the engine infers answers. Prolog is the classic logic language. Useful for AI, expert systems, and constraint-solving problems.
Know how to: define facts, create rules, display solutions.
Key Insight — Paradigms in Unity
Unity's C# scripting uses object-oriented paradigm as its foundation (every script is a class), but also supports functional elements (LINQ queries, lambda expressions) and imperative constructs. Real-world code blends paradigms pragmatically.

Chapter 1 Quiz

Test your understanding of Chapter 1. Select the best answer for each question.

Knowledge Check — Chapter 1 5 Questions
1. Which software development step involves identifying exactly what the software must do and how well it must perform?
A Requirements definition
B Determining specifications
C Design
D Integration
Correct — B: Determining specifications. Requirements definition identifies what the software needs to do. Determining specifications takes it further by precisely defining how it must do it — functional requirements, performance benchmarks, and constraints.
2. A game studio delivers a playable build to their publisher at the end of every two-week cycle and adjusts features based on feedback. Which development approach are they using?
A Waterfall
B Agile
C WAgile
D RAD
Correct — B: Agile. Short cycles (sprints) delivering working increments, with ongoing client/publisher feedback, is the defining characteristic of Agile. Waterfall would complete all design before any development begins.
3. Which of the following is NOT a valid pseudocode keyword according to the NSW HSC course specifications?
A WHILE
B ENDIF
C LOOP
D REPEAT
Correct — C: LOOP. The NSW HSC pseudocode standard uses WHILE/ENDWHILE and REPEAT/UNTIL for iteration. LOOP is not a recognised keyword. Using non-standard keywords in the HSC exam will cost marks.
4. In a flowchart, which symbol represents a decision point (such as an IF condition)?
A Rectangle
B Diamond
C Oval / Ellipse
D Parallelogram
Correct — B: Diamond. The diamond (rhombus) shape represents a decision. Oval/Ellipse = terminator (start/end). Rectangle = process. Parallelogram = input or output.
5. Which programming paradigm treats functions as first-class objects and avoids mutable shared state?
A Imperative
B Object-oriented
C Functional
D Logic
Correct — C: Functional. The functional paradigm treats functions as first-class values and emphasises immutability (no shared mutable state). Logic uses facts and rules. Imperative uses sequential commands. OOP uses objects and message passing.
