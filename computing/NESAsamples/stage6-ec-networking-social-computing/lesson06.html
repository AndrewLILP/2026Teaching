The hook

Devices connect to form a network — you've taken that for granted since Lesson 1. But how they connect to each other is itself a design decision, and it's one with real consequences: what happens if one device fails, how easily the network can grow, and how much it costs to build in the first place. Today you'll meet the handful of patterns enterprises actually choose between.

Part C · Network connectivity

Client-server vs peer-to-peer

Topology describes the shape of the connections. Architecture model describes who's actually providing services to whom.

Client

Sends a request — "show me the latest listings"

→
Server

Processes the request and provides the data

→
Client

Receives the response and displays it

That's client-server architecture: client devices request data or services, and a server provides them. Hatchly's app works exactly this way — a phone (the client) requests data from Hatchly's cloud-hosted backend (the server) — and notice the resemblance to star topology, with the server playing the role of the hub.

Peer-to-peer (P2P) architecture has no dedicated server at all: every device connects to others as an equal, requesting and providing data interchangeably. File-sharing networks and some mesh Wi-Fi systems work this way — and notice the resemblance to mesh topology, with no single central point.

Teacher tip

The topology/architecture pairing isn't a strict rule — a client-server system could technically run over a different physical topology — but the resemblance is a genuinely useful way to build intuition for both ideas at once.
