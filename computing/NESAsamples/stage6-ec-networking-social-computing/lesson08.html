Graph Theory for Network Optimisation: Adjacency, Centrality, Connectedness

Lesson 2 introduced nodes, edges, degree and paths. Today is the same graph language, used for a different purpose — not describing who knows who, but designing the most efficient route for data to travel across Hatchly's infrastructure. Four named concepts extend what you already know into network optimisation: adjacency, centrality, connectedness and weighted graphs.

Part C · Network connectivity

The hook

In Lesson 2, Maya's network answered social questions: who is connected to whom, how many connections does each person have, how do you get from one person to another. Those same questions become engineering questions when the nodes are servers and routers instead of people — and when the answer determines whether data arrives in 90 milliseconds or 105 milliseconds.

The question for today Hatchly's cold-chain sensor at Market B needs to send a stock update to the packing space. There's no direct link between them — the data has to travel via other nodes. How would you find the fastest route without just guessing?

Hatchly's weighted network

Hatchly has four network locations: the cloud server (Cloud), Market A, Market B, and the Packing space. The diagram shows their connections, with each edge labelled with the measured round-trip latency in milliseconds.

Worked example

Finding the optimal path from Market B to Packing

Three simple paths exist from Market B to Packing. Compute each total by adding the edge weights along it:

  1. Market B → Cloud → Packing: 60 + 30 = 90ms
  2. Market B → Market A → Cloud → Packing: 25 + 45 + 30 = 100ms
  3. Market B → Market A → Packing: 25 + 80 = 105ms

The optimal path is route 1 at 90ms — 15ms faster than route 3, even though route 3 uses fewer hops. Fewer hops doesn't always mean less time; weight is what matters.

Adjacency and the matrix

Every graph can be converted into an adjacency matrix — a table where a cell contains 1 if the two nodes share a direct edge, and 0 if they don't. The same diagram above produces this matrix:

Cloud Market A Market B Packing
Cloud 0111
Market A 1011
Market B 1100
Packing 1100

Each row's sum tells you that node's degree directly from the matrix: Cloud = 3, Market A = 3, Market B = 2, Packing = 2. Total = 10, which is exactly double the 5 edges in the diagram — the same handshake check from Lesson 2, confirmed again here. A 0 on the diagonal (top-left to bottom-right) simply means a node doesn't connect to itself — always true in a real network.

Notice the symmetry: the matrix mirrors across the diagonal. If Cloud→Market A is 1, then Market A→Cloud must also be 1, because the edge runs in both directions. This will always be true for an undirected graph, which is what all the diagrams in this unit use.

Part C · Network connectivity

Centrality and connectedness

Degree centrality is easy to read from the matrix: Cloud and Market A each have degree 3, making them the most central nodes in Hatchly's network by this measure. That means if Hatchly's engineers wanted to add a new location, connecting it to Cloud or Market A would give the new node the fastest path to everywhere else.

Why centrality matters for network design A node with high centrality sits on many shortest paths — routing most of Hatchly's traffic through Cloud means Cloud is the most critical point to keep reliable. If Cloud were to fail, Market B and Packing lose their only connections to each other (check the matrix: Market B–Packing = 0, and all paths between them go through Cloud or Market A).

Connectedness can be read from the diagram directly: starting from any node, can you trace a path to every other node? In Hatchly's graph, yes — the graph is connected. But if Market B's two edges were both removed, Market B would become an isolated node and the graph would become disconnected, with no path from Market B to anywhere else.

Teacher tip

The "what if this node failed?" question is a genuine engineering decision — it's why critical infrastructure uses mesh topology rather than star. Students can revisit Lesson 6's topology diagrams and ask which topologies produce a connected graph even after one node is removed.

Evaluate the new network's centrality

Complete the Core task, then re-calculate degree centrality for all five nodes in the updated network and argue whether Cloud is still the most important node to keep reliable — or whether the answer has changed.

Hint for the Core task

The new Market C node has edges to Cloud (55ms) and Market B (35ms). List every simple path from Market C to Packing, add up the weights along each, and pick the smallest total. Check: you should find three paths, with the shortest being Market C → Market B → Cloud → Packing at 35 + 60 + 30 = 125ms — but verify this yourself against all the alternatives before committing to it.
