Data Storage, Workflows and Cloud Computing

Parts B and D established what Hatchly's devices are and how they talk to each other. Part E asks what happens to all the data those devices generate. An order confirmed on a phone, a temperature reading from a delivery van, a buyer's payment detail — each one has to go somewhere, be stored reliably, be retrievable when needed, and be inaccessible to everyone who shouldn't have it. That's the subject of the next two lessons.

The hook

Hatchly's three founders coordinate entirely without an office — every order, every maker listing, every stock update and every payment flows through digital systems. Think about the path a single order takes: a buyer taps "purchase" on their phone, that triggers a confirmation message to the maker, a payment record goes to Hatchly's accounts, a cold-chain sensor note is attached if the item needs refrigeration, and a delivery status update is queued. That's five separate data movements from one action — and at each step, the data has to go somewhere, be stored in the right format, and be accessible to exactly the right person.

The question for today When you send a message on any app and then delete it from your phone, where do you think the message actually goes? Is it gone — or does it still exist somewhere?

Digital workflows

A digital workflow replaces a chain of manual steps — someone emailing an update, someone else copying it into a spreadsheet, a third person approving it over the phone — with software-triggered actions that happen automatically when conditions are met.

Worked example

Hatchly's order workflow

Before Hatchly built a digital workflow for orders, a maker had to check for new orders manually, copy the buyer's address into a delivery app by hand, then message the founders to confirm dispatch. Each step depended on a person being available and remembering to do it. Now:

  1. Buyer places order → payment is confirmed automatically by the payment gateway
  2. Confirmation message is triggered automatically to both buyer and maker
  3. Order details are written automatically to the shared inventory record
  4. If the item is flagged as cold-chain, the sensor system is notified automatically
  5. Delivery status is set to "pending" and becomes visible to the buyer automatically

Steps 1–5 now happen in under two seconds, without anyone doing anything manually after the buyer's tap.

Benefits and limitations — both matter Benefits: speed (steps complete in milliseconds, not minutes), consistency (no step can be forgotten or done differently each time), auditability (every step is logged with a timestamp). Limitations: setup cost is real (workflows take time to design and test), dependency risk (if one system in the chain goes offline, the whole workflow may stall), and data security exposure (data passes through multiple systems, each of which is a potential attack surface).

IaaS, PaaS and SaaS

Lesson 4's cloud-vs-on-premises comparison asked where infrastructure lives. The IaaS/PaaS/SaaS distinction asks a different question: how much of the stack does the provider manage? The more a provider manages, the less control — and the less responsibility — the enterprise retains.

Model Provider manages Enterprise manages Hatchly example
IaaS Hardware, networking, virtualisation OS, runtime, middleware, data, applications Not realistic — managing an OS would need a dedicated IT staff member Hatchly doesn't have
PaaS Hardware, OS, runtime environment Application code and data only The Hatchly marketplace app backend — a developer deploys code; the platform handles everything beneath it
SaaS Everything — hardware through to the application itself Data and personal settings only The messaging app, project board and cloud file storage Hatchly uses daily — sign in and use, nothing to configure

Teacher tip

A useful analogy: IaaS is renting an empty warehouse (you fit it out yourself), PaaS is renting a fitted kitchen (you cook, they maintain the oven), SaaS is ordering delivery (you eat, someone else does everything else).
