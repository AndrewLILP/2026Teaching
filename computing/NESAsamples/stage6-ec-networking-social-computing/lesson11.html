Network Diagrams and Securing Access

Lesson 10 asked where data goes and how it's stored. This lesson asks two follow-on questions: how do you draw the network infrastructure that carries it, in a way anyone in the industry can read? And once data is moving across that infrastructure, how do you make sure only the right people can access it?

The hook

An IT professional from anywhere in the world can pick up a network diagram and immediately understand which device is a router, which is a switch, and how they're connected — even if the diagram uses no English labels at all. That's possible because there's a shared set of symbols everyone in the industry agrees to use. Today you'll use NESA's version of that symbol set to document Hatchly's own network, and then use the graph theory from Lessons 2 and 8 to evaluate what you drew.

The question for today Hatchly's three founders all have laptops, phones, and access to the same cloud storage. If a delivery driver returned a founder's stolen laptop, could they log in as that founder and see Hatchly's order data? What would actually prevent that?

Securing access

Four mechanisms protect access to a network and its data. Tap each card to reveal the definition.

Case study

Securing Hatchly's access points

A founder's laptop holds order records, maker payment details and delivery schedules. Hatchly has three layers protecting it: the laptop requires a biometric scan (fingerprint) before booting, the cloud dashboard requires a strong password plus a one-time code from the founder's phone (two-factor authentication), and Hatchly's maker sign-up form uses a CAPTCHA to block bots from creating fake listings at scale. The laptop itself uses full-disk encryption underpinned by its TPM chip — so the stolen-laptop scenario from the hook ends with an encrypted drive the driver can't read without the founder's fingerprint.

Discuss: Each of these four mechanisms protects against a different kind of threat. Which threat does each one address — and is there any threat that none of the four would stop on its own?

NESA's device symbol set

NESA expects you to label all devices clearly on a network diagram. The following device types appear in the course specifications — you must be able to recognise, name and use all of them.

DeviceRole in the network
RouterConnects a local network to the internet, directing traffic between different networks
SwitchConnects devices within a LAN, sending data only to the specific device it's addressed to
HubConnects devices in a LAN but broadcasts every message to all connected devices — less efficient than a switch
Wireless access point (WAP)Provides Wi-Fi connectivity, bridging wireless devices onto the wired network
RepeaterBoosts a weakening signal over a long cable run or across a large physical space
BridgeConnects two separate network segments, filtering traffic between them
GatewayTranslates between two networks using different protocols — the protocol-bridging role from Lesson 9
NICNetwork Interface Card — the hardware component inside a device that enables it to physically connect to a network
Server (by type)A dedicated device providing a specific service — file server, web server, print server, mail server
WorkstationA desktop or laptop computer used by a person to access network resources

Evaluating a diagram with graph theory

Now apply the concepts from Lessons 2 and 8 to evaluate the network diagram above. The device boxes are the nodes; the dashed lines are the edges.

Graph theory evaluation — worked

Applying adjacency, centrality and connectedness

  1. Adjacency: Which devices are directly adjacent to the Router? Count the dashed lines touching its box: Internet, Switch, WAP, Cloud Server — four direct connections, so the Router's degree is 4. The Smartphone and Workstation A are not adjacent to the Router; they connect only through intermediate nodes.
  2. Centrality: The Router has degree 4 — the highest of any node in the diagram. This confirms it as the most central device: every message between a workstation and the cloud server passes through it. This mirrors Lesson 8's finding about Cloud in Hatchly's location network — the node with the highest degree centrality is always the single most critical point to keep reliable.
  3. Connectedness: Can every device reach every other via some path? Trace from Smart Lock: Smart Lock → IoT Gateway → WAP → Router → Switch → Workstation A. Yes — every device can reach every other, so the graph is connected. But: if the Router were removed, the graph would split into at least two disconnected components (wired side and wireless side). This is the same single-point-of-failure risk Lesson 6 identified for star topology — and Hatchly's network is effectively a double-star (switch-star and WAP-star, both feeding into the Router).

Teacher tip

Ask students to formally draw the network above as a graph (circles and labelled edges, Lesson 2 style) and compute the full adjacency matrix. It directly connects all three Part E activities — diagram, security analysis, and graph theory evaluation — into one integrated task.

Foundation · D–E

Label the diagram

Core · C

Evaluate using graph theory

Identify the two devices with the next-highest degree after the Router, and explain what would happen to the network's connectedness if the WAP were removed.

Extension · A–B

Redesign for resilience

Complete the Core task, then propose one specific change to Hatchly's network that would prevent the loss of any single device from disconnecting the rest of the network. Justify your proposal using the graph theory concepts from this lesson.
