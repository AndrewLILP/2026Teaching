NESA Sample Unit · Software Engineering 11–12 · Programming Fundamentals (Year 11)

Practical Programming in Context

Across fourteen lessons, you'll build a Python program that solves Sudoku puzzles — and use that one project as the thread connecting everything else in the Programming Fundamentals focus area. Software development models, number systems, data types and structures, algorithm design, debugging and testing all attach to the same build, rather than being taught as separate topics that happen to share a syllabus.

SE-11-01 SE-11-02 SE-11-03 SE-11-04 SE-11-06

14 lessons · part of the 40-hour Programming Fundamentals focus area

How this unit fits

Programming Fundamentals is the first of three Year 11 Software Engineering focus areas — The Object-Oriented Paradigm and Programming Mechatronics follow it, both building directly on the algorithm design, data structure and debugging skills developed here. This unit deliberately keeps a light touch on object-oriented programming (a single "experiment with paradigms" task in Lesson 11) because OOP gets its own dedicated focus area next. Everything here is scoped to procedural Python, built around one project, so that the conceptual thread — sequence, selection, iteration, decomposition, backtracking — stays visible from Lesson 1 to Lesson 14 rather than resetting with every new topic heading.

Your journey through this unit

Each lesson sits in one of four phases — building the foundations you need (identifying the problem, researching and designing), then producing the actual solver, then testing and comparing what you've built.

Identifying & Defining

Meeting the problem

Lesson 1 — Meet the Problem: Solving Sudoku

The hook, the project brief, and a first hands-on attempt at solving a Sudoku puzzle by hand.

Researching & Designing

Building the foundations

Lesson 2 — Software Development Models: Waterfall vs Agile

The fundamental software development steps, and comparing the Waterfall and Agile approaches.

Lesson 3 — Number Systems and Data Representation

Binary, decimal and hexadecimal, plus representing integers using two's complement.

Lesson 4 — Data Types and Data Structures

Choosing how to represent a Sudoku grid in Python, starting from a simpler 1D word-jumble warm-up.

Lesson 5 — Decomposition and Pattern Recognition

Breaking the Sudoku problem into smaller pieces and spotting the patterns that repeat.

Lesson 6 — Modelling and Desk-Checking Algorithms

Structure charts, abstraction and refinement diagrams, and checking an algorithm before you trust it.

Lesson 7 — Loops and Branching: Scanning the Grid

Using iteration and selection to scan rows, columns and 3×3 boxes for valid moves.

Lesson 8 — Divide and Conquer, and Backtracking: The Idea

The core algorithmic idea behind the solver, introduced with a non-Sudoku example first.

Lesson 9 — Backtracking, Applied to Sudoku

Stepping through how the solver tries a value, hits a dead end, and backs up to try again.

Producing & Implementing

Building the solver

Lesson 10 — Coding the Solver: Part 1

Turning the algorithm design into working Python, function by function.

Lesson 11 — Coding the Solver: Part 2

Finishing the build, plus an optional paradigm-comparison task for early finishers.

Lesson 12 — Debugging: Breakpoints, Stepping, and Output Statements

Using an IDE debugger and print-statement debugging to find what's actually going wrong.

Lesson 13 — Debugging: Watches and Interfaces Between Functions

Tracking variable state over time and checking how your functions actually talk to each other.

Testing & Evaluating

Finishing up

Lesson 14 — Testing, Evaluating, and Comparing Peer Solutions

Boundary, path and faulty-data testing, classifying errors, and comparing your solver against a classmate's for functionality and usability.

Tools for this unit

All coding activities use Python. Lessons 10–13 are written around VS Code with the Python extension, since its debugger covers every debugging skill named in the syllabus — breakpoints, single-line stepping, watches, and inspecting the interfaces between functions. PyCharm or Spyder work as direct substitutes if your school already uses one of them; a browser-only runner (Replit, online-python.com) is fine for the earlier coding-only lessons but won't support the debugging lessons, since it has no breakpoint or watch panel.
