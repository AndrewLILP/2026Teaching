Meet the Problem: Solving Sudoku

Over the next fourteen lessons, you're going to teach a computer how to solve a puzzle that gives most humans a genuine workout. Today is about the hook, the plan, and your first real attempt at the puzzle itself — before any code gets written.

Identifying & Defining

The hook

Here's the problem, stated plainly: a Sudoku grid is 9×9 squares. Some are already filled in. Your job — whether you're a human with a pencil or a computer with an algorithm — is to fill in the rest so that every row, every column and every 3×3 box contains the digits 1 to 9, with no repeats.

The question for today Could you write down, in plain English, exactly what you do when you solve one of these — step by step, no skipped thoughts — clearly enough that someone else could follow your exact process?

That last question is harder than it sounds, and it's the real subject of this entire unit. Humans solve Sudoku using intuition built from pattern recognition we can't always explain. A computer has none of that — it can only do exactly, and only, what you tell it to do. Turning "I just sort of see it" into a precise set of instructions a machine can follow is the core skill of software engineering, and Sudoku is going to be your training ground for it.

Identifying & Defining

What we're building

By the end of this unit, you'll have built a working Python Sudoku solver — a program that takes a partially-filled grid and fills in the rest automatically. Along the way, every other topic in this unit attaches directly to that one build:

Everything connects back to the solver How software gets planned (Waterfall vs Agile) · how the grid gets represented in memory (number systems, data types, data structures) · how the solving logic gets designed (decomposition, loops, backtracking) · how it gets built, debugged and tested in Python

At the very end of the unit, you'll swap solvers with a classmate and evaluate each other's work for functionality and usability — does it actually solve puzzles correctly, and is it pleasant to use? That comparison is your evidence that the thinking you did in Lesson 1 turned into something that actually works.

Teacher tip

Resist the urge to open an IDE today. The whole point of this lesson is to sit with the problem unplugged first — Lesson 10 is where typing actually starts.

Identifying & Defining

Try it yourself

Before you write a single instruction for a computer, solve a Sudoku puzzle yourself — on paper, or using a free online Sudoku puzzle if one's available in the room. While you solve it, pay attention to your own thinking. Every time you write a number in, ask yourself: how did I know that one was safe? What would I have done if it turned out to be wrong two moves later?

Choose your task

Foundation · D–E

Spot one decision

Solve an easy puzzle (or as much of one as you can in the time). Write down one specific moment where you had to decide what number went in a square, and how you decided.

Core · C

Track three decisions

Solve an easy-to-medium puzzle, and note down at least three different moments where you made a decision, naming the reasoning you used each time (e.g. "this row already has every other digit, so this square must be...").

Extension · A–B

Find a dead end

Solve a medium-to-hard puzzle, and deliberately watch for a moment where a number you placed turned out to be wrong later — forcing you to erase it and try again. Describe exactly what tipped you off that something was wrong.

Teacher tip

The Extension task is deliberately fishing for a real backtracking moment. Not everyone will hit one in a short puzzle — that's fine, it previews Lesson 9 either way.
