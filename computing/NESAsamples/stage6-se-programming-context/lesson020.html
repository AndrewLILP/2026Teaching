Software Development Models: Waterfall vs Agile

Before you design a single algorithm for your Sudoku solver, you need a plan for how you're going to build it. Today's lesson is about the steps every software project moves through, and the two very different ways teams order those steps in practice.

Why does order matter?

Imagine starting to type Python code for your Sudoku solver right now, before you'd decided what a "finished" solver even needs to do, or how you'd represent the grid in memory. You'd almost certainly end up rewriting most of it once those decisions finally got made.

The question for today Every software project — yours included — moves through the same basic set of steps. The real question isn't whether you do them, but in what order, and how rigidly.

That question has two very different standard answers, called Waterfall and Agile. By the end of this lesson, you'll be able to explain both, and you'll have chosen which one (or which mix) actually fits a short, well-defined project like a Sudoku solver.

Researching & Designing

The eight steps

Whichever model a team uses, these are the steps the work actually has to pass through somewhere along the way.

Word bank

Requirements definition

Working out what the program actually needs to do — e.g. "must accept a partially-filled 9×9 grid and return a completed one."

Determining specifications

Turning those requirements into specific, checkable details — exactly what counts as a "valid" grid, what input format is expected.

Design

Planning the algorithm and structure of the solution before any code gets written.

Development

Actually writing the code.

Integration

Connecting the separate pieces — functions, modules — into one program that works together.

Testing and debugging

Checking the program actually works, and fixing what doesn't.

Installation

Getting the finished program into the hands of whoever will actually use it.

Maintenance

Keeping the program working and updated after it's "finished" — bugs found later, new features requested.

Waterfall vs Agile

The Waterfall approach

Waterfall takes those eight steps and runs through them once, in strict order — each stage finishes completely before the next one starts. The name comes from the image of water falling from one level to the next; it never flows back uphill.

Waterfall, in practice Requirements definition finishes → specifications get locked in → the whole design gets planned → all the code gets written → everything gets integrated → the complete program gets tested → it's installed → it's maintained

The Agile approach

Agile takes the same eight steps but runs through a small slice of all of them, repeatedly, in short cycles. Each cycle produces a small working piece of the solution, gets feedback, and adjusts before the next cycle starts.

Agile, in practice Cycle 1: build and test "can the program read a grid in?" → Cycle 2: build and test "can it check whether a number is valid in a square?" → Cycle 3: build and test "can it try a value and back up if it's wrong?" → and so on, each cycle adding a working piece

Comparing the two

AspectWaterfallAgile
Order of workEach stage finishes before the next beginsStages repeat in short cycles
Best suited toProjects with clear, stable requirements from the startProjects where requirements may shift as you learn more
Handling changing requirementsDifficult and costly once a stage is locked inBuilt in — each cycle can adjust based on feedback
DocumentationHeavy emphasis, produced upfrontLighter, produced as needed

Choosing an approach for our solver

Let's reason through this together first

A Sudoku solver is a small, short project with a requirement that's actually pretty clear from the start: "take a partially-filled grid, return a completed one." That stability is exactly what Waterfall handles well — and for a project this size, the planning overhead of full Waterfall documentation isn't a real burden either. But the algorithm design part of the project (decomposition, backtracking) genuinely benefits from an Agile-style "build a small piece, test it, then build the next piece" rhythm, since you won't fully understand how backtracking behaves until you've actually tried coding a slice of it.

Online code collaboration tools

Whichever development model a team uses, most modern software is built by more than one person at once — which means the team needs a way to work on the same code without overwriting each other's changes.

Let's evaluate one together first: GitHub

  • What is it? A platform for storing code online and tracking every change made to it over time.
  • How does it support collaboration? Multiple people can work on the same codebase, see exactly what changed and who changed it, and merge their work together instead of overwriting each other.
  • How popular is it? Extremely — it's one of the most widely used code collaboration platforms in the software industry.
  • Who monitors or administers it? The platform itself is run by a company; individual teams manage their own projects and decide who has access.
  • How does it contribute to efficiency? It lets several people work on different parts of a project at the same time, instead of taking turns on one shared file.
