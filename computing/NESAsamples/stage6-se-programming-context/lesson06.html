Researching & Designing

From breakdown to blueprint

A list of steps in plain English is a great start, but it's still loose enough to hide mistakes. "Try the next value" — which value, exactly? What happens if none of them work? Today's tools exist to close those gaps: pseudocode and flowcharts force you to be precise, and desk-checking forces you to prove your precision is actually correct, before you've spent a single minute debugging real code.

Researching & Designing

Pseudocode and flowcharts

Pseudocode is English-like, structured writing that describes an algorithm precisely, without committing to any one programming language's exact syntax. Here's one piece of our decomposition — "try a value in this square" — written as pseudocode for the first time:

Pseudocode

value = 1
WHILE value <= 9
    IF value NOT IN row AND value NOT IN column AND value NOT IN box THEN
        PLACE value
        EXIT WHILE
    ELSE
        value = value + 1
    ENDIF
ENDWHILE

A flowchart shows the same kind of logic as connected shapes instead of lines of text — useful when you want to see a decision's two branches side by side at a glance.

Structure charts and refinement diagrams

Pseudocode and flowcharts describe one piece of logic. Two other modelling tools help you see the whole program's shape.

A structure chart shows which subprograms call which other subprograms — the "org chart" of your code:

Structure chart for our solver solve_puzzle()
  ├── find_empty_square()
  ├── is_valid()
  ├── place_value()
  └── backtrack()

An abstraction and refinement diagram shows the same problem written at progressively more detailed levels — useful for top-down design, where you start broad and refine downward:

Refinement diagram for "solve the puzzle" Level 0 (most abstract): Solve the puzzle.
Level 1: Find an empty square → try a value → check it → place or move on → backtrack if stuck.
Level 2 (most detailed): for each value 1–9, check row/column/box membership; if all clear, place and recurse; if none work, undo and return.

Desk-checking and peer-checking

Before trusting any algorithm, ask three questions: what is its purpose? What are its inputs and outputs? And how does it connect to other subprograms around it? Once you can answer those, you're ready to desk-check it — tracing through it by hand, line by line, with real sample values, to see whether it actually does what you think it does.
