Coding the Solver: Part 2

You have three working functions — a grid, a way to find empty squares, and a way to check whether a value is allowed. Today you write the one function that uses all three together, calls itself, and actually solves the puzzle: solve().

Producing & Implementing

The missing piece

find_empty() finds a square. is_valid() checks a value. Neither one, on its own, solves anything. What's missing is the function that ties them together — tries a value, places it, moves on, and undoes its own decision if things go wrong further down the line. That's solve(), and it's the only function in this entire unit that calls itself.

Producing & Implementing

Designing solve()

Before any code, the logic — and it's almost exactly Lesson 5's six-step breakdown, written as a function for the first time:

Pseudocode

FUNCTION solve(grid):
    find the next empty square
    IF there is no empty square THEN
        RETURN True          // solved!
    ENDIF

    FOR value = 1 TO 9
        IF value is valid at this square THEN
            place value
            IF solve(grid) THEN
                RETURN True
            ENDIF
            remove value      // backtrack
        ENDIF
    ENDFOR

    RETURN False              // trigger backtracking one level up

The strange-looking part is solve(grid) appearing inside its own definition. This is called recursion — a function calling itself to solve a smaller version of the same problem. Each call handles one square, then hands the rest of the grid to another call of itself. If that next call eventually returns False, it means every value it tried led nowhere — exactly the dead end from Lessons 8 and 9 — so the current call undoes its own value and tries the next one.

Producing & Implementing

Coding solve()

STEP 5 Tie it all together
// ── NEW IN STEP 5 ──
grid = [ ... ]  # same 9×9 grid as Lesson 10, Step 1

def print_grid(grid):
    for row in grid:
        print(row)

def find_empty(grid):
    for row in range(9):
        for col in range(9):
            if grid[row][col] == 0:
                return (row, col)
    return None

def is_valid(grid, row, col, value):
    for c in range(9):
        if grid[row][c] == value:
            return False
    for r in range(9):
        if grid[r][col] == value:
            return False
    box_row_start = (row // 3) * 3
    box_col_start = (col // 3) * 3
    for r in range(box_row_start, box_row_start + 3):
        for c in range(box_col_start, box_col_start + 3):
            if grid[r][c] == value:
                return False
    return True

def solve(grid):
    empty = find_empty(grid)
    if empty is None:
        return True

    row, col = empty

    for value in range(1, 10):
        if is_valid(grid, row, col, value):
            grid[row][col] = value

            if solve(grid):
                return True

            grid[row][col] = 0

    return False
What's new: if empty is None: return True is the "puzzle is finished" check from Lesson 5, step 1. The for value in range(1, 10) loop tries every digit, same as Lesson 7's scans. grid[row][col] = value followed by grid[row][col] = 0 are the exact place-and-undo pair from the Lesson 9 widget — except now they're real code, not a replayed trace.
Producing & Implementing

Running it

Add this underneath everything else and run the file:

if solve(grid): print_grid(grid) else: print("No solution found")

On the puzzle from Lesson 10, this prints a fully completed 9×9 grid — every row, column and box a valid permutation of 1–9. Behind that single line of output, the computer actually made 4,157 separate backtracks before reaching the final answer. The Lesson 9 widget needed exactly one. That gap is the whole reason this had to be a computer's job rather than yours: the logic is identical at both scales, but only a computer can try and undo a choice thousands of times in a fraction of a second.

Teacher tip

If a student is curious how the 4,157 figure was found, it's a global counter incremented every time grid[row][col] = 0 runs — worth mentioning as a preview of the kind of instrumentation real debugging sometimes uses, ahead of Lessons 12–13.
