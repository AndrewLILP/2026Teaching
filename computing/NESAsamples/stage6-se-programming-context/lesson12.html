Debugging: Breakpoints, Stepping, and Output Statements

Your solver works — but "it works" and "I understand exactly what it's doing at every moment" are different things. Today you get two genuine ways to look inside a running program: scattering print statements through your own code, and letting the IDE itself pause and show you everything.

Producing & Implementing

When desk-checking isn't enough

Lesson 6 desk-checked a few lines of pseudocode by hand. Lesson 10 desk-checked real functions by running three lines and reading the output. Neither approach works once a program calls itself recursively thousands of times in a fraction of a second, the way solve() does. You need tools built for exactly that.

Producing & Implementing

Two ways to look inside

Both of today's techniques answer the same question — "what is my program actually doing right now?" — but in very different ways. Debugging output statements means temporarily adding extra print() calls to your own code, so it reports on itself as it runs. Breakpoints and stepping mean using tools built into the IDE itself to pause your program mid-run and inspect it directly, without changing your code at all.

Producing & Implementing

Debugging output statements

The simplest debugging tool there is: add a print() wherever you want visibility, run the program, read the output, then delete the print statement once you're done.

def solve(grid): empty = find_empty(grid) if empty is None: return True row, col = empty for value in range(1, 10): print(f"Trying {value} at row {row}, col {col}") # debugging output statement if is_valid(grid, row, col, value): grid[row][col] = value if solve(grid): return True grid[row][col] = 0 return False

Running this on the Lesson 10 puzzle, the first few lines of output look exactly as expected:

Trying 1 at row 0, col 2 Trying 1 at row 0, col 3 Trying 2 at row 0, col 3 Trying 1 at row 0, col 5 Trying 2 at row 0, col 5 Trying 3 at row 0, col 5 Trying 4 at row 0, col 5 Trying 1 at row 0, col 6

So far, so readable. But that print statement runs once for every value tried at every square, for the entire solve — and the entire solve tries a value 37,652 times before it's done. That's 37,652 lines of console output for one puzzle. Print-statement debugging is fast to set up and easy to understand, but it doesn't scale — past a certain point, the output itself becomes the thing you'd need to debug.

Producing & Implementing

Breakpoints and stepping

A breakpoint is a marker you place on a specific line, telling the IDE: "pause the program right before this line runs." Instead of flooding a console with thousands of lines, you choose one exact moment to freeze and look around.

📷Screenshot needed — setting a breakpoint

Capture: VS Code with solve() open, cursor on the line if is_valid(grid, row, col, value): inside the for loop, with a breakpoint set — the red dot visible in the gutter to the left of the line number.

  1. Circle the red dot in the gutter.
  2. Label it: "breakpoint — execution will pause here every time this line is about to run."

Once a breakpoint is set, running the program in debug mode (in VS Code: pressing F5, or the green "Run and Debug" arrow) doesn't run straight through — it pauses the instant it reaches that line, with the rest of the program frozen exactly as it was.

📷Screenshot needed — paused at the breakpoint

Capture: the same file, now running in debug mode and paused at the breakpoint — the current line highlighted (usually yellow), the debug toolbar visible (Continue, Step Over, Step Into, Step Out, Restart, Stop), and the Variables panel in the sidebar showing the current values of row, col and value.

  1. Box the highlighted current line and label it "execution is paused here, right now."
  2. Circle the debug toolbar and label each button (Continue / Step Over / Step Into / Step Out).
  3. Box the Variables panel and label it "every variable's current value, live."

Single-line stepping is what happens next: instead of clicking Continue (which runs until the next breakpoint), clicking Step Over advances exactly one line and pauses again — letting you watch values change one line at a time.

📷Screenshot needed — stepping forward

Capture: the same paused program, after clicking Step Over two or three times — the highlighted line has moved down, and the Variables panel shows an updated value (for example, value incremented, or grid now showing a newly placed digit).

  1. Circle the Step Over button specifically.
  2. Draw an arrow from the previous screenshot's highlighted line to this one's, showing the movement.
  3. Circle the changed value in the Variables panel and label it "this just changed."

Notice what you get that print statements can't give you: every variable's value at once, updating live, with no need to decide in advance what to print or to clean anything up afterward.

Producing & Implementing

Comparing the two

AspectDebugging output statementsBreakpoints and stepping
SetupAdd print() lines directly into your codeClick in the gutter; no code changes needed
What you seeOnly what you chose to print, in orderEvery variable's current value, at the exact moment you choose
Scales to thousands of calls?No — output becomes unreadableYes — you choose exactly when to pause
Cleanup required?Yes — remove the print statements afterwardNo — breakpoints don't change the saved code

Neither tool is strictly better — a quick print statement is often faster for a simple check, while breakpoints earn their setup cost on anything complex enough to need watching closely. Knowing both means choosing the right one for the bug in front of you.
