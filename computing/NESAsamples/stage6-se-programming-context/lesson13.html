Debugging: Watches and Interfaces Between Functions

Lesson 12 paused your program and let you look around. Today you get a way to track one value continuously without hunting for it every time — and a way to actually follow execution across the boundary when one function calls another, which is exactly where some of the sneakiest bugs hide.

Producing & Implementing

Two more debugger tools

Stepping one line at a time is powerful, but tiring if you're only interested in one value buried in a long Variables panel. And Step Over, from Lesson 12, deliberately skips over the inside of any function it's called on — which is a problem the moment a bug might actually be hiding inside the function being called, not in the line that calls it.

Producing & Implementing

Watches

A watch is an expression you add once, which the debugger then keeps displayed and automatically up to date every single time the program pauses — no more scrolling through the Variables panel looking for it.

📷Screenshot needed — adding a watch expression

Capture: VS Code's Run and Debug sidebar with the WATCH panel expanded (below Variables), showing the "+" (Add Expression) button being used to add row, col, value as a watch, with its current value displayed underneath.

  1. Circle the "+" Add Expression button.
  2. Box the WATCH panel and label it "tracked automatically, every time the program pauses."
  3. Underline the displayed value next to the watch expression.

Watches matter most for exactly the kind of program you've built: solve() pauses at the same breakpoint thousands of times across one run. Without a watch, you'd be hunting for row, col and value in the Variables panel after every single pause. With a watch, they're just always there.

Producing & Implementing

Interfaces between functions

Every time solve() calls is_valid() — or calls itself — there's a boundary being crossed: specific arguments go in, and a specific value comes back out. That boundary is the interface between the two functions, and it's exactly where mismatched expectations cause bugs that are invisible from inside either function alone.

Step Over vs Step Into Step Over treats a function call as a black box — it runs the whole call and lands you on the next line, without showing you what happened inside. Step Into does the opposite: it follows execution across the boundary, landing you on the very first line inside the function being called.
📷Screenshot needed — stepping into a function call

Capture: paused on the line if is_valid(grid, row, col, value): inside solve(), immediately after clicking Step Into — execution should now be paused on the very first line inside is_valid(), with the Call Stack panel showing is_valid at the top and solve listed beneath it as the caller.

  1. Circle the Step Into button.
  2. Box the highlighted line, now inside is_valid(), and label it "we crossed the interface."
  3. Box the Call Stack panel and draw an arrow from "solve" to "is_valid" labelled "caller → called function."

The Call Stack panel lists every function call currently in progress, most recent at the top. Because solve() calls itself recursively, this becomes a genuinely useful way to see recursion rather than just reason about it abstractly.

📷Screenshot needed — the call stack during recursion

  1. Box the whole stack of "solve" entries.
  2. Label the bottom entry "the very first call" and the top entry "the most recent call, still waiting on the one above it."
  3. Count and label how many levels deep the stack has grown.

Step Out does the reverse of Step Into: it finishes the current function completely and lands you back at the caller, right where its return value gets used — letting you watch exactly what crosses back over the interface in the other direction.

Producing & Implementing

Tying it together

"Interfaces between functions" isn't new vocabulary — it's Lesson 6's procedures, functions and parameters, looked at from the debugger's point of view. A parameter is what crosses the interface going in; a return value is what crosses it coming back out; and the Call Stack is simply a live picture of every interface currently open at once.
