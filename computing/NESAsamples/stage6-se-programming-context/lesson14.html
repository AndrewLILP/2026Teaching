Testing, Evaluating, and Comparing Peer Solutions

Your solver works on one puzzle. That's not the same as knowing it works in general. Today's the last lesson of the unit — choosing real test data to genuinely challenge your solver, naming the kinds of errors that slip through, evaluating quality beyond "does it run," and finally comparing your finished solver against a classmate's.

Testing & Evaluating

Almost the finish line

One puzzle solving correctly proves your solver can solve that puzzle. It proves nothing about every other puzzle you haven't tried yet — including the ones specifically designed to break things. Today is about deliberately going looking for the cases your solver hasn't met yet, before a user does.

Choosing test data

Let's run these against our solver together

Boundary value 1 — a fully solved grid (zero empty squares): solve() returns True immediately, since find_empty() finds nothing and returns None straight away.

Boundary value 2 — one empty square left: solve() correctly fills in exactly the one missing value needed to complete the grid.

Path coverage — two different puzzles: the Lesson 10 puzzle needed 4,157 backtracks to solve; a different, easier puzzle might solve with zero. Testing only one of these means an entire branch of your own logic — the backtracking branch — might never actually run during testing.

Faulty data — a grid with a duplicate already among the given clues (no empty squares, but invalid from the very start): running solve() on it returns True — it reports the puzzle as solved, even though it was never a valid puzzle to begin with.

That last result is a genuine gap, not a typo: solve() only ever checks whether empty squares remain. It never checks whether the given clues were valid in the first place. Faulty test data just found a real limitation in code that "worked" on every test we'd tried until now.

Typical errors

Tap each card to reveal what it means.

Why this ordering matters Syntax errors are the easiest to catch — Python refuses to even start. Runtime errors announce themselves with a crash and a traceback. Logic errors are the hardest, because the program runs, produces output, and gives no error message at all — exactly the kind of bug Lessons 12 and 13's debugging tools exist to catch.
Testing & Evaluating

Evaluating quality

"Does it work" is only one of four things worth judging.

AspectWhat to check
FunctionalityDoes it correctly solve every valid puzzle you test it with?
PerformanceHow many backtracks or how long does it take? (Recall Lesson 11's 4,157.)
ReadabilityAre variable and function names clear? Is the logic decomposed sensibly, the way Lessons 5–6 designed it?
DocumentationDo comments explain what each function does and why, for someone reading it for the first time?
Testing & Evaluating

Comparing with a partner

Swap solvers with a classmate, and run their code against the same test data from today: a solved grid, a one-empty-square grid, a puzzle needing backtracking, and a grid with a faulty given. Then give structured feedback using these three prompts:

Structured peer feedback

  • What worked: which of the four test cases did their solver handle correctly, and was the output easy to read?
  • What to improve: did any test reveal a gap — like the faulty-data result earlier — and what would you suggest?
  • One question: something genuinely unclear from reading their code that you'd want them to explain.

Checkpoint

Three quick questions before you go.

1. Which test best demonstrates "path coverage" for our solver?
2. What did testing with an already-solved grid (a boundary value) reveal about solve()?
3. What category of error is grid[9][0] — accessing an index that doesn't exist in a 9×9 grid (valid indices 0–8)?
