// ── Get references to HTML elements ──
const form      = document.getElementById('task-form');
const input     = document.getElementById('task-input');
const taskList  = document.getElementById('task-list');
const emptyMsg  = document.getElementById('empty-msg');

// ── Task counter — gives each task a unique ID ──
let nextId = 1;

// ── Listen for the form submit event ──
form.addEventListener('submit', function (event) {
  event.preventDefault();       // stop the page from reloading
  addTask();
});

// ── addTask: validate input, create list item, append to DOM ──
function addTask() {
  const text = input.value.trim();

  if (text === '') {
    alert('Please enter a task.');
    return;
  }

  // Build the list item
  const li       = document.createElement('li');
  li.className = 'task-item';
  li.dataset.id = nextId;

  const span    = document.createElement('span');
  span.textContent = text;

  li.appendChild(span);
  taskList.appendChild(li);

  // Reset for the next entry
  input.value = '';
  nextId = nextId + 1;
  emptyMsg.style.display = 'none';
}
Identifying & Defining

Step 1 — Annotate the HTML

Click each highlighted group below to reveal what those lines do. Try to work out each answer yourself first — then click to check.

Group A — the <form> element and its children. What is the role of id="task-form"? Why does the button use type="submit"? What does aria-label do on the input?
id="task-form"
The id lets JavaScript find this element using getElementById('task-form'). Without the id, the JS has no way to attach its event listener to this specific form.
type="submit"
A submit button triggers the form's submit event when clicked — or when Enter is pressed in the input. This is what fires the JS event listener. If the type were button, it would do nothing by default.
aria-label="New task description"
Screen readers announce labels so blind and low-vision users know what to type. The placeholder text disappears once typing begins — the aria-label is always available to assistive technology.
Group B — <ul id="task-list" aria-live="polite"> and <p id="empty-msg">. Why is the list empty in the HTML? What does aria-live="polite" announce?
Empty <ul> in HTML
The list starts empty because tasks don't exist until the user creates them. JavaScript adds <li> elements to this <ul> at runtime — this is called dynamic DOM manipulation. The HTML defines the container; the JS fills it.
aria-live="polite"
When new content is added to a region with aria-live, screen readers announce the change without the user having to navigate to it. "Polite" means it waits for a natural pause — it won't interrupt the user mid-sentence.
<p id="empty-msg">
This paragraph is a fallback visible only when no tasks exist. The JS hides it (emptyMsg.style.display = 'none') the moment the first task is added. A common UX pattern — never show a blank white space with no explanation.
Identifying & Defining

Step 2 — Annotate the JavaScript

The JavaScript file contains four distinct parts. Click each to reveal an explanation. After reading all four, you should be able to trace exactly what happens when a user types a task and presses Add.

Part 1 — the four const declarations at the top. Why are these stored in variables? What happens if you skip this step?
const form = document.getElementById('task-form')
getElementById searches the entire HTML document and returns the element with that id. Storing it in a const means we only search once — if we called getElementById every time inside a function, the browser would search the DOM repeatedly, which is slower and harder to read.
const vs let
const means the variable cannot be reassigned — the reference to the HTML element won't change. nextId uses let because it increases with every task added. Using const for things that don't change is a defensive coding habit — it prevents accidental reassignment.
Part 2 — form.addEventListener('submit', function(event) { ... }). What is an event listener? Why does it call event.preventDefault()?
addEventListener('submit', ...)
An event listener is code that waits for something to happen, then runs when it does. Here, we're telling the browser: "When this form fires a submit event, run this function." The function is called a callback — it runs in response to the event, not immediately.
event.preventDefault()
By default, submitting a form sends data to a server and reloads the page. Since there's no server, that would wipe every task from the screen. preventDefault() cancels the default browser behaviour so we can handle the submit ourselves in JavaScript.
Part 3 — inside addTask(): const text = input.value.trim() and the if (text === '') check. Why trim? What's the risk of skipping validation?
.trim()
.trim() removes whitespace from both ends of a string. Without it, pressing the spacebar five times and clicking Add would create a task that appears empty in the list — a blank white card with no text. Invalid data entering the system is a bug source.
if (text === '') { return; }
This is an early return guard — if the input is empty after trimming, we alert the user and stop the function immediately with return. Nothing else in addTask() runs. This prevents empty tasks being added to the list.
Part 4 — createElement, appendChild, and nextId. Trace through this block with the input value "Buy milk". What does the DOM look like after this runs?
document.createElement('li')
Creates a new <li> element in memory only — it doesn't appear on screen yet. We then set its class and a data-id attribute. The data-id will be used in later sprints to find and update specific tasks.
span.textContent = text
Setting textContent (not innerHTML) is deliberate. innerHTML parses the string as HTML — if a user typed <script>alert('hacked')</script>, it could execute. textContent treats everything as plain text — a basic XSS (cross-site scripting) defence.
taskList.appendChild(li)
This line moves the element from memory onto the page. After this runs with input "Buy milk", the DOM contains: <li class="task-item" data-id="1"><span>Buy milk</span></li> inside #task-list. The user sees it immediately.
nextId = nextId + 1
After adding "Buy milk" (id=1), the counter increments to 2. The next task will get id=2, and so on. This ensures every task has a unique identifier — essential for finding and deleting specific tasks in Sprint 4.

Step 4 — Write pseudocode for what you see

Before writing any new code, developers write pseudocode — a plain-English description of the algorithm using NSW structured English keywords. Below is the NSW pseudocode for the addTask() function you just annotated. Study it carefully, then attempt the extension task below.

NSW pseudocode rules: Use BEGIN / END, INPUT / OUTPUT, IF / THEN / ENDIF, ← for assignment. Never use Python or JavaScript syntax. Write for a human reader, not a compiler.
// NSW pseudocode — addTask() function // Matches the starter app's app.js exactly PROCEDURE addTask() // Get the text the user typed and remove leading/trailing spaces taskText ← INPUT from task input field taskText ← TRIM(taskText) // Validate — stop if empty IF taskText = "" THEN OUTPUT "Please enter a task." RETURN ENDIF // Create a new task item with a unique ID newTask ← CREATE list item with text = taskText AND id = nextId // Add it to the task list on screen APPEND newTask TO task list // Reset the input field and increment the counter CLEAR input field nextId ← nextId + 1 HIDE empty message ENDPROCEDURE
Your turn: The starter app currently has no way to remove a task once added. In your project notebook, write NSW pseudocode for a deleteTask(taskId) procedure that would: find the task with the matching ID, remove it from the screen, and show the empty message if no tasks remain. You won't implement it yet — that's Sprint 4. But thinking through the algorithm now will make Sprint 4 much faster.
