Pseudocode first — the full algorithm

Before touching a code editor, read through the complete algorithm for this sprint in NSW pseudocode. This describes everything your app needs to do — three procedures working together.

// Sprint 1 algorithm — NSW pseudocode // Three procedures: initialise, handleSubmit, addTask PROCEDURE initialise() // Get references to the HTML elements we need form ← GET element with id "task-form" input ← GET element with id "task-input" taskList ← GET element with id "task-list" emptyMsg ← GET element with id "empty-msg" nextId ← 1 // Register an event listener on the form ATTACH "submit" event on form → CALL handleSubmit ENDPROCEDURE PROCEDURE handleSubmit(event) // Prevent the default page reload CANCEL default behaviour of event CALL addTask() ENDPROCEDURE PROCEDURE addTask() // Read and clean the input value taskText ← VALUE of input taskText ← TRIM(taskText) // Validate — reject empty strings IF taskText = "" THEN OUTPUT "Please enter a task." Producing & Implementing

Step 1 — Build the HTML structure

Start with index.html. The HTML defines what exists on the page — the form, the input, the button, and the empty list. JavaScript won't be able to find elements that don't exist in the HTML first.

Step 1 · index.html — complete file 
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Task Tracker</title>
  <link rel="stylesheet" href="style.css">
</head>
<body>

  <header class="app-header">
    <h1>Task Tracker</h1>
    <p class="subtitle">Your tasks, your way.</p>
  </header>

  <main class="app-main">
    <form id="task-form" autocomplete="off">
      <input
        type="text"
        id="task-input"
        placeholder="Add a new task…"
        aria-label="New task description"
        maxlength="120"
      >
      <button type="submit">Add</button>
    </form>

    <ul id="task-list" aria-live="polite" aria-label="Task list">
    </ul>

    <p id="empty-msg">No tasks yet. Add one above.</p>
  </main>

  <script src="app.js"></script>
</body>
</html>
This is identical to the Sprint 0 starter — you already know it. The key structural decisions: each element has an id so JavaScript can find it; the <ul> starts empty because JS will fill it; the <script> tag is last so the HTML exists before the JS runs.

Why const for elements, let for nextId? The four element references never change — once found, the same HTML element is always meant. const makes that intention explicit and prevents accidental reassignment. nextId must increase with every task, so it needs let.
Producing & Implementing

Step 3 — Attach the event listener

Add the event listener below the element references. This is the moment the app stops being static HTML and becomes a program that responds to the user.

Step 3 · app.js — complete file
NEW IN STEP 3 — lines 10–14
// ── Get references to HTML elements ──
const form     = document.getElementById('task-form');
const input    = document.getElementById('task-input');
const taskList = document.getElementById('task-list');
const emptyMsg = document.getElementById('empty-msg');

// ── Unique ID counter ──
let nextId = 1;

// ── Listen for form submission ──
form.addEventListener('submit', function (event) {
  event.preventDefault();  // stop page reload
  addTask();
});
What is an event listener? It's an instruction that says: "wait for this event on this element, then run this function." The submit event fires when the user clicks the Add button or presses Enter — both are covered automatically because the button uses type="submit". Without preventDefault(), the browser would try to send the form data to a server and reload the page, wiping every task from the screen.

Now add the addTask function. It reads the input, validates it, and — for now — just logs the value to the console. We'll connect it to the DOM in Step 5.

const form     = document.getElementById('task-form');
const input    = document.getElementById('task-input');

  event.preventDefault();
  addTask();
});

// ── addTask: read, validate, (render in Step 5) ──
function addTask() {
  const taskText = input.value.trim();

  // Guard: reject empty strings
  if (taskText === '') {
    alert('Please enter a task.');
    return;
  }

  // Temporary: prove the function is working
  console.log('Task received:', taskText);
}
Test checkpoint: Open the browser console (F12 → Console tab) and type a task. You should see "Task received: [your task]" appear. If you click Add with an empty input, the alert fires and nothing is logged. If you see a page reload, check that preventDefault() is present.

Why .trim()? A string of five spaces passes the !== '' check without trimming — producing an invisible empty task. .trim() removes whitespace from both ends first.

// ── Get references to HTML elements ──
const form     = document.getElementById('task-form');
const input    = document.getElementById('task-input');
const taskList = document.getElementById('task-list');
const emptyMsg = document.getElementById('empty-msg');

// ── Unique ID counter ──
let nextId = 1;

// ── Listen for form submission ──
form.addEventListener('submit', function (event) {
  event.preventDefault();
  addTask();
});

// ── addTask: read, validate, build, append ──
function addTask() {
  const taskText = input.value.trim();

  if (taskText === '') {
    alert('Please enter a task.');
    return;
  }

  // Build the <li> element in memory
  const li   = document.createElement('li');
  li.className  = 'task-item';
  li.dataset.id = nextId;

  const span = document.createElement('span');
  span.textContent = taskText;   // textContent, not innerHTML — XSS safe

  li.appendChild(span);
  taskList.appendChild(li);

  // Reset state for the next entry
  input.value           = '';
  nextId                = nextId + 1;
  emptyMsg.style.display = 'none';
}
Three stages inside addTask():
  1. Read & validate — get the trimmed value, reject empty strings early.
  2. Build — create the <li> and <span> in memory, set their content and attributes, nest them.
  3. Append & reset — move the element from memory onto the page, clear the input, increment the counter, hide the empty message.

Bug detective

Each code snippet below contains a deliberate bug. Read the code, identify the problem, then predict what the user would experience before revealing the answer.

🐛 Bug 1 — event listener
form.addEventListener('click', function (event) { event.preventDefault(); addTask(); });
What is wrong? What does the user experience?
Bug: The event type is 'click' instead of 'submit'. A click listener on a <form> fires whenever anywhere inside the form is clicked — including inside the text input. So the user experiences addTask() firing the moment they click to start typing, before they've typed anything. The alert fires immediately with "Please enter a task." Fix: change 'click' to 'submit'.
🐛 Bug 2 — missing preventDefault
form.addEventListener('submit', function (event) { addTask(); });
What happens when the user clicks Add? Why?
Bug: event.preventDefault() is missing. When a form submits without this, the browser performs its default action — sending a GET or POST request and reloading the page. The user sees the page flash and all tasks disappear instantly. The addTask() call may run briefly before the reload, but the result is never visible. Fix: add event.preventDefault(); as the first line inside the callback.
🐛 Bug 3 — innerHTML instead of textContent
const span = document.createElement('span'); span.innerHTML = taskText;
This works most of the time — but what specific risk does it introduce?
Bug: innerHTML parses the assigned string as HTML. If a user types <img src=x onerror="alert('XSS')">, the browser renders it as a real image tag and executes the onerror JavaScript. This is a Cross-Site Scripting (XSS) vulnerability — one of the most common web security flaws. textContent treats everything as plain text — tags are displayed as characters, never executed. Always use textContent for user-supplied data.
🐛 Bug 4 — wrong element reference
const taskList = document.getElementById('task-list'); // ... inside addTask() ... const list = document.getElementById('task-list'); list.appendChild(li);
This produces the correct visual output — so why is it still a problem?
Bug: Calling getElementById inside addTask() means the browser searches the entire DOM every time a task is added. With 10 tasks added quickly, that's 10 unnecessary DOM searches. The const taskList at the top of the file already holds a reference — it should be used instead. This is a performance and readability issue, not a correctness bug. In large apps with thousands of elements, repeated DOM queries become measurable. Fix: delete the const list line inside addTask() and use the outer taskList variable.
