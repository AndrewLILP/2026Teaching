Objects & Local Storage

Sprint 1's app loses every task the moment you close the browser. In this sprint you fix that — by storing tasks as structured JavaScript objects in an array, and persisting that array using localStorage. You'll also introduce a critical architectural shift: the array becomes the single source of truth, and the DOM becomes just a view of it.

Identifying & Defining Producing & Implementing
The big shift in Sprint 2: In Sprint 1, tasks existed only in the DOM. Deleting a task meant removing an HTML element. In Sprint 2, tasks live in a JavaScript array first. The DOM is rebuilt from the array whenever something changes. This "data first, view second" pattern is how every professional web framework — React, Vue, Angular — works.
Identifying & Defining

Key concepts

Three ideas power this sprint. Make sure you understand each before reading the pseudocode.

Object literal

A collection of related values grouped under one name. Each value has a key (property name) and a value. Objects let you represent a task as a single thing — not three separate variables.

// One task as an object { id: 1, text: "Buy milk", completed: false, createdAt: "2026-06-17" }
Array of objects

Multiple task objects stored in order inside an array. The array is the source of truth — to find, update, or remove a task, you work with the array, then rebuild the DOM from it.

// The tasks array let tasks = [ { id: 1, text: "Buy milk", completed: false }, { id: 2, text: "Walk dog", completed: true } ];
localStorage

A browser API that stores key-value string pairs on the user's device. Data survives page refreshes and browser restarts. Stores strings only — so objects must be converted using JSON.stringify to save and JSON.parse to load.

// Save localStorage.setItem('tasks', JSON.stringify(tasks)); // Load const raw = localStorage.getItem('tasks'); tasks = raw ? JSON.parse(raw) : [];
JSON — JavaScript Object Notation

A text format for representing structured data. JSON.stringify converts a JS object into a JSON string. JSON.parse converts it back. Used to store objects in localStorage (which only holds strings) and to transfer data between systems.

// JS object → JSON string JSON.stringify({ id: 1, text: "Buy milk" }) // → '{"id":1,"text":"Buy milk"}' // JSON string → JS object JSON.parse('{"id":1,"text":"Buy milk"}') // → { id: 1, text: "Buy milk" }

Pseudocode first — the Sprint 2 algorithm

Sprint 2 adds three new procedures to the Sprint 1 codebase: saveTasks, loadTasks, and renderTasks. The existing addTask is also updated to push to the array before rendering.

// Sprint 2 additions — NSW pseudocode // tasks is the global array; nextId tracks the next unique ID PROCEDURE saveTasks() // Convert the array to a JSON string and write to localStorage jsonString ← CONVERT tasks TO JSON STORE jsonString IN local storage WITH key "tasks" ENDPROCEDURE PROCEDURE loadTasks() // Read from localStorage; if nothing found, start with an empty array stored ← READ from local storage WITH key "tasks" IF stored EXISTS THEN tasks ← PARSE stored FROM JSON nextId ← HIGHEST id IN tasks + 1 ELSE tasks ← EMPTY ARRAY nextId ← 1 ENDIF ENDPROCEDURE PROCEDURE renderTasks() // Clear the list and rebuild it entirely from the tasks array CLEAR contents of taskList IF LENGTH of tasks = 0 THEN SHOW emptyMsg RETURN ENDIF HIDE emptyMsg FOR EACH task IN tasks li ← CREATE <li> with data-id = task.id span ← CREATE <span> with text = task.text ADD span INSIDE li ADD li TO END of taskList NEXT ENDPROCEDURE // Updated addTask — now pushes to array, saves, then renders PROCEDURE addTask() taskText ← VALUE of input taskText ← TRIM(taskText) IF taskText = "" THEN OUTPUT "Please enter a task." RETURN ENDIF newTask ← { id: nextId, text: taskText, completed: FALSE } ADD newTask TO END of tasks nextId ← nextId + 1 CALL saveTasks() CALL renderTasks() SET input VALUE ← "" ENDPROCEDURE // Updated initialise — load saved tasks before attaching events PROCEDURE initialise() // ... (element references as Sprint 1) ... CALL loadTasks() CALL renderTasks() ATTACH "submit" event on form → CALL handleSubmit ENDPROCEDURE CALL initialise()
Notice the order in addTask: push to array → save → render → clear input. The array is always updated before the DOM. If you render first and save second, a crash between those two steps leaves the screen and storage out of sync.

Producing & Implementing

Step 3 — loadTasks()

Add loadTasks. This runs once when the app starts — it reads any previously saved data and restores tasks and nextId to their last known state.

Step 3 · app.js — complete file
NEW IN STEP 3 — loadTasks function
// ── Element references ──
const form     = document.getElementById('task-form');
const input    = document.getElementById('task-input');
const taskList = document.getElementById('task-list');
const emptyMsg = document.getElementById('empty-msg');

// ── Data ──
let tasks  = [];
let nextId = 1;

// ── saveTasks ──
function saveTasks() {
  localStorage.setItem('tasks', JSON.stringify(tasks));
}

// ── loadTasks: restore array from localStorage on startup ──
function loadTasks() {
  const stored = localStorage.getItem('tasks');
  if (stored) {
    tasks  = JSON.parse(stored);
    // nextId must be higher than every existing id
    nextId = Math.max(...tasks.map(t => t.id)) + 1;
  }
  // If nothing stored: tasks stays [], nextId stays 1
}
Why recalculate nextId? If we always started nextId at 1, every session would re-use IDs — a task added after a refresh could get id=1, colliding with a task that already exists in the saved data. Sprint 4's delete logic finds tasks by ID, so IDs must be unique across sessions.

What is Math.max(...tasks.map(t => t.id))? tasks.map(t => t.id) produces an array of all current IDs e.g. [1, 2, 3]. The spread operator ... unpacks that array into separate arguments. Math.max(1, 2, 3) returns 3. Adding 1 gives nextId = 4.

Add renderTasks. Instead of appending one item at a time (Sprint 1), this function clears the list and rebuilds the entire DOM from the array. It will be called after every change.

// ── Element references ──
const form     = document.getElementById('task-form');
const input    = document.getElementById('task-input');
const taskList = document.getElementById('task-list');
const emptyMsg = document.getElementById('empty-msg');

// ── Data ──
let tasks  = [];
let nextId = 1;

// ── saveTasks ──
function saveTasks() {
  localStorage.setItem('tasks', JSON.stringify(tasks));
}

// ── loadTasks ──
function loadTasks() {
  const stored = localStorage.getItem('tasks');
  if (stored) {
    tasks  = JSON.parse(stored);
    nextId = Math.max(...tasks.map(t => t.id)) + 1;
  }
}

// ── renderTasks: wipe the list and rebuild from the array ──
function renderTasks() {
  taskList.innerHTML = '';  // clear existing items

  if (tasks.length === 0) {
    emptyMsg.style.display = '';  // show "No tasks yet"
    return;
  }

  emptyMsg.style.display = 'none';

  tasks.forEach(function (task) {
    const li   = document.createElement('li');
    li.className  = 'task-item';
    li.dataset.id = task.id;

    const span = document.createElement('span');
    span.textContent = task.text;

    li.appendChild(span);
    taskList.appendChild(li);
  });
}
Why taskList.innerHTML = '' here but textContent for user data? Setting innerHTML to an empty string is safe — we are not inserting any user content, just clearing children. The XSS risk from Sprint 1 only applies when user-supplied strings are written into innerHTML. The task text still uses span.textContent.

forEach vs a FOR loop: Both work. forEach is idiomatic for iterating arrays in modern JS and matches what you'll see in professional code. The pseudocode used FOR EACH — the pattern is identical.

// ── Element references ──
const form     = document.getElementById('task-form');
const input    = document.getElementById('task-input');
const taskList = document.getElementById('task-list');
const emptyMsg = document.getElementById('empty-msg');

// ── Data ──
let tasks  = [];
let nextId = 1;

// ── saveTasks ──
function saveTasks() {
  localStorage.setItem('tasks', JSON.stringify(tasks));
}

// ── loadTasks ──
function loadTasks() {
  const stored = localStorage.getItem('tasks');
  if (stored) {
    tasks  = JSON.parse(stored);
    nextId = Math.max(...tasks.map(t => t.id)) + 1;
  }
}

// ── renderTasks ──
function renderTasks() {
  taskList.innerHTML = '';
  if (tasks.length === 0) { emptyMsg.style.display = ''; return; }
  emptyMsg.style.display = 'none';
  tasks.forEach(function (task) {
    const li   = document.createElement('li');
    li.className  = 'task-item';
    li.dataset.id = task.id;
    const span = document.createElement('span');
    span.textContent = task.text;
    li.appendChild(span);
    taskList.appendChild(li);
  });
}

// ── addTask: push to array → save → render → clear input ──
function addTask() {
  const taskText = input.value.trim();
  if (taskText === '') { alert('Please enter a task.'); return; }

  const newTask = { id: nextId, text: taskText, completed: false };
  tasks.push(newTask);
  nextId++;
  saveTasks();
  renderTasks();
  input.value = '';
}

// ── handleSubmit ──
function handleSubmit(event) {
  event.preventDefault();
  addTask();
}

// ── initialise: load saved data, render, then listen ──
function initialise() {
  loadTasks();
  renderTasks();
  form.addEventListener('submit', handleSubmit);
}

Bug detective

Each snippet has a deliberate bug. Read, predict the failure, then reveal the answer.

🐛 Bug 1 — storing directly without stringify
function saveTasks() { localStorage.setItem('tasks', tasks); }
What does localStorage actually store? What happens on the next page load?
Bug: When an array is coerced to a string without JSON.stringify, JavaScript calls .toString() on it — producing something like "[object Object],[object Object]". On the next load, JSON.parse("[object Object]") throws a SyntaxError and the app crashes. Fix: always use JSON.stringify(tasks) before storing.
🐛 Bug 2 — parse without null check
function loadTasks() { tasks = JSON.parse(localStorage.getItem('tasks')); }
What happens when a user opens the app for the very first time (no data saved)?
Bug: When no data is saved, getItem('tasks') returns null. JSON.parse(null) does not throw an error — it returns null. So tasks becomes null. Then renderTasks() calls tasks.forEach(…) — and null.forEach throws a TypeError: Cannot read properties of null, crashing the app on first use. Fix: add the if (stored) guard before parsing.
🐛 Bug 3 — render before save
function addTask() { const taskText = input.value.trim(); if (taskText === '') { return; } const newTask = { id: nextId, text: taskText, completed: false }; tasks.push(newTask); nextId++; renderTasks(); // ← render BEFORE save saveTasks(); input.value = ''; }
The app appears to work fine. So why is this still a bug?
Bug: Under normal conditions this works. But if saveTasks() throws an error (e.g. localStorage is full — a real browser limit around 5 MB), the DOM has already been updated to show the new task, but it was never persisted. On the next refresh, the task is gone — but the user saw it and believes it was saved. This creates a data integrity problem that is very hard to debug. The correct order is always: mutate data → persist data → update view. This is the same pattern used in databases (write first, confirm, then notify the user).
🐛 Bug 4 — hardcoded nextId after load
function loadTasks() { const stored = localStorage.getItem('tasks'); if (stored) { tasks = JSON.parse(stored); nextId = 1; // always reset to 1 } }
You have three saved tasks (ids 1, 2, 3). You refresh the page and add a fourth task. What ID does it get? Why is this a problem now — and a bigger problem in Sprint 4?
Bug: nextId resets to 1, so the new task gets id: 1 — colliding with the first existing task. In Sprint 2 this is invisible (nothing uses IDs yet). But in Sprint 4, the delete function will find a task "by ID" — and with two tasks sharing id: 1, deleting one may delete both, or neither, depending on the implementation. Fix: set nextId to Math.max(...tasks.map(t => t.id)) + 1 after loading.
