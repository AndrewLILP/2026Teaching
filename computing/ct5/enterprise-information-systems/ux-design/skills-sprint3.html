Filtering & Searching

Your task tracker now persists data — but every task is always visible, all at once. In this sprint you add two ways to narrow the view: filter buttons (All / Active / Done) and a live search field. You'll learn the most important architectural principle in front-end development — never mutate the data to change the view. The array stays unchanged; only the rendered output changes.

Identifying & Defining Producing & Implementing
The key insight of Sprint 3: When a user clicks "Done", you do not delete tasks from the array. You set a variable (activeFilter) that tells renderTasks() which tasks to show. The array is always complete. The view is always a subset. This is how spreadsheet filters, database queries, and every search engine works.
Identifying & Defining

Key concepts

Data layer — never changes for display
tasks[ ] array

All tasks, always intact. Modified only when tasks are added, edited, or deleted.

→
filter()
+ search
View layer — rebuilt on every change
Rendered <ul>

A filtered, searched subset of the array. Thrown away and rebuilt whenever renderTasks() runs.

array.filter()

Returns a new array containing only the elements for which the callback returns true. The original array is never modified. The callback receives each element and must return a boolean.

// Keep only incomplete tasks const active = tasks.filter(function (t) { return t.completed === false; }); // tasks is unchanged; active is a new array
String.includes()

Returns true if the string contains the given substring. Case-sensitive by default — so we convert both strings to lowercase before comparing, giving us a case-insensitive search.

// Case-insensitive search const query = "milk"; "Buy Milk".toLowerCase().includes(query) // → true ("buy milk".includes("milk")) "Walk dog".toLowerCase().includes(query) // → false
Chained .filter()

Because filter() returns an array, you can chain a second filter() directly onto the result. Status filter first, then search filter — both applied in a single expression.

const visible = tasks .filter(t => statusMatch(t)) .filter(t => t.text .toLowerCase() .includes(query)); // tasks is still unchanged
State variable

A variable that records the current mode of the UI — not data, but a description of what the user has selected. activeFilter holds the current filter: 'all', 'active', or 'done'. Changing it triggers a re-render.

let activeFilter = 'all'; // When user clicks "Done": activeFilter = 'done'; renderTasks(); // re-render with new state

Identifying & Defining

Pseudocode first — the Sprint 3 algorithm

Sprint 3 adds two new state variables and updates renderTasks() to use them. No other existing procedures change.

// Sprint 3 additions — NSW pseudocode // New state variables (added to the top of app.js) activeFilter ← "all" // "all" | "active" | "done" searchQuery ← "" // current search string (lowercase) PROCEDURE renderTasks() // replaces the Sprint 2 version // Step 1: apply status filter IF activeFilter = "active" THEN visible ← tasks WHERE completed = FALSE ELSE IF activeFilter = "done" THEN visible ← tasks WHERE completed = TRUE ELSE visible ← ALL tasks ENDIF // Step 2: apply search filter on top of status filter IF searchQuery ≠ "" THEN visible ← visible WHERE LOWERCASE(text) CONTAINS searchQuery ENDIF // Step 3: clear the DOM and rebuild from visible subset CLEAR taskList IF LENGTH of visible = 0 THEN SHOW emptyMsg RETURN ENDIF HIDE emptyMsg FOR EACH task IN visible li ← CREATE <li> with data-id = task.id span ← CREATE <span> with text = task.text ADD span INSIDE li ADD li TO taskList NEXT ENDPROCEDURE PROCEDURE setFilter(filterValue) activeFilter ← filterValue CALL renderTasks() ENDPROCEDURE PROCEDURE handleSearch() searchQuery ← LOWERCASE(VALUE of search input) CALL renderTasks() ENDPROCEDURE
Notice: setFilter and handleSearch do the same two things — update one state variable, then call renderTasks(). This is the pattern: change state, then re-render. The render function always reads current state fresh. It never needs to know what changed — only what the state is now.

data-filter attributes: Each button stores its filter value in a data-* attribute. JavaScript reads this with btn.dataset.filter — no need for separate IDs per button.

aria-pressed: Toggle buttons need aria-pressed="true" when active so screen readers announce their state. The "All" button starts pressed; the others start unpressed. JavaScript will update these values when a button is clicked.

type="search": Tells the browser this is a search field — it may render a clear (✕) button automatically and announces it correctly to assistive technology.

Step 3 — Update renderTasks() to use filters

Replace the Sprint 2 version of renderTasks(). The structure is the same — clear and rebuild — but now it filters before rendering.

Step 3 · app.js — renderTasks() replacement
NEW IN STEP 3 — filter logic replaces the old renderTasks body
// ── renderTasks: filter → search → clear → rebuild ──
function renderTasks() {

  // 1. Apply status filter
  let visible = tasks.filter(function (t) {
    if (activeFilter === 'active') { return !t.completed; }
    if (activeFilter === 'done')   { return  t.completed; }
    return true;  // 'all' — include everything
  });

  // 2. Apply search filter on top
  if (searchQuery) {
    visible = visible.filter(function (t) {
      return t.text.toLowerCase().includes(searchQuery);
    });
  }

  // 3. Clear and rebuild DOM from visible subset
  taskList.innerHTML = '';
  if (visible.length === 0) {
    emptyMsg.style.display = '';
    return;
  }
  emptyMsg.style.display = 'none';
  visible.forEach(function (task) {
    const li   = document.createElement('li');
    li.className  = 'task-item';
    li.dataset.id = task.id;
    const span = document.createElement('span');
    span.textContent = task.text;
    li.appendChild(span);
    taskList.appendChild(li);
  });
}
Why filter twice instead of one combined filter? Two chained filters are clearer and easier to change independently. If you later want to add a third filter (e.g. by priority), you add a third .filter() call — you don't rewrite the existing logic. Each filter has one responsibility.

Note on the empty state: If the user is in "Done" mode and there are no completed tasks, the empty message appears — even though the full tasks array has items. This is correct: "no results for this filter" is different from "no tasks at all." Consider updating emptyMsg's text to reflect this in the Extension task.

Step 4 — Filter button event handlers

Add setFilter() and attach it to each button. When a button is clicked, update activeFilter, update the active button's visual state, and re-render.

Step 4 · app.js — setFilter and button loop
NEW IN STEP 4 — setFilter function
// ── setFilter: update state, update button UI, re-render ──
function setFilter(filterValue) {
  activeFilter = filterValue;

  // Update aria-pressed and active class on all buttons
  filterBtns.forEach(function (btn) {
    const isActive = btn.dataset.filter === filterValue;
    btn.classList.toggle('active', isActive);
    btn.setAttribute('aria-pressed', isActive);
  });

  renderTasks();
}
classList.toggle(className, force): When force is true, the class is added; when false, it is removed. This replaces an if/else — one line handles both states.

Why update aria-pressed? The CSS .active class changes the visual style, but screen readers don't see CSS. aria-pressed communicates the toggle state to assistive technology — both must be updated together.

Step 5 — Live search handler

Add handleSearch(). The search updates on every keystroke using the input event — not change, which only fires when the field loses focus.

Step 5 · app.js — handleSearch function
NEW IN STEP 5 — handleSearch function
// ── handleSearch: update searchQuery state and re-render ──
function handleSearch() {
  searchQuery = searchInput.value.toLowerCase().trim();
  renderTasks();
}
Why .toLowerCase() here? We store the query in lowercase so the comparison in renderTasks() is simple: task.text.toLowerCase().includes(searchQuery). Both sides are lowercase — the match is case-insensitive without needing a regex.

input event vs keyup: input fires on every character change — typing, paste, delete, autocomplete. keyup only fires on keyboard events, missing paste and speech input. Always use input for live search.

const form        = document.getElementById('task-form');

// ── UI state ──
let activeFilter = 'all';

// ── saveTasks / loadTasks (unchanged from Sprint 2) ──
function saveTasks() {
  localStorage.setItem('tasks', JSON.stringify(tasks));
}
function loadTasks() {
  const stored = localStorage.getItem('tasks');
  if (stored) {
    tasks  = JSON.parse(stored);
    nextId = Math.max(...tasks.map(t => t.id)) + 1;
  }
}

// ── renderTasks (updated Sprint 3 version with filters) ──
function renderTasks() {
  let visible = tasks.filter(function (t) {
    if (activeFilter === 'active') { return !t.completed; }
    if (activeFilter === 'done')   { return  t.completed; }
    return true;
  });
  if (searchQuery) {
    visible = visible.filter(t => t.text.toLowerCase().includes(searchQuery));
  }
  taskList.innerHTML = '';
  if (visible.length === 0) { emptyMsg.style.display = ''; return; }
  emptyMsg.style.display = 'none';
  visible.forEach(function (task) {
    const li = document.createElement('li');
    li.className = 'task-item'; li.dataset.id = task.id;
    const span = document.createElement('span');
    span.textContent = task.text;
    li.appendChild(span); taskList.appendChild(li);
  });
}

// ── setFilter / handleSearch (Sprint 3) ──
function setFilter(filterValue) {
  activeFilter = filterValue;
  filterBtns.forEach(function (btn) {
    const isActive = btn.dataset.filter === filterValue;
    btn.classList.toggle('active', isActive);
    btn.setAttribute('aria-pressed', isActive);
  });
  renderTasks();
}
function handleSearch() {
  searchQuery = searchInput.value.toLowerCase().trim();
  renderTasks();
}

// ── addTask / handleSubmit (unchanged from Sprint 2) ──
function addTask() {
  const taskText = input.value.trim();
  if (taskText === '') { alert('Please enter a task.'); return; }
  tasks.push({ id: nextId, text: taskText, completed: false });
  nextId++;
  saveTasks(); renderTasks();
  input.value = '';
}
function handleSubmit(e) { e.preventDefault(); addTask(); }

// ── initialise: load, render, attach all listeners ──
function initialise() {
  loadTasks();
  renderTasks();
  form.addEventListener('submit', handleSubmit);
  filterBtns.forEach(function (btn) {
    btn.addEventListener('click', function () {
      setFilter(btn.dataset.filter);
    });
  });
  searchInput.addEventListener('input', handleSearch);
}

Bug detective

Each snippet has a deliberate bug. Identify the problem before revealing the answer.

🐛 Bug 1 — mutating the array to filter
function setFilter(filterValue) { if (filterValue === 'active') { tasks = tasks.filter(t => !t.completed); // reassign tasks! } renderTasks(); }
The filtered view looks correct. But what happens when the user clicks "All"? And what data has been lost?
Bug: tasks = tasks.filter(...) permanently overwrites the tasks array — removing the completed tasks from the data itself, not just from the view. When the user clicks "All", renderTasks() rebuilds from tasks — which now only contains incomplete tasks. The completed tasks are gone forever (unless localStorage still has the old version). The fix: always filter into a local visible variable and never reassign tasks for display purposes.
🐛 Bug 2 — case-sensitive search
function handleSearch() { searchQuery = searchInput.value.trim(); // no toLowerCase() renderTasks(); } // Inside renderTasks: return t.text.includes(searchQuery); // no toLowerCase()
A user types "milk". The task "Buy Milk" exists. What happens? Why?
Bug: "Buy Milk".includes("milk") returns false because JavaScript string comparison is case-sensitive. "Milk" ≠ "milk". The user types in lowercase but the stored task text has a capital M — so no match is found and the task disappears from the filtered view. Fix: convert both sides to lowercase before comparing: t.text.toLowerCase().includes(searchQuery.toLowerCase()). In our implementation we store searchQuery already lowercased, so only t.text.toLowerCase() is needed at the filter site.
🐛 Bug 3 — search on keyup instead of input
searchInput.addEventListener('keyup', handleSearch);
This mostly works. Describe two specific scenarios where it fails.
Bug: keyup only fires for physical keyboard events. It misses: (1) paste — if the user pastes text with Ctrl+V or right-click → Paste, keyup does not fire and the search field updates visually but the task list doesn't filter. (2) voice input / speech-to-text — text inserted by speech recognition triggers input but not keyup. The input event covers all content change mechanisms and is the correct choice for any live field.
🐛 Bug 4 — aria-pressed not updated
function setFilter(filterValue) { activeFilter = filterValue; filterBtns.forEach(function (btn) { btn.classList.toggle('active', btn.dataset.filter === filterValue); // aria-pressed line is missing }); renderTasks(); }
The buttons look correct visually. Who is affected by this bug, and why?
Bug: Screen reader users are affected. The .active CSS class changes the visual appearance of the button, but screen readers don't read CSS — they read the accessibility tree. Without updating aria-pressed, a screen reader user hears "button, All" with no indication of whether it is currently selected or not. Sighted users see the highlight; screen reader users are left guessing. Fix: always pair visual state changes with the corresponding ARIA attribute update.
