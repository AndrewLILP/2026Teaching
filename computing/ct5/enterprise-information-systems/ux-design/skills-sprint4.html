Delete, Edit & Complete

This sprint closes the CRUD loop. You already have Create (Sprint 1) and Read (Sprint 3's filtered view). Now you add Update — marking tasks complete and editing text in-place — and Delete, with a confirmation step. This is also the first sprint with a formal testing phase: you write test cases before testing, then desk-check edge cases before running the app.

Producing & Implementing Testing & Evaluating
The data mutation pattern: Every operation in this sprint follows the same three steps — find the task object by id, mutate it (change a property or remove it from the array), then save and re-render. The DOM is never directly modified — only the array, then the DOM is rebuilt from scratch.

array.findIndex()

Returns the index of the first element where the callback returns true, or -1 if not found. Used to locate a task by its id before mutating the array.

// Find index of task with id 2 const idx = tasks.findIndex(t => t.id === 2); // idx = 1 (second element, zero-indexed) // If not found: idx = -1
array.splice()

Removes and/or inserts elements at a given index, mutating the original array. For deletion: splice(index, 1) removes exactly one element at that position.

// Remove element at index 1 tasks.splice(1, 1); // tasks now has one fewer element // Original array IS modified (unlike filter)
In-place property mutation

To update a task, find its index, then directly reassign a property on that object. Because tasks[idx] is a reference to the object, the change affects the array immediately.

const idx = tasks.findIndex(t => t.id === id); if (idx !== -1) { tasks[idx].completed = true; // mutate tasks[idx].text = newText; // mutate }
event.target + dataset

Instead of attaching a listener to each button, attach one listener to the parent <ul>. When a button inside it is clicked, event.target is that button. dataset.id reads its data-id attribute. This is called event delegation.

taskList.addEventListener('click', function(e) { const btn = e.target.closest('button'); if (!btn) return; const id = +btn.closest('li').dataset.id; // id is now the task's numeric id });
Why event delegation? Tasks are added and removed at runtime, so attaching listeners to each button inside renderTasks() means re-attaching on every render. One listener on the stable parent element handles all current and future buttons automatically.

Pseudocode first — the Sprint 4 algorithm

Four new procedures, all following the same pattern: find → mutate (or remove) → save → re-render.

// Sprint 4 — NSW pseudocode // All four procedures follow: find → mutate → save → render PROCEDURE toggleComplete(taskId) idx ← INDEX of task in tasks WHERE id = taskId IF idx ≠ -1 THEN tasks[idx].completed ← NOT tasks[idx].completed CALL saveTasks() CALL renderTasks() ENDIF ENDPROCEDURE PROCEDURE deleteTask(taskId) idx ← INDEX of task in tasks WHERE id = taskId IF idx ≠ -1 THEN confirmed ← ASK user "Delete this task?" IF confirmed THEN REMOVE element at idx from tasks CALL saveTasks() CALL renderTasks() ENDIF ENDIF ENDPROCEDURE PROCEDURE startEdit(taskId) // Replace the span text with an inline input field idx ← INDEX of task in tasks WHERE id = taskId IF idx ≠ -1 THEN li ← DOM element with data-id = taskId span ← SPAN inside li REPLACE span WITH <input> pre-filled with tasks[idx].text FOCUS the <input> ENDIF ENDPROCEDURE PROCEDURE saveEdit(taskId, newText) newText ← TRIM(newText) IF newText = "" THEN CALL renderTasks() // cancel edit — restore original RETURN ENDIF idx ← INDEX of task in tasks WHERE id = taskId IF idx ≠ -1 THEN tasks[idx].text ← newText CALL saveTasks() CALL renderTasks() ENDIF ENDPROCEDURE
Before coding: Copy each procedure into your notebook. For each one, identify: (a) what it finds, (b) what it mutates, and (c) what would happen if the if (idx !== -1) guard was missing. The guard is your safety net — write it before the mutation, every time.

task.completed ? ' completed' : '' — a ternary expression adds the completed CSS class conditionally. The CSS for .task-item.completed can strike through the text and dim the item.

li.append(span, btnDone, btnEdit, btnDel) — append() accepts multiple arguments, inserting them all in one call. Cleaner than four separate appendChild() calls.

Step 2 — toggleComplete()

Step 2 · app.js — toggleComplete function
NEW IN STEP 2 — toggleComplete
// ── toggleComplete: flip completed flag, save, re-render ──
function toggleComplete(taskId) {
  const idx = tasks.findIndex(function (t) { return t.id === taskId; });
  if (idx === -1) { return; }  // guard: task not found
  tasks[idx].completed = !tasks[idx].completed;
  saveTasks();
  renderTasks();
}
!tasks[idx].completed — the logical NOT operator flips the boolean. If it was true it becomes false; if false it becomes true. One line replaces an if/else.

Test checkpoint: After adding this function (and the event wiring in Step 6), clicking ✓ should add the completed class to the item and change the button to ↩. Clicking ↩ should restore it. Refresh — the completed state should persist.
Producing & Implementing

Step 3 — deleteTask()

Step 3 · app.js — deleteTask function
NEW IN STEP 3 — deleteTask
// ── deleteTask: confirm, remove from array, save, re-render ──
function deleteTask(taskId) {
  const idx = tasks.findIndex(function (t) { return t.id === taskId; });
  if (idx === -1) { return; }
  if (!confirm('Delete "' + tasks[idx].text + '"?')) { return; }
  tasks.splice(idx, 1);
  saveTasks();
  renderTasks();
}
Why confirm()? Deletion is irreversible. A confirmation step prevents accidental loss — a basic usability principle (Nielsen heuristic: error prevention). The task text is included in the message so the user confirms they're deleting the right item.

tasks.splice(idx, 1) — removes exactly one element at position idx. The array length decreases by 1. Unlike filter(), splice() mutates the original array — which is exactly what we want here.

Step 4 — startEdit()

In-place editing replaces the task's <span> with an <input> — without re-rendering the whole list. The input is pre-filled and focused so the user can edit immediately.

Step 4 · app.js — startEdit function
NEW IN STEP 4 — startEdit
// ── startEdit: swap span for an inline input field ──
function startEdit(taskId) {
  const idx = tasks.findIndex(function (t) { return t.id === taskId; });
  if (idx === -1) { return; }

  // Find the live DOM element for this task
  const li   = taskList.querySelector(`[data-id="${taskId}"]`);
  const span = li.querySelector('span');

  // Build the inline edit input
  const editInput = document.createElement('input');
  editInput.type        = 'text';
  editInput.value       = tasks[idx].text;
  editInput.className   = 'task-edit-input';
  editInput.maxLength   = 120;
  editInput.setAttribute('aria-label', 'Edit task text');

  // Commit edit on Enter; cancel on Escape
  editInput.addEventListener('keydown', function (e) {
    if (e.key === 'Enter')  { saveEdit(taskId, editInput.value); }
    if (e.key === 'Escape') { renderTasks(); }
  });
  editInput.addEventListener('blur', function () {
    saveEdit(taskId, editInput.value);
  });

  li.replaceChild(editInput, span);
  editInput.focus();
  editInput.select();  // select all text for easy replacement
}
Three ways to commit the edit: Enter key, clicking away (blur), or calling saveEdit directly. Escape cancels by calling renderTasks(), which rebuilds the list from the unchanged array — discarding the input.

editInput.select() highlights the existing text when the input appears — the user can type immediately to replace it, or use arrow keys to position the cursor. A small UX detail that makes editing feel fast.

Why blur? If the user clicks elsewhere (e.g. another task) without pressing Enter, the blur event fires and the edit is saved — the change is never silently lost.

Step 5 — saveEdit()

Step 5 · app.js — saveEdit function
NEW IN STEP 5 — saveEdit
// ── saveEdit: validate, update array, save, re-render ──
function saveEdit(taskId, newText) {
  const trimmed = newText.trim();
  if (trimmed === '') {
    renderTasks();  // cancel — restore original text
    return;
  }
  const idx = tasks.findIndex(function (t) { return t.id === taskId; });
  if (idx === -1) { return; }
  tasks[idx].text = trimmed;
  saveTasks();
  renderTasks();
}
Empty edit → cancel, not delete: If the user clears the field and blurs or presses Enter, an empty string would replace the task text. Instead, we call renderTasks() which rebuilds from the original (unchanged) array — the edit is silently discarded. This is safer than deleting or creating an empty task.

// ── (all Sprint 1–3 code unchanged above this point) ──
// toggleComplete, deleteTask, startEdit, saveEdit defined above

// ── handleTaskClick: one listener handles all task buttons ──
function handleTaskClick(event) {
  const btn = event.target.closest('button[data-action]');
  if (!btn) { return; }  // click was not on an action button

  const li     = btn.closest('li');
  const taskId = +li.dataset.id;   // + coerces string to number

  switch (btn.dataset.action) {
    case 'toggle': toggleComplete(taskId); break;
    case 'edit':   startEdit(taskId);      break;
    case 'delete': deleteTask(taskId);     break;
  }
}

// ── initialise: updated to add task list delegation ──
function initialise() {
  loadTasks();
  renderTasks();
  form.addEventListener('submit', handleSubmit);
  filterBtns.forEach(function (btn) {
    btn.addEventListener('click', function () { setFilter(btn.dataset.filter); });
  });
  searchInput.addEventListener('input', handleSearch);
  taskList.addEventListener('click', handleTaskClick);  // delegation
}

initialise();
event.target.closest('button[data-action]') — closest() walks up the DOM tree from the clicked element until it finds a match. If the user clicks the text inside a button, event.target is the text node; closest finds the parent button. Returns null if no match — the guard if (!btn) return handles clicks on the list item background.

+li.dataset.id — the unary + operator converts the string "2" to the number 2. findIndex uses strict equality (===), so comparing "2" === 2 would return false. Type coercion is the most common source of subtle ID-matching bugs.

Bug detective

Each snippet has a deliberate bug. Identify the problem before revealing the answer.

🐛 Bug 1 — no idx guard in toggleComplete
function toggleComplete(taskId) { const idx = tasks.findIndex(t => t.id === taskId); tasks[idx].completed = !tasks[idx].completed; // no guard saveTasks(); renderTasks(); }
Under what specific condition does this crash? What error does it produce?
Bug: If findIndex returns -1 (task not found), tasks[-1] is undefined in JavaScript. Accessing undefined.completed throws a TypeError: Cannot read properties of undefined, crashing the function. While this is rare in normal use, it can happen if the task was deleted from another tab, or if taskId is incorrectly coerced (e.g. string vs number mismatch). The guard if (idx === -1) return; prevents the crash entirely.
🐛 Bug 2 — string vs number id comparison
function handleTaskClick(event) { const btn = event.target.closest('button[data-action]'); if (!btn) { return; } const li = btn.closest('li'); const taskId = li.dataset.id; // no + conversion toggleComplete(taskId); }
dataset.id returns a string. findIndex uses ===. What happens?
Bug: dataset.id returns the string "2". Inside toggleComplete, findIndex(t => t.id === taskId) compares the number 2 (stored in the array) with the string "2" using strict equality. 2 === "2" is false, so findIndex returns -1 — the task is never found and nothing happens. The fix is to coerce with the unary +: const taskId = +li.dataset.id;. This is one of the most common JavaScript bugs in DOM-to-data interactions.
🐛 Bug 3 — attaching listeners inside renderTasks
// Inside forEach in renderTasks: btnDel.addEventListener('click', function() { deleteTask(task.id); });
The app works correctly. But what performance problem occurs after 20 task additions and deletions?
Bug: Every time renderTasks() runs, it clears the DOM with innerHTML = '' and recreates all elements — attaching fresh event listeners each time. In a short session with many operations this accumulates no obvious issue. But the pattern scales badly: with hundreds of tasks and frequent re-renders (e.g. live search), hundreds of listeners are created and the old ones linger in memory (even if their elements are gone), causing a memory leak. Event delegation — one listener on the stable parent — runs once and handles all buttons forever, regardless of how many tasks are added or removed.
🐛 Bug 4 — empty edit saves empty string
function saveEdit(taskId, newText) { const idx = tasks.findIndex(t => t.id === taskId); if (idx === -1) { return; } tasks[idx].text = newText; // no trim, no empty check saveTasks(); renderTasks(); }
The user opens a task for editing, accidentally clears the field, and clicks away. What does the user see? What is in localStorage?
Bug: The empty string (or a string of spaces) is saved as the task text. The task now renders with an invisible or blank label — it exists in the list and in localStorage, but the user cannot see it. They cannot easily delete it because the text in the confirm dialog will also be blank. Fix: trim the value and if empty, call renderTasks() to discard the edit and restore the original text from the array.
