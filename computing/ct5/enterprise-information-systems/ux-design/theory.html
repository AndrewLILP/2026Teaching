Designing for User Experience

Every screen you've ever used was designed — or failed to be. This focus area explores how interfaces are created to serve real people: from understanding user needs and accessibility requirements, through to designing, prototyping, and evaluating interactive media products that communicate effectively across diverse audiences.

📋 Teaching advice

Students apply computing skill domains to develop projects using human-centred design. This focus area can be combined with Developing Apps and Web Software (design the UI/UX for your app) or Creating Games and Simulations (develop an educational game using interactive and multimedia products).

Key vocabulary

These four terms are the foundation of this focus area. Study each Frayer card — definition, characteristics, examples, and non-examples — before moving to the content below.

📖 Word bank

User Experience (UX)
Definition
The overall quality of a person's interaction with a product — how it feels to use, not just how it looks.
Characteristics
Subjective; considers emotion, efficiency, satisfaction; shaped by user goals and context.
Examples
Checking out in 2 taps; finding a song without searching; a form that explains its errors clearly.
Non-examples
A beautiful homepage that takes 12 seconds to load; a confusing checkout flow despite nice colours.
User Interface (UI)
Definition
The visual and interactive elements a user directly sees and uses — buttons, menus, icons, forms, screens.
Characteristics
Visual; tangible; includes layout, colour, typography, and interactive controls.
Examples
Navigation bar; login form; hamburger menu; toggle switch; modal dialog.
Non-examples
The server-side code running behind the screen; the database storing user data; the API handling requests.
Accessibility
Definition
Designing products so they can be used by the widest possible range of people, including those with disability.
Characteristics
Perceivable, Operable, Understandable, Robust (WCAG POUR principles); often a legal requirement.
Examples
Captions on videos; alt text on images; keyboard-only navigation; sufficient colour contrast.
Non-examples
Colour-only error indicators; images with no alt text; videos with no captions; mouse-only interactions.
Usability
Definition
How effectively, efficiently, and satisfactorily a specific user can achieve a specific goal with a product.
Characteristics
Measurable; tied to specific tasks; can be tested with real users; shaped by learnability, efficiency, errors.
Examples
A new user completes sign-up in under 60 seconds; error messages explain what went wrong and how to fix it.
Non-examples
A password reset that requires users to check three places; navigation that changes between pages.
Identifying & Defining

Identifying & Defining

Before designing anything, UX practitioners identify the problem they are solving, who they are solving it for, and what constraints shape the solution. This phase produces the requirements that every design decision is later measured against.

User interfaces and interactive media exist to allow people and computing systems to communicate. Before designing:

  • Describe the purpose of the UI or media product — what does it allow users to do?
  • Identify how inputs, storage, transmission, processes and outputs operate within the interface
  • Consider how the interface affects the user experience — not just whether it works, but how it feels
Think about it: The same data (a weather forecast) can be delivered as a text list, a radar map, an animated widget, or a voice assistant response. Each is a different UI — and each creates a different user experience.

A requirements specification defines what a UI or interactive media product must do and how well it must do it.

  • Functional requirements — specific features and behaviours: "The user can filter results by category."
  • Non-functional requirements — quality attributes: accessibility level (WCAG AA), load time, branding consistency, cognitive load minimisation
  • Use cases — scenarios describing how a specific user achieves a specific goal
  • Test cases — defined inputs, expected outputs, and pass conditions used during evaluation
Non-functional examples for UX: Branding and marketing consistency; minimising cognitive load and physical movement; meeting WCAG 2.1 Level AA; response to input within 100ms.
  • Consider the social impacts of UI/interactive media products on diverse communities
  • Analyse Terms of Use policies and End User Licence Agreements (EULAs) — how do they affect data protection and user rights?
  • Investigate the influence of communication methods on audiences, including the perspectives of:
    • Aboriginal and Torres Strait Islander Peoples
    • Culturally and linguistically diverse communities
    • People of different ages, genders, and abilities
Example: Subtitles benefit people who are d/Deaf or hard of hearing — and also people watching in noisy environments, EAL/D learners, and anyone without headphones. Accessibility improvements benefit everyone.

Interactive media products use large amounts of data — images, video, audio. Compression reduces file size to improve performance.

  • Lossy compression — permanently removes data to achieve smaller file sizes. Quality degrades with repeated compression. Used for JPEG images, MP3 audio, MP4 video.
  • Lossless compression — reduces file size without losing any data. Original can be perfectly reconstructed. Used for PNG images, FLAC audio, ZIP archives.
UX trade-off: A heavily compressed JPEG loads faster but looks pixelated. A lossless PNG looks perfect but may slow page load — increasing cognitive load through frustration and delay. UX designers must balance quality and performance.
Researching & Planning

Researching & Planning

UX designers don't invent from nothing — they analyse existing systems, research how interfaces have evolved, and evaluate alternatives before committing to a design direction.

User interfaces have changed dramatically as hardware, software, and human expectations evolved. Understanding this history helps designers recognise patterns and anticipate what's next.

1950s–60s
Punch cards & batch processing

No real-time interaction — instructions punched onto cards, submitted to an operator, results returned hours later. Zero visual feedback.

1970s–80s
Command-line interfaces (CLI)

Text typed at a prompt; results printed to screen. Powerful but required memorising commands. No mouse, no icons.

1984–2000s
Graphical user interfaces (GUI)

Windows, icons, menus, pointer (WIMP). The mouse made computing accessible to non-experts. Popularised by Macintosh, then Windows.

2007–present
Touch & gesture interfaces

Smartphone touchscreens replaced physical keyboards. Swipe, pinch, tap. Designed for thumbs and small screens — a complete rethink of UI conventions.

Now & emerging
Voice, AR/MR/VR & intelligent interfaces

Voice assistants (Siri, Alexa), augmented reality overlays, virtual reality environments, and AI-driven interfaces that adapt to the user. Augmentative and alternative communication (AAC) technology expanding access.

Designers use established principles to evaluate and improve interfaces. Nielsen's 10 Usability Heuristics are the most widely used in the industry:

  • Visibility of system status — always keep users informed about what's happening
  • Match between system and real world — use language and concepts the user knows
  • User control and freedom — provide undo, redo, and easy exits
  • Consistency and standards — follow platform conventions; don't reinvent familiar patterns
  • Error prevention — design to prevent problems before they occur
  • Recognition over recall — make options visible; don't rely on user memory
  • Flexibility and efficiency of use — support both novice and expert users
  • Aesthetic and minimalist design — don't include irrelevant information; reduce cognitive load
  • Help users recognise and recover from errors — plain-language error messages with solutions
  • Help and documentation — provide support that is searchable, task-focused, and concise
  • Augmented Reality (AR) — digital information overlaid onto the real world (e.g. Pokémon GO, IKEA Place, surgical navigation systems)
  • Mixed Reality (MR) — digital and physical objects coexist and interact in real time (e.g. Microsoft HoloLens, Apple Vision Pro)
  • Virtual Reality (VR) — fully immersive digital environment replacing the real world (e.g. Meta Quest, training simulations, therapy applications)

Each creates a different type of immersive experience — and each introduces unique UX challenges around motion sickness, physical safety, cognitive load, and accessibility.

  • Define a real-world problem or need that can be solved by a UI or interactive media product
  • Break it into manageable parts — decomposition is a computational thinking skill
  • Describe the users of the solution — their needs, constraints, devices, and contexts
  • Generate alternative designs and evaluate them against the requirements to select a preferred direction
Evaluate existing UIs: Before designing, assess whether existing solutions could be modified or adapted. Not every problem requires a new product — sometimes redesigning one screen is enough.
Producing & Implementing

Producing & Implementing

This phase is where design decisions become reality — through prototypes, interactive media, data structures, and working products. Every production decision should be traceable back to a requirement.

UX practitioners use a range of tools to plan and communicate design decisions before building:

  • Fishbone diagrams — identify and organise causes of a usability problem for fault-finding
  • Stories (user stories) — plan user journeys through a website or app: "As a [user], I want to [action] so that [outcome]"
  • Storyboards — test the fluency of an animation or interaction sequence frame-by-frame
  • Prototypes — create low- or high-fidelity models of input interfaces for testing before full production

When producing:

  • Design and author an interactive media product considering functionality, accessibility, usability, and aesthetics
  • Apply information architecture elements — logical structure, navigation, labelling
  • Create a record of iterative design and evaluation throughout development
  • Consider privacy and cybersecurity in collecting and interpreting data
  • Specify what data is collected, who owns it, and how it will be protected
  • Access datasets and collect data from users or the environment — considering personally identifiable information (PII)
  • Validate that data is correct and investigate the impact of invalid data
  • Model entities, events and attributes using structured data
  • Represent, store and display data to facilitate computation — selecting appropriate data types, understanding their limitations, and structuring data systematically
  • Summarise data using a spreadsheet — including complex formulas, aggregate functions, and lookup functions
  • Filter, group and sort data — using conditional formatting, grouping, and pivot tables
  • Present data and make predictions using a spreadsheet — decision formulas and optimisation
  • Visualise data to identify trends and outliers using a range of tools
Testing & Evaluating

Testing & Evaluating

Evaluation is not the end of the process — it feeds back into the next iteration. Good evaluation is systematic, uses the requirements as a benchmark, and includes input from real users.

  • Evaluate your own project and that of your peers using the previously developed functional and non-functional requirements
  • Evaluate whether solutions meet social, ethical and legal responsibilities and cybersecurity principles
  • Test and evaluate a UI to improve usability of an interactive media product
  • Test and evaluate a UI to improve overall efficiency
  • Evaluate the functionality of the UI to convey information to a variety of audiences

Exploring careers is part of this focus area. The UX and interactive media field includes:

  • UX Designer — researches users, maps journeys, designs interactions
  • UI Designer — creates the visual layer: typography, colour, layout, component systems
  • Interaction Designer (IxD) — focuses on how users interact — micro-animations, transitions, feedback
  • Accessibility Specialist — audits and remediates products for WCAG compliance
  • Content Strategist — structures and writes the information architecture
  • Motion Designer / Multimedia Producer — creates animated and interactive media content
  • Front-end Developer — implements designs in code, bridging design and engineering

Social & ethical considerations

UX designers make choices that affect millions of people. Those choices carry ethical weight — particularly around who is included, whose data is taken, and what cognitive burdens are placed on users.

Diverse audiences

Investigate the influence of communication methods on different audiences — including Aboriginal and Torres Strait Islander Peoples, CALD communities, people of different ages and genders, and people with disability. Design for the full spectrum, not just the average user.

ATSI cultural protocols

When working with content that may relate to First Nations Peoples, follow ICIP (Indigenous Cultural and Intellectual Property) protocols. This includes permissions for displaying images of Aboriginal people who are deceased and correct attribution of cultural knowledge.

Privacy and data ownership

Specify what data your product collects, who owns it, and how it will be protected. Analyse Terms of Use and EULAs — do they genuinely protect the user, or do they transfer rights away from them?

Cognitive load and equity

Minimising cognitive load is both a usability principle and an equity issue. Complex navigation, dense text, or confusing flows disadvantage users with cognitive disabilities, EAL/D learners, and users under stress. Simplicity is access.

Analysis activity — Accessibility audit

Each card below describes a real interface design decision that has gone wrong. Read the scenario, then identify which WCAG principle it violates. You'll get immediate feedback with an explanation.

WCAG POUR principles: The Web Content Accessibility Guidelines organise around four principles — content must be Perceivable (users can sense it), Operable (users can interact with it), Understandable (users can make sense of it), and Robust (content works with current and future technologies). Which principle is violated in each scenario?
Scenario 1

A news website uses red text on a dark red background to indicate breaking news. No other visual cue distinguishes breaking news from regular articles. A user with red-green colour blindness cannot tell the difference.

✓ Perceivable. When colour is the only means of conveying information, users who cannot distinguish those colours are excluded. WCAG requires a minimum contrast ratio of 4.5:1 for text, and that colour never be the sole visual cue for meaning.
Scenario 2

An e-commerce checkout has a "Continue" button that only becomes active after all fields are filled. However, the button cannot be reached or activated using the Tab key — only by clicking with a mouse. A user who cannot use a mouse is stuck.

✓ Operable. WCAG requires that all functionality be available from a keyboard. Users with motor disabilities, repetitive strain injuries, or who rely on switch access depend on keyboard-operable interfaces. Mouse-only designs exclude them entirely.
Scenario 3

A government health portal displays an error message when a form field is left empty: "Error: invalid input — code 422." There is no explanation of which field is missing or how to fix it. Users are left confused about what they did wrong.

✓ Understandable. WCAG requires that error messages identify the item that is in error and describe the problem in text — in plain language. Technical codes like "422" are unintelligible to most users and provide no path to recovery.
Scenario 4

A school website embeds a promotional video that auto-plays without controls — no pause button, no captions, and no transcript. A student who is d/Deaf cannot access the content, and a student with vestibular disorder is triggered by the motion.

✓ Perceivable. Pre-recorded audio-visual content requires captions and a transcript. Auto-playing media with no pause mechanism also violates WCAG — users must be able to control time-based media. Both issues mean the content is not perceivable by all users.
Scenario 5

A mobile banking app uses a custom dropdown built with a <div> and JavaScript instead of a native <select> element. It looks identical to a standard dropdown — but screen readers cannot identify it as interactive, so blind users cannot use it.

✓ Robust. WCAG requires content to be compatible with current and future assistive technologies. Custom components built without proper ARIA roles are invisible to screen readers. Semantic HTML (using elements for their intended purpose) is the most reliable foundation for robustness.
Scenario 6

A news article page has a slideshow of images that automatically advances every 3 seconds with no pause button. A user with a cognitive disability needs more time to read the captions. A keyboard user cannot stop the motion or navigate between slides.

✓ Operable. WCAG requires that moving, blinking, or auto-updating content can be paused, stopped, or hidden by the user. Auto-advancing carousels without controls are a common WCAG failure — and one of the most frequently cited usability problems in user testing.

Tiered tasks

Choose the task that matches your current level — or challenge yourself with the next tier up.

Foundation

UI evolution timeline

Create a labelled timeline showing five stages in the evolution of user interfaces — from punch cards to voice interfaces. For each stage, name one example product or system and describe one key change in how users interact with it.

Core

WCAG accessibility audit

Choose a real website or app. Evaluate it against all four WCAG POUR principles. For each principle, identify one example of where the site succeeds and one where it fails. Write a requirements specification — two functional and two non-functional requirements — that would guide a redesign.

Extension

UX redesign and justification

Select a publicly available interface with at least three identifiable usability or accessibility failures. Document the failures using Nielsen's heuristics and WCAG POUR. Produce annotated wireframes for a redesigned version. Write a justification that maps each design decision back to a specific heuristic, WCAG criterion, or design principle — and identifies one remaining trade-off you could not fully resolve.
