📋 Functional requirements
All

Researching & Planning
Pseudocode & Flowcharts
Control logic is the brain of your system. Write and diagram it before you code it — catching logical errors on paper is much faster than debugging a physical robot.
📝 Main control loop pseudocode
All
  • 01Write NSW pseudocode for your main control loop — the code that runs continuously and makes decisions based on sensor readings.
  • 02Trace through your pseudocode with two specific examples: what happens in a normal scenario, and what happens when a sensor detects something unexpected.
NSW Pseudocode — example: obstacle-avoiding robot
// Main control loop — runs continuously
BEGIN ControlLoop
  WHILE running = true
    distance ← readSensor(frontSensor)

    IF distance < 20 THEN
      stopMotors()
      turnLeft(90)           // avoid obstacle
    ELSE IF distance < 50 THEN
      setSpeed(motors, 30)    // slow approach
    ELSE
      setSpeed(motors, 80)    // full speed ahead
    END IF
  END WHILE
END ControlLoop
⬆ To reach Grade B–A: your pseudocode should reflect the layered decision-making of a real system. Threshold values, multiple conditions, and graceful handling of edge cases are all marks of technically concise algorithm design.
write pseudocode in your notebook document using NSW conventions
🔀 Flowchart
All

A flowchart makes branching logic visible. For a physical system that must respond to unpredictable environments, seeing all the decision paths laid out is essential.

  • 01Draw a flowchart for your main control algorithm. Use standard symbols: rectangles for processes, diamonds for decisions, ovals for start/end. Include every branch, including what happens when a condition is not met.
  • 02Does your flowchart include a loop? Most control systems run continuously — show where the loop starts and what condition (if any) ends it.
  • 03Trace through your flowchart using a desk-check table. For at least two scenarios (normal operation and an edge case), follow each step and record what happens.

Project Plan
Mechatronic projects have physical and software components that must be developed in the right order. A plan that ignores dependencies will stall your build.
📅 Build sequence & sprints
All
  • 01Break your project into 3–5 build sprints. Each sprint should end with something you can physically test — not just code that compiles.
  • 02Order your sprints by dependency. What must be wired and working before you can test anything else? (e.g. controller + one motor working before adding sensors; sensors reading correctly before writing decision logic.)

⬆ To reach Grade B–A: the order of your sprints reveals your systems thinking. Starting with "wire everything and run the full program" is a red flag. Starting with "get one sensor reading reliably, then add one actuator, then add decision logic" shows iterative, testable thinking.

💻 Pseudocode → code comparison
Y10
  • 01For each key algorithm from your pseudocode, paste the implemented code alongside it in your notebook document.

Example — annotated control code
// Read front distance sensor — port 1
float distance = distSensor.objectDistance(mm);

// 200mm threshold — tested empirically; below this
// the robot doesn't have enough room to stop cleanly
if (distance < 200) {
  leftMotor.stop();
  rightMotor.stop();
  // brief pause before turning to avoid sensor bounce
  wait(0.3, seconds);
  leftMotor.spin(reverse, 40, pct);
  rightMotor.spin(forward, 40, pct);
  wait(0.8, seconds);   // approx 90° turn at this speed
}

Testing & Evaluating
Final Evaluation
A good evaluation of a mechatronic system is grounded in test results and physical evidence. It doesn't say the system is good or bad — it says what the data shows, and what it would take to improve.
📊 Against requirements
All
  • 01Return to your success criteria. For each one, state whether your system meets it — and cite the specific test result (pass rate, measured values) as evidence.
  • 02How reliably does your system perform? A system that works 8/10 times is not the same as one that works 10/10 times. Be honest about consistency.
  • 03Were safety and ethical responsibilities met? Does the system behave safely in all tested conditions? Are there any conditions you didn't test that could be dangerous?
⬆ To reach Grade A: every evaluative claim needs physical evidence. "My system detects obstacles reliably" needs test data — measured distances across multiple runs, across multiple surfaces. Assertions without data are Grade D evidence.
