Identifying & Defining
Problem & Purpose
The best apps and websites solve a real problem for real people. Start by being clear about what problem you're solving, who has it, and why software is the right solution.
🎯 The problem or opportunity
All
  • 01What problem does your app or website solve? Describe it in plain language — not "I want to make an app" but "people who [do X] currently have to [struggle with Y], and my solution will [do Z] instead."
  • 02Are you building an app, a website, or a web app? Explain the distinction and justify your choice. (A native app vs a browser-based solution have different strengths.)
  • 03Why is software the right solution? What could software do here that a spreadsheet, a notebook, or a conversation couldn't?
  • 04What is the core value your solution provides? If a user could only keep one feature, what would it be?
⬆ To reach Grade B–A: your problem statement should be specific enough that someone else could read it and understand exactly what gap your solution fills — not a generic description of a category of app.

Audience & Requirements
You can't design a good UI without knowing who will use it. Define your user precisely, then translate their needs into testable requirements.
👤 Target user
All
  • 01Describe your primary user. Age, background, technical experience, context of use (on a phone at home? on a desktop at work? in a hurry?).
  • 02What does your user already know? What can you assume they understand, and what needs to be explained in the interface?
  • 03What device will they most likely use? Mobile, desktop, or both? How does this shape your design decisions?
  • 04What accessibility needs might your users have? Consider vision, motor, cognitive, and language differences.

Data & System Design
Apps and web software are fundamentally data systems. Before designing the interface, understand what data flows through your solution and how it is stored, processed, and displayed.
⚙️ Inputs, storage, processes & outputs
All
  • 01What are the inputs to your app? (User-typed data, button clicks, file uploads, data from an API or database, device sensors?)
  • 02How and where is data stored? (Local storage, a database, a file, a cloud service?) Does it need to persist between sessions?
  • 03What processes transform or act on that data? (Validation, calculation, filtering, sorting, formatting, sending to an API?)
  • 04What are the outputs? (Displayed values, charts, downloadable files, emails, notifications, updated records?)
  • 05Draw or diagram this as an IPO or data flow diagram. Include it as an image in your notebook document.
⬆ To reach Grade B–A: show how data moves through the whole system — not just what goes in and comes out, but what happens to it along the way. Include storage and transmission in your diagram.
write your response and include a diagram in your notebook document

🔔 Event-driven design
All

Apps and web software respond to events — things the user does (clicks, typing, form submission) or things that happen in the system (data loading, a timer firing, a network response arriving). Identifying these events early shapes your code structure.

  • 01List the key user events in your app. (e.g. "User clicks Save", "User selects a category", "User submits the login form".) What should happen in response to each?
  • 02List any system events. (e.g. "Page loads", "Data fetched from database", "Timer expires".) What triggers these and what do they do?
  • 03Draw a simple event flow for your most important interaction — from the moment the user does something to the moment the interface updates.
⬆ To reach Grade B–A: understanding event-driven programming is what separates apps that react intelligently from ones that feel clunky. Show that you've thought about when things happen, not just what things happen.

Pseudocode & Logic
Write the logic before you write the code. Pseudocode forces you to think through what your app actually does — before syntax gets in the way.
📝 Core feature pseudocode
All
  • 01Choose your 2–3 most important features. Write NSW pseudocode for each one before implementing it.
  • 02Trace through your pseudocode with a specific example. What happens when a user submits a form? When data is retrieved? When validation fails?
  • 03After implementing, return to this section and compare your pseudocode to your actual code. What changed, and why?
NSW Pseudocode — example: save a new entry
// User clicks 'Save' on the entry form
BEGIN SaveEntry
  IF form.amount = empty OR form.category = empty THEN
    display "Please fill in all required fields"
    STOP
  END IF
  newEntry ← { date: today, amount: form.amount,
                category: form.category, note: form.note }
  entries.add(newEntry)
  saveToStorage(entries)
  refreshDisplay()
  clearForm()
END SaveEntry
⬆ To reach Grade B–A: your pseudocode should handle failure cases — what happens when data is invalid, missing, or when storage fails. Robust pseudocode leads to robust code.

🗺️ User story map
Y9

A user story map organises what your app does from the user's perspective. It helps you see the whole project before building, and identify what must come first.

  • 01Write user stories for your app using the format: As a [user], I want to [do something] so that [I get some outcome]. Aim for 8–12 stories covering the core functionality.
  • 02Group stories into activities (e.g. "managing my account", "adding data", "viewing my history") and arrange them left-to-right in the order a user encounters them.
  • 03Draw a horizontal MVP line through your map. What is the minimum set of stories that makes a usable product?
  • 04Include your map as an image in your notebook document.
⬆ To reach Grade B–A: stories written in user voice show design empathy. "As a user, I want clear confirmation that my entry was saved so I don't submit it twice" is stronger than "add a success message."

🏃 Scrum board & sprint planning
Y9

Scrum breaks a project into fixed-length sprints, each with a committed goal. A Scrum board shows tasks for the current sprint moving through states as work progresses.

  • 01Plan three sprints for your proposed project. For each sprint: write the sprint goal (what should the user be able to do at the end?), list the committed tasks, and define "done."
  • 02How long is each sprint? Justify the length — too short and you produce nothing; too long and you lose focus.
  • 03What would you demonstrate in a sprint review? Who would you show it to?
⬆ To reach Grade B–A: each sprint should end with something the user can actually interact with and give feedback on — not just code that compiles.
write your sprint plan as a table in your notebook document
📊 Gantt chart
Y9

A Gantt chart maps tasks against a timeline, showing when each task runs and how tasks depend on each other. It makes it immediately clear whether your plan is realistic.

  • 01Create a Gantt chart spanning the full production term. Each major task is a bar across the weeks you'd spend on it.
  • 02Show dependencies — which tasks must finish before others begin. (e.g. data model must be defined before the display layer can be built.)
  • 03Mark milestones — end of each sprint, any prototype reviews, and the final submission deadline.
  • 04Include a buffer period. Where in your plan have you left time to catch up if something takes longer than expected?
  • 05Reflect: What is the most time-pressured section of your plan? What is the consequence if it slips?
Tools you could use

A colour-coded table in your notebook, a spreadsheet, an online Gantt tool, or a hand-drawn chart. Label weeks clearly.

⬆ To reach Grade B–A: a credible Gantt chart acknowledges that not everything takes equal time, and that the hardest parts need the most space. A chart where everything finishes a week early is not credible.

Example — annotated JS event handler
// Runs when the save button is clicked
saveBtn.addEventListener('click', () => {
  const amount = parseFloat(amountInput.value);

  // Validate before doing anything — don't store bad data
  if (isNaN(amount) || amount <= 0) {
    showError('Please enter a valid amount');
    return;  // early return keeps nesting flat
  }

  const entry = { amount, category: categorySelect.value, date: new Date() };
  entries.push(entry);
  saveToStorage(entries);  // persist so data survives page refresh
  renderList();            // re-render from source of truth, not DOM
  amountInput.value = '';  // clear field after save
});
