Algorithms & Programming

You design algorithms in NSW pseudocode, implement them in Python or JavaScript, and verify they work by tracing and testing. Every example on this page follows the same pattern: pseudocode first, then code — never the other way around.

📖 OPL-01 in plain English

Designs algorithms
Write the logic in NSW pseudocode before writing any code. The algorithm is the plan; the code is the construction.
Produces algorithms
Implement the algorithm in a real programming language — Python, JavaScript, or both — so it actually runs.
Evaluates algorithms
Test the implementation: desk check it on paper, run test cases, debug errors, and assess whether it meets requirements.
General-purpose language
Python — used for backend logic, scripts, data processing. Can run anywhere without a browser.
Object-oriented language
Python (Stage 5) and JavaScript both support OOP — organising code into classes with attributes and methods.
Modular design
Splitting a program into functions that each do one thing. Easier to read, test, and extend than one giant block.
Example 01

To-do list — add, complete, remove

A to-do list is one of the clearest examples of CRUD operations: Create (add a task), Read (display tasks), Update (mark complete), Delete (remove). These four operations appear in almost every real app.

Functions Conditionals Lists & iteration 
// ── To-do list algorithm
// Data: tasks is a list of records {text, done}
// Operations: add, complete, remove, display

MODULE TodoApp

  tasks ← []           // empty list — stores all tasks

  FUNCTION addTask(text)
    IF text ≠ "" THEN
      newTask ← {text: text, done: FALSE}
      tasks ← tasks + [newTask]
      RETURN TRUE
    ELSE
      RETURN FALSE      // reject empty task text
    ENDIF
  ENDFUNCTION

  FUNCTION completeTask(index)
    IF index ≥ 0 AND index < LENGTH(tasks) THEN
      tasks[index].done ← TRUE
      RETURN TRUE
    ELSE
      RETURN FALSE      // index out of range
    ENDIF
  ENDFUNCTION

  FUNCTION removeTask(index)
    IF index ≥ 0 AND index < LENGTH(tasks) THEN
      tasks ← tasks WITHOUT tasks[index]
      RETURN TRUE
    ELSE
      RETURN FALSE
    ENDIF
  ENDFUNCTION

  FUNCTION displayTasks()
    FOR EACH task IN tasks DO
      IF task.done THEN
        OUTPUT "[✓] " + task.text
      ELSE
        OUTPUT "[ ] " + task.text
      ENDIF
    NEXT task
  ENDFUNCTION

ENDMODULE
# todo.py — To-do list implementation

class TodoApp:
    """OOP implementation — tasks stored as list of dicts."""

    def __init__(self):
        self.tasks = []          # ← pseudocode: tasks ← []

    def add_task(self, text):
        """Add a task. Returns True if added, False if text empty."""
        if text.strip():          # ← pseudocode: IF text ≠ ""
            self.tasks.append({"text": text.strip(), "done": False})
            return True
        return False

    def complete_task(self, index):
        """Mark a task complete by index. Returns False if out of range."""
        if 0 <= index < len(self.tasks):
            self.tasks[index]["done"] = True
            return True
        return False

    def remove_task(self, index):
        """Remove a task by index."""
        if 0 <= index < len(self.tasks):
            self.tasks.pop(index)   # ← pseudocode: tasks WITHOUT tasks[index]
            return True
        return False

    def display_tasks(self):
        """Print all tasks with completion status."""
        if not self.tasks:
            print("No tasks yet.")
            return
        for i, task in enumerate(self.tasks):
            marker = "✓" if task["done"] else " "
            print(f"{i}. [{marker}] {task['text']}")


# ── Usage ────────────────────────────────────
if __name__ == "__main__":
    app = TodoApp()
    app.add_task("Buy groceries")
    app.add_task("Finish assignment")
    app.add_task("")           # rejected — empty string
    app.complete_task(0)
    app.display_tasks()
    # Output:
    # 0. [✓] Buy groceries
    # 1. [ ] Finish assignment
// todo.js — To-do list with DOM rendering

const tasks = [];   // ← pseudocode: tasks ← []

function addTask(text) {
  if (!text.trim()) return false;
  tasks.push({ text: text.trim(), done: false });
  renderTasks();
  return true;
}

function completeTask(index) {
  if (index < 0 || index >= tasks.length) return false;
  tasks[index].done = true;
  renderTasks();
  return true;
}

function removeTask(index) {
  if (index < 0 || index >= tasks.length) return false;
  tasks.splice(index, 1);   // ← pseudocode: tasks WITHOUT tasks[index]
  renderTasks();
  return true;
}

// Renders the tasks array into the DOM — separation of data and display
function renderTasks() {
  const list = document.getElementById("task-list");
  list.innerHTML = "";

  tasks.forEach(function(task, i) {
    const li = document.createElement("li");
    li.textContent = (task.done ? "✓ " : "○ ") + task.text;
    if (task.done) li.style.textDecoration = "line-through";

    const doneBtn = document.createElement("button");
    doneBtn.textContent = "Done";
    doneBtn.onclick = () => completeTask(i);

    const delBtn = document.createElement("button");
    delBtn.textContent = "Delete";
    delBtn.onclick = () => removeTask(i);

    li.appendChild(doneBtn);
    li.appendChild(delBtn);
    list.appendChild(li);
  });
}

// Event listener — wires the Add button to addTask()
document.getElementById("add-btn").addEventListener("click", function() {
  const input = document.getElementById("task-input");
  addTask(input.value);
  input.value = "";
  input.focus();
});
Function / procedure Lists & arrays Conditional (if/else) Iteration (for each) Return values (True/False) OOP (class + methods) Modular design Event-driven (JS)

Pseudocode → Python: The Python version uses a class to group the tasks list and its operations together — this is encapsulation. The pseudocode uses a MODULE for the same reason. self.tasks.append() is the Python equivalent of tasks ← tasks + [newTask].

Python → JavaScript: JavaScript doesn't need a class here because the tasks array and functions can live at module scope. The key difference is renderTasks() — a function the Python version doesn't need because Python prints to a terminal, while JS must update the DOM. This is the event-driven layer: every data change triggers a re-render.

Modular design: Notice that addTask, completeTask, removeTask, and renderTasks each do exactly one thing. You can test each independently — if removing breaks, you only need to look at removeTask, not search the entire program.

  • Index bounds check: 0 <= index < len(tasks) prevents a crash when an invalid index is passed. This is defensive programming — check before acting.
  • Empty string check: if text.strip() in Python / if (!text.trim()) in JS — both reject whitespace-only input for the same reason as the form validator.
Example 02

Search & filter — find items in a list

Search and filter algorithms appear in every app that manages more than a handful of items — product catalogues, contact lists, file managers. The core algorithm is iteration + conditional: loop through every item, keep the ones that match.

Iteration String matching Filter function 
// ── Search and filter algorithm
// Searches a list of product records by name substring

FUNCTION searchProducts(products, query)
  results ← []
  query ← LOWERCASE(query)           // case-insensitive search

  FOR EACH product IN products DO
    name ← LOWERCASE(product.name)
    IF name CONTAINS query THEN
      results ← results + [product]
    ENDIF
  NEXT product

  RETURN results
ENDFUNCTION


FUNCTION filterByCategory(products, category)
  // Empty category string = return all (no filter applied)
  IF category = "" THEN
    RETURN products
  ENDIF

  results ← []
  FOR EACH product IN products DO
    IF product.category = category THEN
      results ← results + [product]
    ENDIF
  NEXT product

  RETURN results
ENDFUNCTION
# search.py — Search and filter functions

# Sample data — list of product dicts
products = [
    {"name": "Mechanical Keyboard", "category": "peripherals", "price": 149},
    {"name": "USB-C Hub",           "category": "peripherals", "price": 49},
    {"name": "Monitor 27in",        "category": "displays",    "price": 399},
    {"name": "Webcam HD",          "category": "peripherals", "price": 89},
]


def search_products(products, query):
    """Return all products whose name contains query (case-insensitive)."""
    query = query.lower()                     # ← LOWERCASE(query)
    results = []
    for product in products:               # ← FOR EACH product IN products
        if query in product["name"].lower():  # ← name CONTAINS query
            results.append(product)
    return results


def filter_by_category(products, category):
    """Filter products by exact category match. Empty = return all."""
    if not category:
        return products               # ← IF category = "" THEN RETURN products
    return [p for p in products if p["category"] == category]


# ── Usage ────────────────────────────────────
if __name__ == "__main__":
    found = search_products(products, "key")
    for p in found:
        print(f"{p['name']} — ${p['price']}")
    # Output: Mechanical Keyboard — $149

    peripherals = filter_by_category(products, "peripherals")
    print(len(peripherals), "peripherals found")  # 3
// search.js — Live search with DOM update on keypress

const products = [
  { name: "Mechanical Keyboard", category: "peripherals", price: 149 },
  { name: "USB-C Hub",           category: "peripherals", price: 49  },
  { name: "Monitor 27in",        category: "displays",    price: 399 },
  { name: "Webcam HD",           category: "peripherals", price: 89  },
];

function searchProducts(products, query) {
  const q = query.toLowerCase();                    // case-insensitive
  return products.filter(function(p) {
    return p.name.toLowerCase().includes(q);       // ← name CONTAINS query
  });
}

function filterByCategory(products, category) {
  if (!category) return products;                 // empty = show all
  return products.filter(function(p) {
    return p.category === category;
  });
}

function renderResults(results) {
  const container = document.getElementById("results");
  container.innerHTML = "";

  if (results.length === 0) {
    container.textContent = "No products found.";
    return;
  }

  results.forEach(function(p) {
    const div = document.createElement("div");
    div.textContent = p.name + " — $" + p.price;
    container.appendChild(div);
  });
}

// Live search — fires on every keystroke (input event)
document.getElementById("search-input").addEventListener("input", function() {
  const query    = this.value;
  const category = document.getElementById("category-select").value;

  let results = searchProducts(products, query);
  results = filterByCategory(results, category);
  renderResults(results);
});
Iteration (for each / .filter) String methods (CONTAINS / .includes) Case normalisation Return values (filtered list) List comprehension (Python) Event: input (live search) Chained function calls

The algorithm is identical in all three: loop through items, check if name contains query, collect matches. What changes is the syntax and how the result is rendered.

Python list comprehension: [p for p in products if p["category"] == category] is shorthand for the FOR EACH loop in the pseudocode — more concise, same logic. Both forms are valid Python.

JavaScript .filter(): products.filter(function(p) { return ... }) is a higher-order function — it takes a function as an argument and applies it to every element. The result is a new array containing only the elements where the function returned true. This is the array equivalent of the pseudocode FOR EACH + results list pattern.

Live search (JS only): The input event fires on every keystroke — not just on submit. This is a UX decision. The same searchProducts function is called every time; the event listener is what makes it feel "live."

  • Case normalisation: .toLowerCase() on both the query and the product name before comparing — so "keyboard", "KEYBOARD", and "Keyboard" all match. Always normalise before comparing strings.
  • Empty query: if the user clears the search box, "" is contained in every string — so all products show. This is intentional behaviour, not a bug.
Example 03

Login flow — authenticate a user

Authentication is the algorithm that checks whether someone is who they claim to be. This example shows a simple username + password check — the kind that would underpin a school portal, a members area, or a simple admin panel.

Boolean logic Functions Data structures 
// ── Login authentication algorithm
// Users stored as list of records {username, password_hash, role}
// IMPORTANT: passwords are never stored or compared as plain text

FUNCTION findUser(users, username)
  FOR EACH user IN users DO
    IF LOWERCASE(user.username) = LOWERCASE(username) THEN
      RETURN user
    ENDIF
  NEXT user
  RETURN NULL           // user not found
ENDFUNCTION


FUNCTION attemptLogin(users, username, password)
  // Rule 1: fields must not be empty
  IF username = "" OR password = "" THEN
    RETURN {success: FALSE, message: "Please enter username and password"}
  ENDIF

  // Rule 2: find the user record
  user ← findUser(users, username)
  IF user = NULL THEN
    RETURN {success: FALSE, message: "Invalid username or password"}
  ENDIF
  // Note: same message for wrong username AND wrong password
  // This prevents attackers from knowing which one failed

  // Rule 3: verify password
  IF HASH(password) = user.password_hash THEN
    RETURN {success: TRUE, role: user.role, message: "Login successful"}
  ELSE
    RETURN {success: FALSE, message: "Invalid username or password"}
  ENDIF
ENDFUNCTION
# login.py — Authentication with password hashing
import hashlib


def hash_password(password):
    """SHA-256 hash of a password. Never store plain text."""
    return hashlib.sha256(password.encode()).hexdigest()


# Simulated user database — in production this would be a real DB
users = [
    {"username": "alice",   "password_hash": hash_password("Pass1234"), "role": "admin"},
    {"username": "bob",     "password_hash": hash_password("Hello999"), "role": "user"},
]


def find_user(users, username):
    """Find user by username (case-insensitive). Returns user or None."""
    for user in users:
        if user["username"].lower() == username.lower():
            return user
    return None               # ← pseudocode: RETURN NULL


def attempt_login(users, username, password):
    """Authenticate user. Returns dict with success, role, message."""
    # Rule 1: empty fields
    if not username.strip() or not password:
        return {"success": False, "message": "Please enter username and password"}

    # Rule 2: find user
    user = find_user(users, username)
    if user is None:
        return {"success": False, "message": "Invalid username or password"}

    # Rule 3: verify password hash
    if hash_password(password) == user["password_hash"]:
        return {"success": True, "role": user["role"], "message": "Login successful"}

    return {"success": False, "message": "Invalid username or password"}


# ── Test suite ────────────────────────────────────────────────
def run_tests():
    cases = [
        ("alice",   "Pass1234", True,  "Valid login"),
        ("ALICE",   "Pass1234", True,  "Case-insensitive username"),
        ("alice",   "wrong",    False, "Wrong password"),
        ("unknown", "Pass1234", False, "Unknown user"),
        ("",        "Pass1234", False, "Empty username"),
    ]
    for username, password, expected, desc in cases:
        result = attempt_login(users, username, password)
        status = "PASS" if result["success"] == expected else "FAIL"
        print(f"[{status}] {desc}")

run_tests()
// login.js — Client-side login handler
// NOTE: password hashing must happen server-side in production
// This JS example shows the algorithm structure only

const users = [
  { username: "alice", passwordHash: "hashed_alice_pass", role: "admin" },
  { username: "bob",   passwordHash: "hashed_bob_pass",   role: "user"  },
];

function findUser(username) {
  return users.find(function(u) {
    return u.username.toLowerCase() === username.toLowerCase();
  }) || null;
}

function attemptLogin(username, password) {
  // Rule 1: empty fields
  if (!username.trim() || !password) {
    return { success: false, message: "Please enter username and password" };
  }

  // Rule 2: find user
  const user = findUser(username);
  if (!user) {
    return { success: false, message: "Invalid username or password" };
  }

  // Rule 3: In production, send to server for hash comparison
  // Never compare passwords client-side — this is for demonstration only
  if (user.passwordHash === "hashed_" + user.username + "_pass") {
    return { success: true, role: user.role, message: "Login successful" };
  }

  return { success: false, message: "Invalid username or password" };
}

// Event listener — form submit triggers login attempt
document.getElementById("login-form").addEventListener("submit", function(e) {
  e.preventDefault();
  const username = document.getElementById("username").value;
  const password = document.getElementById("password").value;
  const result   = attemptLogin(username, password);
  const feedback = document.getElementById("feedback");

  feedback.textContent = result.message;

  if (result.success) {
    feedback.style.color = "#34d399";
    document.getElementById("login-form").style.display = "none";
    document.getElementById("welcome").textContent =
      "Welcome! You are logged in as: " + result.role;
  } else {
    feedback.style.color = "#f87171";
  }
});
Boolean return values Iteration + early return Dictionary / object as return type Password hashing (hashlib) Vague error messages (security) Case-insensitive comparison NULL / None sentinel value

The vague error message is intentional: "Invalid username or password" is the same message whether the username doesn't exist or the password is wrong. This is a deliberate security design — if the error said "username not found," an attacker could enumerate valid usernames. This is the information disclosure principle: don't reveal more than necessary.

Password hashing: Passwords are never stored or compared as plain text. The Python version uses hashlib.sha256() to convert "Pass1234" into a fixed-length hex string. Even if someone stole the user database, they'd get hashes, not passwords.

JS caveat: Password verification must happen server-side. The JS example shows the algorithm structure only — in a real app, the JS would fetch() to a Python/Flask endpoint that does the comparison. Comparing passwords in the browser exposes hashes to anyone who opens DevTools.

  • Early return pattern: Each rule returns immediately on failure rather than nesting more if statements. This keeps the logic flat and readable — a key code quality principle.
  • .find() in JS: Returns the first matching element, or undefined. The || null converts it to null for consistency with the pseudocode's RETURN NULL.
Example 04

Render data to page — display a list of records

Almost every web app reads data and renders it as HTML — a product catalogue, a contact list, search results, a dashboard. This algorithm takes an array of objects and builds the DOM elements to display them. It's the bridge between data and interface.

Iteration Modular DOM / OOP 
// ── Render data to page algorithm
// Converts a list of records into displayable output
// Works for any record type — the algorithm is the same

FUNCTION renderItems(items, containerID)
  IF LENGTH(items) = 0 THEN
    showEmptyState(containerID)
    RETURN
  ENDIF

  clearContainer(containerID)       // remove previous content

  FOR EACH item IN items DO
    card ← buildCard(item)           // create one display unit
    appendToContainer(containerID, card)
  NEXT item
ENDFUNCTION


FUNCTION buildCard(item)
  // Each card shows: name, category, price, and an action button
  card.title    ← item.name
  card.subtitle ← item.category
  card.price    ← "$" + item.price
  card.action   ← BUTTON("Add to cart")
  RETURN card
ENDFUNCTION


FUNCTION showEmptyState(containerID)
  clearContainer(containerID)
  appendToContainer(containerID, "No items to display.")
ENDFUNCTION
# render.py — Flask template rendering (server-side)
# Python generates HTML using Jinja2 templates
from flask import Flask, render_template

app = Flask(__name__)

products = [
    {"name": "Mechanical Keyboard", "category": "Peripherals", "price": 149},
    {"name": "Monitor 27in",        "category": "Displays",    "price": 399},
    {"name": "USB-C Hub",           "category": "Peripherals", "price": 49},
]


def build_card_data(item):
    """Prepare display data for one item — mirrors buildCard() in pseudocode."""
    return {
        "title":    item["name"],
        "subtitle": item["category"],
        "price":    f"${item['price']}",
    }


@app.route("/")
def index():
    # Build card data for every product — FOR EACH item IN items
    cards = [build_card_data(p) for p in products]
    return render_template("products.html", cards=cards)

// render.js — Dynamic DOM rendering (client-side)
// No page reload — JavaScript builds the HTML from data

const products = [
  { name: "Mechanical Keyboard", category: "Peripherals", price: 149 },
  { name: "Monitor 27in",        category: "Displays",    price: 399 },
  { name: "USB-C Hub",           category: "Peripherals", price: 49  },
];

// buildCard — mirrors pseudocode: returns one DOM element
function buildCard(item) {
  const card = document.createElement("div");
  card.className = "product-card";

  const title = document.createElement("h3");
  title.textContent = item.name;             // textContent — never innerHTML

  const sub = document.createElement("p");
  sub.textContent = item.category;

  const price = document.createElement("p");
  price.textContent = "$" + item.price;

  const btn = document.createElement("button");
  btn.textContent = "Add to cart";
  btn.addEventListener("click", function() {
    alert("Added: " + item.name);
  });

  card.appendChild(title);
  card.appendChild(sub);
  card.appendChild(price);
  card.appendChild(btn);
  return card;
}

// renderItems — mirrors pseudocode: loops and appends each card
function renderItems(items, containerId) {
  const container = document.getElementById(containerId);
  container.innerHTML = "";           // ← clearContainer()

  if (items.length === 0) {            // ← IF LENGTH(items) = 0
    container.textContent = "No items to display.";
    return;
  }

  items.forEach(function(item) {      // ← FOR EACH item IN items
    const card = buildCard(item);
    container.appendChild(card);
  });
}

// Initial render on page load
renderItems(products, "product-grid");
FOR EACH → .forEach() DOM creation (createElement) Separation: data vs display Empty state handling Server-side render (Jinja2) List comprehension (Python) textContent (XSS safe)

The algorithm is language-agnostic: "loop through items, build a display unit, append it to a container." Whether the container is a terminal (Python CLI), a Jinja2 template (Flask), or the browser DOM (JS), the algorithm is identical. Only the rendering mechanism differs.

Python (Flask) vs JavaScript: Flask renders server-side — the HTML is built once, sent to the browser, and the page loads fully formed. JavaScript renders client-side — the browser loads an empty container and JS populates it after. Both approaches are valid; the choice depends on performance, interactivity, and whether the data changes after the page loads.

buildCard() as a function: Separating "build one card" from "render all cards" is a key modular design decision. If you need to change how a card looks — you change one function in one place. If buildCard() were inlined inside renderItems(), every visual change requires finding and editing code buried inside a loop.

  • Empty state: Always handle the case where the list is empty. A blank white area with no explanation is a UX failure — "No items to display" is the minimum.
  • Jinja2 template syntax: {% for card in cards %} is the template equivalent of FOR EACH card IN cards. Jinja2 runs on the server before the HTML reaches the browser.
