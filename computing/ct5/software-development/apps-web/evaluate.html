Functional Requirements

Does your app do what it was meant to do?

Evaluate your project against the functional requirements you specified in the Identifying & Defining phase. Each requirement should have been tested with at least one test case. CT5-EVL-01 CT5-DPM-01

I have listed all functional requirements I specified at the start of the project
Each requirement should be a clear, testable statement of what the app must do — not how it looks or how fast it runs. E.g. "A user can submit a registration form with name, email, and password."
Evidence to record in notebook Copy your original requirements list here. Highlight any that changed during the project and explain why.
C
I have tested each functional requirement using a defined test case
For each requirement, you need: a specific input, the expected output, and the actual output. If actual ≠ expected — record the discrepancy and note whether you fixed it.
  • Form validation: does it reject empty name? Invalid email? Short password?
  • UI: does every button do what its label says?
  • Edge cases: what happens with spaces-only input, very long strings, or special characters?
Evidence to record in notebook Complete a test case table: Requirement | Input | Expected output | Actual output | Pass/Fail | Action taken.
B
I have evaluated my app against a peer's using the same requirement criteria
Peer evaluation using predetermined criteria is more structured than "did you like it?" Ask your peer to run specific test cases and report what they found — not just general opinions.
Evidence to record in notebook Summarise 2–3 pieces of peer feedback. For each: what was tested, what the peer found, and what you changed (or decided not to change and why).
B
I have documented which requirements were fully met, partially met, and not met — with reasons
An honest evaluation acknowledges limitations. If a requirement wasn't met, explain what prevented it and what you would do differently with more time.
Evidence to record in notebook Write 1–2 sentences for each unmet or partially met requirement. Avoid vague statements like "ran out of time" — be specific about the technical or design challenge.

Non-Functional Requirements

How well does your app perform?

Non-functional requirements describe quality attributes — usability, accessibility, performance, security. These were defined in your Identifying & Defining phase. CT5-EVL-01 CT5-SAF-01

Usability: a first-time user can navigate my app without help
Watch someone use your app without explaining anything. Where do they hesitate? Where do they click in the wrong place? This is the most honest usability test you can run.
Evidence to record in notebook Describe 1–2 usability observations. What did you discover? What did you fix or what would you change?
C
Accessibility: my app does not exclude users based on ability or device
Check at least three of the following:
  • All form inputs have associated <label> elements
  • Error messages are announced to screen readers (aria-live)
  • Colour contrast ratio is at least 4.5:1 for body text
  • The app works with keyboard-only navigation (no mouse)
  • Text scales up without breaking layout when browser zoom is at 150%
Evidence to record in notebook Record which accessibility checks you ran and what you found. Reference the WCAG principle each check relates to (Perceivable, Operable, Understandable, Robust).
B
HTML and CSS are separated — content is not mixed with presentation
Open your HTML file. Are there inline style="" attributes on elements? Is presentational logic (colours, fonts, spacing) in the HTML rather than a CSS file or <style> block? The separation principle is a non-functional requirement in itself.
Evidence to record in notebook List any inline styles you found. Note which ones you removed (or explain why they remain).
C
My non-functional requirements from the planning phase have been tested with specific criteria
If you wrote "the app must be easy to use" — what did you measure to determine whether that's true? If you wrote "the form must load quickly" — did you test it? Non-functional requirements are only useful if they are measurable.
Evidence to record in notebook For each non-functional requirement: state the measurable criterion you used and whether your app met it.

Algorithm & Code Testing

Does your code behave correctly?

Use selected data to check your algorithm and programming code. This section maps directly to the Skills page desk check and trace table activities. CT5-OPL-01

I wrote NSW pseudocode for my main algorithm before implementing it in code
The pseudocode should exist in your project notebook from the Researching & Planning phase. If you coded first and wrote pseudocode after — note that in your evaluation and explain the difference between designing and reverse-engineering.
Evidence to record in notebook Reference the page/section of your notebook where your pseudocode appears. Note any differences between the pseudocode and the final implementation.
C
I have tested my validation algorithm with at least 6 distinct inputs covering normal, boundary, and invalid cases
Your test data should include:
  • Normal: valid inputs that should pass (e.g. "Alex", "alex@example.com", "Pass1234")
  • Boundary: inputs at the exact limit (e.g. an 8-character password — the minimum)
  • Invalid: inputs that should fail (empty name, missing @, 7-character password)
  • Edge: whitespace-only name, password with only digits, email with multiple @ signs
Evidence to record in notebook Complete the test case table from the Skills page (or your own version): Input | Expected | Actual | Pass/Fail.
B
I have debugged at least one error in my code and documented the fix
What was the bug? Was it a syntax error (code won't run), a logic error (wrong result), or a runtime error (crash during use)? How did you find it — testing, tracing, reading the error message?
Evidence to record in notebook Describe the bug: what it was, how you found it, what the fix was, and how you verified the fix worked.
B
My code uses functions/procedures to separate concerns — not one monolithic block
Modular programming means each function does one thing. Check: can you describe what each function does in one sentence? If a function does more than one thing — it should probably be two functions.
Evidence to record in notebook List the functions in your code and write a one-sentence description of each. Identify any function that could be broken into smaller parts.
B
My code handles errors gracefully — it never crashes silently or gives the user no feedback
Test what happens when: the user submits a completely empty form, inputs a very long string (500+ characters), or (if using the combo stack) the server is unreachable. The app should always give the user meaningful feedback.
Evidence to record in notebook Describe 2 error scenarios you tested. What happened before you fixed it? What happens now?

Tracing & Test Cases

Can you prove the algorithm is correct by hand?

Validate your algorithm by tracing it manually — not just running it. This is the desk check process from the Skills page applied to your own project code. CT5-OPL-01 CT5-EVL-01

I have completed a desk check (trace table) for my main algorithm using at least two different inputs
A trace table shows every variable's value at each step. Pick one input that should pass validation and one that should fail. Trace both — if the algorithm produces the right output on paper, it should produce the right output in code.
Evidence to record in notebook Include your completed trace table(s). If the trace revealed a bug — describe it and show the fixed algorithm.
C
My test case table links each test to a specific requirement
Each test case should be traceable back to a requirement from your Identifying & Defining phase. "Test 3 — Password length check" links to "Requirement 3 — Password must be at least 8 characters." If a test doesn't link to a requirement, ask why you're testing it.
Evidence to record in notebook Add a "Requirement" column to your test case table. Confirm every requirement has at least one test case.
B
My pseudocode and my implemented code produce the same outputs for the same inputs
Run your test cases through the pseudocode (on paper) and through your actual code. Do the outputs match? If they don't — one of them is wrong. Identify which and explain why they diverge.
Evidence to record in notebook Table comparing pseudocode output vs code output for each test input. Note any discrepancies and their cause.

Social, Ethical & Legal

Does your app meet its social responsibilities?

Evaluate whether your solution meets social, ethical and legal responsibilities. These requirements were identified in your Identifying & Defining phase. CT5-SAF-01 CT5-EVL-01

My app does not make cultural assumptions about users' names, language, location, or identity
Check for common assumptions:
  • Does your form require a "first name" AND "last name" as separate fields? (Not all cultures use this structure)
  • Does your app assume a particular date format, phone format, or address structure?
  • Are there any labels or messages that assume the user's gender?
  • Does the app work in multiple character sets, or does it break on non-ASCII names?
Evidence to record in notebook Identify one cultural assumption that exists in your app (even if you couldn't fix it). Explain what the impact would be and how you would address it with more time.
B
Any images, text, or media in my app respects copyright and intellectual property
If your app uses any images, icons, fonts, or code libraries — you must be able to identify the licence under which you're using them. "I found it on Google Images" is not sufficient. Indigenous Cultural and Intellectual Property (ICIP) applies to digital content involving Aboriginal and Torres Strait Islander communities.
Evidence to record in notebook List all third-party assets used (images, libraries, fonts). For each, record the source and licence.
C
My app considers the needs of diverse users, including people with disability and EAL/D learners
Have you considered: users who rely on keyboard navigation, users with visual impairments using screen readers, users for whom English is not their first language, users on low-bandwidth connections or older devices?
Evidence to record in notebook Describe one design decision you made (or one you would make with more time) specifically to support a user group that is often overlooked.
B
My app addresses at least one environmental, lifestyle, societal, or economic challenge
This was part of your Researching & Planning phase. Return to your original research. Does the finished app actually address the challenge you identified? Has your approach changed? Is there a gap between your intention and what you built?
Evidence to record in notebook Sentence starter: "My app was designed to address [challenge]. The finished version [does/partially does/does not] address this because…"

Cybersecurity

Is your app safe for real users?

Evaluate your solution against cybersecurity principles. Cybersecurity is not optional — it is a responsibility of every developer. CT5-SAF-01

My form uses textContent (not innerHTML) when displaying user-controlled input
Setting element.innerHTML = userInput can allow a malicious user to inject executable HTML or JavaScript — a cross-site scripting (XSS) attack. Always use element.textContent = userInput when inserting user data into the page.
Evidence to record in notebook Search your code for every place you output user input to the page. Confirm each uses textContent or equivalent safe method.
C
If using the Python + JS combo: validation runs on the server, not only the client
Client-side JavaScript can be disabled or bypassed in seconds. If your app only validates in the browser, a malicious user can send any data they want directly to your server. Server-side validation is the authoritative security check — it must always run.
Evidence to record in notebook Describe what would happen if someone submitted your form with JavaScript disabled. Does your server still validate the data?
B
My app specifies what data it collects, who owns it, and how it is protected
Even a school project should be able to answer: What data does this app collect? Where is it stored? Who can access it? How long is it kept? This is the starting point of a privacy policy and a legal responsibility in Australia under the Privacy Act.
Evidence to record in notebook Write a 3–4 sentence data statement for your app: what it collects, where it's stored, who can access it.
B
I can explain at least two cybersecurity principles relevant to my project and how my code addresses them
Examples from this project:
  • Input validation — reject bad data before it reaches processing
  • Defence in depth — multiple layers of validation (client + server)
  • Least privilege — only collect the data you actually need
  • Safe output — never inject user data as HTML (textContent vs innerHTML)
Evidence to record in notebook Write 2–3 sentences for each principle: what it means, why it matters, and where/how your code applies it.
