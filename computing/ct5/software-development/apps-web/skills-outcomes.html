In this focus area, CT5-OPL-01 requires you to:

  • Write NSW pseudocode for your algorithm before implementing it in code
  • Implement validation logic using functions in Python or JavaScript
  • Apply data types correctly (strings, booleans, lists/arrays)
  • Use conditional logic (if/elif/else) to check each validation rule
  • Use iteration (for loop / any()) to check characters in a password
  • Design and implement modular programs — separate functions for separate concerns
  • Validate programs using test cases and debug a range of errors
  • Trace an algorithm manually (desk check) to verify correctness before running

From the form validation project — this is the algorithm designed in pseudocode, then implemented as a Python function:

NSW Pseudocode
FUNCTION validateForm(name, email, password)
  errors ← []
  IF name = "" THEN
    errors ← errors + ["Name is required"]
  ENDIF
  IF NOT (email CONTAINS "@") THEN
    errors ← errors + ["Email must be valid"]
  ENDIF
  IF LENGTH(password) < 8 THEN
    errors ← errors + ["Password too short"]
  ENDIF
  RETURN (LENGTH(errors) = 0, errors)
ENDFUNCTION
# Python implementation — mirrors pseudocode exactly def validate_form(name, email, password): errors = [] if name.strip() == "": errors.append("Name is required") if "@" not in email: errors.append("Email must be a valid address") if len(password) < 8: errors.append("Password must be at least 8 characters") return (len(errors) == 0, errors)

From the JS form walkthrough — error feedback designed for real users, not developers:

/* CSS — error state communicated visually AND structurally */ input.invalid { border-color: #dc2626; /* red border — visual cue */ } .error-list { color: #dc2626; margin-top: 1rem; padding-left: 1.2rem; } /* HTML — ARIA live region so screen readers announce errors */ <div id="feedback" aria-live="polite" aria-atomic="true"></div> // JS — move focus to first invalid field after failed submit if (nameEl.value.trim() === "") { nameEl.classList.add("invalid"); nameEl.focus(); // keyboard users land on the problem }

Design decisions visible here: colour alone doesn't convey the error (accessible); aria-live means screen readers announce the error list; moving focus is a keyboard UX decision, not just visual polish.

Form validation is a microcosm of app evolution. Consider how registration forms changed:

-- Evolution of form validation (1995 → present) -- 1995 No validation — forms submitted raw, server crashed on bad data Innovation: CGI scripts — first server-side processing 2000 Basic server-side validation — page reload on every error Enterprise: e-commerce required secure, reliable registration 2005 Client-side JS validation — instant feedback without page reload Innovation: AJAX, DOM manipulation became practical 2010 HTML5 required/type attributes — browser-native validation Automation: browsers handle some checks automatically 2018 Frameworks (React, Vue) — component-based form state Enterprise: scalable validation libraries (Yup, Zod) Now Real-time validation per keystroke; biometric auth; passwordless login (passkeys) — evolving still
