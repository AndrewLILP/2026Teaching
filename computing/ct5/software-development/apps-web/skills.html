Form Validation — Building Safe Inputs

Form validation is the algorithm that runs before any data is sent — checking that inputs are present, correctly formatted, and safe. In this skills walkthrough you'll design the algorithm in NSW pseudocode first, then implement it across three language contexts: Python, vanilla JavaScript, and a combined Python backend / JS frontend.

Key vocabulary

📖 Word bank

Validation
Checking that input data meets defined rules before it is processed or stored.
Sanitisation
Cleaning input data to remove or escape characters that could cause harm (e.g. preventing code injection).
Client-side validation
Validation that runs in the browser (JavaScript) before data is sent to the server — fast feedback for the user.
Server-side validation
Validation that runs on the server (Python/Flask) after data is received — the authoritative security check.
Function / procedure
A named, reusable block of code that performs a specific task. Functions return a value; procedures do not.
Boolean
A data type with only two values: True or False. Validation functions typically return a boolean.
Event listener
JavaScript code that waits for a user action (event) — such as clicking Submit — and then runs.
Desk check
Manually tracing through an algorithm step by step, tracking variable values, before running the code.

Concept recap

On the Theory page you explored how apps specify functional and non-functional requirements, map data using the ISTPO model, and apply safe, secure and ethical practices in data handling. This skills page puts those concepts into code: form validation is one of the most important places where all three intersect. It protects users from submitting bad data, protects the system from malicious input, and gives users clear, respectful feedback when something goes wrong.

Form validation always follows the same logical structure regardless of language:

  1. Collect the user's input
  2. Check each field against its rules (presence, format, length, range)
  3. If any rule fails — record the error and stop
  4. If all rules pass — proceed to process or submit the data
  5. Display meaningful feedback to the user either way

NSW Pseudocode — design before you code

Before writing any code in any language, write the algorithm in NSW pseudocode. This is the design — language-independent, logic-focused. The implementation (Python, JS, or both) comes after.

NSW pseudocode rules used here: FUNCTION / ENDFUNCTION · RETURN · IF / THEN / ELSE / ENDIF · AND / OR / NOT · ← for assignment · // for comments
📋 NSW Pseudocode — Form Validation Algorithm
// ── Form Validation Algorithm
// Validates a registration form: name, email, password
// Returns TRUE if all fields are valid, FALSE otherwise

FUNCTION validateForm(name, email, password)
  errors ← []                              // empty list to collect error messages

  // ── Rule 1: Name must not be empty
  IF name = "" THEN
    errors ← errors + ["Name is required"]
  ENDIF

  // ── Rule 2: Email must contain @ and a dot after it
  IF NOT (email CONTAINS "@") OR NOT (email CONTAINS ".") THEN
    errors ← errors + ["Email must be a valid address"]
  ENDIF

  // ── Rule 3: Password must be at least 8 characters
  IF LENGTH(password) < 8 THEN
    errors ← errors + ["Password must be at least 8 characters"]
  ENDIF

  // ── Rule 4: Password must contain at least one digit
  hasDigit ← FALSE
  FOR EACH character IN password DO
    IF character >= "0" AND character <= "9" THEN
      hasDigit ← TRUE
    ENDIF
  NEXT character
  IF NOT hasDigit THEN
    errors ← errors + ["Password must contain at least one number"]
  ENDIF

  // ── Return result
  IF LENGTH(errors) = 0 THEN
    RETURN TRUE
  ELSE
    OUTPUT errors
    RETURN FALSE
  ENDIF

ENDFUNCTION

Now you have a language-independent design. The walkthroughs below implement this exact algorithm — compare the pseudocode to each language's code as you work through the steps.

  <script>
    // validateForm — mirrors the NSW pseudocode algorithm
    function validateForm(name, email, password) {
      var errors = [];  // ← pseudocode: errors ← []

      // Rule 1: name not empty
      if (name.trim() === "") {
        errors.push("Name is required");
      }

      // Rule 2: valid email

        errors.push("Email must be a valid address");
      }

      // Rule 3: password length
      if (password.length < 8) {
        errors.push("Password must be at least 8 characters");
      }

      // Rule 4: password contains a digit
      if (!/\d/.test(password)) {
        errors.push("Password must contain at least one number");
      }

      return errors;  // empty array = valid
    }
  </script>
What's happening: The JS function directly mirrors the pseudocode — same four rules, same logic. .trim() removes whitespace (like Python's .strip()). /\d/.test(password) is a regular expression — \d means "any digit". The ! inverts it: "if no digit found". JS returns the errors array directly rather than a tuple.

# app.py
from flask import Flask, request, jsonify, render_template

app = Flask(__name__)

def validate_form(name, email, password):
    """Server-side validation — authoritative security check."""
    errors = []
    if not name.strip():
        errors.append("Name is required")
    if "@" not in email or "." not in email.split("@")[-1]:
        errors.append("Email must be a valid address")
    if len(password) < 8:
        errors.append("Password must be at least 8 characters")
    if not any(c.isdigit() for c in password):
        errors.append("Password must contain at least one number")
    return errors

@app.route("/")
def index():
    return render_template("index.html")

if __name__ == "__main__":
    app.run(debug=True)
What's happening: Same validation logic as the Python-only tab, now inside Flask. This is the server's copy — it will run regardless of whether JS validation passed or was bypassed. A malicious user could disable JavaScript and submit bad data directly — the server must always check.
STEP 03 Add the /register API route
── NEW IN STEP 03 — app.py ──
@app.route("/")
def index():
    return render_template("index.html")

@app.route("/register", methods=["POST"])
def register():
    """Receive form data, validate server-side, return JSON response."""
    data     = request.get_json()
    name     = data.get("name", "")
    email    = data.get("email", "")
    password = data.get("password", "")

    errors = validate_form(name, email, password)

    if errors:
        return jsonify({"success": False, "errors": errors}), 400
    else:
        return jsonify({"success": True, "message": "Registration successful!"}), 200

if __name__ == "__main__":
    app.run(debug=True)
What's happening: /register is a POST endpoint — it receives JSON data, validates it, and returns a JSON response. HTTP status 400 means "bad request" (validation failed); 200 means "OK". The frontend reads this JSON to decide what to show the user.

── NEW IN STEP 06 ──

Bug detective

Each code sample below contains one bug. Identify the bug and explain what will go wrong at runtime before revealing the answer.

🐛 Bug #1 — Python

What's wrong with this validation function?

def validate_form(name, email, password):
    errors = []
    if name.strip() == "":
        errors.append("Name is required")
    if "@" not in email or "." not in email.split("@")[-1]:
        errors.append("Email must be a valid address")
    if len(password) < 8:
        errors.append("Password must be at least 8 characters")
    if not any(char.isdigit() for char in password):
        errors.append("Password must contain at least one number")
    return (True, errors)

Hint: What does the return statement always return, regardless of what errors were collected?

Bug found: The return statement always returns True as the first element of the tuple — even when there are errors. It should be return (len(errors) == 0, errors). As written, the calling code will always think validation passed, even if the errors list is full. This is a logic error — no syntax error, no crash, just silently wrong behaviour. The hardest bugs to find are the ones that don't cause a crash.
🐛 Bug #2 — JavaScript

This event listener has a critical flaw. What is it?

document.getElementById("reg-form").addEventListener("submit", function(e) {
  // event.preventDefault() — commented out for now
  var name     = document.getElementById("name").value;
  var email    = document.getElementById("email").value;

Hint: What happens when a form is submitted in a browser without preventDefault()?

Bug found: Without event.preventDefault() (or e.preventDefault()), the browser performs its default form submission behaviour — the page refreshes immediately. The JavaScript runs, but the page reloads so fast the user never sees any error messages or success feedback. The fix is to call e.preventDefault() as the very first line of the event handler — before anything else runs.
🐛 Bug #3 — Python + JS (architecture)

This Flask route has a security problem. Can you spot it?

@app.route("/register", methods=["POST"])
def register():
    data     = request.get_json()
    name     = data.get("name", "")
    email    = data.get("email", "")
    password = data.get("password", "")

    # Client already validated — skip server check for speed
    # Save user to database here...
    return jsonify({"success": True, "message": "Registration successful!"}), 200

Hint: Who controls the data that arrives at this route?

Bug found: The server trusts the client. A malicious user can bypass JavaScript entirely — using a tool like Postman or curl to send any data directly to /register. By skipping server-side validation "for speed," this route accepts empty names, malformed emails, weak passwords, or even malicious code. The fix is to always call validate_form() on the server regardless of what the client sent. Client-side validation is a UX feature. Server-side validation is a security requirement. They are not interchangeable.
