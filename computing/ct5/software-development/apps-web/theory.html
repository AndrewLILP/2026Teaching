Developing Apps & Web Software

Apps and web tools are now the primary way people interact with computing systems. In this focus area you'll explore how software is designed, built, and evaluated — from understanding user needs through to writing object-oriented code and testing your solution against real requirements.

📋 Teaching advice

Students apply computing skill domains to design software projects using a general-purpose or object-oriented programming language (Stage 5). This focus area can be combined with Designing for User Experience (UI/UX design for the app) or Analysing Data (building a data visualisation app or dashboard).

Key vocabulary

These terms appear throughout the focus area. Make sure you can explain each one in your own words before moving to the activities below.

📖 Word bank

Functional requirement
What the app must do — specific features and behaviours (e.g. "user can log in").
Non-functional requirement
How well the app must do it — qualities like speed, security, accessibility, or cost.
Use case
A scenario describing how a specific user interacts with the app to achieve a goal.
User interface (UI)
The visual elements a user interacts with — buttons, forms, menus, screens.
User experience (UX)
The overall quality of a person's interaction with the app — how it feels to use.
Event-driven programming
Code that runs in response to user actions (events) like clicks, key presses, or form submissions.
Object-oriented programming
A style of programming that organises code into objects with properties (data) and methods (actions).
Algorithm
A precise, step-by-step set of instructions to solve a problem or complete a task.
Modular programming
Breaking a program into smaller, reusable functions or modules — easier to build, test, and fix.
Agile / iterative design
A development approach using short cycles (sprints) to design, build, test, and improve — rather than building everything at once.
Test case
A specific input, the action taken, and the expected output — used to check code behaves correctly.
Cybersecurity
Practices and technologies that protect digital systems, data, and users from unauthorised access or harm.
Identifying & Defining

Identifying & Defining

Before writing any code, developers must understand the problem they're solving. This phase is about asking the right questions — who will use this app, what do they need it to do, and what are the boundaries of the project?

Apps and web-based tools are designed to solve real problems for real people. Before building, developers explore:

  • The purpose of the app — what problem does it solve?
  • The uses — who uses it, when, and why?
  • The impact of apps on industries — from healthcare to education to retail
  • How apps changed in response to society's evolving needs and opportunities
Think about it: TikTok, MyFitnessPal, Google Maps — what problem does each one solve? Who is the intended user? How have their features changed over time?

Every app can be described using the ISTPO model. Mapping these five elements for an existing app before you design your own is one of the most powerful analysis tools in software development.

  • Inputs: What data does the user (or system) provide? E.g. username, location, photo.
  • Storage: Where is data kept? Local device, database, cloud?
  • Transmission: How does data move — between device and server, user to user?
  • Processes: What does the app calculate, sort, filter, or generate?
  • Outputs: What does the user receive? A result, a notification, an updated screen?
Syllabus link: "Explore inputs, storage, transmission, processes and outputs in apps or web-based tools" — this is the foundation of systems thinking. You'll use ISTPO to annotate and then design your own app.

Specifying requirements is how a developer defines what "success" looks like before a line of code is written.

  • Functional requirements — what the app must do: "Users can search for recipes by ingredient."
  • Non-functional requirements — how well it must do it: performance, security, accessibility, cost.
Examples (non-functional) Subscription or purchase cost · Motivation or engagement · Load time under 2 seconds · Works on mobile screens · WCAG 2.1 AA accessibility compliance

Requirements come from use cases — stories about how a specific user interacts with the app. From a use case, you define test cases: an input, an expected output, and the result you actually got.

Key skill: "Identify, define and visually represent the function and purpose of elements and features of existing code." Look at a real app's features — can you map each one to a functional requirement?

Software shapes society — and developers have ethical and legal responsibilities as a result. Consider how apps affect different groups, including Aboriginal and Torres Strait Islander Peoples, people of different ages and genders, people with disability, and culturally and linguistically diverse people.

NESA examples Avoiding cultural assumptions — users do not always have a first and last name · Collective ownership of images and artworks, e.g. Indigenous Cultural and Intellectual Property (ICIP) · Protocols and permissions for capturing and displaying images, voice or video of Aboriginal people.
Researching & Planning

Researching & Planning

This phase is about investigating existing solutions, designing alternatives, and selecting the approach you'll build. Good research prevents bad code — developers who skip planning spend twice as long debugging.

Professional software teams use agile methods — short development cycles called sprints — rather than planning everything upfront. This lets teams respond to feedback and fix problems early.

  • Experiment with agile tools: Kanban boards, user story maps, sprint logs
  • Plan and manage a project using an iterative approach — build a small piece, test it, improve it, repeat
  • Define a real-world problem, break it into manageable parts, and interview stakeholders to identify their needs
  • Evaluate existing solutions to the problem or related problems

Before generating your own designs, explore and evaluate existing design principles. Good design is invisible — users don't notice it when it works, but immediately notice when it fails.

Design principle examples Accessibility and inclusion: Web Content Accessibility Guidelines (WCAG) · Augmentative and alternative communication (AAC) · Cognitive load, consistency, hierarchy, meaningful feedback and error messages.
  • Outline how app and web software address environmental, lifestyle, societal and economic challenges, including cyber safety
  • Generate alternative designs and evaluate them against requirements to select a preferred design
Social challenge examples Access, equity, diversity, representation, sensitivities · Stereotyping, empowerment · Demographics, popularity, digital footprint

Algorithms are designed before being coded — using flowcharts, pseudocode, or diagrams. In NSW Computing Technology, we always write NSW pseudocode before writing code.

  • Represent algorithms using flowcharts and pseudocode
  • Design your own and modify existing algorithms for apps
  • Explore sub-programs and code modules (functions) in programming code
  • Compare operators within existing code or HTML, including arithmetic, logical and relational operators
  • Explore the application of code modules that perform discrete functions
  • Validate algorithms using desk checking — trace through the logic manually before running

Modern apps collect, store, and transmit user data. Developers must understand and apply data principles at the planning stage — not as an afterthought.

  • Investigate data collection and interpretation adhering to privacy and cybersecurity principles — specify what data is collected, who owns it, and how it will be protected
  • Represent data and code to facilitate computation: select appropriate data types, structure code systematically, understand data type limitations
  • Identify and define the features of an object-oriented programming language — classes, objects, attributes, methods, inheritance
Producing & Implementing

Producing & Implementing

This is the build phase — turning your design into working software. At Stage 5, apps are built in an object-oriented programming language, with event-driven programming responding to user input and HTML/CSS for web-based presentation.

Object-oriented programming organises code around objects — real-world things modelled in code. Each object has:

  • Attributes (properties/data) — e.g. a User object has a username and email
  • Methods (behaviours/actions) — e.g. a User can login() or updateProfile()

Key OOP features to describe:

  • Class — a blueprint for creating objects
  • Object — an instance of a class
  • Encapsulation — keeping data and methods bundled together
  • Inheritance — one class can extend another

You'll also: interpret and modify existing programs for apps; design and implement modular programs with functions; and extend or implement object-oriented programs.

Apps respond to user actions — clicks, taps, keyboard input, form submissions. This is event-driven programming: code waits for an event, then runs.

  • Develop the UI and UX of an app, including using event-driven programming to respond to user input
  • Develop a web page or app that separates content and presentation using HTML and CSS
  • Select and use specialist terminology in context
HTML/CSS separation: HTML defines the structure and content. CSS controls how it looks. Keeping them separate makes it much easier to change the design without breaking the content — a core principle of professional web development.
  • Apply selected algorithms and data structures for apps
  • Validate programs using test cases and debug a range of errors
  • Document the design and implementation of the solution in a project notebook
  • Create a record of project development demonstrating iterative design and evaluation
Remember: Pseudocode before code. Every algorithm you implement should first be written in NSW pseudocode — this shows your thinking, not just the result.
Testing & Evaluating

Testing & Evaluating

Testing checks that your app does what it's supposed to do. Evaluating goes further — did it solve the original problem? Did it meet social and ethical responsibilities? What would you do differently?

  • Test cases: Use predetermined functional and non-functional requirements as the basis for testing
  • Algorithm checking: Use selected data to check an algorithm and programming code by testing
  • Tracing: Validate algorithms and programs through tracing and test cases — step through the logic manually, tracking variable values at each step
  • Peer evaluation: Evaluate your own project and that of peers using the same predetermined requirements
  • Evaluate whether solutions meet social, ethical and legal responsibilities and cybersecurity principles
  • Relate social, ethical and cybersecurity considerations of a software project
  • Explore careers in software development
Reflection prompt: Does your app work equally well for all intended users? Could it be misused? Does it handle user data responsibly? These are professional software review questions — not just a school checklist.

Analysis activity — examine before you build

Before designing your own app, you'll analyse an existing one. Understanding how real software works gives you far better design instincts than starting from scratch.

🔍 Analysis activity

Annotate a real app

Choose a web app or mobile app you use regularly (e.g. a school portal, a streaming service, a food delivery app). Work through the following steps in your project notebook:

  1. ISTPO mapping: Identify at least two inputs, one storage type, one transmission step, two processes, and two outputs.
  2. Requirements reverse-engineering: Write three functional requirements and two non-functional requirements that the app appears to satisfy.
  3. Design principles audit: Identify one example of good UI design and one area for improvement. Use specific language: cognitive load, hierarchy, consistency, feedback.
  4. Diversity check: Does the app make assumptions about its users? (Names, language, device, ability?) What could be more inclusive?
  5. Data and privacy: What data does the app collect about you? Where might it be stored or transmitted? Is there a privacy policy — and is it actually readable?

Social & ethical considerations

Software developers hold significant power — the choices they make about what an app collects, who it includes, and how it behaves affect real people at scale.

Privacy and data ownership

Specify what data your app collects, who owns it, and how it's protected. Users have the right to know. Developers have a legal and ethical obligation to protect that data.

Accessibility and inclusion

Design for the full range of users — including people with disability, EAL/D learners, and users on older devices. Accessibility is a legal requirement in many contexts, not just good practice.

Cultural sensitivity

Avoid encoding cultural assumptions in code — not all users have a "first name / last name", a single timezone, or access to the same services. ICIP applies to digital content.

Cybersecurity responsibilities

Developers are responsible for the security of the systems they build. Weak authentication, plain-text passwords, and unencrypted transmissions put real users at risk.

Tiered tasks

Choose the task that matches your current level — or challenge yourself with the next tier up.

Foundation

ISTPO annotation

Choose one app from the list provided. Using the ISTPO model, identify and list at least one example of each element. Draw a simple diagram showing how data flows through the system.

Core

Requirements specification

Choose an app you use regularly. Write a requirements specification with three functional requirements, two non-functional requirements, and two use cases. For each use case, write one test case with a defined input, expected output, and a "pass" condition.

Extension

Design critique and redesign

Critically evaluate an existing app's UI against at least three WCAG accessibility principles and two design principles (e.g. cognitive load, hierarchy). Identify two design failures. Produce an annotated wireframe for a revised screen with a written justification referencing specific design principles.
