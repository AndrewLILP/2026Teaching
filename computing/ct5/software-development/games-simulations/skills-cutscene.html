Cutscenes, Timeline & Game State

Almost every game has moments where you take control away from the player — a story introduction, a level transition, a boss arrival. These are cutscenes. On this page you'll build one using Unity's Timeline tool, then write a game state machine in C# to manage the transition from cutscene into gameplay. You'll also meet enum — a safer and more expressive way to represent state than a plain string.

Producing & Implementing

Key Vocabulary

These terms are all new on this page. Several of them describe Unity tools rather than C# syntax — read the definitions before opening Unity.

Word Bank — Cutscenes & Game State

Timeline
A Unity editor window for sequencing animations, audio, and camera movements over time — like a video editing timeline. You arrange clips on tracks; the Playable Director plays them back. No C# required to build the clip content.
Playable Director
A Unity component that controls playback of a Timeline asset. You add it to a GameObject in your scene, assign a Timeline asset to it, and call director.Play() or director.Stop() from C# to control it.
Animation Track
One row in the Timeline window. You bind it to a GameObject (e.g. the camera) and add animation clips to it. The Director plays those clips in sequence when the Timeline runs.
game state
A variable that records what mode the game is currently in — "cutscene", "playing", "paused", "game over". Code reads this variable to decide what to allow: player input is blocked during a cutscene, enabled during play.
enum
A C# keyword that defines a named set of constant values. enum GameState { Cutscene, Playing, Paused } creates a type where the only valid values are exactly those names. Safer than strings because typos cause compile errors instead of silent bugs.
event subscription
The pattern of connecting a method to an event using +=. director.stopped += OnCutsceneComplete tells Unity: "when the director stops, call my OnCutsceneComplete method." The opposite — disconnecting — uses -=.
enabled
A property on Unity components. Setting playerMovement.enabled = false stops the script from running its Update() without destroying it. Setting it back to true resumes it. Used to freeze player input during cutscenes.
SignalEmitter
A Unity Timeline marker that fires a signal at a specific time in the Timeline. A SignalReceiver component on a GameObject listens for it and calls a C# method — letting Timeline trigger your code without polling.

What is Timeline? — A No-Code Sequencer

Unity's Timeline window lets you arrange clips on tracks and play them back in sequence — similar to a video editor. Each row is a track bound to a specific object; each coloured block is a clip that animates, plays audio, or triggers an event.

The Signal marker (📍 on the third track) fires at a specific point in the timeline — in this case, near the end. It calls your C# method (OnCutsceneComplete) to hand control back to the player.

🎮 Set up Timeline in Unity

  1. Open Window → Sequencing → Timeline.
  2. Create an empty GameObject in the Hierarchy. Name it CutsceneDirector.
  3. With it selected, click Create in the Timeline window. Unity adds a Playable Director component and saves a new .playable asset.
  4. In the Timeline window, click + Add → Animation Track. Drag your Main Camera from the Hierarchy into the track's binding slot on the left.
  5. Right-click on the track → Add From Animation Clip — or record a clip directly by pressing the red record button and moving the camera.
  6. Press Play in the Timeline window (not the game) to preview. Press Play in the game to run it live.

Game State — Tracking What the Game is Doing

A game state is a variable that records the current mode of the game. During a cutscene the player should be frozen — input disabled, camera under Timeline control. When the cutscene ends, the player is unfrozen and takes control.

Without a state variable, your code has no reliable way to ask "am I currently in a cutscene?" — and things like the skip button or pause logic become impossible to implement correctly.

Producing & Implementing

Enums — Safer Than Strings

The simplest way to represent game state is a string: string gameState = "cutscene". This works — but it's fragile. A single typo ("Cutscene" vs "cutscene") creates a bug that compiles fine and fails silently at runtime.

An enum is a named set of constant values. The compiler enforces valid values — a typo is a compile error, not a hidden bug.

❌ String state — fragile
string gameState = "cutscene";

// Later in code:
if (gameState == "Cutscene")  // typo — capital C
{
    // This never runs — silent bug
}
✓ Enum state — safe
enum GameState { Cutscene, Playing, Paused }
GameState gameState = GameState.Cutscene;

// Later in code:
if (gameState == GameState.Cutscen)  // typo
{
    // Compile error — caught immediately
}
Enum is a conceptual step up. Before writing any C#, draw the state machine in your project notebook. Label each state (box) and each transition (arrow with the trigger that causes it). This diagram is your design evidence for the D&P framework — keep it.
Producing & Implementing

Algorithm: Starting the Game via Cutscene

This algorithm runs when the scene loads. It sets the state, disables player input, plays the cutscene, then waits for it to finish before handing control back.

NSW Pseudocode — StartGame algorithm
BEGIN StartGame
  gameState ← "cutscene"
  playerInput.enabled ← FALSE
  CALL director.Play()
  WAIT UNTIL director.state ≠ PlayState.Playing
  gameState ← "playing"
  playerInput.enabled ← TRUE
END StartGame
Reading the pseudocode — the WAIT UNTIL challenge WAIT UNTIL director.state ≠ PlayState.Playing cannot be written as a simple loop in C# — it would block the main thread and freeze Unity. Instead, we use an event subscription: director.stopped += OnCutsceneComplete. Unity calls our method when the director stops, rather than us checking every frame. This is the same event-driven pattern from Page 2.

STEP 3 Subscribe to the director.stopped event in Start()
✦ NEW IN STEP 3 — event subscription in Start(), unsubscription in OnDestroy()
using UnityEngine;
using UnityEngine.Playables;

public enum GameState { Cutscene, Playing, Paused, GameOver }

public class GameManager : MonoBehaviour
{
    [SerializeField] private PlayableDirector director;
    [SerializeField] private PlayerMovement   playerMovement;
    private GameState currentState;

    void Start()
    {
        director.stopped += OnCutsceneComplete;   // ── NEW IN STEP 3 ──
    }

    void OnDestroy()                              // ── NEW IN STEP 3 ──
    {                                             // ── NEW IN STEP 3 ──
        director.stopped -= OnCutsceneComplete;   // ── NEW IN STEP 3 ──
    }                                             // ── NEW IN STEP 3 ──

    void Update()
    {

    }
}
director.stopped += OnCutsceneComplete — the += operator subscribes a method to an event. When the director finishes playing, Unity calls OnCutsceneComplete automatically. The OnDestroy() method uses -= to unsubscribe when the GameManager object is destroyed. This is essential: if you don't unsubscribe, Unity may try to call a method on a destroyed object and crash. Always subscribe in Start() and unsubscribe in OnDestroy().
STEP 4 Implement StartCutscene() — disable input, play director
✦ NEW IN STEP 4 — StartCutscene() sets state and plays the Timeline
using UnityEngine;
using UnityEngine.Playables;

public enum GameState { Cutscene, Playing, Paused, GameOver }

public class GameManager : MonoBehaviour
{
    [SerializeField] private PlayableDirector director;
    [SerializeField] private PlayerMovement   playerMovement;
    private GameState currentState;

    void Start()
    {
        director.stopped += OnCutsceneComplete;
        StartCutscene();   // ── NEW IN STEP 4 — call on scene load
    }

    void OnDestroy()
    {
        director.stopped -= OnCutsceneComplete;
    }

    void Update()
    {

    }

    private void StartCutscene()                      // ── NEW IN STEP 4 ──
    {                                                  // ── NEW IN STEP 4 ──
        currentState = GameState.Cutscene;             // ── NEW IN STEP 4 ──
        playerMovement.enabled = false;                // ── NEW IN STEP 4 ──
        director.Play();                               // ── NEW IN STEP 4 ──
    }                                                  // ── NEW IN STEP 4 ──
}
currentState = GameState.Cutscene — we set the enum value before doing anything else, so the state is accurate from the start. playerMovement.enabled = false — stops the PlayerMovement script's Update() from running, so the player cannot move during the cutscene. The script is not destroyed — just paused. director.Play() — starts the Timeline. Compare to the pseudocode: gameState ← "cutscene", playerInput.enabled ← FALSE, CALL director.Play() — line for line.
STEP 5 Implement OnCutsceneComplete() — re-enable input, update state
✦ NEW IN STEP 5 — OnCutsceneComplete() completes the state transition
using UnityEngine;
using UnityEngine.Playables;

public enum GameState { Cutscene, Playing, Paused, GameOver }

public class GameManager : MonoBehaviour
{
    [SerializeField] private PlayableDirector director;
    [SerializeField] private PlayerMovement   playerMovement;
    private GameState currentState;

    void Start()
    {
        director.stopped += OnCutsceneComplete;
        StartCutscene();
    }

    void OnDestroy()
    {
        director.stopped -= OnCutsceneComplete;
    }

    void Update()
    {

    }

    private void StartCutscene()
    {
        currentState = GameState.Cutscene;
        playerMovement.enabled = false;
        director.Play();
    }

    private void OnCutsceneComplete(PlayableDirector pd)  // ── NEW IN STEP 5 ──
    {                                                      // ── NEW IN STEP 5 ──
        currentState = GameState.Playing;                  // ── NEW IN STEP 5 ──
        playerMovement.enabled = true;                     // ── NEW IN STEP 5 ──
    }                                                      // ── NEW IN STEP 5 ──
}
OnCutsceneComplete(PlayableDirector pd) — the parameter pd is the director that stopped. We don't use it here, but the signature must match what Unity's event passes. currentState = GameState.Playing — update the state first, then re-enable input. The order matters: if another script checks the state in the same frame, it sees Playing before the player can move. playerMovement.enabled = true — the Update() loop in PlayerMovement resumes, and the player regains control. Press Play in Unity — the cutscene should run, then the player should become moveable when it ends.
Producing & Implementing

Event Subscription — The += / -= Pattern

You first saw event-driven programming in Page 2 with OnTriggerEnter — Unity called your method when a collision happened. The director.stopped event is the same idea, but you control the subscription explicitly with += and -=.

event subscription (+=)

What it is

Connecting a method (your code) to an event (Unity's signal) so that Unity calls your method when the event fires.

Example

director.stopped += OnCutsceneComplete;
When the director stops, call OnCutsceneComplete.

Why unsubscribe?

If the GameManager is destroyed but the director still exists, Unity tries to call a method on a null object — NullReferenceException. Always use -= in OnDestroy().

Analogy

Subscribing to a newsletter: += opts you in, -= opts you out. If you leave town (object destroyed) without opting out, letters arrive at an empty house.

Producing & Implementing

Alternative: SignalEmitter and SignalReceiver

Instead of subscribing to director.stopped, you can fire a method at a precise moment during the Timeline using a SignalEmitter marker. This is useful when the cutscene should hand control back before it finishes — for example, the camera finishes panning at 4 seconds but the audio fades out until 5 seconds.

🎮 Add a SignalEmitter to Timeline

  1. In the Timeline window, right-click on the timeline ruler at the point you want the signal — e.g. at 4 seconds.
  2. Select Add Signal Emitter. A small marker (📍) appears on the ruler.
  3. In the Inspector for that marker, click Create Signal. Name it CutsceneEndSignal and save it.
  4. Select your GameManager object in the Hierarchy. Add a Signal Receiver component.
  5. In the Signal Receiver, click +. Drag the CutsceneEndSignal asset into the signal slot. In the reaction field, drag the GameManager object in and choose OnCutsceneComplete.
  6. Now when the Timeline reaches that marker, Unity calls your method — no C# subscription needed in this approach.
Which approach to use? director.stopped += is simpler to code and fires when the entire Timeline finishes. SignalEmitter is more flexible — it fires at any point in the Timeline regardless of when playback ends. For this page, either works. For the Extension task, SignalEmitter becomes essential.
🎨 Shader Lesson Cross-Reference

Want to add a fade-to-black transition?

A common polish effect is fading the screen to black at the start of a cutscene and fading back in when gameplay begins. This is done with a full-screen shader on a UI overlay — a quad that covers the camera, driven by an alpha value your C# code changes over time.

See Shader Lesson 01: Unlit Colour & Transparency — specifically the transparency section — for how to create and control an unlit material's alpha from a script. The fade would be called at the start of StartCutscene() and reversed inside OnCutsceneComplete().

🔍 Bug 1 — Syntax Error

GameManager.cs — wrong event operator

This script won't compile. The error reads: Cannot implicitly convert type 'void' to 'UnityEngine.Playables.PlayableDirector'. The event subscription line is wrong. Find it.

using UnityEngine;
using UnityEngine.Playables;

public enum GameState { Cutscene, Playing, Paused, GameOver }

public class GameManager : MonoBehaviour
{
    [SerializeField] private PlayableDirector director;
    [SerializeField] private PlayerMovement   playerMovement;
    private GameState currentState;

    void Start()
    {
        director.stopped = OnCutsceneComplete;   // ← wrong operator
        StartCutscene();
    }

    void OnDestroy()
    {
        director.stopped -= OnCutsceneComplete;
    }

    private void StartCutscene()
    {
        currentState = GameState.Cutscene;
        playerMovement.enabled = false;
        director.Play();
    }

    private void OnCutsceneComplete(PlayableDirector pd)
    {
        currentState = GameState.Playing;
        playerMovement.enabled = true;
    }
}
Bug found: director.stopped = OnCutsceneComplete uses = (assignment) instead of += (subscribe). Events cannot be assigned with = from outside the class that declares them — it's a C# rule. Fix: change = to +=.
🔍 Bug 2 — Logical Error

GameManager.cs — player input re-enabled before cutscene ends

This script compiles and runs. The cutscene plays — but the player can move during the cutscene from the very first frame. Then when the cutscene ends, nothing changes. What is wrong?

using UnityEngine;
using UnityEngine.Playables;

public enum GameState { Cutscene, Playing, Paused, GameOver }

public class GameManager : MonoBehaviour
{
    [SerializeField] private PlayableDirector director;
    [SerializeField] private PlayerMovement   playerMovement;
    private GameState currentState;

    void Start()
    {
        director.stopped += OnCutsceneComplete;
        StartCutscene();
    }

    void OnDestroy()
    {
        director.stopped -= OnCutsceneComplete;
    }

    private void StartCutscene()
    {
        currentState = GameState.Cutscene;
        playerMovement.enabled = true;   // ← wrong value — should be false
        director.Play();
    }

    private void OnCutsceneComplete(PlayableDirector pd)
    {
        currentState = GameState.Cutscene;   // ← wrong state — should be Playing
        playerMovement.enabled = true;
    }
}
Two logical errors found: (1) StartCutscene() sets playerMovement.enabled = true instead of false — the player is never frozen, so they can move freely during the cutscene. (2) OnCutsceneComplete() sets currentState = GameState.Cutscene instead of GameState.Playing — the state never transitions, so any code that checks for the Playing state will never run. Fix both values.
