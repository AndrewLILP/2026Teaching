Horror Clone — Keys, Doors & NavMesh AI

The final capstone. You'll build a game where the player collects three keys to unlock a door while being pursued by an AI enemy. The enemy uses Unity's NavMesh system — a predefined pathfinding algorithm you configure and control rather than write from scratch. Read the genre guidelines before anything else.

⚠️ Read Before Starting — Genre Scope

This is a Horror-Inspired Game — Not a Horror Game

The "horror" genre is used on this page because its core game logic — collecting items, unlocking paths, being pursued by an AI — is excellent for practising arrays, triggers, and NavMesh. The genre is a design context, not a content directive.

Your game must stay within these boundaries. Age suitability is a non-functional requirement and a legal responsibility for any game developer. Violations of these guidelines are a design failure, not just a rules issue.

✅ Atmosphere through lighting and sound only — darkness, ambient music, footsteps
✅ Enemy is a stylised shape or low-poly character — not a realistic human
✅ Keys, doors, and an exit — puzzle and exploration mechanics
✅ Player is caught and game ends — no damage, no health, no death animation
❌ No gore, no blood, no realistic violence of any kind
❌ No weapons — the player cannot fight the enemy
❌ No distressing content targeting real-world trauma
❌ No realistic human enemy models — stylised only
Identifying & Defining

Project Brief

🎮 Design Brief — Stage 5 Capstone

The Hollow

Design and build a first-person or third-person exploration game in Unity 6. The player moves through a darkened indoor environment, collecting three keys scattered through different rooms. Each key unlocks a door. When all doors are unlocked, the player can reach the exit. An AI pursues the player throughout. If the AI catches the player, the game ends.

Atmosphere is conveyed through lighting, ambient audio, and level design — not graphic content. The game must be designed with age suitability considered from the start, documented in your design brief in the notebook.

Functional Requirements

  • Player moves through a multi-room environment
  • 3 keys placed in different rooms — player collects by walking over them
  • Each key collected: HUD updates (e.g. 🗝️ 🗝️ ☐)
  • All 3 keys collected: final door unlocks
  • Player reaches the exit: win panel appears, input disabled
  • AI enemy patrols when player not in sight; chases when visible
  • AI catches player: game-over panel appears, input disabled

Non-Functional Requirements

  • Age suitability: specify target rating (e.g. PG) — justify in notebook
  • Atmosphere: tension through lighting and sound, not content
  • Enemy: must be stylised — no realistic human model
  • Performance: NavMesh baked correctly — no teleporting enemy
  • Accessibility: subtitles or visual cues for any audio warning
  • Code quality: all scripts documented, modular structure

Researching & Planning

NavMesh — Using a Predefined Algorithm

🧠 R&P-1 — Predefined Algorithms & AI

You don't write the pathfinding. You use it.

The NESA syllabus asks you to "explore how predefined algorithms and artificial intelligence (AI) control computer components." NavMesh is your primary example on this page.

NavMesh (Navigation Mesh) is a baked representation of the walkable surfaces in your scene. Unity's NavMeshAgent component uses an A* (A-star) pathfinding algorithm to find the shortest walkable route between any two points. You set a destination; the algorithm calculates the path and moves the agent along it, avoiding obstacles automatically.

This is identical to how game studios use physics engines — you don't write the rigid-body simulation, you configure it and control it. Using predefined algorithms efficiently is a professional skill. Understanding what the algorithm does and what can go wrong when it's misconfigured is what the Bug Detective section on this page tests.

🎮 Baking a NavMesh in Unity

  1. Build your level geometry (floors, walls, corridors). All static geometry must have the Static checkbox ticked in the Inspector — specifically Navigation Static.
  2. Open Window → AI → Navigation. The Navigation window opens.
  3. Click the Bake tab. Leave default settings for a first bake. Click Bake.
  4. A blue overlay appears on the floor in Scene view — this is the NavMesh. Blue = walkable. The AI can only navigate the blue areas.
  5. Select your enemy GameObject. In the Inspector, click Add Component → NavMesh Agent.
  6. Set Speed, Stopping Distance, and Angular Speed to match your game's feel.
  7. Critical: if you move any walls or add geometry after baking, you must rebake. Un-rebaked NavMeshes cause the Bug Detective's third bug — the enemy teleports.
What is A*? A* (A-star) is a graph search algorithm that finds the shortest path between two nodes. It balances the actual distance travelled so far with an estimated distance remaining (the heuristic). Unity's NavMesh pre-computes the graph from your geometry; the agent runs A* on it at runtime. You'll study graph algorithms in Stage 6 — for now, understand it as a smart, efficient map reader.

Pseudocode — Four Core Algorithms

Write all four algorithms in your project notebook before Sprint 2. Annotate each with the C# concept it maps to.

NSW Pseudocode — Key & Door System
ARRAY keysCollected[3] ← [FALSE, FALSE, FALSE]
CONST KEYS_NEEDED ← 3

FUNCTION CollectKey(index : integer)
  keysCollected[index] ← TRUE
  CALL UpdateKeyHUD()
  IF AllKeysCollected() THEN
    CALL UnlockFinalDoor()
  ENDIF
ENDFUNCTION

FUNCTION AllKeysCollected() : boolean
  FOR i ← 0 TO 2
    IF keysCollected[i] = FALSE THEN
      RETURN FALSE
    ENDIF
  NEXT i
  RETURN TRUE
ENDFUNCTION

PROCEDURE OnTriggerEnter(other : Collider)   // enemy catches player
  IF other.tag = "Enemy" THEN
    gameState ← "caught"
    CALL ShowGameOver()
  ENDIF
ENDPROCEDURE
Reading AllKeysCollected() carefully The logic is: assume all keys are collected, return FALSE the moment one is not. If the loop completes without returning FALSE, all three must be TRUE — so return TRUE. This pattern is called an early exit. Bug 2 in the Bug Detective has this logic inverted — study it carefully before you implement.
NSW Pseudocode — NavMesh Enemy AI
// Enemy AI — uses predefined NavMesh pathfinding (A* algorithm)
PROCEDURE UpdateEnemy()
  IF CanSeePlayer() THEN
    agent.destination ← player.position
    agent.speed ← chaseSpeed
  ELSE
    CALL Patrol()
  ENDIF
ENDPROCEDURE

PROCEDURE Patrol()
  IF agent.remainingDistance < 0.5 THEN
    nextWaypoint ← waypoints[currentIndex]
    agent.destination ← nextWaypoint.position
    currentIndex ← (currentIndex + 1) MOD waypoints.length
  ENDIF
ENDPROCEDURE
MOD — modulo operator (currentIndex + 1) MOD waypoints.length wraps the index back to 0 after the last waypoint, creating a loop. In C#: currentIndex = (currentIndex + 1) % waypoints.Length;. Without it, the index would exceed the array bounds and Unity would throw an IndexOutOfRangeException.

Key C# reference patterns

C# EnemyAI.cs — core pattern (write your own full version)
📋 REFERENCE — not a complete file. NavMeshAgent requires using UnityEngine.AI;
using UnityEngine;
using UnityEngine.AI;   // ← REQUIRED — NavMeshAgent lives here

public class EnemyAI : MonoBehaviour
{
    [SerializeField] private Transform   player;
    [SerializeField] private Transform[] waypoints;
    [SerializeField] private float       chaseSpeed  = 4f;
    [SerializeField] private float       patrolSpeed = 2f;
    [SerializeField] private float       sightRange  = 10f;

    private NavMeshAgent agent;
    private int currentIndex = 0;

    void Start()
    {
        agent = GetComponent<NavMeshAgent>();
    }

    void Update()
    {
        if (CanSeePlayer())
        {
            agent.speed = chaseSpeed;
            agent.destination = player.position;
        }
        else
        {
            agent.speed = patrolSpeed;
            Patrol();
        }
    }

    private bool CanSeePlayer()
    {
        // Check distance first — Raycast only within sightRange
        if (Vector3.Distance(transform.position, player.position) > sightRange)
            return false;

        // Fire a ray from this enemy toward the player
        Vector3 direction = (player.position - transform.position).normalized;
        if (Physics.Raycast(transform.position, direction, out RaycastHit hit, sightRange))
        {
            return hit.transform.CompareTag("Player");   // true only if ray hits player first
        }
        return false;
    }

    private void Patrol()
    {
        if (waypoints.Length == 0) return;

        if (agent.remainingDistance < 0.5f)
        {
            currentIndex = (currentIndex + 1) % waypoints.Length;   // MOD wrap
            agent.destination = waypoints[currentIndex].position;
        }
    }
}
Physics.Raycast fires an invisible ray. out RaycastHit hit captures what it hit. We check hit.transform.CompareTag("Player") — if the first thing the ray hits is tagged Player, the enemy can see them. If a wall is in the way, the ray hits the wall first and returns false — the enemy loses sight and returns to patrol. In the Inspector, drag the Player object into the Player slot and fill the Waypoints array with your empty patrol point GameObjects.

Bug Detective

Three bugs — but only two are code errors. The third is a workflow error: a problem caused by how the editor was used, not what the code says. Find all three.

A new type: Workflow Error Not every problem in game development is a bug in your code. Some problems come from how you configure the Unity editor — baking, prefab settings, Inspector connections. These don't appear in the script and won't be caught by a compiler. Learning to distinguish code bugs from configuration bugs is a professional skill.
🔍 Bug 1 — Syntax Error

EnemyAI.cs — missing namespace

This script won't compile. Error: The type or namespace name 'NavMeshAgent' could not be found. Are you missing a using directive?

using UnityEngine;   // ← something is missing above or below this

public class EnemyAI : MonoBehaviour
{
    private NavMeshAgent agent;

    void Start()
    {
        agent = GetComponent<NavMeshAgent>();
    }

    void Update()
    {
        agent.destination = player.position;
    }
}
Bug found: using UnityEngine.AI; is missing. NavMeshAgent lives in the UnityEngine.AI namespace — the same pattern as SceneManager requiring UnityEngine.SceneManagement. Fix: add using UnityEngine.AI; as the second line of the file.
🔍 Bug 2 — Logical Error

HorrorGameManager.cs — AllKeysCollected() returns true too early

This script compiles and runs. The player collects the first key — and the door immediately unlocks. The remaining two keys have no effect. What is wrong with the condition?

using UnityEngine;

public class HorrorGameManager : MonoBehaviour
{
    private bool[] keysCollected = new bool[3];

    public void CollectKey(int index)
    {
        keysCollected[index] = true;
        if (AllKeysCollected())
            UnlockFinalDoor();
    }

    private bool AllKeysCollected()
    {
        for (int i = 0; i < keysCollected.Length; i++)
        {
            if (keysCollected[i] == true)   // ← inverted condition
                return true;                  // ← returns too early
        }
        return false;
    }

    private void UnlockFinalDoor() { /* ... */ }
}
Bug found: The condition is inverted. The current code returns true the moment it finds any key that has been collected. The correct logic is the opposite: return false immediately if any key has not been collected; if the loop completes without finding a false, return true. Fix: change if (keysCollected[i] == true) return true; to if (keysCollected[i] == false) return false; and change the final return to return true;.
🔍 Bug 3 — Workflow Error
⚙️ Not a code error — a configuration error

Enemy teleports instead of navigating — no code changed

All scripts compile. The enemy was navigating correctly yesterday. Today, the level designer moved two walls to widen a corridor. Now the enemy skips from room to room instantly, passing through walls it shouldn't be able to cross. No code was changed between yesterday and today. What happened?

// EnemyAI.cs — unchanged, no errors
// NavMeshAgent — attached, no errors
// Scene geometry — walls moved in the editor this morning

// Error observed at runtime:
// Enemy moves to destination in a straight line through walls
// Enemy appears to "teleport" across rooms
// No errors in Console
Workflow error found — not a code bug: The NavMesh was not rebaked after the walls were moved. The NavMesh is a snapshot of the scene's walkable geometry taken at bake time. When walls move, the NavMesh still reflects the old layout — the agent navigates the old map, not the actual scene, producing straight-line movement through what are now solid walls. Fix: open Window → AI → Navigation → Bake tab → click Bake. The NavMesh will update to reflect the new wall positions. This is why NavMesh baking must be part of your workflow whenever you change static geometry — it is not automatic.
