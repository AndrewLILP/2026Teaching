Project Brief

🎮 Design Brief — Stage 5 Platformer

Coin Rush

Design and build a 3D platformer in Unity 6 inspired by the mechanics of classic coin-collection games. The player moves through a level, collecting coins to reach a target. When enough coins are collected, a star spawns. Collecting the star completes the level and shows a win screen.

The game must be designed for a broad audience — consider age suitability, accessible controls, and clear visual feedback at every stage. You are responsible for all implementation decisions not specified below.

Functional Requirements

  • Player moves using WASD or arrow keys
  • 10 coins placed in the scene — player collects by walking over them
  • Each coin collected: coin disappears, score increases by 1
  • When all 10 coins are collected: a star spawns at a marked position
  • Player collects star: win panel appears, player input disabled
  • HUD shows current coin count at all times

Non-Functional Requirements

  • Age suitability: suitable for all ages (G rating — justify in notebook)
  • Visual feedback: score updates must be immediate and clearly visible
  • Performance: no frame-rate drops from coin collision detection
  • Accessibility: controls must be remappable via Unity Input Manager
  • Code quality: all scripts documented with comments, modular structure
NESA Syllabus connection Specifying functional and non-functional requirements (I&D-4, I&D-5) · Developing prototypes to communicate design ideas (P&I-4) · Planning and managing a project using an iterative approach (P&I-13) · Evaluating whether solutions meet social, ethical and legal responsibilities (T&E-2).

Class Structure

You will write three C# scripts this project, plus update the GameManager from Page 3. Each has a single, clearly defined responsibility. Keeping concerns separate makes each script easier to test and change.

CoinCollector.cs
  • OnTriggerEnter(Collider other)
  • private void CollectCoin()
  • private void CheckWinCondition()
  • [SerializeField] GameManager gm
StarPickup.cs
  • OnTriggerEnter(Collider other)
  • calls gm.OnStarCollected()
  • [SerializeField] GameManager gm
HUDManager.cs
  • public void UpdateScore(int count)
  • public void ShowWinPanel()
  • public void HideWinPanel()
  • [SerializeField] TMP_Text scoreText
  • [SerializeField] GameObject winPanel
GameManager.cs
  • public int coinCount
  • public GameState currentState
  • public void OnStarCollected()
  • [SerializeField] HUDManager hud
  • [SerializeField] PlayerMovement pm
  • [SerializeField] GameObject star
Design decision — why separate scripts? Each script handles one thing. CoinCollector.cs handles coin logic; it doesn't know about the HUD. HUDManager.cs handles display; it doesn't know about coins. This is the Single Responsibility Principle — an OOP pattern the Extension task will take further with an interface. In your notebook, explain why this structure makes it easier to add a new coin type later.
Producing & Implementing

Pseudocode — Three Core Algorithms

Write each of these in your project notebook before opening Visual Studio. The pseudocode is your plan — note where each algorithm maps to a class above.

NSW Pseudocode — Coin Collection system
ARRAY coins[10]
coinCount ← 0
CONST COINS_NEEDED ← 10

FUNCTION CollectCoin(index : integer)
  coins[index].active ← FALSE      // deactivate the collected coin
  coinCount ← coinCount + 1
  CALL UpdateHUD(coinCount)
  CALL CheckWinCondition()
ENDFUNCTION

PROCEDURE CheckWinCondition()
  IF coinCount ≥ COINS_NEEDED THEN
    CALL SpawnStar()
  ENDIF
ENDPROCEDURE

PROCEDURE OnStarCollected()
  gameState ← "win"
  CALL ShowWinScreen()
ENDPROCEDURE
Three things to notice (1) coinCount ≥ COINS_NEEDED uses ≥ not = — this handles the edge case where two coins are collected in the same frame. Using = would require exactly 10 and could be skipped. (2) coins[index].active ← FALSE maps to gameObject.SetActive(false) in C# — the coin disappears without being destroyed, so it can be re-enabled if needed. (3) CALL SpawnStar() — the star is already in the scene, inactive. Spawning it just calls star.SetActive(true).

Key C# patterns you'll need

C# CoinCollector.cs — core pattern (write your own full version)
📋 REFERENCE — not a complete file. Use this as a starting point.
using UnityEngine;

public class CoinCollector : MonoBehaviour
{
    [SerializeField] private GameManager gameManager;
    private int coinCount = 0;
    private const int COINS_NEEDED = 10;

    private void OnTriggerEnter(Collider other)
    {
        if (other.CompareTag("Coin"))
        {
            other.gameObject.SetActive(false);   // coins[index].active ← FALSE
            coinCount++;                          // coinCount ← coinCount + 1
            gameManager.hud.UpdateScore(coinCount);
            CheckWinCondition();
        }
    }

    private void CheckWinCondition()
    {
        if (coinCount >= COINS_NEEDED)            // IF coinCount ≥ COINS_NEEDED
        {
            gameManager.SpawnStar();
        }
    }
}
This is a reference pattern, not a handout. Notice that the trigger is on the player this time — the player walks into coins, not the other way around. Alternatively, you can put CoinCollector.cs on each coin and have the coin deactivate itself on trigger entry. Discuss the trade-offs with your teacher or in your notebook: which approach is easier to manage with 10 coins? Which is more reusable if you add more coins later?
C# HUDManager.cs — TMP text update pattern
📋 REFERENCE — add using TMPro; or TMP_Text won't compile
using UnityEngine;
using TMPro;   // ← REQUIRED for TextMeshPro — don't forget this

public class HUDManager : MonoBehaviour
{
    [SerializeField] private TMP_Text scoreText;
    [SerializeField] private GameObject winPanel;

    void Start()
    {
        winPanel.SetActive(false);               // hide win panel on scene load
        UpdateScore(0);                          // show "Coins: 0" from the start
    }

    public void UpdateScore(int count)
    {
        scoreText.text = "Coins: " + count + " / 10";
    }

    public void ShowWinPanel()
    {
        winPanel.SetActive(true);
    }
}
using TMPro; is easy to forget — it's the most common compile error on this project. Without it, TMP_Text is an unknown type. In the Inspector, drag your Canvas → Score Text object into the Score Text slot, and your Canvas → Win Panel into the Win Panel slot.

🔍 Bug 2 — Logical Error

CoinCollector.cs — win condition only fires at exactly 10

This script compiles and runs. Coins collect correctly. But when the student collects the 10th coin, the star never spawns — even though the counter shows 10. What is wrong with the condition?

using UnityEngine;

public class CoinCollector : MonoBehaviour
{
    [SerializeField] private GameManager gameManager;
    private int coinCount = 0;
    private const int COINS_NEEDED = 10;

    private void OnTriggerEnter(Collider other)
    {
        if (other.CompareTag("Coin"))
        {
            other.gameObject.SetActive(false);
            coinCount++;
            gameManager.hud.UpdateScore(coinCount);
            CheckWinCondition();
        }
    }

    private void CheckWinCondition()
    {
        if (coinCount == COINS_NEEDED)   // ← wrong operator
        {
            gameManager.SpawnStar();
        }
    }
}
Bug found: coinCount == COINS_NEEDED requires the count to equal exactly 10. If two coins are collected on the same physics frame (the player stands between two overlapping coins), the count can jump from 9 to 11, skipping 10 entirely — and the star never spawns. Fix: change == to >= so the condition fires at 10 or above.
🔍 Bug 3 — Run-time Error

CoinCollector.cs — MissingReferenceException on Destroy

This script compiles. When the player collects a coin, it usually works. But occasionally — especially when walking quickly through multiple coins — Unity logs: MissingReferenceException: The object of type 'GameObject' has been destroyed but you are still trying to access it. What is causing this?

using UnityEngine;

public class CoinCollector : MonoBehaviour
{
    [SerializeField] private GameManager gameManager;
    private int coinCount = 0;
    private const int COINS_NEEDED = 10;

    private void OnTriggerEnter(Collider other)
    {
        if (other.CompareTag("Coin"))
        {
            Destroy(other.gameObject);   // ← using Destroy instead of SetActive
            coinCount++;
            gameManager.hud.UpdateScore(coinCount);
            CheckWinCondition();
        }
    }

    private void CheckWinCondition()
    {
        if (coinCount >= COINS_NEEDED)
        {
            gameManager.SpawnStar();
        }
    }
}
Bug found: Destroy(other.gameObject) queues the destruction for the end of the frame — but the coin's collider remains active until then. If the player's collider overlaps the coin across multiple physics updates in the same frame, OnTriggerEnter can fire a second time before the destroy completes, attempting to call methods on an object that's been marked for destruction. Fix: replace Destroy(other.gameObject) with other.gameObject.SetActive(false). The collider is immediately disabled, so no second trigger is possible.
