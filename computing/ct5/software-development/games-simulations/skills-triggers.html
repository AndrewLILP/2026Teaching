Collisions, Triggers & Portals

On this page you'll meet the most important idea in event-driven game programming: a method that Unity calls for you. You don't put it in Update(). You don't call it yourself. When a player walks into a trigger zone, Unity fires OnTriggerEnter — and your code runs. Understanding why is more important than memorising the syntax.

Producing & Implementing

Key Vocabulary

These terms are new on this page — they do not appear in Page 1. Read them before continuing.

Word Bank — Triggers & Portals

Collider
A Unity component that gives a GameObject a physical shape for the physics engine. When two colliders meet, Unity applies physics — objects push each other, bounce, or stop. The player stands on the floor because both have colliders.
Trigger
A collider with the Is Trigger checkbox ticked. Objects pass through it as if it isn't there — but Unity fires an event when they enter. Used for invisible zones: collectibles, portals, checkpoints, area-of-effect.
OnTriggerEnter
A Unity event function (callback) called automatically by the engine when another collider enters this trigger zone. You write the method body — Unity decides when to call it. This is event-driven programming.
Collider other
The parameter passed into OnTriggerEnter. It represents whatever entered the trigger. You check other.CompareTag("Player") to verify it was the player and not a random physics object.
CompareTag()
A Unity method that checks whether a GameObject's tag matches a string. Faster and safer than using == with other.tag — it avoids memory allocation on every call and throws a useful error if the tag doesn't exist.
SceneManager
A Unity class in the UnityEngine.SceneManagement namespace. SceneManager.LoadScene() loads a different scene by name or index. Requires a using statement at the top of the file.
bool
A data type that holds only true or false. Used here as a gate — bool portalActive = true lets us disable the portal after it fires once, preventing it from triggering again on the next frame.
Mathf.Sin()
A Unity maths function that returns a value oscillating smoothly between –1 and 1 over time. Multiplied by a small number and added to a scale, it makes an object gently breathe or pulse — no animation system required.
Producing & Implementing

Colliders vs Triggers — What's the Difference?

Every Unity physics interaction starts with a Collider component. The single checkbox labelled Is Trigger changes its behaviour entirely.

Collider (Is Trigger: off)

  • Objects cannot pass through — they collide physically
  • Fires OnCollisionEnter / OnCollisionExit
  • Unity applies forces, friction, and bounce
  • Used for: floors, walls, character bodies

Trigger (Is Trigger: on)

  • Objects pass through — no physical interaction
  • Fires OnTriggerEnter / OnTriggerExit
  • No physics — only the event fires
  • Used for: portals, collectibles, checkpoints, zones

🎮 Set up a trigger in Unity

  1. Create a Cube (GameObject → 3D Object → Cube) and position it where the portal should be.
  2. In the Inspector, find the Box Collider component.
  3. Tick the Is Trigger checkbox. The collider outline in the Scene view turns green.
  4. The cube's Mesh Renderer is still on — it's visible but passable. You can remove the renderer entirely for an invisible zone, or keep it styled as a portal frame.
  5. Add a Rigidbody to the portal cube and tick Is Kinematic — this tells Unity to include it in trigger detection without simulating physics on the portal itself.
NESA Syllabus connection Implementing an event loop to respond to user input (P&I-5) and developing game UI / UX using event-driven programming (P&I-15). OnTriggerEnter is a Unity event callback — it responds to a physics event, not direct user input, but the pattern is identical.

What Makes This Event-Driven?

In Page 1, you called your own logic from Update() — checking input 60 times a second, applying movement if a key was held. That is polling: your code constantly asks "did anything happen?"

OnTriggerEnter is different. You write the method and wait. Unity watches the physics engine, and when an overlap is detected, it calls your method for you. This is an event-driven callback — the engine notifies your code, not the other way around.

Algorithm 1: Trigger Zone Teleport

The first portal teleports the player to a new position in the same scene — for example, from one end of a level to the other. Write the algorithm in NSW pseudocode before reading the code.

NSW Pseudocode — Teleport algorithm
// Trigger zone — detects player entry and teleports

FUNCTION Teleport(destination : Vector3)
  player.position ← destination
ENDFUNCTION

PROCEDURE OnTriggerEnter(other : Collider)
  IF other.tag = "Player" THEN
    CALL Teleport(exitPoint.position)
  ENDIF
ENDPROCEDURE
Reading the pseudocode Notice the structure: OnTriggerEnter is a PROCEDURE — it does not return a value, it just runs. Teleport is a FUNCTION in NSW pseudocode terms, though in this case it also returns nothing (in C# it will be void). The key line is the IF check — without it, any physics object entering the trigger would teleport.

STEP 4 Add OnTriggerEnter — the event-driven callback
✦ NEW IN STEP 4 — OnTriggerEnter: Unity calls this, you don't
using UnityEngine;

public class PortalController : MonoBehaviour
{
    [SerializeField] private Transform exitPoint;
    [SerializeField] private bool portalActive = true;

    void Start()
    {

    }

    void Update()
    {

    }

    private void Teleport(Transform player)
    {
        player.position = exitPoint.position;
    }

    private void OnTriggerEnter(Collider other)                // ── NEW IN STEP 4 ──
    {                                                          // ── NEW IN STEP 4 ──
        if (!portalActive) return;                             // ── NEW IN STEP 4 ──
        if (other.CompareTag("Player"))                        // ── NEW IN STEP 4 ──
        {                                                      // ── NEW IN STEP 4 ──
            portalActive = false;                              // ── NEW IN STEP 4 ──
            Teleport(other.transform);                         // ── NEW IN STEP 4 ──
        }                                                      // ── NEW IN STEP 4 ──
    }                                                          // ── NEW IN STEP 4 ──
}
OnTriggerEnter(Collider other) — Unity calls this automatically whenever another collider enters this trigger. You never call it yourself. The first line if (!portalActive) return; exits immediately if the portal has already fired — the ! means "not". CompareTag("Player") is safer than other.tag == "Player": it's faster (no string allocation) and throws a useful error in the console if the "Player" tag hasn't been created in Unity's Tag Manager. In the Inspector, drag an empty GameObject (your exit point) into the Exit Point slot.
Press Play. Walk the player capsule into the portal cube. The player should snap to the exit point's position. If nothing happens: check the Player tag is set, the Box Collider has Is Trigger ticked, and the Exit Point Transform is assigned in the Inspector.

Algorithm 2: Load a New Scene on Trigger Entry

The second portal loads an entirely different Unity scene — for example, a level transition. The algorithm is simpler than the teleport, but requires a new tool: SceneManager.

NSW Pseudocode — Scene Loader algorithm
// Scene loader — load new scene on trigger entry
PROCEDURE OnTriggerEnter(other : Collider)
  IF other.tag = "Player" THEN
    CALL SceneManager.LoadScene(sceneName)
  ENDIF
ENDPROCEDURE
New concept: namespaces SceneManager lives in UnityEngine.SceneManagement — a separate namespace from the default UnityEngine. Without the correct using statement at the top of your file, C# will not know what SceneManager means and will refuse to compile. This is Bug 1 in the Bug Detective section.
Producing & Implementing

Code Walkthrough: Scene Loader Version

Setup: In Unity, go to File → Build Settings → Add Open Scenes. Add both Scene 1 (your main scene) and Scene 2 (a second scene you create — it can be an empty room). Scenes must be in the Build Settings list for SceneManager.LoadScene() to find them.

✦ NEW IN STEP 1 — the using statement that unlocks SceneManager
using UnityEngine;
using UnityEngine.SceneManagement;   // ── NEW IN STEP 1 ──

public class PortalController : MonoBehaviour
{
    [SerializeField] private Transform exitPoint;
    [SerializeField] private bool portalActive = true;

    void Start()
    {

    }

    void Update()
    {

    }

    private void Teleport(Transform player)
    {
        player.position = exitPoint.position;
    }

    private void OnTriggerEnter(Collider other)
    {
        if (!portalActive) return;
        if (other.CompareTag("Player"))
        {
            portalActive = false;
            Teleport(other.transform);
        }

}
One new line at the top unlocks the entire SceneManagement namespace. Without it, C# doesn't know SceneManager exists. This is where the syntax bug in the Bug Detective comes from — the code looks correct but won't compile because the namespace is missing.

The Teleport() method remains in the file — you can use one portal for in-scene teleports and another for scene loading by commenting out whichever call you don't need.

Bonus: Making the Portal Breathe

A portal that just sits still looks inactive. One line of code in Update() makes the cube gently scale up and down — a breathing pulse that signals "I'm interactive" to the player.

NSW Pseudocode — Portal pulse
BEGIN PulsePortal
  WHILE gameIsRunning DO
    pulseScale ← 1 + (Mathf.Sin(Time.time) × 0.1)
    portal.localScale ← Vector3.one × pulseScale
  ENDWHILE
END PulsePortal
FINAL Complete PortalController.cs — with pulse animation
✦ NEW IN FINAL — pulse animation added to Update()
using UnityEngine;
using UnityEngine.SceneManagement;

public class PortalController : MonoBehaviour
{
    [SerializeField] private Transform exitPoint;
    [SerializeField] private string sceneName = "Scene2";
    [SerializeField] private bool portalActive = true;

    void Start()
    {

    }

    void Update()
    {
        // Pulse: breathe gently between 90% and 110% of base scale ── NEW IN FINAL ──
        float pulse = 1f + Mathf.Sin(Time.time) * 0.1f;              // ── NEW IN FINAL ──
        transform.localScale = Vector3.one * pulse;                   // ── NEW IN FINAL ──
    }

    private void Teleport(Transform player)
    {
        player.position = exitPoint.position;
    }

    private void OnTriggerEnter(Collider other)
    {
        if (!portalActive) return;
        if (other.CompareTag("Player"))
        {
            portalActive = false;
            SceneManager.LoadScene(sceneName);
            // To teleport instead of loading a scene, replace the line above with:
            // Teleport(other.transform);
        }
    }
}
Mathf.Sin(Time.time) returns a value that oscillates between –1 and 1 as time passes — a smooth wave. Multiplying by 0.1f makes the swing small (10%). Adding 1f centres the range at 1, so the scale moves between 0.9 and 1.1. Vector3.one is shorthand for new Vector3(1, 1, 1) — multiplying it by pulse scales all three axes equally. The portal cube breathes continuously without an Animator component or animation clip.

🔍 Bug 2 — Logical Error

PortalController.cs — teleport fires every frame

This script compiles and runs, but the player teleports continuously and can never stand near the portal without being thrown to the exit point over and over. What is wrong?

using UnityEngine;

public class PortalController : MonoBehaviour
{
    [SerializeField] private Transform exitPoint;
    [SerializeField] private bool portalActive = true;

    void Start() { }

    void Update()                                  // ← teleport logic is in the wrong method
    {
        if (portalActive)                          // ← wrong method
        {                                          // ← wrong method
            portalActive = false;                  // ← wrong method
            transform.position = exitPoint.position; // ← wrong method
        }                                          // ← wrong method
    }

    private void OnTriggerEnter(Collider other) { }
}
Bug found: The teleport logic is inside Update(), which runs every frame. It fires immediately when the scene loads — not when the player enters the trigger. The fix: move the teleport logic into OnTriggerEnter(Collider other) with a CompareTag("Player") check. Update() should be empty (or contain only the pulse animation).
🔍 Bug 3 — Run-time Error

PortalController.cs — portal fires for all objects

This script compiles and runs. The portal works correctly when the player walks in — but it also fires when a coin rolls into the trigger zone, or when any physics prop touches the portal. The result is coins disappearing and random objects being teleported. What is missing?
