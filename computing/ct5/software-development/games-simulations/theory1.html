Understanding Games & Simulations

Before you build, you analyse. This page covers how games and simulations have evolved, how they are structured, what makes them engaging, and the social and ethical responsibilities that come with creating them.

Identifying & Defining

Key Vocabulary

These terms appear throughout this focus area. Read them before continuing — they are the language of games and simulations.

Word Bank — Games & Simulations

algorithm
A precise, step-by-step set of instructions a computer follows to complete a task — the logic behind every game mechanic.
game loop
The repeating cycle at the heart of every game: process input → update game state → render output. Runs tens of times per second.
event-driven
Code that runs in response to an event (key press, mouse click, timer) rather than in a fixed order from top to bottom.
simulation
A software model of a real-world system, used to predict outcomes, train people, or test ideas without real-world risk or cost.
iteration
Repeating a block of code a set number of times or while a condition is true — used in game loops, physics updates, and AI.
branching
Code that takes different paths depending on a condition (IF/ELSE). Game AI, collision checks, and menu choices all use branching.
non-functional requirements
Qualities a system must have beyond what it does — e.g. age suitability, performance, accessibility, and visual appeal.
object-oriented
A programming style where code is organised into objects — e.g. a Player object with data (health, position) and behaviours (move, jump).
mechanics
The rules and systems that define how a player interacts with a game — scoring, movement, winning conditions, and resource management.
aesthetics
The sensory experience of a game — visuals, sound, feel, and the emotions it creates in the player.
usability
How easy a game or simulation is to learn and use — intuitive controls, clear feedback, and accessible design all improve usability.
data type
The kind of value stored in a variable — integer (whole number), float (decimal), string (text), boolean (true/false).
Identifying & Defining

How Games & Simulations Have Evolved

Games and simulations evolve in response to changing technology and society's needs. Understanding this history helps you identify what problems your own project could solve.

NESA Syllabus — Identifying & Defining Explore how the changing needs of society have influenced the development of games and simulations, including the impact of simulations and games on a range of industries.

A Brief History of Games

1950s–60s

Early Simulations & Arcade Origins

The first computer programs included simple games like Nim (1951) and Spacewar! (1962). Military and aerospace simulations ran on mainframes, training pilots and modelling battle scenarios.

1970s–80s

Arcade, Console & Home Computing

Pong (1972), Pac-Man (1980), and the Atari 2600 brought games into public spaces and homes. Limited memory forced designers to be creative within tight constraints.

1990s

3D Graphics, CD-ROM & the Internet

Hardware acceleration enabled 3D worlds (Doom, Quake, Mario 64). The internet enabled multiplayer and changed game distribution forever.

2000s

Online Gaming, Mobile & Indie Revolution

MMORPGs, mobile gaming, and engines like Unity democratised game development. Anyone with a laptop could release a game globally.

2010s–now

Cloud, VR/AR, AI & Simulation for Good

Streaming games from the cloud, VR/AR immersive experiences, AI-driven procedural content, and serious simulations for medicine, climate, architecture, and accessibility.

Simulations Solving Real Problems

Simulations aren't just games — they model reality to solve problems, train people, and test ideas safely.

Engineering

Wave tank testing

Virtual fluid simulations replace physical wave tanks, letting engineers test hull designs without expensive prototypes — improving efficiency.

Sustainability

Virtual prototyping

Testing designs in simulation before manufacture reduces material waste and energy use — a direct impact on sustainability.

Education

Strategy games & reasoning

Games requiring planning and resource management develop abstract reasoning — a recognised benefit for STEM education.

Accessibility

VR sign language

VR environments that incorporate sign language interpretation make training and entertainment accessible to Deaf players and learners.

Training

Hazardous environments

Simulated environments train workers for situations too dangerous, expensive, or remote to practise in person.

Privacy

Data exposure

Modern games collect vast amounts of player data. Privacy simulations can reveal how personal data moves without user knowledge.

Identifying & Defining

How Games Are Structured

Every game shares the same fundamental structural building blocks. Understanding these lets you analyse any game and plan your own.

The Core Loop

At its heart, every game runs a continuous game loop. The computer executes this loop tens or hundreds of times per second. Each cycle is called a frame.

🎮
1. Input
Detect what the player did — key press, mouse move, controller button, touch
⚙️
2. Update
Apply game logic — move objects, check collisions, run AI, update score and state
🖥️
3. Render
Draw the current state to screen — sprites, models, UI, particle effects

This Input → Update → Render loop maps directly to Unity's Update() method, which you'll use throughout your project.

Logic Building Blocks

NESA Syllabus — Identifying & Defining Investigate the representation of logic when designing games and simulations, including sequences, branching and iteration, including logical and relational operators. Describe inputs, storage, transmission, processes and outputs in games and/or simulations.

All game logic is built from three programming constructs:

  • →
    Sequence — instructions that run one after another in order. Loading a scene is a sequence: load assets → initialise player → start game loop.
  • ⑂
    Branching (Selection) — code that takes different paths. IF health ≤ 0 THEN end game. Collision detection, AI decision trees, and menu navigation all use branching.
  • ↺
    Iteration (Loops) — code that repeats. The game loop itself is an infinite loop. Checking every enemy for collision uses a FOR loop. Spawning waves uses a WHILE loop.

Game Types & Genres

Genre shapes the mechanics, aesthetics, and audience. Knowing the genre helps you analyse design decisions.

👾
Action / Arcade
Fast reflexes, simple rules. Physics and collision detection are key. e.g. Space Invaders, Breakout
🧩
Puzzle
Logic and pattern recognition. State management and undo systems matter. e.g. Tetris, Portal
⚔️
RPG
Character stats, inventory, branching narrative. Complex data structures. e.g. Pokémon, Minecraft
🌍
Strategy / Simulation
Resource management and AI opponents. System modelling mirrors real simulations. e.g. Cities: Skylines
🏎️
Sports / Racing
Physics engines, timing, leaderboards. Real-time precision input. e.g. Mario Kart
🎲
Serious Simulation
Models real-world systems for training, research, or accessibility. e.g. flight sims, surgical trainers

Design Principles in Game Design

NESA Syllabus — Researching & Planning Explore design principles and issues relevant to game design · Evaluate an existing game or simulation in terms of its mechanics, aesthetics and usability · Generate alternative designs and evaluate them against the requirements to select a preferred design.

A positive feedback loop rewards success with more power — getting coins makes you faster, which helps you get more coins. These create momentum but can snowball into the game becoming too easy.

A negative feedback loop helps struggling players catch up — a racing game that gives slower players a speed boost. These improve fairness and keep multiple players competitive.

Design question: What feedback loops exist in the game you analysed? Are they positive, negative, or both?

Too easy → boredom. Too hard → frustration. Great games keep players in a flow state where challenge matches their growing skill level.

Techniques include: gradual introduction of new mechanics, checkpoints that reduce the cost of failure, and procedurally generated difficulty (e.g. Tetris speeds up as you clear more lines).

Intrinsic motivation (playing because it's fun, satisfying, or curious) is stronger and more lasting than extrinsic motivation (playing to earn a badge or reward).

Self-Determination Theory identifies three needs games can satisfy: competence (getting better at something), autonomy (making meaningful choices), and relatedness (playing with or against others).

NESA requires you to consider games from the perspectives of diverse groups — including Aboriginal and Torres Strait Islander Peoples, culturally and linguistically diverse communities, people of different ages and genders, and people with disability.

  • Cultural sensitivity: seek permissions before using cultural sites or symbols
  • Representation: reflect diverse characters and stories authentically
  • Accessibility: colourblind modes, subtitle options, remappable controls
  • Use of augmentative and alternative communication (AAC) in game interfaces

Augmented Reality (AR) overlays digital content on the real world (Pokémon GO). Virtual Reality (VR) immerses the player in a fully digital environment. Mixed Reality (MR) blends real and virtual so they interact.

These technologies create deeply immersive experiences, but raise questions about motion sickness, social isolation, and the blurring of real and simulated environments.

Predefined algorithms and AI are increasingly used to control characters and environments in VR/AR — from physics-based AI opponents to procedurally generated worlds.

Identifying & Defining

Social, Ethical & Legal Responsibilities

Creating a game or simulation is not ethically neutral. As a developer, you have responsibilities to your players, your community, and the law.

NESA Syllabus Consider the social impacts and ethical and legal responsibilities in games or simulations · Investigate data collection and interpretation adhering to privacy and cybersecurity principles, including specify what data is collected, who owns it, and how it will be protected · Explore games and simulations considering the perspectives of diverse groups.

Age Classification

Australia's Classification system (G, PG, M, MA15+, R18+) is a legal requirement for commercial games. As a designer, your non-functional requirements must specify age suitability, and your design choices must reflect it.

Data & Privacy

Even a simple game can collect play time, in-game purchases, location, and device identifiers. Under Australia's Privacy Act, you must specify what is collected, who owns it, and how it's secured — especially for games played by minors.

Cybersecurity

Online games are targets for account theft, cheating, and harassment. As a developer you apply cybersecurity principles during design — not as an afterthought. This includes secure login, anti-cheat systems, and reporting mechanisms.

Cultural Respect

Games that use cultural symbols, religious imagery, or traditional stories require permission and sensitivity. NESA specifically names Aboriginal and Torres Strait Islander Peoples. Misrepresentation causes real harm.

Environmental Impact

Game development and play consume significant computing power and energy. Simulations can model environmental impact before decisions are made — virtual testing before manufacturing avoids physical waste.

Intellectual Property

Game assets — art, music, code, character designs — are protected by copyright. Using someone else's work without a licence is illegal. Open licences (Creative Commons, MIT) allow reuse with conditions you must follow.

🤔 Ethical Design Challenge

Loot boxes and addictive design

Many commercial games use loot boxes (random in-game rewards purchased with real money) and dark patterns (design that manipulates players into spending more time or money than intended). Some countries have classified loot boxes as gambling and banned them.

For your notebook: Is it ethical for a game aimed at children to use loot boxes? Where is the line between engaging game design and manipulation? What responsibilities do you have as a developer?

🔒 Cybersecurity Scenario

Privacy in multiplayer games

In 2020, a popular online game exposed the usernames, play histories, and email addresses of millions of players due to an unsecured API endpoint. The developer had not considered what data was collected, who could access it, or how it would be protected.

Applying cybersecurity principles during the design phase — before a single line of code is written — is far more effective than adding security after the fact.
