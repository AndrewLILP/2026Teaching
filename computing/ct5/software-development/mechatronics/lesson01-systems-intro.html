What Is a Mechatronic System?

Identifying & Defining CT5-DPM-01 CT5-THI-01

Week 1, Lesson 1 of Designing Mechatronic Solutions. Today's question: what actually makes something a "mechatronic" system, and how do we describe one using the same language an engineer would?

Defining mechatronic systems

A mechatronic system combines mechanical parts, electronics, and code so that a device can sense its environment and respond to it. A robotic system usually adds movement and manipulation. An automated system carries out a task with little or no ongoing human input once it's running.

Teacher tip

Ask students to name a device at home that "notices something and does something about it" (e.g. a motion-sensor light). That's the simplest possible mechatronic system.

✎ "Mechatronic" = mechanical + electronic + code, working together.

Word bank

Sensor
A component that detects a condition (light, distance, sound, touch) and turns it into a signal.
Actuator
A component that creates movement or a physical effect (motor, servo, solenoid).
End effector
The "hand" of a system — the part that directly interacts with the task (gripper, wheel, nozzle).
Microcontroller
A small computer that reads sensors and controls actuators by running code.
Autonomous
Operating on its own, making decisions without a human directly controlling each action.
Human-controlled
A person directly drives the system's actions in real time (e.g. a remote-controlled rover).

The IPO model: tracing a system end-to-end

Every mechatronic or automated system can be described using five stages. Learning to spot these five stages is the single most useful skill for this unit — you'll use it on every system you investigate, including your own project later in the term.

Input

Something is sensed or entered.

→
Storage

A value is held for later use.

→
Transmission

A signal travels somewhere.

→
Process

A decision is made.

→
Output

Something happens in the real world.

Worked example — automatic hand sanitiser dispenser
Input

Infrared sensor detects a hand nearby.

→
Storage

A short timer value, so it can't trigger twice in a row.

→
Transmission

Sensor signal travels to the microcontroller.

→
Process

"Is a hand close enough, and has enough time passed?"

→
Output

Pump motor runs for a fixed time.

Human-controlled vs autonomous control

Not every mechatronic system makes its own decisions. Some are directly controlled by a person the whole time; others are autonomous and only need a human to set them up or supervise. Some systems sit in between — semi-autonomous — switching between the two.

Classify each example as human-controlled, autonomous, or semi-autonomous — and be ready to justify the tricky ones: a forklift driven by an operator · a robot vacuum that maps and cleans a room on its own · a drone flown by a hobbyist with a remote · a self-checkout scale that detects an unscanned item · a prosthetic hand that responds to muscle signals · a factory arm welding the same joint on every car that passes · a delivery drone that flies itself to your street, then hands control to a person to land precisely on the porch.

Mini investigation — spot the system

Over the next two weeks you'll meet three kinds of real-world problem that mechatronic and automated systems can solve — a wellbeing device, a social-issue data-capture system, and an autonomous response system. Today, just practise noticing how systems work — you won't need to commit to a project topic yet.

Identifying & defining task

Choose one real-world system and break it down

Pick a real device that fits one of the three problem-types above.

  1. Name the system and the real-world problem it solves.
  2. Describe its inputs, process, and outputs (storage and transmission if you can identify them).
  3. Decide: is it human-controlled, autonomous, or semi-autonomous? Justify your answer.
  4. Note one social, ethical, or legal consideration this system raises.
