Shared Algorithms

Good algorithms transfer across platforms. The patterns on this page — sensor-decision-action loops, state machines, modular function design, desk checking, and test tables — apply equally to VEX and Micromelon. Master them here and they carry into every project you build.

Researching & Planning

Algorithm patterns — the reusable toolkit

Most robot programs are combinations of a small number of core patterns. Recognising which pattern to reach for — and knowing how to adapt it — is what separates engineers who solve problems from engineers who google until something works.

🔄
Pattern 01

Sensor-Decision-Action loop

The foundation of all autonomous robot control. Read sensor → make decision → command actuator → repeat. Every sprint in this unit is built on this loop.

🔀
Pattern 02

State machine

The robot has a finite set of behaviours (states). Sensor conditions trigger transitions between states. Only one state is active at a time, making complex behaviour manageable.

📦
Pattern 03

Modular functions

Each distinct behaviour is a named function. The main loop calls functions — it doesn't contain logic directly. Functions can be tested independently and reused across sprints.

📋
Pattern 04

Named constants

Threshold values, speeds, and durations are named constants at the top of the file. Change a value once and it updates everywhere. Never scatter raw numbers through code.

🔍
Pattern 05

Desk checking

Trace pseudocode manually with sample values before writing code. Catch logic errors when they cost nothing to fix — not after the robot has driven into a wall.

🧪
Pattern 06

Systematic testing

Test cases based on requirements. Every functional requirement generates at least one test. Include boundary values (exactly at threshold), not just comfortable middle values.

Producing & Implementing

Pattern 01 — Sensor-Decision-Action loop

Every autonomous robot repeats the same cycle: sense, decide, act. The only thing that changes between programs is what you're sensing, what decisions you make, and what actions follow. Understanding this pattern lets you design any autonomous behaviour by filling in the blanks.

NSW Pseudocode — Sensor-Decision-Action loop (general form)
BEGIN setup
  ← Initialise variables and constants
END setup

WHILE True DO           ← run forever

  ← ① SENSE: read all sensors needed this iteration
  sensorValue ← sensor.read()

  ← ② DECIDE: use sensor data to determine what to do
  IF sensorValue < THRESHOLD THEN
    action ← "avoid"
  ELSE
    action ← "proceed"
  ENDIF

  ← ③ ACT: command actuators based on decision
  IF action = "avoid" THEN
    avoidObstacle()
  ELSE
    driveForward(SPEED)
  ENDIF

  wait(LOOP_DELAY)      ← always include a loop delay
ENDWHILE
NSW Pseudocode Castle Crasher — SDA pattern applied 
CONSTANT THRESHOLD ← 30    ← cm
CONSTANT SPEED     ← 50    ← %

WHILE True DO
  distance ← sensor.readDistance()   ← ① SENSE

  IF distance < THRESHOLD THEN       ← ② DECIDE
    action ← "avoid"
  ELSE
    action ← "proceed"
  ENDIF

  IF action = "avoid" THEN            ← ③ ACT
    avoidObstacle()
  ELSE
    driveForward(SPEED)
  ENDIF

  wait(20ms)
ENDWHILE
The two-step decision (DECIDE then ACT) can be collapsed into one IF block for simple cases. Keeping them separate is good practice for complex programs — it makes the decision logic readable independently of the action logic.
VEX VR Python Castle Crasher — SDA pattern 
THRESHOLD = 30   # cm
SPEED     = 50   # percent

def avoid_obstacle():
    drivetrain.stop()
    wait(200, MSEC)
    drivetrain.turn(RIGHT)
    wait(0.6, SECONDS)
    drivetrain.stop()

drivetrain.set_drive_velocity(SPEED, PERCENT)

while True:
    # ① SENSE
    distance = distance_sensor.get_distance(MM) / 10

    # ② DECIDE + ③ ACT (combined for simplicity)
    if distance < THRESHOLD:
        avoid_obstacle()
    else:
        drivetrain.drive(FORWARD)

    wait(20, MSEC)
VEX note: Distance returns in mm — divide by 10 to match your cm threshold. drivetrain.drive() is non-blocking; the loop delay handles timing. Set velocity once before the loop, not inside it.
Micromelon Python Colour-based SDA pattern 
from micromelon import *

COLLECT = "red"     # tile colour to act on
AVOID   = "black"   # boundary colour

def collect_debris():
    sleep(1000)
    rover.setLED("red")

while True:
    # ① SENSE
    colour = rover.getColour()

    # ② DECIDE + ③ ACT
    if colour == COLLECT:
        collect_debris()         # act: collect
    elif colour == AVOID:
        rover.right(90)          # act: turn away
    else:
        rover.forward(1)         # act: keep moving
Micromelon note: rover.forward(1) is blocking — it completes before the loop repeats. No separate wait() needed after movement. Colour strings are lowercase — always compare with "red", not "Red".

The basic SDA loop handles one sensor and one decision. Real systems extend this in predictable ways:

  • Multiple sensors: Read all sensors at the top of the loop, then combine their values in the decision step — e.g. IF frontDistance < 30 AND leftDistance < 20 THEN emergencyStop()
  • Priority ordering: When multiple conditions can be true simultaneously, the order of IF/ELIF determines which takes priority. Always put emergency conditions first.
  • Hysteresis: Use different thresholds for entering and leaving a state to prevent the robot from rapidly switching back and forth (e.g. enter avoid at 25cm, resume driving at 40cm).
  • Sensor fusion: Combine multiple readings to make a better decision — e.g. average the last three distance readings before deciding, to filter out sensor noise.
NSW Pseudocode — hysteresis example
CONSTANT ENTER_AVOID ← 25    ← start avoiding closer than 25cm
CONSTANT EXIT_AVOID  ← 40    ← stop avoiding when clear beyond 40cm
avoiding ← FALSE

WHILE True DO
  distance ← sensor.readDistance()

  IF avoiding = FALSE AND distance < ENTER_AVOID THEN
    avoiding ← TRUE
    stopMotors()
  ELSE IF avoiding = TRUE AND distance > EXIT_AVOID THEN
    avoiding ← FALSE
  ENDIF

  IF avoiding THEN
    turnRight(speed: 50)
  ELSE
    driveForward(speed: 50)
  ENDIF

  wait(20ms)
ENDWHILE

Pattern 02 — State machines

A state machine gives your robot a memory — it knows not just what it's sensing right now, but what it was doing before. This lets you model complex multi-step behaviours that a simple IF/ELSE chain can't handle cleanly.

Three rules of state machines: (1) The robot is always in exactly one state. (2) Only one state's logic runs each loop iteration. (3) State transitions happen when a condition is met — the robot doesn't transition partway through an action.
  1. List all distinct behaviours your robot needs. Each major behaviour is a candidate state.
  2. Identify the trigger for entering each state — what sensor condition or event causes the robot to transition into it?
  3. Identify the exit condition — what causes the robot to leave each state and transition to the next?
  4. Draw the state diagram — circles for states, arrows for transitions, label each arrow with its trigger condition.
  5. Write pseudocode — one branch per state in the main IF/ELIF chain, one function per state's logic.
  6. Desk check — trace a realistic scenario through the state machine. Does every state have a way out? Can the robot reach every state?
NSW Pseudocode — generalised state machine template
state ← "initial_state"

FUNCTION stateA()
  ← behaviours for state A
  IF exitConditionA THEN
    state ← "stateB"
  ENDIF
ENDFUNCTION

FUNCTION stateB()
  ← behaviours for state B
  IF exitConditionB THEN
    state ← "stateA"
  ENDIF
ENDFUNCTION

BEGIN
  WHILE True DO
    ← Read all sensors first
    sensorValue ← sensor.read()

    ← Route to the active state
    IF state = "stateA" THEN
      stateA()
    ELSE IF state = "stateB" THEN
      stateB()
    ENDIF

    wait(20ms)
  ENDWHILE
END
NSW Pseudocode Three-state patrol robot 
← Robot patrols, collects objects when close, returns to base

CONSTANT COLLECT_RANGE ← 15   ← cm — collect if object this close
CONSTANT BASE_COLOUR   ← "white"

state ← "patrolling"

WHILE True DO
  distance ← sensor.readDistance()
  colour   ← sensor.readColour()

  IF state = "patrolling" THEN
    IF distance < COLLECT_RANGE THEN
      state ← "collecting"
    ELSE IF colour = "black" THEN
      turnRight(speed: 50, duration: 0.5)
    ELSE
      driveForward(speed: 50)
    ENDIF

  ELSE IF state = "collecting" THEN
    collectObject()
    state ← "returning"

  ELSE IF state = "returning" THEN
    IF colour = BASE_COLOUR THEN
      depositObject()
      state ← "patrolling"
    ELSE
      navigateToBase()
    ENDIF
  ENDIF

  wait(20ms)
ENDWHILE
This three-state machine handles patrol, collect, and return. Notice: sensor reads happen outside the state branches — every state has access to current sensor data. State transitions always lead to a concrete next state — there is no undefined transition.
VEX VR Python Three-state patrol robot

BASE_COLOUR   = Color.WHITE

state = "patrolling"

    drivetrain.stop()
    wait(1, SECONDS)

def deposit_object():
    wait(0.5, SECONDS)

    if eye_sensor.get_color() == Color.BLACK:

        wait(0.5, SECONDS)

Micromelon Python Three-state patrol robot 
from micromelon import *

COLLECT_RANGE = 15    # cm
BASE_COLOUR   = "white"

state = "patrolling"

def collect_object():
    sleep(1000)
    rover.setLED("green")

def deposit_object():
    sleep(500)
    rover.setLED("off")

def navigate_to_base():
    colour = rover.getColour()
    if colour == "black":
        rover.right(90)
    else:
        rover.forward(1)

while True:
    distance = rover.getDistance()
    colour   = rover.getColour()

    if state == "patrolling":
        if distance < COLLECT_RANGE:
            state = "collecting"
        elif colour == "black":
            rover.right(90)
        else:
            rover.forward(1)

    elif state == "collecting":
        collect_object()
        state = "returning"

    elif state == "returning":
        if colour == BASE_COLOUR:
            deposit_object()
            state = "patrolling"
        else:
            navigate_to_base()
Micromelon note: No wait() at the end of the loop — rover.forward(1) and rover.right(90) are blocking, so the loop naturally paces itself. If your state machine doesn't move the rover in a given iteration (e.g. during a state transition), add a small sleep(50) to prevent a busy loop.

Pattern 03 — Modular function design

A well-designed robot program looks like a table of contents at the top level — the main loop reads like plain English, and the implementation details live inside clearly named functions. This makes programs easier to read, test, debug, and reuse across sprints.

Main loop
while True
Coordinates everything — no logic, only calls
↙
↓
↘
Function
get_sensor_data()
All sensor reads in one place
Function
avoid_obstacle()
Complete avoidance behaviour
Function
collect_debris()
Complete collection behaviour
  • One job per function: avoid_obstacle() avoids an obstacle. It does not also read sensors or update state. If a function does two unrelated things, split it.
  • Name the job, not the mechanism: turn_right_600ms() tells you how. avoid_obstacle() tells you why. Use the why name.
  • Parameters for variation: If you need to avoid at different speeds, pass speed as a parameter — avoid_obstacle(speed) — rather than hardcoding it inside.
  • Return values for data: Functions that calculate or read data should return their result — get_distance_cm() returns a number. Functions that command actuators typically return nothing.
  • Write a docstring: One sentence describing what the function does and what it returns. Future you will thank present you.
  • Test functions in isolation: Before wiring functions into the state machine, test each one independently. A function that works alone is much easier to debug than a function buried in a 200-line program.
Before modularisation (hard to test): 
while True:
    d = distance_sensor.get_distance(MM) / 10
    if d < 30:
        drivetrain.stop()
        wait(200, MSEC)
        drivetrain.turn(RIGHT)
        wait(0.6, SECONDS)
        drivetrain.stop()
    else:
        drivetrain.drive(FORWARD)
    wait(20, MSEC)
After modularisation (easy to test): 
def get_distance_cm():
    return distance_sensor.get_distance(MM) / 10

def avoid_obstacle():
    """Stop, pause, and turn right to clear detected obstacle."""
    drivetrain.stop()
    wait(200, MSEC)
    drivetrain.turn(RIGHT)
    wait(0.6, SECONDS)
    drivetrain.stop()

while True:
    if get_distance_cm() < STOP_DISTANCE:
        avoid_obstacle()
    else:
        drivetrain.drive(FORWARD)
    wait(LOOP_DELAY, MSEC)
Testing & Evaluating

Pattern 05 — Desk checking

Desk checking is the practice of manually tracing through pseudocode or code step by step, tracking the value of each variable as you go. It catches logic errors before you spend time typing code that doesn't work. It is also a required assessment skill in NSW Computing Technology.

The desk checking process

Follow these six steps every time you desk check an algorithm:

Set up a variable tracking table

Create a table with one column per variable, plus columns for the current line and any output. Leave a row for each step. This is your trace table.

Choose test input values

Select values that exercise different paths through the code. Include: a typical normal value, a boundary value (exactly at a threshold), a value just above the boundary, and an edge case (zero, negative, empty). Don't just use comfortable middle values.

Execute one line at a time

Work through the code line by line. At each line, update the variable values in your table. Never skip lines — every assignment, condition, and function call changes something.

Evaluate conditions explicitly

When you reach an IF condition, substitute current variable values and write out whether it evaluates to TRUE or FALSE — don't assume. IF distance < 30 with distance = 30 is FALSE, not TRUE.

Record every output and state change

Note any output produced (print, motor command, LED change) and any state variable that changes. The trace table captures what the robot does, not just what variables hold.

Compare expected to predicted behaviour

After tracing, check: does the output match what the algorithm is supposed to do? If not, find the line where the trace diverges from expectation — that's where the bug is.

Pattern 06 — Systematic testing with test tables

A test table links each functional requirement to at least one test case. It records the input condition, the expected behaviour, the actual behaviour, and whether the test passed or failed. Test tables are evidence that you validated your system — not just that you ran it once and it worked.

  • Normal / typical: Values in the expected range — the most common scenario. If your obstacle threshold is 30cm, test with distance = 50 (clear) and distance = 15 (obstacle).
  • Boundary / edge: Values at the exact threshold. If the condition is distance < 30, test with distance = 29, 30, and 31. These three catch most off-by-one errors.
  • Extreme: Values at the sensor's limits — distance = 0 (touching the wall), distance = 300 (maximum range). Does the algorithm fail gracefully?
  • Invalid / unexpected: What if the sensor returns -1 (no reading)? What if two contradictory conditions are true simultaneously? Good testing anticipates failure modes.

Tiered tasks

These tasks apply the patterns from this page to your sprint programs.

Foundation

Complete the trace table

Complete rows D and E of the hysteresis desk check trace table above. Then create a test table for the Micromelon Coral Reef Clean-Up program with at least five test cases — one normal, two boundary (red tile detected at different distances), one extreme (boundary tile immediately ahead at start), and one system test (complete run).

Core

Design a new algorithm from scratch

Design an algorithm for a new scenario: a robot that must patrol a rectangular room, detect when an object enters the room (distance drops below 50cm), and trigger an alarm (flash LED red five times) before resuming patrol. Write full NSW pseudocode, identify the states, draw a state diagram, desk check with two scenarios, then implement on either platform.

Extension

Refactor and evaluate

Take your current Castle Crasher or Coral Reef program and refactor it using all six patterns: (1) identify and name the SDA loop, (2) convert to a formal state machine if not already, (3) ensure every behaviour is a named function, (4) replace all magic numbers with named constants, (5) desk check two scenarios with a trace table, (6) create a complete test table with boundary and extreme cases. Document every refactoring decision in your project notebook.
