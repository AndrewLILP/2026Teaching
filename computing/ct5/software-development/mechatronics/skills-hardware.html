Hardware Concepts

Before writing a single line of code, you need to understand the hardware your program will control. This page maps the physical components of a mechatronic system — sensors, actuators, motors, and microcontrollers — and compares the two platforms you'll be working with: VEX and Micromelon.

Identifying & Defining

How a robot thinks

Every mechatronic system — no matter how complex — follows the same pattern. Something in the environment changes. A sensor detects it. The microcontroller reads that data, runs your algorithm, and sends a command to a motor or output device. Understanding this signal path is the foundation of everything else in this unit.

Trace it on your robot: Pick one behaviour your robot performs (e.g. stopping before a wall). Identify each stage of the signal flow for that behaviour: what triggers it, which sensor detects it, what the brain decides, which motor responds, and what physically happens. Write this out before you touch any code.

Mechatronic systems sit on a spectrum from fully remote-controlled to fully autonomous:

  • Human-controlled: A joystick or controller sends commands directly — the human is the decision-maker. The robot has no program logic of its own.
  • Semi-autonomous: The robot handles some decisions itself (e.g. stopping before a wall) while the human handles others (e.g. direction and speed).
  • Fully autonomous: The robot reads sensors, makes all decisions, and acts without any human input during operation. Your algorithm is the only decision-maker.
In your sprints: Sprint 0 (Treasure Hunt) used mostly human-controlled driving with simple sensor responses. Sprint 1 (Castle Crasher) and beyond push toward autonomous behaviour — the robot must detect and respond to its environment without you steering it.
Researching & Planning

Sensors

Sensors are the robot's way of experiencing the world. Without sensors, your robot is blind — it can't react to anything in its environment. Choosing the right sensor for a task is an engineering decision with real trade-offs.

Distance / Sonar sensor Both

Emits ultrasonic pulses and measures the time for the echo to return. Converts time to a distance in centimetres. Essential for obstacle detection and wall-following.

Typical range: 3 – 300 cm Output: distance in cm (integer) Use case: Stop before wall, navigate maze
Colour / Vision sensor VEX

Detects the colour and brightness of surfaces or objects in front of it. In VEX, the Vision Sensor can identify and track coloured objects. Essential for sorting challenges like Coral Reef Clean-Up.

Output: hue, saturation, brightness values Use case: Identify objects, line detection
Colour / Line sensor Micromelon

The Micromelon Rover has a downward-facing colour sensor for detecting coloured tiles and tracking lines. Also used for identifying objects placed directly under the rover.

Output: colour name or RGB value Use case: Line following, colour sorting
Bumper / Touch sensor VEX

A simple digital switch — pressed (1) or not pressed (0). Useful for collision detection and physical triggers. The VEX Bumper Switch detects direct contact with an object.

Output: Boolean (True / False) Use case: Detect collisions, trigger events
Gyroscope / IMU Both

Measures rotation and orientation. Allows precise turning to a specific angle rather than guessing by timing. Inertial Measurement Units (IMUs) also measure acceleration.

Output: heading in degrees (float) Use case: Precise turns, orientation tracking
Motor encoder Both

Built into smart motors — counts rotations to report exact position and speed. Allows the robot to move a specific distance by counting wheel rotations instead of timing.

Output: rotation count (integer) Use case: Move exact distances, speed control
Selecting sensors for your project: Don't just use every sensor available. Identify what information your algorithm needs to make each decision, then select the minimum set of sensors that provides that information. Over-sensing creates complexity and data you don't know how to use.

Real sensors don't always return exactly what you expect. Understanding sensor limitations is critical to writing robust code:

  • Noise: Sensor readings fluctuate slightly even when nothing is moving. An ultrasonic sensor might return 47, 49, 47, 48 when the wall hasn't moved. Your code needs to handle this.
  • Range limits: Ultrasonic sensors have a minimum distance (too close = no reading) and a maximum range. Test your sensor at boundary distances before writing your algorithm.
  • Angle sensitivity: Ultrasonic sensors work best when facing surfaces head-on. A wall at an angle reflects the pulse away — you may get no reading or a wildly incorrect one.
  • Interference: Two ultrasonic sensors running simultaneously can interfere with each other. Space them in time if you're using multiple.
Design response: Use a threshold band rather than an exact value. Instead of IF distance = 30, write IF distance < 35. This accounts for sensor noise and gives you a reliable trigger.

Actuators & motors

Actuators are how your robot affects the physical world. In your projects, motors are the primary actuators — they drive wheels, arms, and mechanisms. Understanding how to control motor speed and direction is fundamental to all three sprints.

Drive motor

DC / Smart motor

Spins continuously in either direction at a controllable speed. Paired on left and right sides to drive the robot forward, backward and turn. Speed is set as a percentage (−100 to +100).

VEX: Smart Motor (11W) · Micromelon: built-in drive motors
Position motor

Servo motor

Rotates to a specific angle and holds there. Used for arms, claws, and mechanisms that need to move to a precise position and stay. Controlled by a target angle (0°–270°).

VEX: Servo (attached via 3-wire port) · Not available on basic Micromelon
Output device

LED / Display

Not all outputs are motion. LEDs signal robot state, displays show sensor values during testing. Critical for debugging — a flashing LED can confirm your code reached a certain point.

VEX Brain screen · Micromelon: onboard LED strip
End effector

Gripper / Claw

The tool at the end of an arm that interacts directly with objects — picking, placing, pushing or sorting. Driven by a servo or motor. In Coral Reef Clean-Up, this is what collects debris.

Custom-built on VEX · Attachment accessories on Micromelon

Both platforms control drive motors using a speed value from −100 to +100:

  • Positive value (+): forward rotation
  • Negative value (−): reverse rotation
  • Zero (0): motor stops
  • Differential steering: to turn, set the two drive motors to different speeds. A tank turn (spinning on the spot) sets one motor positive and one motor negative at the same value.
Tank turn example: Left motor: +50 · Right motor: −50 → spins clockwise on the spot
Left motor: −50 · Right motor: +50 → spins anti-clockwise on the spot

Motor encoders let you control distance rather than time. Instead of "run motors for 2 seconds", you can tell the robot to move until the encoder reads a specific rotation count — far more precise and repeatable.

Physical robots have port assignments that your code must match exactly:

  • VEX V5 / EXP: Smart ports numbered 1–21. Each motor and smart sensor is assigned to a port. Your program references the port number to communicate with each device.
  • Micromelon Rover: Motors and sensors are built in — no wiring required. The API uses named references (rover.forward(), rover.distance()) rather than port numbers.
  • VEX VR Simulator: Uses the same port conventions as the physical V5 brain. Configure your virtual devices in the simulator to match the port numbers in your code.
Common mistake: Plugging a motor into port 3 but telling the code it's on port 4 — the motor does nothing and the code shows no error. Always verify port assignments before testing, both on physical robots and in the simulator's device panel.

Microcontrollers

The microcontroller is the brain of your system — the computer that runs your program, reads sensor data, and sends commands to motors. Understanding what it can and can't do shapes how you write your code.

A microcontroller is a self-contained computer on a chip. Unlike a general-purpose computer (your laptop), it's designed to run one program repeatedly and interact with the physical world in real time.

  • CPU: Executes your program instructions — comparisons, calculations, function calls.
  • Memory (RAM): Stores variables while the program runs. Very limited compared to a laptop — don't store large datasets on a microcontroller.
  • Flash storage: Where your program lives. Written once when you upload your code.
  • I/O ports: Inputs and outputs — the connections to sensors and motors. Reading a port gets you sensor data; writing to a port commands a motor.
  • Real-time execution: Your loop runs continuously, typically hundreds of times per second, reading sensors and updating motor commands.

Almost all robot programs follow the same fundamental structure: setup once, then loop forever.

  • Setup / initialise: Run once at the start — configure ports, set initial motor speeds to zero, calibrate sensors.
  • Main loop: Runs continuously — read sensors, make decisions, update motor commands, repeat. This is where your autonomous behaviour lives.
  • Wait / sleep: A small delay inside the loop (wait(20, MSEC)) prevents the CPU from running at 100% and allows sensor readings to stabilise.
Pseudocode structure of almost every robot program:

BEGIN setup
  // Initialise motors and sensors
END setup

WHILE True DO
  distance ← sensor.read()
  IF distance < 30 THEN
    // Avoid obstacle
  ELSE
    // Drive forward
  ENDIF
  wait(20ms)
ENDWHILE

VEX vs Micromelon — platform comparison

You'll work with two different platforms in this unit. They solve similar problems in different ways, and understanding those differences will make you a better engineer — you'll know which tool to reach for, and why.

Feature 🤖 VEX V5 / VEX VR 🟠 Micromelon Rover
Controller / Brain VEX V5 Brain — large touchscreen, 21 smart ports, runs VEXcode Micromelon onboard controller — compact, pre-wired, battery-powered
Programming VEXcode — Blocks or Python. VEX VR uses the same IDE in-browser. Micromelon IDE — Blocks or Python. Simulator is a downloadable desktop app.
Sensors (built-in) Gyro/IMU in brain; encoders in smart motors. Other sensors added via smart ports. Distance sensor, colour/line sensor, gyro — all built in, no wiring needed.
Sensors (add-on) Ultrasonic, Vision, Optical, Bumper, GPS, Rotation sensors via smart ports Limited add-on support — the rover is designed as a self-contained unit
Motors / Drive 11W Smart Motors — independent speed/direction, encoder feedback Built-in differential drive — controlled by named methods, not port numbers
Simulator VEX VR — browser-based, no install, rich environments including maze, field, and playground worlds Micromelon Simulator — desktop app download required; mirrors the physical rover's API exactly
Build / customise Fully customisable — build any robot structure from VEX metal components Fixed chassis — accessories and attachments for specific tasks
Best for Complex builds, competition-style challenges, multi-sensor autonomous systems Quick prototyping, colour-sensing tasks, beginner-friendly platform
API style leftMotor.spin(FORWARD, 50, PERCENT) rover.forward(50)
Why both? Each platform exposes you to a different API design philosophy. VEX is explicit — you name every device and specify every parameter. Micromelon abstracts the hardware — the rover object handles the complexity. Both approaches appear in real-world embedded systems engineering.

Using simulators

Simulators let you write and test robot programs without needing physical hardware — and in many cases, they let you test scenarios that would be dangerous or impractical on a physical robot. Both VEX VR and the Micromelon Simulator are first-class development environments, not toys.

  • Access: vr.vex.com — browser-based, no install. Sign in with your school Google account.
  • Environments: Multiple worlds available — Maze, Disk Mover, Dynamic Environment, and more. Your teacher will specify which world to use for each sprint.
  • Coding: Same VEXcode IDE as the physical V5 — Blocks or Python. Code that works in VEX VR transfers directly to a physical V5 robot.
  • Sensors in VR: The VR robot has a downward-facing eye sensor (colour detection), distance sensors (front and sides), and a gyro. These are configured via the device panel — match port numbers to your code.
  • Debugging in VR: Use the brain's screen (brain.screen.print()) to display sensor values in real time while your program runs. Essential for understanding what your sensors are actually seeing.
VR vs physical: VEX VR sensors are idealised — no noise, no angle sensitivity, no interference. This makes them easier to work with but means physical robots require more robust code. Always test physical behaviour separately.
  • Access: Desktop app — download from the Micromelon website. Install on your school device or laptop before your first Micromelon session.
  • API parity: The simulator uses exactly the same Python API as the physical rover. Code that works in the simulator runs on the physical rover without modification.
  • Environments: Grid-based worlds with coloured tiles, walls and objects. You'll use these for the Coral Reef Clean-Up sprint.
  • Debugging: Print statements appear in the IDE console. Use print(rover.distance()) to see what your sensors are reading in real time.
Connecting to a physical rover: When you're ready to test on hardware, connect the Micromelon Rover via USB or Bluetooth — the same code uploads directly. Your teacher will walk you through the connection process.
  • Sensor noise: Simulators return clean values. Physical robots return noisy ones. Code that works perfectly in a simulator may need threshold bands to work on hardware.
  • Motor slippage: Wheels slip on physical surfaces, especially when turning. An encoder-based distance command may overshoot or undershoot on carpet vs tiles.
  • Timing: Physical robots have real-world delays — communication lag, motor inertia. A wait() that works in simulation may need adjustment on hardware.
  • Environment variation: Lighting affects colour sensors on physical robots. Simulate in ideal conditions, but test in your actual classroom environment.
Best practice: Develop and test logic in the simulator. When moving to physical hardware, test each sensor and motor independently before running the full program. Never assume simulator behaviour matches physical behaviour without testing.

Compare and justify

Extension

Design a hardware specification

You are designing a robot for the Coral Reef Clean-Up challenge. Write a hardware specification document that lists: (a) all required sensors with justification, (b) all required actuators with justification, (c) any physical constraints or trade-offs (power, size, cost).
