Micromelon Skills

The Micromelon Rover is a compact, self-contained robot built for colour sensing, grid navigation, and rapid prototyping. This page covers the Micromelon simulator, the Python API, and Sprint 2 — Coral Reef Clean-Up — where your robot collects ocean debris by reading colour from the tiles beneath it.

Identifying & Defining

Getting started with Micromelon

Simulator: Download the Micromelon app from micromelon.com.au and install it on your device. The simulator is self-contained — no internet required once installed. Sign in with your school account to save your code.
  • Code editor (left panel): Write your Blocks or Python program here. Toggle the language using the Blocks/Python switch at the top. Your code is saved automatically per world.
  • Simulator view (right panel): The grid world your rover navigates. Coloured tiles represent ocean debris, obstacles, or empty ocean floor. The rover icon shows your robot's current position and heading.
  • World selector: Choose or load a world layout. For Coral Reef Clean-Up your teacher will provide a specific world file — load it before coding.
  • Run / Stop: Green play button starts your program. The rover resets to its starting tile each run.
  • Console: print() output appears below the editor — watch colour readings and state changes here as your program runs.
  • Connecting to physical rover: Once your program works in the simulator, connect the physical Micromelon Rover via USB (or Bluetooth if available) — your teacher will walk you through this step.
Key difference from VEX VR: Micromelon uses a grid-based world — the rover moves tile-by-tile rather than continuously. One rover.forward(1) call moves the rover exactly one tile forward. This makes distance management simpler: you count tiles, not centimetres.

Researching & Planning

Micromelon vs VEX — the same idea, different syntax

The programming concepts are identical across both platforms — variables, selection, iteration, functions. What changes is the API. Compare these side-by-side to see the pattern: Micromelon abstracts more, VEX is more explicit.

🤖 VEX VR — Python
# Drive forward
drivetrain.drive(FORWARD)
wait(1, SECONDS)
drivetrain.stop()

# Turn right 90°
drivetrain.turn_for(RIGHT, 90, DEGREES)

# Read distance
d = distance_sensor.get_distance(MM) / 10

# Read colour below
c = eye_sensor.get_color()
🟠 Micromelon — Python
# Drive forward 1 tile
rover.forward(1)

# Turn right 90°
rover.right(90)

# Read distance ahead
d = rover.getDistance()

# Read colour below
c = rover.getColour()
Same logic, different words: The pseudocode you write is the same regardless of platform. distance ← sensor.readDistance() becomes d = distance_sensor.get_distance(MM) / 10 on VEX and d = rover.getDistance() on Micromelon. The algorithm doesn't change — only the translation to code does.

Producing & Implementing

Colour sensing — the key skill for Coral Reef

The Micromelon Rover's downward-facing colour sensor reads the tile directly beneath the rover. This is the primary input for the Coral Reef Clean-Up challenge — your algorithm makes every decision based on what colour the sensor reads.

NSW Pseudocode — basic colour-based decision
BEGIN
  colour ← rover.readColour()

  IF colour = "red" THEN
    collectDebris()
  ELSE IF colour = "black" THEN
    avoidBoundary()
  ELSE IF colour = "green" THEN
    avoidSeagrass()
  ELSE
    rover.moveForward(1)     ← safe tile — keep moving
  ENDIF
END
  • Read colour after moving, not before: The colour sensor reads the tile the rover is currently on, not the tile ahead. Move first, then read and decide.
  • Case sensitive strings: rover.getColour() returns lowercase — always compare with "red", not "Red". A string comparison that differs in case will always return False.
  • Unknown colours: Add an ELSE branch for any colour you haven't explicitly handled — the rover should have a safe default behaviour (e.g. stop and flash an LED) rather than undefined behaviour.
  • Physical rover vs simulator: Colour readings are perfect in the simulator. On a physical rover, ambient lighting can shift colour values — test your colour reading in the actual lighting conditions of your classroom.
Debug technique: Add print(rover.getColour()) at the start of every loop to watch colour readings in the console as the rover moves. This lets you see exactly what the sensor is reading at each tile — essential before writing your full decision logic.

For Coral Reef, your rover needs to switch between distinct behaviours depending on what it's doing. A state machine is a clean way to organise this — the rover is always in exactly one state, and transitions happen when sensor conditions change.

NSW Pseudocode — state machine structure
state    ← "searching"
carrying ← FALSE

WHILE True DO
  colour ← rover.readColour()
  print(state, colour)        ← debug

  IF state = "searching" THEN
    IF colour = "red" THEN
      state ← "collecting"
    ELSE IF colour = "black" THEN
      avoidBoundary()
    ELSE
      rover.moveForward(1)
    ENDIF

  ELSE IF state = "collecting" THEN
    collectDebris()
    carrying ← TRUE
    state ← "depositing"

  ELSE IF state = "depositing" THEN
    IF colour = "white" THEN
      depositDebris()
      carrying ← FALSE
      state ← "searching"
    ELSE
      navigateToDeposit()
    ENDIF
  ENDIF
ENDWHILE
Why states? Without states, your program has one big IF/ELIF chain that tries to handle every situation simultaneously. With states, you separate concerns — the SEARCHING logic is only active when searching, the DEPOSITING logic only when depositing. Code is easier to read, test, and extend.

Sprint 2 · Design challenge

Coral Reef Clean-Up

The Great Barrier Reef is under threat from plastic debris. Your Micromelon Rover must autonomously navigate the ocean floor, detect debris (red tiles), collect it, and deposit it at the collection station (white tile) — all without crossing seagrass beds (green) or the reef boundary (black).

📋 Challenge brief

Coral Reef Clean-Up

Marine biologists have identified several patches of plastic debris scattered across a section of coral reef. Your robot must systematically search the reef floor, collect each debris item, and return it to the collection point before continuing the search.

The reef is a grid of coloured tiles. Your rover starts at the collection station (white tile). Debris tiles (red) are scattered across the blue ocean floor. Seagrass beds (green) and the reef boundary (black) must not be crossed.

Functional requirements

  • Detect red debris tiles using the colour sensor
  • Collect debris when detected (pause 1 second)
  • Navigate to white deposit tile after collecting
  • Deposit debris at the collection station
  • Avoid black boundary tiles
  • Avoid green seagrass tiles
  • Collect all debris items in the world

Non-functional requirements

  • Complete clean-up in under 5 minutes
  • Zero boundary crossings during run
  • Zero seagrass tile crossings
  • Program uses at least two functions
  • NSW pseudocode in project notebook
  • State machine approach used or justified
  • Sprint log completed after testing
World setup: Your teacher will load the Coral Reef world in the Micromelon simulator before the sprint. If working at home, ask for the world file. Verify you can see coloured tiles in the simulator before writing any code — run print(rover.getColour()) in a simple loop and move the rover manually to confirm colour readings.

DEPOSIT  = "white"
BOUNDARY = "black"

state    = "searching"

def navigate_to_deposit():

    Extension: replace with smarter pathfinding."""
    colour = rover.getColour()

Step 04 Main state machine loop — complete program
── NEW IN STEP 4 ──
from micromelon import *

COLLECT  = "red"
DEPOSIT  = "white"
BOUNDARY = "black"
SEAGRASS = "green"

state    = "searching"
carrying = False

def collect_debris():
    """Pause on tile to simulate debris collection. Light LED red."""
    sleep(1000)
    rover.setLED("red")

def deposit_debris():
    """Deposit collected debris at station. Turn off LED."""
    sleep(500)
    rover.setLED("off")

def navigate_to_deposit():
    """Move toward deposit tile — simple forward strategy."""
    colour = rover.getColour()
    if colour == BOUNDARY:
        rover.right(90)
    elif colour == SEAGRASS:
        rover.left(90)
    else:
        rover.forward(1)

# ── Main state machine loop ───────────────────────────────────
while True:
    colour = rover.getColour()
    print(f"state={state}  colour={colour}  carrying={carrying}")

    if state == "searching":
        if colour == COLLECT:
            state = "collecting"
        elif colour == BOUNDARY:
            rover.right(90)
        elif colour == SEAGRASS:
            rover.left(90)
        else:
            rover.forward(1)      # blue tile — continue search

    elif state == "collecting":
        collect_debris()
        carrying = True
        state = "depositing"

    elif state == "depositing":
        if colour == DEPOSIT:
            deposit_debris()
            carrying = False
            state = "searching"
        else:
            navigate_to_deposit()
The f-string debug line print(f"state={state} colour={colour} carrying={carrying}") prints all three key variables each loop. This is the most useful debugging tool in this program — you can watch the state machine transition in real time and immediately see when a transition doesn't happen as expected. Remove or comment out before submitting.

Trace & test

Trace the state machine through a complete collect-and-deposit cycle. Predict each output in your notebook before comparing to the simulator.

2 blue searching False rover.forward(1) Still searching

🔍 Bug Hunt — State machine error

The rover that collects but never deposits

A student's rover correctly detects red tiles and sets state to "collecting" — but then the LED lights up and the state never changes to "depositing". The rover just stays on the red tile forever. Find the bug:

elif state == "collecting":
    collect_debris()
    carrying = True               # line 3
    state = "collecting"          # line 4
Bug on line 4: state = "collecting" should be state = "depositing". The student accidentally set the state back to "collecting" instead of advancing it — so after collect_debris() runs, the loop immediately enters the collecting branch again, re-running collect_debris() forever. This is a classic copy-paste error. The trace table shows step 5 clearly: after collecting, state must change to "depositing".
