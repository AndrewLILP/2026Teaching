Researching & Planning

Pseudocode first — always

In NSW Computing Technology, the rule is simple: write NSW pseudocode before you write code in any language. Pseudocode lets you think through the logic without worrying about syntax. It's also the format used in NESA assessments.

The rule in this unit: For every new robot behaviour, write the pseudocode in your project notebook first. Desk check it with at least two test values. Only then open VEXcode or the Micromelon IDE.
NSW Pseudocode — structure rules
BEGIN                          ← starts the algorithm
  variable ← value            ← assignment uses ←
  INPUT variable               ← for user input (rare in robotics)
  OUTPUT variable              ← print / display

  IF condition THEN            ← selection
    statements
  ELSE
    statements
  ENDIF

  WHILE condition DO           ← pre-test loop
    statements
  ENDWHILE

  FOR variable ← start TO end DO   ← count-controlled loop
    statements
  ENDFOR

  FUNCTION name(parameters)    ← defining a function
    statements
    RETURN value
  ENDFUNCTION
END
NSW pseudocode vs Python: The ← arrow for assignment, THEN/ENDIF/ENDWHILE keywords, and FUNCTION/ENDFUNCTION wrappers are all specific to NSW pseudocode. Python uses =, colons, and indentation instead. Both describe the same logic — the pseudocode describes what, the Python describes how.
Producing & Implementing

Variables & data types

A variable is a named container that stores a value your program can read and change. In robotics, variables store sensor readings, motor speeds, counters, and state flags — the information your algorithm needs to make decisions.

NSW Pseudocode
BEGIN
  distance ← 0        ← integer: sensor reading in cm
  speed    ← 50       ← integer: motor speed (%)
  turning  ← FALSE    ← boolean: is the robot currently turning?
  colour   ← "none"   ← string: last detected colour
END
Set Variable blocks (Variables category, orange)
set distance to 0
set speed to 50
set turning to false
set colour to "none"
To use a variable in another block, drag it from the Variables panel into a value slot
Blocks tip: Create variables in the Variables category before using them. All variables in VEXcode Blocks are stored as either numbers or strings — there's no explicit Boolean type; use 1 (true) and 0 (false) as flags instead.
Variables in Python Assignment, types, and naming conventions 
# Variables — Python infers the type from the value
distance = 0        # int: sensor reading in cm
speed    = 50       # int: motor speed (−100 to +100)
turning  = False    # bool: is the robot currently turning?
colour   = "none"   # str: last detected colour
= not ←: Python uses = for assignment, not ←. The type is inferred — Python knows 0 is an integer and False is a boolean without you declaring it. Use descriptive names in snake_case — current_distance is better than d.
  • Integer (int): Whole numbers — sensor distances in cm, motor speed percentages, loop counters. Most robot sensor readings are integers.
  • Float: Decimal numbers — heading angles from a gyro (e.g. 90.4°), calculated distances, PID control values. Use float when precision matters.
  • Boolean: True/False — flags for robot state (is_obstacle_detected, has_completed_task). Essential for controlling program flow.
  • String: Text — colour names returned by colour sensors ("red", "blue"), display messages.
Type mismatch bug to watch for: If a sensor returns a float (e.g. 47.3) and your comparison uses an integer threshold (if distance < 30), Python handles this fine. But if you accidentally compare a string to a number (if "47" < 30), you'll get a TypeError. Always check what type your sensor API actually returns.

Sequence

Sequence is the simplest concept — instructions execute in order, one after another. In robotics, a sequenced program runs a fixed series of movements with no decisions or loops. It's useful for scripted demos, but brittle in dynamic environments.

NSW Pseudocode — drive forward, turn, stop
BEGIN
  driveForward(speed: 50, duration: 2000)
  wait(2000)
  turnRight(speed: 50, duration: 500)
  wait(500)
  stopMotors()
END
Motion and Wait blocks (blue = drive, orange = control)
drive FORWARD for 2 seconds at 50%
wait 2 seconds
turn RIGHT for 500 ms at 50%
stop driving
In Blocks, "drive FORWARD for X seconds" already includes the wait. Check whether your chosen block waits automatically or needs an explicit wait block below it.
Sequence in Python VEX VR drive commands 
# VEX VR — sequence example
drivetrain.drive(FORWARD)
wait(2, SECONDS)
drivetrain.turn(RIGHT)
wait(0.5, SECONDS)
drivetrain.stop()
drive() is non-blocking: drivetrain.drive(FORWARD) starts the motors and returns immediately — the program continues to the next line while the robot is still moving. wait(2, SECONDS) pauses execution for 2 seconds before the next command runs. Forgetting the wait is the single most common sequencing mistake.

Selection — IF / ELSE

Selection lets your robot make decisions based on sensor data. Without selection, your robot is just a sequence of fixed movements. With selection, it can respond to its environment.

NSW Pseudocode — obstacle avoidance decision
BEGIN
  distance ← sensor.readDistance()

  IF distance < 30 THEN
    stopMotors()
    turnRight(speed: 50, duration: 600)
  ELSE
    driveForward(speed: 50)
  ENDIF
END
If/Else block (Control — orange). Sensor block inside the condition slot (yellow).
set distance to [Distance Sensor · cm]
if distance < 30 then
stop driving
turn RIGHT for 600 ms at 50%
else
drive FORWARD at 50%
end if
Drag the Distance Sensor reporter block from the Sensing category into the condition hexagon of the If/Else block.
Selection in Python if / elif / else — obstacle avoidance 
distance = distance_sensor.get_distance(MM) / 10  # convert mm → cm

if distance < 30:
    drivetrain.stop()
    drivetrain.turn(RIGHT)
    wait(0.6, SECONDS)
else:
    drivetrain.drive(FORWARD)
Colon + indent: Python uses a colon after the condition and indentation (4 spaces) to define the block — there is no ENDIF keyword. elif adds extra conditions between if and else. Note the unit conversion — VEX VR returns distances in mm, so dividing by 10 converts to cm to match your pseudocode threshold.

When a robot needs to handle more than two situations, you can nest IFs or use ELIF/ELSE IF chains:

NSW Pseudocode — three-zone obstacle response
IF distance < 10 THEN
  stopMotors()                ← emergency stop
ELSE IF distance < 30 THEN
  reduceSPeed(speed: 20)      ← slow zone
ELSE
  driveForward(speed: 50)     ← clear path
ENDIF
if distance < 10:
    drivetrain.stop()          # emergency stop
elif distance < 30:
    drivetrain.drive(FORWARD)
    drivetrain.set_drive_velocity(20, PERCENT)
else:
    drivetrain.drive(FORWARD)
    drivetrain.set_drive_velocity(50, PERCENT)
Order matters: Python evaluates conditions top-to-bottom and stops at the first True condition. Always put the most restrictive condition first (distance < 10 before distance < 30) — otherwise the < 30 block will catch everything, including the emergency case.

Iteration — loops

Loops let your robot repeat a behaviour — either a fixed number of times (FOR loop) or until a condition changes (WHILE loop). In robotics, WHILE loops are far more common because the robot usually needs to keep running until something happens.

NSW Pseudocode — WHILE loop (main control loop)
WHILE True DO
  distance ← sensor.readDistance()

  IF distance < 30 THEN
    stopMotors()
    turnRight(speed: 50, duration: 600)
  ELSE
    driveForward(speed: 50)
  ENDIF

  wait(20ms)
ENDWHILE
Forever block (Control — orange) — runs everything inside it continuously
forever
set distance to [Distance Sensor · cm]
if distance < 30 then
stop driving
turn RIGHT for 600 ms
else
drive FORWARD at 50%
end if
wait 20 milliseconds
The Forever block is VEXcode's equivalent of WHILE True. All your autonomous decision logic goes inside it.
WHILE loop in Python The robot main control loop 
while True:
    distance = distance_sensor.get_distance(MM) / 10

    if distance < 30:
        drivetrain.stop()
        drivetrain.turn(RIGHT)
        wait(0.6, SECONDS)
    else:
        drivetrain.drive(FORWARD)

    wait(20, MSEC)   # small delay — let sensors stabilise
while True: runs forever — this is the standard pattern for autonomous robot control. The wait(20, MSEC) at the bottom of the loop is essential: without it, the loop runs at maximum CPU speed, which can cause sensors to behave erratically and drains battery faster on physical hardware.

FOR loops repeat a fixed number of times. In robotics, they're useful for scripted tasks — collecting exactly 3 objects, flashing an LED 5 times, or making a square (4 turns of 90°).

NSW Pseudocode — drive a square
FOR side ← 1 TO 4 DO
  driveForward(speed: 50, duration: 1000)
  wait(1000)
  turnRight(speed: 50, duration: 600)
  wait(600)
ENDFOR
for side in range(4):      # range(4) → 0, 1, 2, 3 (four iterations)
    drivetrain.drive(FORWARD)
    wait(1, SECONDS)
    drivetrain.turn(RIGHT)
    wait(0.6, SECONDS)
range(4) vs range(1, 5): range(4) gives 0, 1, 2, 3 — four iterations starting from zero. range(1, 5) gives 1, 2, 3, 4 — same count, different start. Either works for a square. The NSW pseudocode equivalent is FOR side ← 1 TO 4.

Functions & modularity

Functions group related instructions under a single name. Instead of writing the same obstacle-avoidance logic three times across your program, you write it once as a function and call it whenever needed. Functions make code readable, reusable, and easier to test.

NSW Pseudocode — defining and calling a function
FUNCTION avoidObstacle()
  stopMotors()
  wait(200)
  turnRight(speed: 50, duration: 600)
  wait(600)
ENDFUNCTION

FUNCTION driveForwardSafe(speed)
  distance ← sensor.readDistance()
  IF distance < 30 THEN
    avoidObstacle()
  ELSE
    driveForward(speed)
  ENDIF
ENDFUNCTION

BEGIN
  WHILE True DO
    driveForwardSafe(50)
    wait(20)
  ENDWHILE
END
Custom functions use "define" blocks (My Blocks / Functions category — dark blue/purple)
define avoidObstacle
stop driving
wait 200 milliseconds
turn RIGHT for 600 ms at 50%

define driveForwardSafe speed: ( )
set distance to [Distance Sensor · cm]
if distance < 30 then
avoidObstacle
else
drive FORWARD at speed%
end if
Create custom blocks in the My Blocks category. You can add input parameters (numbers or strings) — they appear as rounded slots in the function definition.
Functions in Python def, parameters, and calling functions 
def avoid_obstacle():
    """Stop, wait, then turn right to clear an obstacle."""
    drivetrain.stop()
    wait(200, MSEC)
    drivetrain.turn(RIGHT)
    wait(0.6, SECONDS)

def drive_forward_safe(speed):
    """Drive forward at speed% — avoid if obstacle closer than 30cm."""
    distance = distance_sensor.get_distance(MM) / 10
    if distance < 30:
        avoid_obstacle()
    else:
        drivetrain.drive(FORWARD)
        drivetrain.set_drive_velocity(speed, PERCENT)

# Main loop — calls functions instead of writing logic inline
while True:
    drive_forward_safe(50)
    wait(20, MSEC)
def keyword: Functions are defined with def and a colon; the body is indented. Parameters (like speed) are listed in parentheses — they act as local variables inside the function. The triple-quoted string ("""...""") is a docstring — a brief description of what the function does. Write docstrings as habit: they appear in IDE tooltips and in your project notebook.

Functions can also calculate and return a value to the code that called them. This is useful when you want to package a sensor read + conversion into a clean, reusable call:

NSW Pseudocode — function with RETURN
FUNCTION getDistanceCM()
  raw ← sensor.readDistance()   ← returns mm
  RETURN raw / 10               ← convert to cm
ENDFUNCTION

BEGIN
  distance ← getDistanceCM()
  IF distance < 30 THEN
    avoidObstacle()
  ENDIF
END
def get_distance_cm():
    """Return distance sensor reading converted from mm to cm."""
    return distance_sensor.get_distance(MM) / 10

# Now all distance reads use the same, clean function call
distance = get_distance_cm()
if distance < 30:
    avoid_obstacle()
Blocks limitation: Custom blocks in VEXcode Blocks don't support return values. To calculate a result and use it elsewhere, store the result in a variable inside the function instead of returning it.

The robot main loop — complete structure

Now that you have all the building blocks — variables, selection, iteration, functions — here's how they combine into the complete structure of an autonomous robot program. This is the pattern every program in this unit will follow.

NSW Pseudocode — full robot program structure
BEGIN setup
  ← Initialise all variables
  ← Configure sensors and motors
  speed    ← 50
  distance ← 0
END setup

FUNCTION avoidObstacle()
  stopMotors()
  wait(200)
  turnRight(speed: 50, duration: 600)
ENDFUNCTION

BEGIN main
  WHILE True DO
    distance ← sensor.readDistance()

    IF distance < 30 THEN
      avoidObstacle()
    ELSE
      driveForward(speed)
    ENDIF

    wait(20)
  ENDWHILE
END main
Two hat blocks: "when started" (setup) + "forever" (main loop)
when started ▶
set speed to 50
set distance to 0

when started ▶
forever
set distance to [Distance Sensor · cm]
if distance < 30 then
avoidObstacle
else
drive FORWARD at speed%
end if
wait 20 milliseconds
In VEXcode Blocks you can use multiple "when started" hat blocks — they run as parallel threads. Use one for setup and one for the main loop.
Complete program in Python VEX VR — full autonomous structure 
# ── Imports (VEX VR provides these automatically) ──────────────
from vex import *

# ── Device setup ───────────────────────────────────────────────
brain            = Brain()
distance_sensor  = Distance(Ports.PORT1)
left_motor       = Motor(Ports.PORT8)
right_motor      = Motor(Ports.PORT3, True)  # True = reversed
drivetrain       = SmartDrive(left_motor, right_motor, Gyro(Ports.PORT7))

# ── Constants ─────────────────────────────────────────────────
DRIVE_SPEED     = 50    # percent
STOP_DISTANCE   = 30    # cm
TURN_DURATION   = 0.6   # seconds
LOOP_DELAY      = 20    # milliseconds

# ── Functions ─────────────────────────────────────────────────
def get_distance_cm():
    """Return front distance in centimetres."""
    return distance_sensor.get_distance(MM) / 10

def avoid_obstacle():
    """Stop and turn right to clear a detected obstacle."""
    drivetrain.stop()
    wait(200, MSEC)
    drivetrain.turn(RIGHT)
    wait(TURN_DURATION, SECONDS)

# ── Main loop ─────────────────────────────────────────────────
while True:
    distance = get_distance_cm()

    if distance < STOP_DISTANCE:
        avoid_obstacle()
    else:
        drivetrain.drive(FORWARD)
        drivetrain.set_drive_velocity(DRIVE_SPEED, PERCENT)

    wait(LOOP_DELAY, MSEC)
Named constants: STOP_DISTANCE = 30 at the top means you change the threshold in one place and it updates everywhere in the program. Compare to scattering 30 across 6 different if statements — one wrong edit leaves bugs. This is the professional standard for any robotics program.

Bug hunt

Find the bugs before you run the code. Each program below has one or more logical errors — read carefully, trace through the logic, and explain what's wrong.

🔍 Bug Hunt #1 — Selection error

The robot that always stops

This program is supposed to drive forward when clear and stop when an obstacle is detected at less than 30cm. But the robot always stops, even when the path is clear. What's wrong?

while True:
    if distance > 30:          # line 2
        drivetrain.stop()
    else:
        drivetrain.drive(FORWARD)
    wait(20, MSEC)
Bug on line 2: The condition is backwards. distance > 30 is True when the path is clear — so the robot stops on a clear path and drives when there's an obstacle. The condition should be distance < 30. Also notice: distance is never updated inside the loop — it should read distance = get_distance_cm() at the top of the loop before the IF check.
🔍 Bug Hunt #2 — Missing wait

The robot that teleports

This program should drive forward for 2 seconds then turn right. But the robot seems to jump straight to the turn without moving forward. Why?

drivetrain.drive(FORWARD)
drivetrain.turn(RIGHT)         # line 2
Bug on line 2: drivetrain.drive(FORWARD) is non-blocking — it starts the motors and immediately moves to the next line. Without a wait(2, SECONDS) between the two commands, the turn command fires almost instantly, overriding the forward motion before the robot moves any meaningful distance. Fix: add wait(2, SECONDS) after the drive command.
🔍 Bug Hunt #3 — FOR loop count

The robot that draws a pentagon

This program should make the robot drive a square (4 sides). But it draws a shape with 3 sides instead. What's the error?

for side in range(3):          # line 1
    drivetrain.drive(FORWARD)
    wait(1, SECONDS)
    drivetrain.turn(RIGHT)
    wait(0.6, SECONDS)
Bug on line 1: range(3) iterates 3 times (0, 1, 2) — producing a triangle, not a square. A square requires 4 sides, so the loop should be range(4). The NSW pseudocode equivalent would be FOR side ← 1 TO 4.
