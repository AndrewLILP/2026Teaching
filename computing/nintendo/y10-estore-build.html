Producing & Implementing

Build Steps

Work through these in order. Each step builds on the last.

1 JSON Catalogue & Product Grid Produce

Create a products.json file — an array of objects, each with an id, name, price, category, and image path. Fetch it and render one product card per entry on your Home screen.

Pseudocode 
BEGIN LoadCatalogue()
  products = FetchJSON("products.json")
  FOR EACH product IN products
    RenderProductCard(product)
  END FOR
END
JavaScript 
async function loadCatalogue() {
  const response = await fetch('products.json');
  const products = await response.json();
  products.forEach(renderProductCard);
}

fetch returns a Promise, which is why await is needed twice — once to wait for the network request, once to wait for the response body to finish parsing as JSON.

2 Filter & Sort Produce

Add a category dropdown and a price sort control. On change, re-derive the list to display from the full products array — never delete or mutate the original data, just filter and sort a copy of it each time.

Pseudocode 
BEGIN ApplyFilters()
  filtered = products WHERE category matches selected
  sorted = filtered SORTED BY price, selected direction
  ClearGrid()
  FOR EACH product IN sorted
    RenderProductCard(product)
  END FOR
END
JavaScript 
function applyFilters() {
  let result = products.filter(p =>
    selectedCategory === 'all' || p.category === selectedCategory);

  result.sort((a, b) =>
    sortOrder === 'price-asc' ? a.price - b.price : b.price - a.price);

  grid.innerHTML = '';
  result.forEach(renderProductCard);
}

products.filter(...) returns a brand new array — it never changes products itself. That matters because every time a filter or sort changes, you need the full original list to filter from again, not whatever was left over from the last filter.

3 Cart with localStorage Produce

Store the cart as an array of { id, qty } objects in localStorage, so it survives a page reload. Before adding a new entry, check whether that product is already in the cart — if it is, increase its quantity instead of adding a duplicate row.

Pseudocode 
BEGIN AddToCart(productId)
  cart = LoadCart()
  existing = FindInCart(cart, productId)
  IF existing THEN
    existing.qty = existing.qty + 1
  ELSE
    cart.Add({ id: productId, qty: 1 })
  END IF
  SaveCart(cart)
  UpdateCartDisplay()
END
JavaScript 
function addToCart(productId) {
  const cart = JSON.parse(localStorage.getItem('cart') || '[]');
  const existing = cart.find(item => item.id === productId);

  if (existing) {
    existing.qty += 1;
  } else {
    cart.push({ id: productId, qty: 1 });
  }

  localStorage.setItem('cart', JSON.stringify(cart));
  updateCartDisplay();
}
Recognise this guard?

Check-then-act shows up again: confirm something is true (existing found or not) before deciding what to do. You've now seen this shape in a collectible pickup, a checkpoint, a game-state transition, and here in a shopping cart — same underlying idea, fourth different context.

4 Checkout Confirmation Produce

Build a confirmation screen that lists every cart item and the total, then clears the cart from localStorage once the player confirms. This is a simulated checkout — it never collects or stores real payment details of any kind, only a mock "Confirm Order" action.

Pseudocode 
BEGIN ConfirmCheckout()
  RenderOrderSummary(cart)
  ClearCart()
  ShowConfirmationScreen()
END
JavaScript 
function confirmCheckout() {
  renderOrderSummary(cart);
  localStorage.removeItem('cart');
  showConfirmationScreen();
}

Why no real payment details, ever

This also connects to CT5-SAF-01 — safe, secure and ethical practices in the use of data. A classroom project is never an appropriate place to collect real card numbers, addresses, or personal details, even for practice. Simulating "Confirm Order" with no real data collection isn't a shortcut — it's the correct, safe design choice a real developer would also make for a demo environment.

Build Checkpoint

Before this counts as your final e-Store build, all of the following must be true:

  • All four screens from your wireframes exist and are reachable: Home, Product Detail, Cart, Checkout Confirmation.
  • Filter and sort correctly narrow and reorder the product grid without altering the original data.
  • Adding an already-cart item increases its quantity rather than duplicating the entry.
  • The cart persists correctly through a full page reload.
  • Checkout clears the cart completely and collects no real payment or personal details.
