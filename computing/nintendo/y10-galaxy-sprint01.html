Gravity & Orbital Movement

Your GDD pseudocode from Lessons 1–2 described a player who can shift which surface counts as "down." This sprint makes that real: player movement, a custom gravity system you control (instead of Unity's default), the gravity-shift trigger itself, and platforms that orbit a central point. Nothing here is decorative yet — this sprint is the physical rules everything else in the game will sit on top of.

Producing & Implementing

Sprint Steps

Work through these in order. Each step builds on the last — don't skip to Step 3 with a Step 1 that doesn't work yet.

1 Read Input & Apply Movement Force Produce

Add a Rigidbody to your player capsule. In a script, read horizontal/vertical input each physics step and apply it as a force — not a direct position change, since you want Unity's physics engine handling collisions for you.

  • Use FixedUpdate(), not Update(), for anything touching Rigidbody — physics runs on a fixed timestep, separate from the render frame rate.
  • Store moveSpeed as a public/serialized field so you can tune it in the Inspector without editing code.
2 Build a Custom Gravity Vector Produce

Turn off the Rigidbody's built-in gravity (Use Gravity unticked in the Inspector). Instead, store your own gravityDirection as a Vector3 (starting pointing down), and apply it manually with AddForce() every physics step. This is what makes shifting gravity possible — you can't rotate Unity's built-in gravity, but you can rotate a variable you control yourself.

Why bother?

This is a case where understanding why matters more than copying the code. Unity's default gravity always points world-down. Your game needs "down" to be changeable — so the solution is to stop using Unity's gravity entirely and simulate your own. Every gravity-shift game (and there are commercial examples worth researching) uses some version of this idea.

3 Implement the Gravity-Shift Trigger Produce

On a button press, rotate gravityDirection by 90 degrees using a Quaternion, and rotate the player's visual orientation so their "up" matches the new "away from gravity" direction. Predict your pseudocode first, then compare against the annotated example below — it's a simplified version of the real pattern, not a copy-paste solution.

Pseudocode 
BEGIN FixedUpdate()
  ApplyMovementForce()
  ApplyGravityForce(gravityDirection)
  IF ShiftButtonPressed() THEN
    gravityDirection = Rotate(gravityDirection, 90)
    AlignPlayerUpTo(-gravityDirection)
  END IF
END
C# 
private void FixedUpdate()
{
    ApplyMovementForce();
    rb.AddForce(gravityDirection * gravityStrength);

    if (Input.GetButtonDown("ShiftGravity"))
    {
        gravityDirection = Quaternion.Euler(0, 0, 90) * gravityDirection;
        AlignPlayerUp(-gravityDirection);
    }
}

Quaternion.Euler(0, 0, 90) * gravityDirection rotates the vector 90° around the Z-axis. If your gravity-shift feels like it's rotating the wrong way, this is the line to check first — try a different axis or a negative angle before assuming the whole approach is broken.

4 Orbital Platform Script Produce

Create a script for your orbiting platforms that rotates them around a fixed pivot point using Transform.RotateAround() every frame. Attach it to a Prefab so every platform in every zone can reuse the same behaviour — the same automation idea from Lessons 1–2.

  • Expose orbitSpeed and pivotPoint as serialized fields so each platform instance can orbit differently without new code.
  • Test with at least two platforms orbiting at different speeds before moving on — a single working platform can hide bugs that only show up with more than one.

Sprint Checkpoint

Before you move to Sprint 2, all of the following must be true:

  • Player moves smoothly in all directions using force, not direct position changes.
  • Gravity is entirely custom (Rigidbody's Use Gravity is off) and applied every physics step via AddForce.
  • Pressing the shift button rotates gravity 90° and the player's visual orientation updates to match.
  • Rapid repeated shifting doesn't break or freeze the gravity direction.
  • At least two orbiting platforms run at independently-set speeds from the same Prefab.
  • All five test cases above have recorded actual results — not left blank.

Sprint 1 — Notebook Entry

Sprint 1 crosses two D&P phases — Producing & Implementing and Testing & Evaluating. Your notebook entry needs evidence of both.

Your sprint notebook entry must include:

  1. Sprint log: For each of the four steps, one sentence — what you implemented and whether your pseudocode prediction matched the real behaviour.
  2. Test table: The completed test table from the Testing section, with your actual results recorded.
  3. Bug report: Describe at least one thing that didn't work first try — especially around the gravity-shift rotation. What was wrong, and what did you change?
  4. Design reflection: Now that gravity can shift, play through a few shifts in a row. Does the rotation feel readable, or disorienting? What's one change that would make it clearer to a player experiencing it for the first time?

Your completed test table must be in your project notebook (physical or digital) before you move to Sprint 2. Untested gravity-shift logic doesn't go to production.
