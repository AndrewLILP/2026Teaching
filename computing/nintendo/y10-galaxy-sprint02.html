Collectibles & Feedback

Sprint 1 gave you movement and gravity — a game a player can control, but not yet play. This sprint gives them something to do: star fragments that award score, a shared score system every collectible can talk to, and the sound and visual feedback that makes picking one up feel satisfying rather than silent. You'll also confront a classic Unity bug on purpose, so you know how to spot it later.

Producing & Implementing

Sprint Steps

Work through these in order. Each step builds on the last.

1 Build the Collectible Prefab Produce

Create a star fragment GameObject: a Sphere Collider with Is Trigger ticked, a Mesh Renderer, and a pointValue field you can set differently per instance. Save it as a Prefab so every star fragment in every zone reuses the same script.

  • Set up three Prefab variants (or three instances with different pointValue) worth 10, 25, and 50 points, so Sprint 2 testing has something meaningful to check.
2 ScoreManager — One Shared Script Produce

Every collectible needs to add to the same running total — not its own separate score. Build a single ScoreManager script, attached to one GameObject in the scene, that every collectible can reach through a static Instance reference.

Pseudocode 
BEGIN Awake()
  Instance = this
END

BEGIN AddScore(points)
  totalScore = totalScore + points
  UpdateScoreDisplay(totalScore)
END
C# 
public static ScoreManager Instance;
private int totalScore = 0;
public TextMeshProUGUI scoreText;

private void Awake()
{
    Instance = this;
}

public void AddScore(int points)
{
    totalScore += points;
    scoreText.text = "Score: " + totalScore;
}
Why Awake(), not Start()?

Awake() runs before any Start() method in the scene, on every script, in whatever order Unity loads them. Because collectibles might call ScoreManager.Instance from their own Start(), the Instance reference has to already exist by then — which is exactly what putting the assignment in Awake() guarantees.

3 Sound & Particle Feedback Produce

Extend your Lessons 1–2 CollectibleItem.cs so a pickup plays a sound and spawns a short particle burst, on top of the score call you already have.

Pseudocode 
BEGIN OnTriggerEnter(other)
  IF other.tag = "Player" THEN
    ScoreManager.AddScore(pointValue)
    PlaySound(pickupClip)
    SpawnParticles(pickupEffect, position)
    Destroy(this.gameObject)
  END IF
END
C# 
private void OnTriggerEnter(Collider other)
{
    if (other.CompareTag("Player"))
    {
        ScoreManager.Instance.AddScore(pointValue);
        AudioSource.PlayClipAtPoint(pickupSound, transform.position);
        Instantiate(pickupEffect, transform.position, Quaternion.identity);
        Destroy(gameObject);
    }
}

Keep the particle effect short — under a second. A pickup effect that lingers makes rapid collecting feel cluttered rather than satisfying.

4 UI Score Display Produce

Add a TextMeshPro UI element to your Canvas and drag it into your ScoreManager's scoreText field in the Inspector. Confirm it updates live in Play Mode as you collect fragments — a score system nobody can see isn't communicating anything to the player.

Testing & Evaluating

Test the Loop

Test case 4 below is a genuine, well-known Unity bug source. Finding it now — deliberately — means you'll recognise the pattern instantly if it shows up again in Sprint 3.

Test CaseStepsExpected ResultActual Result
Basic pickup Walk into a single star fragment Score increases by pointValue, sound plays, particles spawn, orb disappears Record here
Score accuracy Collect all three point-value fragments (10, 25, 50) in any order Displayed score equals the exact sum — 85 Record here
Rapid double-trigger (negative test) Approach a fragment at high speed so the player might overlap it across two physics steps Score increases by pointValue exactly once, not twice Record here
Two fragments touched near-simultaneously (negative test) Position two fragments so the player can touch both in the same frame Both register — the ScoreManager correctly adds both amounts Record here
UI reflects reality Compare the displayed score against a manually tallied total after 5+ pickups Displayed number matches your manual tally exactly Record here
If test case 3 fails

Destroy(gameObject) doesn't remove the object instantly — it's queued and happens at the end of the frame. If your collider can be triggered again before that frame ends, AddScore can fire twice for one pickup. A common fix is disabling the collider (or an "already collected" flag) the instant the pickup is detected, before calling Destroy.

Sprint Checkpoint

Before you move to Sprint 3, all of the following must be true:

  • All Sprint 1 behaviours (movement, gravity-shift, orbiting platforms) still work.
  • At least three collectible instances, with different point values, share one ScoreManager via Instance.
  • Pickup triggers sound, a short particle effect, and score update together, every time.
  • The on-screen score display updates live and matches a manual tally.
  • Rapid or simultaneous pickups do not double-count or drop score.
  • All five test cases above have recorded actual results.

Sprint 2 — Notebook Entry

Sprint 2 crosses two D&P phases — Producing & Implementing and Testing & Evaluating. Your notebook entry needs evidence of both.

Your sprint notebook entry must include:

  1. Sprint log: For each of the four steps, one sentence — what you implemented and whether it worked first time.
  2. Test table: The completed test table from the Testing section, with your actual results recorded.
  3. Bug report: Did you hit the double-scoring bug? Describe what you saw and what fix you applied — or, if you avoided it entirely, explain what design choice prevented it.
  4. Design reflection: Play through a short collecting sequence. Does the feedback (sound, particles, score) feel rewarding, or does something feel flat or overwhelming? What's one specific change you'd make?

Your completed test table must be in your project notebook (physical or digital) before you move to Sprint 3. Untested scoring logic doesn't go to production.
