Producing & Implementing

Your Brief

Studio context: "This is the last sprint before the game ships. I need to see a beginning, middle, and end — multiple zones, a clear way to win, a clear way to lose, and menus so a player who's never seen this game before knows what's happening. Test the whole thing end to end, not just what's new."

Before starting, confirm every Sprint 1 and Sprint 2 checkpoint still passes. Sprint 3 adds a state machine that other scripts (your platforms, your collectibles) may need to check before acting — if movement or scoring breaks now, it's often because a system isn't respecting currentState yet.

Identifying & Defining

Game States

Your game now needs exactly one thing to be true at any moment: is the player currently playing, have they won, or have they lost? An enum makes this explicit instead of guessing from scattered booleans.

Playing
Movement, gravity-shift, and scoring all active
→ all fragments collected in final zone
Win
Win panel shown, input disabled
Playing
Movement, gravity-shift, and scoring all active
→ player falls into the void
Lose
Lose panel shown, input disabled

Notice both transitions only ever leave Playing — nothing moves directly between Win and Lose. That's a design decision worth stating explicitly in your notebook, not just something the code happens to do.

Producing & Implementing

Sprint Steps

Work through these in order.

1 Multiple Zones Produce

Group each zone's platforms and collectibles under its own parent GameObject. Build at least three zones, increasing in difficulty — more platforms, faster orbit speeds, or gravity-shifts required in quicker succession. A trigger volume at the edge of one zone activates the next and deactivates the previous, so only the current zone's objects are active at once.

  • Reuse your Sprint 1 platform Prefab and Sprint 2 collectible Prefabs — new zones should mean new arrangements, not new scripts.
2 Win & Lose State Machine Produce

Build a GameManager with a GameState enum. Guard every state change so it can only happen once — exactly the same "queued Destroy" lesson from Sprint 2 applies here: without a guard, a fast-falling player could trigger Lose more than once before the panel even appears.

Pseudocode 
BEGIN TriggerLose()
  IF currentState = Playing THEN
    currentState = Lose
    ShowLosePanel()
    DisablePlayerInput()
  END IF
END
C# 
public enum GameState { Playing, Win, Lose }
public GameState currentState = GameState.Playing;

public void TriggerLose()
{
    if (currentState == GameState.Playing)
    {
        currentState = GameState.Lose;
        losePanel.SetActive(true);
        player.enabled = false;
    }
}

Write TriggerWin() yourself, following the same shape as TriggerLose() above. It should be a near-mirror — that similarity is a good sign your design is consistent, not a sign you're repeating yourself unnecessarily.

3 The Void — A Kill Zone Produce

Add a large trigger volume well below your lowest platform. When the player enters it — meaning they've fallen off everything — call GameManager.Instance.TriggerLose(). This reuses the exact OnTriggerEnter + CompareTag("Player") pattern from Lessons 1–2 and Sprint 2, just calling a different method.

4 UI Polish — Menus & Panels Produce

Build three UI panels on your Canvas: a Start screen (shown before play begins), a Win panel, and a Lose panel — each with a short message and a "Play Again" button wired to SceneManager.LoadScene to restart cleanly. Keep every panel hidden by default except Start, and let GameManager control which one shows.

Sprint Checkpoint

Before this counts as your final build, all of the following must be true:

  • At least three zones, increasing in challenge, each activating and deactivating correctly.
  • Win and Lose both work, are guarded against double-triggering, and disable player input.
  • Start, Win, and Lose panels all exist and show at the correct moment, never simultaneously.
  • Play Again reliably restarts the game to a clean state.
  • Every Sprint 1 and Sprint 2 checkpoint behaviour still functions with zones and states in place.
  • All six test cases above have recorded actual results.

Sprint 3 — Notebook Entry

Sprint 3 crosses two D&P phases — Producing & Implementing and Testing & Evaluating. This is also your final Galaxy path entry, so it needs to summarise the whole build, not just this sprint.

Your sprint notebook entry must include:

  1. Sprint log: For each of the four steps, one sentence — what you implemented and whether it worked first time.
  2. Test table: The completed test table from the Testing section, including the regression test results.
  3. Bug report: Describe at least one bug you hit while adding zones or the state machine — especially anything that broke a Sprint 1 or Sprint 2 behaviour. What was it, and what fixed it?
  4. Whole-project reflection: Looking back across all three sprints, what's the single design decision you're most proud of, and what's one thing you'd redesign if you started again? This answer feeds directly into your studio panel pitch.

Your completed test table must be in your project notebook (physical or digital) before this build is considered finished. Untested win/lose logic doesn't go to production.
