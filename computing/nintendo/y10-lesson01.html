Identifying & Defining

Deconstruct the Project

Open the shared starter project (Galaxy-Kart-Starter) in Unity 6. It's a small 3D scene with a player capsule, a collectible orb, gravity, and a basic camera rig — deliberately generic, because both Galaxy and Kart are built from the same underlying ideas. Work through the Hierarchy panel first, before you open a single script.

  • ▾ SampleScene
    • ▸ Player Transform Rigidbody — physics: mass, gravity, drag Capsule Collider — shape used for physical contact PlayerMovement.cs — reads input, applies force
    • ▸ CollectibleOrb (Prefab) Transform Sphere Collider — "Is Trigger" ticked CollectibleItem.cs — detects pickup, adds score Mesh Renderer + Material — ShaderGraph "UnlitGlow"
    • ▸ Main Camera CameraFollow.cs — tracks Player transform each frame
    • ▸ GameManager ScoreManager.cs — holds score, exposed to every collectible

Notice the pattern: every GameObject is a container, and everything it can do comes from the Components attached to it. A Rigidbody doesn't know how to detect a pickup, and a Collider doesn't know how to apply gravity — each component does one job, and a script (like CollectibleItem.cs) is just another component that adds custom behaviour. This is the same "one job per part" thinking you used in Godot's scene tree — the vocabulary has changed, the underlying design idea hasn't.

Identifying & Defining

Scene & Script Anatomy

Open CollectibleItem.cs. Before reading the C#, write what you think it must do in NSW structured English pseudocode, based only on what you saw in the Hierarchy. Then compare it against the real script below.

Pseudocode 
BEGIN OnTriggerEnter(other)
  IF other.tag = "Player" THEN
    ScoreManager.AddScore(pointValue)
    PlayPickupSound()
    Destroy(this.gameObject)
  END IF
END
C# 
private void OnTriggerEnter(Collider other)
{
    if (other.CompareTag("Player"))
    {
        ScoreManager.Instance.AddScore(pointValue);
        AudioSource.PlayClipAtPoint(pickupSound, transform.position);
        Destroy(gameObject);
    }
}

Two things worth annotating in your notebook:

  • CompareTag("Player") is a guard condition — without it, the orb would disappear if anything touched it, including an enemy or a stray wall. Reading this line tells you the designer deliberately restricted pickup to one tag.
  • ScoreManager.Instance is one script reaching out to another. This is your first look at scripts communicating — a pattern you'll rely on constantly once your game has more than one system (score, lives, UI) that all need to talk to each other.

Now open PlayerMovement.cs and CameraFollow.cs and repeat the same process: predict in pseudocode, then compare. Add both annotated pairs to your notebook before Lesson 2.

Enterprise, Innovation & Automation

Studio Discussion

Every real game studio — including the teams behind Nintendo's first-party titles — starts new hires exactly the way you started this lesson: reading an existing project before writing anything new. Prefabs, reusable scripts, and shared systems like ScoreManager are all forms of automation — they let a small team avoid rebuilding the same logic for every object, every level, every game.

Answer in your notebook:

  1. Name one thing in the starter project that is automated — built once, reused everywhere. What would break down if every collectible had its own unique, hand-written script instead?
  2. Studios treat "read before you build" as standard process, not a suggestion. How does today's lesson show that innovation in computing technology is often about working practices, not just new features?

Designing

Game Design Document

Using what you now know about GameObjects, components, and scripts talking to each other, write a short GDD for your chosen path's core loop. Every GDD must include NSW structured English pseudocode for at least one system (movement, scoring, or a win/lose condition) — not prose describing what you'll build, the actual pseudocode.

Tiered GDD Task

Foundation · Grades D–E

One system, described

Choose your path. Describe, in pseudocode, one system: either the player's movement OR the collectible/scoring loop. Label which components (from today's vocabulary) it would need.

Core · Grade C

Two systems, linked

Pseudocode for movement and scoring, showing how they connect (e.g. how a collision event in one leads to a call in the other). Note which GameObjects and components each system touches.

Extension · Grades A–B

Full core loop

Pseudocode for your complete core loop — movement, scoring, and a win/lose condition — plus one justified design decision explaining why you chose this approach over an alternative.
