Producing & Implementing

Setting Up MakeCode Arcade

MakeCode Arcade (arcade.makecode.com) is a browser-based game development environment. It lets you build games using visual blocks that automatically generate JavaScript. You can switch between blocks and JavaScript at any time — and you should, to understand what your blocks are doing.

Create Your Project

  1. Go to arcade.makecode.com and click New Project
  2. Name your project using your game title from Lesson 2
  3. You'll see two views: Blocks (default) and JavaScript — there's a toggle at the top. Switch between them throughout this lesson.
  4. The workspace has two default blocks: on start and forever. We'll use both.

Blocks ↔ JavaScript

Every time you place a block in MakeCode Arcade, it writes JavaScript code for you. Click the JavaScript toggle to see what each block produces. Reading the JavaScript version helps you understand what the code is doing — and prepares you for writing code directly in Lessons 5–6.

Producing & Implementing

Sprint 1 — Five Steps

Follow each step in order. After each step, switch to the JavaScript view and compare the generated code with your pseudocode from Lesson 2. Add a note to your project notebook about any differences.

1
Create Your Player Sprite
Sprites

A sprite is any game object — your player, enemies, coins, and projectiles are all sprites. Each sprite has a position (x, y), a velocity (vx, vy), and an image.

Blocks to use: Inside the on start block, drag in:

🟨 Sprites → set mySprite to sprite [ ] of kind Player

Click the grey square to draw your character image. Keep it simple — 16×16 pixels is a good size.

Your Pseudocode (Lesson 2) 
BEGIN createPlayer
  SET player ← new sprite
  SET player.kind ← Player
END createPlayer
JavaScript (MakeCode generates) 
let player = sprites.create(
  img`your image here`,
  SpriteKind.Player
)

After adding the block, switch to JavaScript view. You'll see sprites.create() — this is a function call that creates the sprite object.

2
Left & Right Movement
Input

The controller.moveSprite block handles left/right movement automatically — it reads the arrow keys (or A/D keys) and updates the sprite's x position every frame.

Block to use: Inside on start, after the sprite creation block:

🎮 Controller → move mySprite with buttons vx 100 vy 0

Set vx to 100 (horizontal speed) and vy to 0 (we handle vertical separately with gravity).

Your Pseudocode 
IF LEFT pressed THEN
  player.x ← player.x - speed
ENDIF
IF RIGHT pressed THEN
  player.x ← player.x + speed
ENDIF
JavaScript 
controller.moveSprite(
  player,
  100,  // vx — horizontal speed
  0     // vy — let gravity handle this
)

Test it now — click the Play button (▶). Can you move left and right? If not, check that moveSprite is inside on start, not forever.

3
Jumping
Input

Jumping works by giving the player sprite a negative vertical velocity (vy) — pushing them upward. Gravity (Step 4) then pulls them back down.

Blocks to use: Add a new event block:

🎮 Controller → on A button pressed

Inside that event, add:

🟨 Sprites → set mySprite vy to -150
Your Pseudocode 
IF A button pressed THEN
  player.velocityY ← -jumpForce
ENDIF
JavaScript 
controller.A.onEvent(
  ControllerButtonEvent.Pressed,
  function () {
    player.vy = -150
    // negative = upward
  }
)

Why negative? In MakeCode Arcade, the y-axis is flipped: y = 0 is the top of the screen, and y increases going down. So a negative vy moves the sprite upward.

Test it — press A to jump. The player will fly off screen because there's no gravity yet. That's expected. Step 4 fixes this.

4
Gravity & a Tilemap
Physics

MakeCode Arcade has a built-in gravity system. We also need a tilemap — a grid of tiles that forms the level layout. Solid tiles act as platforms the player collides with.

Add Gravity

Block: Inside on start, add:

🌍 Scene → set gravity to 500
Your Pseudocode 
WHILE game is running
  player.velocityY ←
    player.velocityY + gravity
  player.y ←
    player.y + player.velocityY
ENDWHILE
JavaScript 
// MakeCode handles this loop
// automatically each frame
  player.ay = .5

// Higher number = falls faster
// Try 1–100 to feel different

Add a Tilemap

Block: Inside on start, add:

🗺️ Scene → set tilemap to [ ]

Click the tilemap editor. Draw a ground row at the bottom and at least two raised platforms. Mark the tiles as Wall (right-click → Wall) so the player collides with them.

What is a tilemap? A tilemap is a 2D grid where each cell contains a tile image. MakeCode Arcade automatically detects when a sprite with physics overlaps a Wall tile and prevents it from passing through. This is how platforms work — no extra code needed.

Test it — the player should now fall onto the ground and be able to jump onto platforms. If the player falls through, check that you marked the tiles as Wall.

5
Camera Follow
Camera

If your level is wider than the screen, you need the camera to follow the player. Without this, the player runs off the visible area.

Block: Inside on start, add:

📷 Scene → camera follow sprite mySprite
Your Pseudocode 
WHILE game is running
  camera.x ← player.x
  camera.y ← player.y
ENDWHILE
JavaScript 
scene.cameraFollowSprite(player)
// MakeCode handles the
// update loop automatically

Final test — move to the right edge of your tilemap. The camera should scroll with you. Make your tilemap wider in the tilemap editor if the level feels too short.

Sprint 1 Checkpoint

Before moving to Lesson 4, tick off every item below in your project notebook. Don't move on until everything is working — problems now become much harder to fix in later sprints.

✓ Sprint 1 — Checklist

Your game must be able to:

  • Display a player sprite with a custom-drawn image
  • Move left and right when arrow keys (or A/D) are pressed
  • Jump when the A button is pressed
  • Fall back down due to gravity after jumping
  • Land on the ground without falling through
  • Land on at least one raised platform without falling through
  • Have the camera follow the player across a level wider than the screen

Debugging Tip

Something Not Working?

Check these in order:

  1. Player falls through ground: Are the tiles marked as Wall (not just drawn)?
  2. Jump doesn't work: Is controller.A.onEvent a separate block — not nested inside on start?
  3. Player flies upward forever: Did you add scene.setGravity()?
  4. Camera not moving: Is scene.cameraFollowSprite(player) inside on start?

If you're still stuck, switch to JavaScript view and compare your code with the examples above line by line.
