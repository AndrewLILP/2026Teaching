Identifying & Defining

Game States — A Program Behaves Differently Over Time

Every game exists in one of several states at any given moment. A state determines what the program does and what input it accepts. Understanding states is a key computing concept — many programs beyond games use the same idea.

PLAYING
Player can move, collect, take damage
→
reach goal
WIN
game.over(true) — celebration screen
→
lives = 0
LOSE
game.over(false) — try again screen

In MakeCode Arcade, game.over(true) triggers the win screen and game.over(false) triggers the lose screen. Both end the PLAYING state. Your job in this sprint is to make sure both transitions work correctly — and test them.

States in the real world State machines appear everywhere in computing. A traffic light cycles through STOP → READY → GO states. A door lock is LOCKED or UNLOCKED. An ATM is IDLE, PROCESSING, or DISPENSING. Game states are just one application of this fundamental idea.
Producing & Implementing

Sprint 3 — Five Steps

Open your MakeCode Arcade project. Sprints 1 and 2 should both be working. Add these steps on top — never remove previous work.

1
Create a Patrolling Enemy
Sprites

An enemy is a sprite of kind Enemy. The simplest patrol behaviour uses a constant horizontal velocity and the BounceOnWall flag — the enemy reverses direction automatically when it reaches a screen edge.

Inside on start, after your existing code:

🟥 Sprites → set enemy to sprite [ ] of kind Enemy
📍 Sprites → set enemy position to x [120] y [80]
➡️ Sprites → set enemy vx to 60
🔁 Sprites → set enemy flag BounceOnWall to true
Your Pseudocode 
BEGIN createEnemy
  SET enemy ← new sprite
  SET enemy.kind ← Enemy
  SET enemy.x ← 120
  SET enemy.vx ← 60
  IF enemy hits wall
  THEN enemy.vx ← enemy.vx × -1
  ENDIF
END createEnemy
JavaScript 
let enemy = sprites.create(
  img`your image`,
  SpriteKind.Enemy
)
enemy.x = 120
enemy.vx = 60
enemy.setFlag(
  SpriteFlag.BounceOnWall,
  true
)
// MakeCode reverses vx
// when screen edge is reached

Draw a clearly threatening sprite image — red, spiky, or any design that signals "danger" to the player. Position the enemy on a platform your player must cross.

Test it — the enemy should move left/right and reverse at screen edges without any extra code.

2
Set Up the Lives Counter
Data

Before detecting enemy contact, set the player's starting lives. MakeCode Arcade manages lives as a built-in data value — similar to how it manages score. When lives reach zero, the game automatically triggers a lose state.

Inside on start, near the top before the sprite creation code:

❤️ Info → set life to 3
Your Pseudocode 
BEGIN setupGame
  SET lives ← 3
  -- displayed top-left
END setupGame
JavaScript 
info.setLife(3)
// Shows as heart icons top-left
// Auto game.over(false) at 0

Three lives is standard — but consider your game's difficulty. If your level is very short, one or two lives might create better tension. Add a note in your project notebook explaining your choice.

3
Player–Enemy Overlap → Lose a Life
Events

When the player touches an enemy, they should lose a life and the enemy should briefly flash (giving the player a moment to recover). This uses the same sprites.onOverlap pattern from Lesson 4 — different sprite kinds, different outcome.

Add a new top-level overlap event block:

💥 Sprites → on sprite of kind Player overlaps otherSprite of kind Enemy

Inside it, add:

❤️ Info → change life by -1
✨ Sprites → sprite start effect spray
Your Pseudocode 
IF player overlaps enemy
THEN
  lives ← lives - 1
  OUTPUT damage effect
  IF lives = 0 THEN
    game.over(lose)
  ENDIF
ENDIF
JavaScript 
sprites.onOverlap(
  SpriteKind.Player,
  SpriteKind.Enemy,
  function(sprite, otherSprite) {
    info.changeLifeBy(-1)
    sprite.startEffect(
      effects.spray
    )
    // MakeCode auto-calls
    // game.over(false) at 0
  }
)

Test it — walk into the enemy. A heart icon should disappear. Run into the enemy three times — the game should automatically show the lose screen. If it does, Step 3 is working.

🔎 Evaluation note

You just deliberately tested a fail path. This is called negative testing — verifying that your failure condition triggers correctly. Good developers test fail cases as rigorously as success cases. Record what happened in your sprint log.

4
Win Condition — Reach the Goal
Events

Every game needs a win condition. Choose the one that matches your Lesson 2 design — reaching a goal sprite, collecting all items, or both. The most common approach for a platformer is a goal sprite at the end of the level.

Create a Goal Sprite

Inside on start:

🌟 Sprites → set goal to sprite [ ] of kind Food

Use a kind of Food (or create a new kind called Goal via Advanced → Sprites). Position it at the far end of your level.

Detect the Goal Overlap

Add a new overlap event for Player and your goal kind:

🏆 Game → game over WIN
Your Pseudocode 
IF player overlaps goal
THEN
  OUTPUT "You win!"
  game.over(win)
ENDIF
JavaScript 
sprites.onOverlap(
  SpriteKind.Player,
  SpriteKind.Food,
  function(sprite, otherSprite) {
    // Only if it's the goal
    game.over(true)
    // true = WIN screen
  }
)

Separating goal from collectibles

If your goal sprite is the same kind as your collectibles (Food), the collect event from Lesson 4 will fire first — destroying the goal before the win event triggers. The cleanest solution is to create a custom kind for your goal. In MakeCode Arcade: open the sprites.onOverlap block → click the kind dropdown → Add a new kind → name it Goal. Then set your goal sprite's kind to Goal and write a separate overlap event for Player + Goal.

5
Win Condition — Collect All Items (Alternative)
Optional

If your game's win condition is collecting all items (not reaching a flag), you need to check whether all collectibles have been destroyed after each collection. Add this inside your existing Lesson 4 collectible overlap event:

Your Pseudocode 
IF player overlaps collectible
THEN
  score ← score + 1
  collectible.destroy()
  IF score = totalCollectibles
  THEN
    game.over(win)
  ENDIF
ENDIF
JavaScript 
// Inside existing overlap event:
info.changeScoreBy(1)
otherSprite.destroy()
// Check if all collected:
if (info.score() >= 5) {
  game.over(true)
}

Replace 5 with however many collectibles your level has. If you add or remove collectibles later, update this number too — otherwise the win condition will never trigger (or trigger too early).

Testing & Evaluating

Test Both Paths — Evaluation Begins Here

This is your first formal testing and evaluation activity. In the Design and Production Framework, Testing & Evaluating means deliberately checking that your solution works as designed — not just playing it and hoping for the best.

Use this test table as a guide. Complete it in your project notebook — write the actual result you observed, not what you expected.

Test case Expected result Your actual result (write in notebook)
Walk into enemy once One heart disappears; player continues Record actual result →
Walk into enemy 3 times All hearts gone; lose screen appears Record actual result →
Collect one item Score increases by 1; item disappears; sound plays Record actual result →
Reach the goal sprite Win screen appears immediately Record actual result →
Collect all items (if collect-all win) Win screen appears after last item Record actual result →
Jump on a platform while enemy patrols below Player safe on platform; enemy doesn't follow Record actual result →
🔎 Evaluation

For each test where the actual result didn't match the expected result, write:

  1. What went wrong (the bug)
  2. What you changed to fix it (the fix)
  3. Whether the fix introduced any new problems (a regression)

This is the process real software developers follow. Recording bugs and fixes is as important as recording what worked.

Sprint 3 — Notebook Entry

Sprint 3 crosses two D&P phases — Producing & Implementing and Testing & Evaluating. Your notebook entry needs evidence of both.

Your sprint notebook entry must include:

  1. Sprint log: For each step, write one sentence — what you implemented and whether it worked first time. Note any differences between your pseudocode and the final code.
  2. Test table: The completed test table from the Testing section (all six test cases, with your actual results written in).
  3. Bug report: Describe at least one thing that didn't work as expected. What was the bug? What did you change? Did the fix cause any regressions?
