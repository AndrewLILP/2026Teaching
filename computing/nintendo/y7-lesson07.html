micro:bit Controller

You built a game in software. Now you'll build a controller in hardware. The micro:bit will be programmed to act as a USB keyboard — tilting it left or right sends left/right arrow key signals to your computer, and pressing button A sends the jump key. Hardware meets software.

Producing & Implementing
Producing & Implementing

Your Brief

Studio context: Your game shipped. Now the hardware team wants to see it played with a physical controller. Your task: program the micro:bit to act as a keyboard controller using its built-in accelerometer and buttons. When you plug it in via USB, your MakeCode Arcade game should be fully playable without touching the keyboard.

This lesson uses makecode.microbit.org — not MakeCode Arcade. You'll write a separate program that runs on the micro:bit itself, turning it into a USB Human Interface Device (HID). The approach is the same as Lessons 3–6: pseudocode before code, analyse before build, test deliberately.

Two devices, two programs At the end of this lesson you'll have two separate programs running simultaneously — your game in MakeCode Arcade (on the computer), and your controller code on the micro:bit. They communicate through a simple channel: the micro:bit pretends to be a keyboard, and MakeCode Arcade responds to keyboard input. No wireless connection needed.
Identifying & Defining

Analyse First — What Does a Controller Actually Send?

Before building anything, you need to understand what a game controller is at a technical level — not as a physical object, but as a data source.

Analysis Task

Map Your Game's Inputs

Open your MakeCode Arcade game from Lesson 6. In your project notebook, make a table with two columns:

  1. List every input your game currently accepts (check the keyboard controls — arrow keys, A button, etc.)
  2. For each input, write what the micro:bit could do to replicate it — which button or which physical motion?

Then answer: are there any inputs your game needs that the micro:bit can't easily replicate? What would you do about that?

This mapping exercise is the design work for your controller. Before you write a single block in MakeCode for micro:bit, you should know exactly what the controller needs to send and when.

Identifying & Defining

Systems Thinking — The Full Signal Chain

Systems thinking means understanding how all the parts of a system work together — not just one component in isolation. Your controller setup is a small but real computing system. Trace the signal from physical movement to on-screen action.

🤲 Physical tilt you tilt left
→
📡 Accelerometer X axis < -300
→
🔲 micro:bit code IF tilt THEN send key
→
🔌 USB HID Left arrow keypress
→
💻 Operating system keyboard event
→
🎮 MakeCode Arcade player moves left

Each node in the chain is a system component. If the chain breaks anywhere — bad USB connection, wrong threshold value, game not in focus — the signal doesn't reach the game. When you debug your controller in Step 4, you'll trace the chain to find where the break is.

What is a HID device?

HID stands for Human Interface Device — a USB standard that includes keyboards, mice, and gamepads. When the operating system sees a HID device, it knows how to interpret its signals as input events without needing special drivers. By programming the micro:bit to act as a HID keyboard, your computer treats it exactly like any other keyboard. MakeCode Arcade doesn't need to know a micro:bit is connected — it just sees keystrokes arriving.

micro:bit v1 vs v2

The keyboard HID extension works on both micro:bit v1 and v2. The accelerometer sensitivity and range are slightly different between versions — you'll set the tilt threshold in Step 3 and calibrate it in Lesson 8. If the controller feels unresponsive or too sensitive, the threshold value is the first thing to adjust.

Producing & Implementing

Setup — Four Steps

Open makecode.microbit.org in a new tab. Keep your MakeCode Arcade game open in another tab — you'll need both. Create a new micro:bit project and name it after your game + "Controller".

1
Add the Keyboard Extension
Setup

The keyboard HID capability isn't built into MakeCode by default — you need to add it as an extension.

  1. In your micro:bit project, click the Extensions button (gear icon or in the toolbox)
  2. Search for keyboard
  3. Click the keyboard extension to add it — new blocks will appear in the toolbox under "Keyboard"
What the extension adds: The keyboard extension lets the micro:bit communicate via USB as a HID keyboard. When you flash a program using it to the micro:bit and plug it in, the computer recognises it as a keyboard device — no drivers needed.

After adding the extension, switch to JavaScript view in MakeCode for micro:bit. You'll see the import at the top of your file. This is the same technique from Lessons 3–6 — understanding what the blocks generate.

2
Button A — Jump
Buttons

Button A on the micro:bit will send the Space key — MakeCode Arcade maps Space to the A button (jump). We need two events: one for when button A is pressed (send keydown) and one for when it's released (send keyup). This ensures the game doesn't register a continuous held jump.

Blocks to add (two separate event blocks):

🔲 Input → on button A pressed → Keyboard → key press Space
🔲 Input → on button A released → Keyboard → key release Space
Pseudocode 
IF button A pressed THEN
  send SPACE keydown
ENDIF
IF button A released THEN
  send SPACE keyup
ENDIF
JavaScript (micro:bit) 
input.onButtonPressed(
  Button.A,
  function () {
    keyboard.key(Key.Space,
      KeyEvent.Press)
  }
)

input.onButtonReleased(
  Button.A,
  function () {
    keyboard.key(Key.Space,
      KeyEvent.Release)
  }
)

Flash and test: Download the .hex file to your micro:bit (or use WebUSB to flash directly). Open your MakeCode Arcade game in the browser. Make sure the browser window is active (click on it), then press the micro:bit's A button. The player should jump.

Why press AND release?

KeyDown vs KeyUp

A keyboard sends two signals for every keypress: a keydown when the key is pressed, and a keyup when it's released. If you only send keydown and never keyup, the computer thinks the key is being held down permanently — the player will jump continuously and uncontrollably. Always send both events.

3
Accelerometer — Left & Right Movement
Accelerometer

The micro:bit's accelerometer measures force in three directions: X (left/right tilt), Y (forward/back tilt), and Z (up/down). For our controller, we only need the X axis. The values range from approximately -1024 (full left tilt) to +1024 (full right tilt), with 0 at flat.

The Dead Zone

We use a threshold to create a dead zone — a range of values near zero that registers as "no movement". Without it, any tiny wobble in your hand would trigger movement constantly.

X-axis Accelerometer Range

LEFT < -300
DEAD ZONE -300 to +300
RIGHT > +300
-1024 (full left) 0 (flat) +1024 (full right)

The threshold value 300 is a starting point — you'll calibrate it to your specific micro:bit in Lesson 8. Smaller values (e.g. 150) make the controller more sensitive; larger values (e.g. 500) require a stronger tilt.

Blocks: Inside a forever block, add logic using the accelerometer reading:

🔁 Basic → forever

Inside it:

📊 Logic → if [acceleration (mg) x] < [-300] then ...
⌨️ Keyboard → key press Left Arrow
⌨️ Keyboard → key release Right Arrow

And mirror this for the else-if (tilt right) and else (no tilt) branches.

Pseudocode 
WHILE true
  SET tilt ← accelerometer.x
  IF tilt < -300 THEN
    send LEFT keydown
    send RIGHT keyup
  ELSE IF tilt > 300 THEN
    send RIGHT keydown
    send LEFT keyup
  ELSE
    send LEFT keyup
    send RIGHT keyup
  ENDIF
  pause 50ms
ENDWHILE
JavaScript (micro:bit) 
basic.forever(function () {
  let tilt =
    input.acceleration(
      Dimension.X
    )
  if (tilt < -300) {
    keyboard.key(
      Key.Left, KeyEvent.Press)
    keyboard.key(
      Key.Right, KeyEvent.Release)
  } else if (tilt > 300) {
    keyboard.key(
      Key.Right, KeyEvent.Press)
    keyboard.key(
      Key.Left, KeyEvent.Release)
  } else {
    keyboard.key(
      Key.Left, KeyEvent.Release)
    keyboard.key(
      Key.Right, KeyEvent.Release)
  }
  basic.pause(50)
})

The basic.pause(50) at the end of the forever loop is important — it limits the polling rate to 20 checks per second. Without it, the loop runs thousands of times per second and can overwhelm the USB connection with key events.

4
Flash, Connect & Test
Testing

Download your micro:bit program and flash it to the device. Then run your MakeCode Arcade game and test the controller.

Flashing via WebUSB (recommended)

  1. In MakeCode for micro:bit, click the Download button
  2. If prompted, choose Download via WebUSB and pair your device
  3. The micro:bit will reboot and run your controller code immediately

Flashing via .hex file

  1. Download the .hex file to your computer
  2. The micro:bit appears as a USB drive called MICROBIT
  3. Drag the .hex file onto the MICROBIT drive — it will flash automatically

Test sequence

  1. Open your MakeCode Arcade game and click the game window to give it focus — keyboard events only go to the active window
  2. Tilt the micro:bit left — the player should move left
  3. Tilt right — the player should move right
  4. Return to flat — the player should stop
  5. Press micro:bit button A — the player should jump

Debugging — Controller Not Working?

Trace the Signal Chain

  1. Nothing responds: Is the MakeCode Arcade browser window in focus? Click it first, then tilt.
  2. Movement is inverted: Swap the conditions — change < -300 to > 300 and vice versa.
  3. Player moves continuously without tilting: Your threshold is too low. Increase from 300 to 400 or 500.
  4. No response at all: Open a plain text editor (e.g. Notepad) and tilt the micro:bit. Do you see arrow keys appearing? If yes, the micro:bit is working — the issue is game focus. If no, re-flash the program.
  5. Jump doesn't stop: Check that your button A released event is present and correctly sends KeyEvent.Release.

✓ Lesson 7 — Checklist

Your controller must:

  • Have the keyboard extension added to the micro:bit project
  • Send Space keydown and keyup when micro:bit button A is pressed and released
