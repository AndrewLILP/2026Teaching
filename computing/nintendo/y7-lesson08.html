Controller Calibration & LED Feedback

Your controller works — but does it feel right? In this lesson you'll calibrate the tilt threshold using a systematic data-collection process, introduce named constants to make future adjustments clean, and add LED arrow feedback so the micro:bit confirms what input it's sending. The goal: a controller that feels as good as it plays.

Producing & Implementing

Named Constants — One Change, One Place

In your Lesson 7 code, the threshold value 300 appears twice in the forever loop — once for left, once for right. During calibration you'll change this value many times. If you change one instance but forget the other, the controller breaks.

The solution is a named constant — a variable declared once at the top that holds a value used throughout the program. Change it in one place, and every reference to it updates automatically.

Without constant (fragile) 
IF tilt < -300 THEN   ← change here
  send LEFT
ELSE IF tilt > 300 THEN  ← AND here
  send RIGHT
ENDIF
// Two places to remember to change.
// Easy to forget one.
With constant (robust) 
const THRESHOLD = 300  ← change here only

if (tilt < -THRESHOLD) {
  // reads from constant
} else if (tilt > THRESHOLD) {
  // reads from constant
}
// One place. Always in sync.

Make this change now — open your Lesson 7 micro:bit project in JavaScript view and add const THRESHOLD = 300 at the very top of your program, then replace both instances of 300 with THRESHOLD (and -THRESHOLD for the left condition). Flash the updated program and confirm it still works before proceeding to calibration.

const vs let We use const (not let) because the threshold is a fixed value we set intentionally — it shouldn't change during the program's runtime. Using const makes this intent clear to anyone reading the code. Constants are typically written in ALL_CAPS by convention, which is why we write THRESHOLD not threshold.
Testing & Evaluating

Calibration — Finding Your Optimal Threshold

Calibration is the systematic process of adjusting a device's settings based on measured evidence to match real-world conditions. It's a form of evaluation — you're comparing actual performance against expected performance and refining until they match.

🔎 Evaluation

Calibration is not guessing. It is:

  1. Baseline — record how the current setting performs
  2. Adjust — change one variable (the threshold)
  3. Measure — observe and record the new performance
  4. Compare — decide whether the change was an improvement
  5. Repeat — until performance is satisfactory

This is the scientific method applied to hardware design. Real game controllers go through thousands of calibration iterations before shipping.

Expected results for each range

At 100: extremely sensitive — the player will move even when you're trying to hold still. Hard to stand in the dead zone. At 200: noticeably sensitive — works for confident, deliberate tilts but may drift. At 300: the default starting point — reasonable for most micro:bits and play styles. At 400: requires a deliberate tilt — good if you have shaky hands or are sitting a distance from the screen. At 500: requires a strong tilt — may feel sluggish for a fast platformer. Every micro:bit and player is different — there's no universally correct answer.

Adding LED Feedback to Your Forever Loop

Open your micro:bit project. Inside the forever loop, add a basic.showArrow() call at the start of each branch — before the keyboard key calls. The LED updates first, then the key is sent.

For the dead zone (else branch), clear the display and show a single centre dot using basic.clearScreen() followed by led.plot(2, 2). This is less disruptive to the display than basic.showLeds() which redraws all 25 pixels.

Updated Pseudocode 
WHILE true
  SET tilt ← accelerometer.x
  IF tilt < -THRESHOLD THEN
    OUTPUT ← arrow
    send LEFT keydown
    send RIGHT keyup
  ELSE IF tilt > THRESHOLD THEN
    OUTPUT → arrow
    send RIGHT keydown
    send LEFT keyup
  ELSE
    OUTPUT · dot
    send LEFT keyup
    send RIGHT keyup
  ENDIF
  pause 50ms
ENDWHILE
JavaScript — Updated Forever Loop 
basic.forever(function () {
  let tilt =
    input.acceleration(Dimension.X)

  if (tilt < -THRESHOLD) {
    basic.showArrow( // ★ NEW
      ArrowNames.West)
    keyboard.key(Key.Left,
      KeyEvent.Press)
    keyboard.key(Key.Right,
      KeyEvent.Release)

  } else if (tilt > THRESHOLD) {
    basic.showArrow( // ★ NEW
      ArrowNames.East)
    keyboard.key(Key.Right,
      KeyEvent.Press)
    keyboard.key(Key.Left,
      KeyEvent.Release)

  } else {
    basic.clearScreen() // ★ NEW
    led.plot(2, 2)     // ★ NEW
    keyboard.key(Key.Left,
      KeyEvent.Release)
    keyboard.key(Key.Right,
      KeyEvent.Release)
  }
  basic.pause(50)
})

Flash the updated program. The LED display should now clearly show the current input direction — left arrow when tilting left, right arrow when tilting right, single dot in the dead zone. Use this to verify your calibration: if the dot appears when you think you're holding flat, your dead zone is working correctly.

Your Complete Controller Program

Here is the full micro:bit controller program combining Lessons 7 and 8. Lines marked ★ are the Lesson 8 additions. Compare this against your own code — they should match in structure, with your chosen threshold value in place of 300.

Complete Controller — Lessons 7 & 8 ★ marks Lesson 8 additions 
// ── CONSTANT — change this to calibrate ── ★
const THRESHOLD = 300  // ★ replace with your chosen value

// ── BUTTON A — Jump ──────────────────────────
input.onButtonPressed(Button.A, function () {
  keyboard.key(Key.Space, KeyEvent.Press)
})

input.onButtonReleased(Button.A, function () {
  keyboard.key(Key.Space, KeyEvent.Release)
})

// ── MOVEMENT + LED FEEDBACK ───────────────────
basic.forever(function () {
  let tilt = input.acceleration(Dimension.X)

  if (tilt < -THRESHOLD) {
    basic.showArrow(ArrowNames.West)  // ★
    keyboard.key(Key.Left,  KeyEvent.Press)
    keyboard.key(Key.Right, KeyEvent.Release)

  } else if (tilt > THRESHOLD) {
    basic.showArrow(ArrowNames.East)   // ★
    keyboard.key(Key.Right, KeyEvent.Press)
    keyboard.key(Key.Left,  KeyEvent.Release)

  } else {
    basic.clearScreen()  // ★
    led.plot(2, 2)        // ★  centre dot = dead zone
    keyboard.key(Key.Left,  KeyEvent.Release)
    keyboard.key(Key.Right, KeyEvent.Release)
  }

  basic.pause(50)  // 20Hz polling rate
})
