Analysis Task

Read the Project Like a Reviewer

Open the project in Godot 4 and answer every question below in your project notebook. Use full sentences — this is your evidence, not a checklist.

  1. Open the main level scene in the Scene panel. What is the root node, and what type is it?
  2. Find the Player node. What node type is it, and what child nodes does it have attached (look for things like a sprite, a collision shape, an animation player)?
  3. Open the script attached to the Player. Find the function that runs every physics frame. What is it called, and what does it control?
  4. Find where gravity is applied in the code. Is it a number typed directly into the script, or does it come from somewhere else in the project?
  5. Find one signal connection in the project — for example, something that happens when the player touches a collectible or an enemy. Which node sends the signal, and which function receives it?
  6. Look at the project's folder structure in the FileSystem panel. What organisation choices has the original developer made — are scenes, scripts, and art kept in separate folders?
  7. Note one design decision in this project you disagree with, or would do differently in your own game. Explain why.
Identifying & Defining

Scenes, Nodes & the Scene Tree

Everything in Godot is built from two ideas: nodes and scenes. A node is a single building block — a sprite, a collision shape, a timer, a piece of UI. A scene is a group of nodes arranged in a tree, saved together as one reusable unit. Your whole game is a tree of scenes, each one a tree of nodes.

Here's a simplified version of the scene tree you should find in the starter project's main level:

Level (Node2D)
├─ Player (CharacterBody2D)
│   ├─ Sprite2D
│   ├─ CollisionShape2D
│   └─ AnimationPlayer
├─ TileMap // the level geometry
├─ Enemy (CharacterBody2D)
│   ├─ Sprite2D
│   └─ CollisionShape2D
├─ Coin (Area2D) // detects overlap, doesn't collide physically
│   ├─ Sprite2D
│   └─ CollisionShape2D
└─ UI (CanvasLayer)
    └─ ScoreLabel (Label)

Notice that CharacterBody2D is used for anything that moves and collides solidly (the Player and Enemy), while Area2D is used for things that only need to detect overlap without physically blocking movement (the Coin). This is a design decision the original developer made deliberately — picking the node type whose built-in behaviour already matches what the object needs to do.

Nodes as a systems-thinking idea A node type is a bit like a class in general programming — it defines a set of built-in behaviours and properties. When you drag a CharacterBody2D into your scene, you're not writing movement and collision code from scratch; you're using an existing system and adding your own logic on top of it. Recognising which built-in system already solves part of your problem is a core systems-thinking skill — and it only comes from reading other people's projects, which is exactly what you're doing right now.

GDScript at a Glance

You won't write GDScript until Sprint 1 — but reading it now, side by side with NSW structured English, will make the starter project's script make sense. GDScript looks close to Python: indentation matters, and there are no curly braces.

Reading player movement

Structured English 
EVERY physics frame
  direction ← GET horizontal input
  IF direction ≠ 0 THEN
    SET velocity.x TO direction × speed
  ELSE
    SET velocity.x TO 0
  END IF
  MOVE player using velocity
GDScript 
func _physics_process(delta):
  var direction = Input.get_axis("move_left", "move_right")
  if direction != 0:
    velocity.x = direction * speed
  else:
    velocity.x = 0
  move_and_slide()

_physics_process(delta) is a built-in function — Godot calls it automatically, every physics frame, without you ever calling it yourself. The delta parameter is the time (in seconds) since the last frame, which is how consistent, frame-rate-independent movement is achieved. Find this exact function in the Player's script now — you already answered which node it's attached to in Question 3.

Reading a signal connection

Structured English 
WHEN player overlaps coin
  ADD 1 TO score
  REMOVE coin
  UPDATE score display
GDScript 
func _on_body_entered(body):
  if body.name == "Player":
    Score.total += 1
    queue_free()

This is a signal — Godot's version of an event. The Coin's Area2D emits a body_entered signal whenever something overlaps it; the function _on_body_entered receives that signal and reacts to it. Signals are how nodes talk to each other without being tightly wired together — the Coin doesn't need to know anything about the Player except that something entered its area.
