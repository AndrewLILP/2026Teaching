Movement & Physics

Your Game Design Document has a core loop written in structured English. Sprint 1 turns the first two lines of it into real GDScript: "IF player presses movement key THEN move player horizontally" and "IF player presses jump AND player is on ground THEN apply upward velocity; apply gravity every frame." By the end of this sprint your player will move, jump, and interact with solid level geometry — the foundation every later sprint builds on.

Producing & Implementing

Your Brief

Studio memo: "Design documents are cheap; working movement is not. Build the Player scene exactly the way you planned it in your Node/Scene Plan, then bring your core loop pseudocode to life one line at a time. Test after every step — don't stack three untested changes on top of each other and hope."

Open your own project in Godot 4 (a new, empty project — not the starter project, which stays untouched as your reference). Have your Lesson 3 GDD open beside the editor. Every step below maps to a specific line of your core loop pseudocode.

Identifying & Defining

Read the Starter Script First

Before writing your own movement code, reopen the starter project from Lessons 1–2 and find the Player's script again. You already identified _physics_process(delta) as the function that runs every physics frame — now read exactly how it handles movement, line by line, before you write your own version from scratch.

Analysis Task

Trace the Starter Project's Movement Logic

  1. What is the exact input method used to read left/right movement — is it Input.get_axis(), two separate Input.is_action_pressed() checks, or something else?
  2. Where is the speed value stored — a plain number typed into the function, or a variable declared elsewhere in the script?
  3. Does the script check is_on_floor() before allowing a jump? What happens if you (hypothetically) removed that check?
  4. Is gravity applied as a fixed number, or does it reference a project-wide physics setting?

You don't need to match the starter project's exact approach in your own game — but every difference should be a deliberate design choice, not an accident from not having read the reference first.

Producing & Implementing

Sprint 1 — Five Steps

Work through these steps in order. Test after each one — don't move on until the current step behaves correctly.

1
Build the Player Scene
Scene Setup

Create a new scene with a CharacterBody2D as the root, named Player. Add a Sprite2D child for the artwork and a CollisionShape2D child sized to match it. This should match the Node/Scene Plan from your GDD — if it doesn't, update your GDD now rather than silently drifting from your own plan.

Player (CharacterBody2D)
├─ Sprite2D
└─ CollisionShape2D

Attach a new script to the Player node. Before writing any functions, declare three @export variables so speed, jump strength, and gravity can be tuned from the Inspector without editing code — the same idea as the Export Variable you defined in Lessons 1–2's vocabulary.

Structured English 
DECLARE speed ← 200
DECLARE jumpStrength ← 400
DECLARE gravity ← 900
GDScript 
extends CharacterBody2D

@export var speed: float = 200.0
@export var jump_strength: float = 400.0
@export var gravity: float = 900.0

Test it — run the scene. Nothing should move yet, but the Inspector should now show Speed, Jump Strength, and Gravity as editable fields under your script's export variables.

2
Horizontal Movement
GDD Core Loop — Line 1

This step implements the first line of your core loop: "IF player presses movement key THEN move player horizontally." Add a _physics_process(delta) function — this is the exact built-in function you found in the starter project.

Structured English 
EVERY physics frame
  direction ← GET horizontal input
  IF direction ≠ 0 THEN
    SET velocity.x TO direction × speed
  ELSE
    SET velocity.x TO 0
  END IF
GDScript 
func _physics_process(delta):
  var direction = Input.get_axis("move_left", "move_right")
  if direction != 0:
    velocity.x = direction * speed
  else:
    velocity.x = 0
  move_and_slide()

Before this works, define move_left and move_right as Input Actions in Project → Project Settings → Input Map, mapped to your arrow keys or A/D.

Test it — the player should slide left and right smoothly, and stop immediately when you release the key.

3
Jump & Gravity
GDD Core Loop — Lines 2–3

This step implements the next two lines of your core loop: "IF player presses jump AND player is on ground THEN apply upward velocity" and "apply gravity every frame." Add these lines inside the same _physics_process(delta) function from Step 2 — don't create a second one.

Structured English 
IF NOT player is on ground THEN
  velocity.y ← velocity.y + gravity × delta
IF jump pressed AND player is on ground THEN
  SET velocity.y TO -jumpStrength
GDScript 
# add inside _physics_process, after direction/velocity.x
  if not is_on_floor():
    velocity.y += gravity * delta
  if Input.is_action_just_pressed("jump") and is_on_floor():
    velocity.y = -jump_strength

Map a jump Input Action to the spacebar in the Input Map, the same way you mapped movement in Step 2. Note that is_on_floor() is checked twice — once to decide whether gravity should pull the player down, and again to stop mid-air jumping. This is exactly the check you should have found in the starter project's script.

Test it — press jump while grounded and the player should rise then fall back down under gravity. Press jump while airborne and nothing should happen — no double-jumping unless you deliberately design for it.

4
Build the Level
Scene Setup

Movement means nothing without something solid to stand on. Add a TileMap node to your main scene and paint a simple level: a floor, at least one raised platform, and one solid wall the player can't walk through.

  • Set the TileMap's physics layer so its tiles register as solid collision, matching what you found in the starter project.
  • Make sure your Player's CollisionShape2D is sized correctly — too large and the player will appear to float above platforms; too small and it will visually clip into the floor.
  • Leave a gap or ledge somewhere in the level — you'll deliberately test falling off it in the next section.

Test it — the player should stand on the floor and platforms without sinking through them, and should be stopped by the wall rather than passing through it.

5
Camera & Screen Boundaries
Polish

Add a Camera2D node as a child of Player and enable its Current property, so the view follows the player automatically as they move. If your level is larger than a single screen, set the camera's Limit properties (left, right, top, bottom) in the Inspector so the camera stops at the level's edges instead of showing empty space beyond it.

Test it — walk to both ends of your level. The camera should follow smoothly and never show the void past the level boundary.

Testing & Evaluating

Test Your Movement

Complete this test table in your project notebook. Write the actual result you observed — not what you expected to happen.

Test case Expected result Your actual result (write in notebook)

Release movement keys Player stops moving immediately Record actual result →

🔍 Evaluation

For each test where the actual result didn't match the expected result, write:

  1. What went wrong (the bug)
  2. What you changed to fix it (the fix)
  3. Whether the fix introduced any new problems (a regression)

A very common bug at this stage: gravity accumulates infinitely while grounded because is_on_floor() isn't resetting velocity.y. If your player sinks slightly into the floor or jumps feel inconsistent, this is the first place to check.

Sprint 1 Checkpoint

Tick off every item in your project notebook. Your project must pass this checkpoint before Sprint 2 begins.

✓ Sprint 1 — Checklist

Your project must:

  • Have a Player scene matching your GDD's Node/Scene Plan (CharacterBody2D, Sprite2D, CollisionShape2D)
  • Move left and right smoothly in response to input, stopping immediately when keys are released
  • Jump only when grounded — no mid-air jumping
  • Fall under consistent gravity when walking off a platform edge
  • Have a level with a floor, at least one platform, and one solid wall the player can't pass through
  • Have a camera that follows the player and respects the level boundaries
  • Have a completed test table in your project notebook (all six tests attempted)

Sprint 1 — Notebook Entry

Sprint 1 crosses two D&P phases — Producing & Implementing and Testing & Evaluating. Your notebook entry needs evidence of both.

Your sprint notebook entry must include:

  1. Sprint log: for each of the five steps, one sentence on what you implemented and whether it worked first time.
  2. Test table: the completed test table from the Testing section, with actual results written in.
  3. Bug report: describe at least one thing that didn't work as expected, what you changed, and whether the fix caused any regressions.
  4. Design reflection: compare your finished movement to your GDD's core loop pseudocode. Does anything feel different from how you imagined it while writing the GDD? Would you change the pseudocode now that you've built it?

Bring a working, tested build to Sprint 2 — enemies and collectibles are added on top of this movement, not instead of it.
