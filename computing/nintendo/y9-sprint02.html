Enemies & Collectibles

Sprint 1 gave your player a body and a level to stand on. Sprint 2 gives the game something to do: the next two lines of your core loop — "IF player overlaps collectible THEN increase score, remove collectible" and "IF player overlaps enemy THEN decrease lives; IF lives = 0 THEN trigger fail condition." Along the way you'll introduce a small state machine for your enemy, and a global place to store score and lives that any scene can read from.

Your Brief

Studio memo: "Movement alone isn't a game — it's a tech demo. Nintendo AU wants to see risk and reward: something to collect, something to avoid. Before you copy the starter project's approach for detecting the player, look again at how it identified who it was colliding with. There's a more robust way to do it, and you're using it from here on."

Sprint 1 must be fully working before you start — the checkpoint from last lesson is not optional. Everything in this sprint builds on top of your existing movement and level, never replacing it.

Producing & Implementing

Sprint 2 — Five Steps

Work through these steps in order. Test after each one.

1
A Global Home for Score & Lives
Autoload

Score and lives need to be readable and changeable from any scene — the Coin, the Enemy, and your UI all need access to the same values. Create a new script called GameState.gd, then register it as an Autoload in Project → Project Settings → Autoload. This makes it a global singleton — accessible from anywhere in your project simply by typing GameState.

Structured English 
DECLARE score ← 0
DECLARE lives ← 3
GDScript — GameState.gd 
extends Node

var score: int = 0
var lives: int = 3

While you're in the Player scene, also add the Player node to a new Group called player (in the Node dock's Groups tab). You'll use this in Step 2 instead of checking the node's name.

Test it — nothing visible changes yet, but GameState should appear with no errors when you run any scene.

2
Build the Coin
GDD Core Loop — Line 4

This step implements "IF player overlaps collectible THEN increase score, remove collectible." Create a new scene: an Area2D root named Coin, with a Sprite2D and CollisionShape2D child — the same pattern you found in the starter project during Lessons 1–2.

Coin (Area2D)
├─ Sprite2D
└─ CollisionShape2D

Connect the Coin's body_entered signal to a new function on its own script. Notice the check below uses is_in_group("player") instead of comparing the node's name — this keeps working even if you rename the Player node later, and correctly ignores the Enemy or any other body that might overlap the coin.

Structured English 
WHEN body enters coin area
  IF body is player THEN
    ADD 1 TO score
    REMOVE coin
GDScript 
func _on_body_entered(body):
  if body.is_in_group("player"):
    GameState.score += 1
    queue_free()

Place two or three instances of your Coin scene around the level. Test it — walk into a coin; it should disappear immediately. Print GameState.score to the Output panel temporarily to confirm it's actually increasing.

3
A Patrolling Enemy with a Simple State Machine
Enemy Scene

Build an Enemy scene the same way you built the Player: a CharacterBody2D root with Sprite2D and CollisionShape2D children. Rather than writing separate patrol logic with no structure, give it a small state machine using an enum and a match statement — the same idea as the game states (Playing, Win, Lose) you used back in Year 7, now written as real code.

Structured English 
DECLARE state ← PATROL

EVERY physics frame
  CASE state OF
    PATROL:
      move in current direction
      IF touching wall THEN reverse direction
    STUNNED:
      stop moving
GDScript 
extends CharacterBody2D

enum State { PATROL, STUNNED }
var state: State = State.PATROL
@export var speed: float = 80.0
var direction: int = 1

func _physics_process(delta):
  match state:
    State.PATROL:
      velocity.x = direction * speed
      if is_on_wall():
        direction *= -1
    State.STUNNED:
      velocity.x = 0
  move_and_slide()

Right now STUNNED is never reached — that's expected. Step 4 is what triggers it. Place the Enemy in your level between two walls (or the level boundary and a wall) so is_on_wall() gives it something to bounce off.

Test it — the enemy should patrol back and forth, correctly reversing direction at each wall, without you needing to specify patrol distances or timers.

4
Player–Enemy Overlap → Lose a Life
GDD Core Loop — Line 5

This step implements "IF player overlaps enemy THEN decrease lives; IF lives = 0 THEN trigger fail condition." Add a child Area2D named HurtArea to your Enemy scene, with its own CollisionShape2D slightly larger than the enemy's body. Connect its body_entered signal on the Enemy's own script.

Enemy (CharacterBody2D)
├─ Sprite2D
├─ CollisionShape2D
└─ HurtArea (Area2D)
   └─ CollisionShape2D
Structured English 
WHEN body enters hurt area
  IF body is player THEN
    SUBTRACT 1 FROM lives
    SET state TO STUNNED
    IF lives = 0 THEN
      trigger fail condition
    START stun timer
GDScript 
func _on_hurt_area_body_entered(body):
  if body.is_in_group("player"):
    GameState.lives -= 1
    state = State.STUNNED
    if GameState.lives <= 0:
      # full fail-state handling arrives in Sprint 3
      print("Fail condition triggered")
    $StunTimer.start()

Add a Timer node named StunTimer as a child of Enemy, set to one-shot with a 1-second wait time. Connect its timeout signal to set state = State.PATROL again, returning the enemy to its normal behaviour.

Test it — walk into the enemy. Lives should decrease by one, the enemy should stop moving briefly, then resume patrolling after about a second.

5
Display Score & Lives
UI

Numbers that only exist in the Output panel don't help a player. Add a CanvasLayer to your main level scene with two Label children — matching the UI structure you found in the starter project during Lessons 1–2.

UI (CanvasLayer)
├─ ScoreLabel (Label)
└─ LivesLabel (Label)
Structured English 
EVERY frame
  SET scoreLabel text TO "Score: " + score
  SET livesLabel text TO "Lives: " + lives
GDScript 
func _process(delta):
  $ScoreLabel.text = "Score: " + str(GameState.score)
  $LivesLabel.text = "Lives: " + str(GameState.lives)

Test it — both labels should update live as you collect coins and take hits from the enemy, with no delay or refresh needed.

Testing & Evaluating

Test Score, Lives & the Enemy

Complete this test table in your project notebook. Include your Sprint 1 tests too — confirm movement and physics still work unchanged.

Test case Expected result Your actual result (write in notebook)

🔍 Evaluation

For each test where the actual result didn't match the expected result, write:

  1. What went wrong (the bug)
  2. What you changed to fix it (the fix)
  3. Whether the fix introduced any new problems (a regression)

A common bug at this stage: lives drop by more than 1 per hit, because the player stays overlapping the HurtArea across several frames before the STUNNED state or invulnerability takes effect. If this happens, check whether body_entered is firing once or repeatedly.

✓ Sprint 2 — Checklist

Your project must:

  • Have a GameState Autoload storing score and lives, accessible from every scene
  • Have at least one working Coin that increases score and removes itself on pickup
  • Have a patrolling Enemy that correctly reverses direction at walls using is_on_wall()
  • Decrease lives and briefly stun the enemy on player contact, using the PATROL/STUNNED state machine
  • Print a fail condition message when lives reach zero
  • Display live-updating Score and Lives labels on screen
  • Have all Sprint 1 movement, jumping, and collision behaviours still working unchanged
  • Have a completed test table in your project notebook (all six tests attempted)

Sprint 2 — Notebook Entry

Sprint 2 crosses two D&P phases — Producing & Implementing and Testing & Evaluating. Your notebook entry needs evidence of both.

Your sprint notebook entry must include:

  1. Sprint log: for each of the five steps, one sentence on what you implemented and whether it worked first time.
  2. Test table: the completed test table from the Testing section, with actual results written in.
  3. Bug report: describe at least one thing that didn't work as expected, what you changed, and whether the fix caused any regressions.
  4. Design reflection: compare your enemy and coin behaviour to your GDD's core loop pseudocode. Does the STUNNED state change how the game feels compared to what you originally imagined? Would you adjust the pseudocode now?
