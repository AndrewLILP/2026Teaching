3
A Second Level
Multiple Levels

Duplicate your main level scene to create a second one — rearrange platforms, add another enemy, and adjust the Goal's position so it isn't identical to Level 1. Store both level paths in GameState so the Win screen knows what to load next.

Structured English 
DECLARE levels ← [level1, level2]
DECLARE currentLevel ← 1

BEGIN loadNextLevel
  currentLevel ← currentLevel + 1
  IF currentLevel ≤ length(levels) THEN
    LOAD levels[currentLevel]
END loadNextLevel
GDScript — GameState.gd 
var levels: Array = [
  "res://levels/level_1.tscn",
  "res://levels/level_2.tscn"
]
var current_level: int = 1

func load_next_level():
  current_level += 1
  if current_level <= levels.size():
    get_tree().change_scene_to_file(levels[current_level - 1])

Connect the Win screen's Next button's pressed signal to call GameState.load_next_level().

Test it — complete Level 1, press Next on the Win screen, and confirm Level 2 loads with your player able to move immediately.

4
Sound Design
Polish

Add an AudioStreamPlayer2D child to the Player, the Coin, and the Enemy, each with a short sound effect assigned. Play each sound at the exact moment its matching event happens — not before, not after.

Structured English 
WHEN coin collected
  PLAY pickup sound
  ADD 1 TO score
GDScript — Coin.gd 
func _on_body_entered(body):
  if body.is_in_group("player"):
    $PickupSound.play()
    GameState.score += 1
    # queue_free() waits until the sound finishes —
    # see the note below
    get_tree().create_timer(0.3).timeout.connect(queue_free)

Notice the coin can no longer call queue_free() immediately — doing so would delete the AudioStreamPlayer2D before its sound finished playing. A short timer delays the removal just long enough for the sound to complete.

Test it — jumping, collecting a coin, and taking damage from the enemy should each produce a distinct sound, and the coin's sound should play in full before it disappears.

5
Saving a High Score Safely
Safe & Secure Data

A high score is the simplest kind of persistent data — but "simple" doesn't mean you skip thinking about it. Godot separates two file locations: res:// is your read-only project itself, and user:// is a writable location on the player's device for data that changes at runtime, like save files.

Structured English 
BEGIN saveHighScore
  IF score > highScore THEN
    highScore ← score
    WRITE highScore TO save file
END saveHighScore
GDScript — GameState.gd 
const SAVE_PATH = "user://highscore.save"
var high_score: int = 0

func save_high_score():
  if score > high_score:
    high_score = score
    var file = FileAccess.open(SAVE_PATH, FileAccess.WRITE)
    file.store_var(high_score)

func load_high_score():
  if FileAccess.file_exists(SAVE_PATH):
    var file = FileAccess.open(SAVE_PATH, FileAccess.READ)
    high_score = file.get_var()
🔒 Safe, Secure & Ethical Data Use

Notice save_high_score() only overwrites the file if the new score is actually higher — without that check, a worse run could silently erase a genuinely better one. This is a small example of a bigger principle: any time your program writes data, decide deliberately what gets kept, what gets overwritten, and under what condition.

If you later added a leaderboard with player names, the same principle scales up significantly — you'd need to consider what's appropriate to collect, how it's stored, and how it could be misused if displayed publicly. Saving a single integer safely is good practice for handling more sensitive data later.

Call load_high_score() once when your first level starts, and save_high_score() the moment trigger_game_over() runs.

Test it — get a score, lose, and check the high score displays correctly. Close Godot entirely, reopen the project, and confirm the high score is still there.
